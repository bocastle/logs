# 커서 기반 페이지네이션 정리

## 핵심 요약

- 커서 기반 페이지네이션은 "몇 번째 페이지"보다 "어디까지 읽었는지"를 기준으로 다음 데이터를 가져오는 방식이다.
- 데이터가 많고, 계속 추가되거나 순서가 자주 바뀌는 서비스에서 offset 방식보다 안정적일 때가 많다.
- 무한 스크롤, 타임라인, 채팅 목록 같은 곳에서 특히 잘 맞는다.
- 면접에서는 offset 방식과 비교해서 성능, 일관성, 구현 난이도를 같이 말하면 좋다.

## 페이지네이션을 왜 나누어 생각하나

페이지네이션은 단순히 목록을 잘라 보여주는 문제가 아니다.  
데이터가 많아질수록 조회 비용, 사용자 경험, 중복/누락 가능성까지 같이 봐야 한다.

작은 테이블에서는 offset 방식도 충분히 잘 동작한다.  
하지만 데이터가 커지고 실시간으로 추가되는 목록에서는 offset 방식의 약점이 드러난다.

## offset 기반 페이지네이션이 뭐냐

가장 익숙한 방식은 `limit 20 offset 40`처럼 "앞에서 몇 개를 건너뛸지" 지정하는 것이다.

장점은 단순하다.

- 구현이 쉽다.
- "3페이지", "10페이지"처럼 번호 기반 UI와 잘 맞는다.

하지만 뒤로 갈수록 DB가 앞부분을 많이 건너뛰어야 해서 부담이 커질 수 있고, 중간에 데이터가 추가/삭제되면 중복이나 누락이 생길 수 있다.

## 커서 기반 페이지네이션이 뭐냐

커서 기반 페이지네이션은 마지막으로 본 데이터의 기준값을 들고 다음 데이터를 가져온다.

예를 들어 `createdAt desc, id desc`로 정렬한다면,

- 첫 요청: 최신 글 20개 조회
- 다음 요청: "마지막 글의 createdAt과 id"를 커서로 전달
- 이후 요청: 그 커서보다 뒤에 있는 데이터 20개 조회

즉 "40개 건너뛰기"가 아니라 "이 지점 다음부터 가져오기"에 가깝다.

## 왜 성능에 유리하다고 하나

핵심은 앞부분을 계속 스캔해서 버리지 않아도 된다는 점이다.

적절한 인덱스가 있으면 DB는 커서 조건 이후 구간부터 바로 읽기 쉬워진다.  
그래서 데이터가 커질수록 offset보다 더 안정적으로 동작하는 경우가 많다.

## 왜 일관성에도 유리하다고 하나

타임라인 데이터는 조회하는 동안에도 새 글이 계속 들어올 수 있다.  
이때 offset 방식은 앞쪽 데이터가 밀리면서 사용자가 보지 않았던 데이터가 건너뛰어지거나, 이미 본 데이터가 다시 보일 수 있다.

커서 방식은 "마지막으로 본 지점 이후"를 기준으로 잡기 때문에 이런 문제가 상대적으로 적다.

물론 정렬 기준이 불안정하면 커서 방식도 꼬일 수 있으니, 정렬 컬럼은 일관되게 잡아야 한다.

## 구현할 때 중요한 점

### 1. 정렬 기준이 명확해야 한다

`createdAt`만으로는 동률이 생길 수 있다.  
그래서 보통 `createdAt + id`처럼 유니크하게 순서를 보장할 수 있는 조합을 쓴다.

### 2. 커서는 정렬 기준과 맞물려야 한다

내림차순 정렬이면 "이 값보다 작은 것"을 가져오는 식으로 조건이 정렬과 정확히 맞아야 한다.

### 3. 다음 페이지 존재 여부를 같이 설계해야 한다

보통 `limit + 1`개를 조회해서 다음 페이지가 있는지 판단하거나, 응답에 `hasNext`와 `nextCursor`를 함께 준다.

### 4. 임의 페이지 이동과는 잘 안 맞는다

커서 방식은 1페이지, 2페이지, 3페이지보다 "계속 다음으로 보기"에 맞는 방식이다.  
그래서 관리자 화면처럼 바로 57페이지로 가야 하는 UI에는 offset이 더 자연스러울 수 있다.

## 예시로 이해하기

최신순 게시글 목록 API를 만든다고 하면,

1. 첫 요청에는 커서 없이 호출한다.
2. 서버는 최신 글 20개와 `nextCursor`를 준다.
3. 클라이언트는 마지막 글의 기준값을 다음 요청에 넣는다.
4. 서버는 그 커서보다 오래된 글 20개를 내려준다.

이 구조는 무한 스크롤과 궁합이 좋다.

## 면접에서 비교해서 말하면 좋은 포인트

### offset 방식

- 구현이 단순하다.
- 페이지 번호 기반 UI에 잘 맞는다.
- 큰 데이터셋이나 실시간 변경이 많은 목록에서는 비효율과 정합성 문제가 생길 수 있다.

### 커서 방식

- 대량 데이터와 무한 스크롤에 잘 맞는다.
- 실시간 목록에서 중복/누락 위험을 줄이기 좋다.
- 구현과 응답 설계가 조금 더 복잡하다.

## 면접 답변 예시

> 커서 기반 페이지네이션은 몇 번째 페이지를 조회하는 방식이 아니라, 마지막으로 본 데이터의 위치를 기준으로 다음 데이터를 가져오는 방식입니다. 예를 들어 최신순 목록이라면 마지막 데이터의 createdAt, id 같은 값을 커서로 넘겨서 그 이후 구간만 조회할 수 있습니다. 그래서 데이터가 크거나 실시간으로 계속 추가되는 타임라인에서는 offset 방식보다 성능과 일관성 측면에서 유리한 경우가 많습니다. 대신 임의 페이지 이동에는 불편하고, 정렬 기준과 인덱스를 같이 설계해야 한다는 점은 주의해야 합니다.

## 장점

- 큰 목록에서 offset보다 안정적인 조회 성능을 기대하기 좋다.
- 실시간 데이터 추가가 있는 환경에서 중복/누락을 줄이기 쉽다.
- 무한 스크롤 UX와 잘 맞는다.

## 단점

- 구현과 테스트가 offset보다 복잡하다.
- "몇 페이지" 같은 번호 기반 탐색과는 잘 맞지 않는다.
- 정렬 기준이 애매하면 오히려 버그가 생기기 쉽다.

## 주의사항 / 실무 팁

- 커서 값은 외부에 그대로 노출하기보다 필요한 경우 인코딩해서 다루기도 한다.
- 정렬 컬럼과 인덱스 설계를 같이 봐야 진짜 성능 이점이 난다.
- `createdAt` 단독보다 `createdAt + id`처럼 tie-breaker를 두는 편이 안전하다.
- 면접에서는 "offset이 나쁘다"가 아니라 "관리자형 UI와 타임라인형 UI의 요구가 다르다"는 식으로 말하면 더 실무적으로 들린다.
