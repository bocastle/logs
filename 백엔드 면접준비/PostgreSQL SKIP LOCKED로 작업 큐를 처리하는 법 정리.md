# PostgreSQL SKIP LOCKED로 작업 큐를 처리하는 법 정리

## 핵심 요약

- 이 주제는 정상 동작보다 실패했을 때 어떤 보장을 남길지 묻는 백엔드 면접 주제다.
- 면접에서는 개념, 필요한 이유, 운영 지표, 장애 대응 순서까지 연결해서 말하는 편이 좋다.
- 단순히 설정값을 외우기보다 어떤 위험을 줄이고 어떤 trade-off를 받아들이는지 설명해야 한다.

## 개념 설명

`PostgreSQL SKIP LOCKED로 작업 큐를 처리하는 법`은 서비스가 커지고 장애 조건이 다양해질 때 더 중요해지는 주제다. 백엔드에서는 한 번 성공하는 코드보다, 느려지거나 실패하거나 중복 실행될 때도 피해를 제한하는 설계가 더 오래 간다.

이런 질문을 받으면 먼저 문제 상황을 좁혀 말하는 것이 좋다. 예를 들어 중복 처리 문제인지, 지연 문제인지, 정합성 문제인지, 운영자가 원인을 보기 어려운 문제인지 나눠야 한다. 그다음 로그, 메트릭, 알림, 롤백 기준을 붙이면 답변이 현실적으로 들린다.

## 예시

```sql
SELECT id
FROM jobs
WHERE status = 'ready'
ORDER BY created_at
FOR UPDATE SKIP LOCKED
LIMIT 10;
```

예시는 구조를 설명하기 위한 작은 형태다. 실무에서는 timeout, retry, idempotency, 권한 검증, 모니터링을 함께 봐야 한다.

## 면접 답변 예시

> `PostgreSQL SKIP LOCKED로 작업 큐를 처리하는 법`은 단순 구현보다 장애와 운영 기준을 같이 봐야 하는 주제라고 생각합니다. 저는 먼저 이 설계가 막으려는 문제가 중복 실행인지, 지연 전파인지, 데이터 불일치인지 나누겠습니다. 그다음 실패했을 때 재시도 가능한지, 같은 작업이 두 번 처리돼도 안전한지, 어떤 지표로 이상을 감지할지 정하겠습니다. 마지막으로 변경 범위를 작게 잡고 롤백이나 완화 전략까지 준비하겠다고 답하겠습니다.

## 장점

- 장애 상황에서 영향 범위를 줄이는 데 도움이 된다.
- 운영 지표와 함께 설계하면 문제를 더 빨리 찾을 수 있다.
- 팀 안에서 같은 유형의 실패를 다루는 기준을 만들기 쉽다.

## 단점

- 설계가 세밀해질수록 구현과 운영 비용이 늘어난다.
- 잘못된 기준을 잡으면 장애를 줄이기보다 숨길 수 있다.
- 강한 정합성을 선택하면 지연이나 처리량 손해가 생길 수 있다.

## 주의사항 / 실무 팁

- 정상 흐름보다 실패 흐름을 먼저 그려 보면 답변이 좋아진다.
- 재시도와 timeout을 말할 때는 idempotency와 중복 처리 기준을 같이 말한다.
- 운영에서는 단일 지표보다 로그, 메트릭, trace를 함께 보고 원인을 좁힌다.
