# 자바 프로그램 실행 흐름 정리

## 핵심 요약

자바 프로그램은 `.java` 소스를 `javac`로 컴파일해 **바이트코드(`.class`)** 를 만들고, 이후에는 JVM이 **클래스 로딩(Loading → Linking → Initialization)** 과 **실행(Interpreter + JIT)** 을 담당하며 최종적으로 기계어로 변환해 실행합니다.

## 전체 실행 흐름

1. **컴파일**
   - 개발자가 작성한 `.java` 파일을 `javac(java compiler)`로 컴파일
   - JVM이 이해할 수 있는 **바이트코드**로 변환되어 `.class` 파일 생성

2. **클래스 로더(Class Loader)가 바이트코드를 동적으로 로드**
   - `.class`를 JVM 메모리에 로드하고, 로드된 클래스는 **Method Area**에 저장
   - 로딩 과정에서 **Loading → Linking → Initialization** 단계를 거침

3. **실행 엔진(Execution Engine)이 바이트코드 실행**
   - 바이트코드는 그대로는 CPU가 실행할 수 없으므로,
   - JVM이 **인터프리터(Interpreter)** 와 **JIT 컴파일러(Just-In-Time Compiler)** 를 함께 사용해 기계어(네이티브 코드)로 변환하여 실행

## 클래스 로더의 “동적 로드” 의미

프로그램 시작 시 모든 클래스를 한꺼번에 올리지 않고, **런타임에 필요한 시점에만** 클래스를 로드하는 것을 의미합니다.

### 클래스 로드가 발생하는 대표 시점

- 인스턴스를 생성할 때
- `static` 메서드/변수를 사용할 때
- `static` 변수에 값을 할당할 때

## 로딩, 링킹, 초기화 단계

### 로딩(Loading)

- 클래스 로더가 `.class` 파일을 읽어 **JVM 메모리에 로드**
- 로드된 클래스는 **Method Area**에 저장

### 링킹(Linking)

로드된 클래스가 실행될 수 있도록 준비하는 단계이며, 다음 3단계로 구성됩니다.

1. **Verification**
   - `.class` 파일이 구조적으로 올바른지 확인
2. **Preparation**
   - `static` 변수를 메모리에 할당하고 **기본값으로 초기화**
3. **Resolution**
   - 런타임 상수 풀의 **심볼릭 레퍼런스**를 실제 메모리 레퍼런스로 교체

### 초기화(Initialization)

- `static` 변수를 **사용자가 지정한 값**으로 초기화
- `static` 블록을 실행

## 인터프리터와 JIT를 함께 쓰는 이유

### 인터프리터(Interpreter)

- 바이트코드를 **한 줄씩 읽어 실행**
- 장점: **초기 실행 속도가 빠름**
- 단점: 반복 실행 시 매번 해석해야 하므로 **반복 실행 성능이 떨어질 수 있음**

### JIT 컴파일러(Just-In-Time Compiler)

- 자주 실행되는 메서드(Hotspot)를 감지하면, 해당 메서드 전체를 **네이티브 코드로 변환**하여 **캐싱**
- 장점: 반복 실행 시 **매우 빠른 성능**
- 단점: 컴파일 과정 자체가 비용이 있어 **초기 오버헤드**가 발생할 수 있음

### 결론

JVM은 두 방식을 함께 사용해 **초기 실행 속도**와 **반복 실행 성능**을 모두 확보합니다.

## 장점

- (동적 로딩) 필요할 때만 클래스를 로드해 **불필요한 로드/메모리 사용을 줄일 수 있음**
- (JIT) 반복 실행되는 코드에 대해 **실행 성능을 크게 향상**시킬 수 있음

## 단점

- (JIT) 컴파일 작업으로 인해 **초기 실행 시 오버헤드**가 발생할 수 있음
- (동적 로딩) 런타임에 로드/링킹/초기화가 일어나므로, 특정 시점에 **지연이 체감**될 수 있음

## 주의사항 및 실무 팁

- `static` 초기화 블록/초기화 로직이 무거우면 **클래스 초기화 시점의 지연**으로 이어질 수 있으니 주의합니다.
- “동적 로드”는 편리하지만, 의존 관계가 복잡한 환경에서는 특정 시점에 **클래스 로딩 실패(ClassNotFound 등)** 가 발생할 수 있어 배포/클래스패스 관리가 중요합니다.
- 성능 분석 시 “느린 지점”이 바이트코드 실행 자체인지, **클래스 로딩/초기화**인지, **JIT 워밍업**인지 분리해서 보는 것이 도움이 됩니다.
