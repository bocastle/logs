# Hedged Requests 정리

## 핵심 요약
- `Hedged Requests`는 응답이 늦어질 조짐이 보이면 동일 요청을 다른 복제본이나 다른 경로로 한 번 더 보내 꼬리 지연을 줄이는 기법이다.
- 평균 지연보다 `p95`, `p99` 같은 느린 구간을 줄이는 데 초점이 있다.
- 면접에서는 "모든 요청을 중복 전송하는 방식"이 아니라, 늦는 요청만 제한적으로 복제해 tail latency를 낮추는 전략이라고 설명하면 좋다.
- 중복 요청이 생기기 때문에 멱등성, 비용 증가, 부하 전파를 함께 고려해야 한다.

## 개념 설명
분산 시스템에서는 평균 응답 속도보다 일부 아주 느린 요청이 사용자 경험을 크게 망치는 경우가 많다. 예를 들어 대부분 50ms 안에 끝나도, 가끔 2초씩 걸리는 요청이 있으면 사용자는 서비스가 불안정하다고 느낀다.

`Hedged Requests`는 이런 꼬리 지연을 줄이기 위한 전략이다. 처음 요청을 보낸 뒤 일정 시간 안에 응답이 없으면, 같은 요청을 다른 서버나 복제본에 한 번 더 보낸다. 그리고 먼저 도착한 응답을 사용하고 나머지는 취소하거나 무시한다.

즉, 핵심은 "평균을 더 빠르게"보다 "아주 느린 소수를 덜 아프게"이다. 면접에서는 tail latency, 복제 요청, 멱등성 이 세 가지를 함께 묶어 설명하면 답변이 선명해진다.

## 예시
검색 API가 여러 읽기 전용 복제본에 분산되어 있다고 가정해보자.

- 평소에는 한 복제본으로 요청을 보낸다
- 그런데 80ms 안에 응답이 없으면 다른 복제본에 같은 읽기 요청을 한 번 더 보낸다
- 둘 중 먼저 끝난 응답을 사용자에게 반환한다

이 방식은 평소 비용은 크게 늘리지 않으면서도, 특정 복제본의 순간적인 GC 지연이나 네트워크 흔들림 때문에 생기는 긴 꼬리를 완화하는 데 도움이 된다.

## 면접 답변 예시
- 질문: "`Hedged Requests`는 무엇인가요?"  
  답변: "`Hedged Requests`는 요청이 비정상적으로 늦어질 때 동일 요청을 다른 서버나 복제본에 한 번 더 보내서, 먼저 도착한 응답을 사용하는 지연 완화 기법입니다. 평균 지연보다 tail latency를 줄이는 데 목적이 있습니다."
- 질문: "아무 요청에나 적용해도 되나요?"  
  답변: "아닙니다. 읽기 요청이나 멱등성이 보장된 요청처럼 중복 실행 부작용이 적은 곳에 더 적합합니다. 쓰기 요청에 무분별하게 적용하면 중복 처리나 부하 증가 문제가 생길 수 있습니다."

## 장점
- `p95`, `p99` 같은 꼬리 지연을 줄이는 데 효과적이다
- 순간적인 장애나 특정 노드의 느려짐에 덜 민감해질 수 있다
- 읽기 복제본이 있는 구조에서 비교적 자연스럽게 적용하기 좋다

## 단점
- 중복 요청 때문에 네트워크와 서버 자원 사용량이 늘어난다
- 잘못 적용하면 오히려 전체 시스템 부하를 키울 수 있다
- 멱등성이나 취소 처리 설계가 부족하면 부작용이 생긴다

## 주의사항 / 실무 팁
- 모든 요청에 기본 적용하기보다, tail latency가 실제 문제인 읽기 경로에 제한적으로 적용하는 편이 안전하다.
- 임계 시간은 무작정 짧게 두기보다 실제 지연 분포를 보고 정해야 한다.
- 면접에서는 `timeout`, `retry`, `load balancing`과 차이를 함께 말하면 더 설득력 있다. `retry`는 실패 후 재시도이고, `hedged request`는 느려질 때 병렬로 우회한다는 점이 다르다.
- 실무에서는 취소 신호 전파, 중복 요청 모니터링, 비용 증가 여부를 함께 관찰해야 한다.
