# Read Repair 정리

## 핵심 요약
- `Read Repair`는 분산 저장소에서 읽기 시점에 복제본 간 데이터 불일치를 발견하면 바로 고쳐주는 방식이다.
- 핵심은 쓰기 경로가 아니라 읽기 경로에서 일관성을 조금씩 회복한다는 점이다.
- 면접에서는 `쿼럼 읽기`, `리드 리플리카`, `eventual consistency`와 연결해서 설명하면 좋다.
- 읽기 요청이 많을수록 불일치가 자연스럽게 정리될 수 있지만, 읽기 지연이 늘 수 있다는 점도 같이 말해야 한다.

## 개념 설명
분산 데이터베이스나 키-값 저장소에서는 하나의 데이터를 여러 노드에 복제해 둔다. 그런데 일부 노드에만 최신 쓰기가 반영되거나, 네트워크 문제 때문에 복제 시점이 어긋나면 복제본 간 값이 달라질 수 있다.

이때 `Read Repair`는 읽기 요청을 처리하는 과정에서 여러 복제본 값을 비교하고, 오래된 복제본이 보이면 최신 값으로 다시 맞춰주는 기법이다. 즉, 사용자가 조회를 한 순간이 데이터 정합성을 회복하는 계기가 되는 셈이다.

이 방식은 모든 복제본을 쓰기 순간에 강하게 맞추기 어려운 환경에서 자주 등장한다. 그래서 면접에서는 "완벽한 즉시 일관성을 보장하는 기술"이 아니라, "eventual consistency 환경에서 정합성을 점진적으로 회복하는 장치"라고 설명하는 것이 정확하다.

## 예시
세 개의 복제 노드 A, B, C가 있고, 주문 상태가 원래 모두 `PAID`여야 한다고 가정해보자. 그런데 네트워크 지연 때문에 C만 아직 `PENDING` 상태를 들고 있을 수 있다.

이 상태에서 읽기 요청이 들어와 A, B, C를 함께 조회했더니 A와 B는 `PAID`, C는 `PENDING`을 반환했다면 다음처럼 동작할 수 있다.

- 다수결 또는 최신 버전 기준으로 `PAID`를 정답으로 판단한다
- 클라이언트에는 `PAID`를 응답한다
- 동시에 C 노드에 `PAID`로 다시 써서 복제본을 맞춘다

이렇게 하면 다음 읽기에서는 불일치가 줄어든다.

## 면접 답변 예시
- 질문: "Read Repair는 무엇인가요?"  
  답변: "분산 저장소에서 읽기 요청 시 여러 복제본의 값을 비교해 오래된 복제본을 최신 값으로 보정하는 방식입니다. 즉, 읽기 경로를 이용해 복제본 간 불일치를 점진적으로 회복하는 기법입니다."
- 질문: "장점과 단점은 무엇인가요?"  
  답변: "백그라운드 동기화가 늦더라도 실제 트래픽을 통해 정합성을 회복할 수 있다는 장점이 있습니다. 반면 읽기 시 비교와 보정이 들어가므로 지연이 늘 수 있고, 읽히지 않는 데이터는 오래 불일치 상태로 남을 수 있다는 단점이 있습니다."

## 장점
- 읽기 트래픽을 활용해 복제본 정합성을 회복할 수 있다
- eventual consistency 환경에서 실용적인 보완책이 된다
- 일부 노드가 뒤처졌더라도 클라이언트에는 더 정확한 값을 응답할 가능성이 높다

## 단점
- 읽기 시 비교와 보정 작업이 추가돼 응답 시간이 늘 수 있다
- 자주 읽히지 않는 데이터는 오래된 상태가 지속될 수 있다
- 충돌 해결 기준이 불명확하면 오히려 잘못된 값이 퍼질 위험이 있다

## 주의사항 / 실무 팁
- `Read Repair`는 충돌 해결 전략과 함께 설명해야 답변이 탄탄해진다.
- `Last Write Wins`, 버전 벡터, 타임스탬프 같은 기준이 없으면 어떤 값이 최신인지 판단하기 어렵다.
- 면접에서는 `쿼럼 읽기`와 연결해 "읽기 중 여러 복제본을 비교할 수 있을 때 효과적"이라고 설명하면 좋다.
- 실무에서는 읽기 지연 증가를 감수할지, 백그라운드 안티엔트로피와 병행할지 같이 고민해야 한다.
