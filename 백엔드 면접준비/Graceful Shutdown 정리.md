# Graceful Shutdown 정리

## 핵심 요약
- `Graceful Shutdown`은 서버를 종료할 때 들어오던 요청과 사용 중인 자원을 최대한 안전하게 정리하고 내려가는 방식이다.
- 핵심은 "빨리 끄는 것"이 아니라 "중간 작업을 덜 망가뜨리고 종료하는 것"이다.
- 쿠버네티스, 오토스케일링, 배포 자동화 환경에서는 특히 자주 나오는 운영 면접 주제다.
- 면접에서는 "새 요청은 막고, 진행 중인 요청은 가능한 한 마무리한 뒤 커넥션과 리소스를 정리하고 종료하는 절차"라고 설명하면 된다.

## 개념 설명
백엔드 서버는 단순히 프로세스 하나만 떠 있는 것이 아니라 DB 커넥션, 메시지 소비, 스레드풀, 외부 API 호출 같은 여러 자원을 함께 사용한다. 이 상태에서 프로세스를 즉시 강제 종료하면 어떤 문제가 생길까. 처리 중인 요청이 중간에 끊길 수 있고, 메시지를 읽어 놓고 처리 완료 전에 죽을 수 있으며, DB 작업이 애매한 상태로 끝날 수도 있다.

그래서 운영 환경에서는 서버를 종료할 때 `Graceful Shutdown` 절차를 둔다. 먼저 로드밸런서나 서비스 디스커버리에서 해당 인스턴스를 트래픽 대상에서 빼고, 새 요청을 받지 않게 한다. 그다음 이미 처리 중인 요청은 일정 시간 안에 마무리하도록 기다리고, 메시지 컨슈머나 스케줄러도 안전하게 멈춘다. 마지막으로 커넥션과 자원을 정리하고 프로세스를 종료한다.

이 개념은 무중단 배포나 오토스케일링과도 연결된다. 종료를 우아하게 하지 않으면 인스턴스 교체가 곧 장애나 데이터 불일치로 이어질 수 있다.

## 예시
트래픽을 받는 API 서버가 배포로 교체된다고 가정해보자.

- 서버가 종료 신호를 받는다
- readiness를 내려 새 요청 유입을 막는다
- 이미 처리 중인 요청은 제한 시간 안에 끝내도록 기다린다
- DB 커넥션 풀, 메시지 컨슈머, 백그라운드 작업을 정리한다
- 정리 완료 후 프로세스를 종료한다

이 흐름이 없으면 배포 순간 일부 요청이 끊기거나, 컨슈머가 메시지를 처리하다 말고 내려가 재처리 문제가 생길 수 있다.

## 면접 답변 예시
- 질문: "`Graceful Shutdown`은 왜 중요한가요?"  
  답변: "서버 종료 시 진행 중인 요청과 자원을 안전하게 정리하지 않으면 요청 손실, 중복 처리, 연결 누수 같은 문제가 생길 수 있기 때문입니다. 특히 배포나 스케일링 환경에서는 안정적인 종료 절차가 필수입니다."
- 질문: "어떻게 구현하나요?"  
  답변: "종료 신호를 받아 새 요청을 막고, 처리 중인 작업을 일정 시간 기다린 뒤, DB 커넥션이나 메시지 컨슈머 같은 자원을 정리하고 종료합니다. 로드밸런서나 쿠버네티스 readiness/lifecycle과 함께 보는 경우가 많습니다."

## 장점
- 운영 관점이 있는 백엔드 답변으로 보이기 좋다
- 배포, 스케일링, 메시지 처리, 커넥션 관리와 연결해서 설명할 수 있다
- 단순 기능 구현이 아니라 서비스 안정성 관점을 보여 주기 좋다

## 단점
- 프레임워크와 인프라 환경에 따라 구현 방식이 달라 일반화가 어렵다
- "기다린다"는 표현만 하면 timeout과 강제 종료 전략이 빠져 답변이 약해질 수 있다
- 애플리케이션 종료와 데이터 일관성을 섞어 말하면 설명이 산만해질 수 있다

## 주의사항 / 실무 팁
- 종료 시 무한정 기다릴 수는 없으므로 timeout과 fallback 종료 전략을 함께 둬야 한다.
- HTTP 요청뿐 아니라 메시지 컨슈머, 스케줄러, 비동기 작업도 종료 대상에 포함해야 한다.
- 면접에서는 `Zero-Downtime Deployment`, `Connection Pool`, `메시지 큐의 전달 보장 방식`과 연결하면 좋다.
- 실무에서는 readiness 해제 시점과 실제 프로세스 종료 시점 사이의 간격을 반드시 검증해야 한다.
