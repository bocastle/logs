# MVCC 정리

## 핵심 요약

- `MVCC(Multi-Version Concurrency Control)`는 데이터를 한 버전만 보지 않고 여러 버전으로 관리해서 읽기와 쓰기 충돌을 줄이는 방식이다.
- 핵심은 `읽는 쪽이 잠금에 덜 막히고, 동시성을 높이면서도 일관성을 유지하려는 전략`이라고 설명하면 된다.
- 면접에서는 락이 아예 없어지는 기술이 아니라, `락 의존도를 줄이는 기술`이라고 말하는 편이 정확하다.
- 트랜잭션 격리 수준, Undo Log, 스냅샷 읽기와 연결해서 설명하면 답변이 깊어진다.

## MVCC가 뭐냐

전통적으로 동시성 제어를 단순하게 하면 읽기와 쓰기가 서로 많이 기다리게 된다.  
그런데 서비스 트래픽이 커질수록 "읽기 요청이 쓰기 때문에 계속 막히는 구조"는 병목이 되기 쉽다.

MVCC는 데이터를 수정할 때 기존 값을 바로 덮어쓰는 대신 버전을 나눠 관리하고,  
트랜잭션마다 자신이 볼 수 있는 시점의 데이터를 읽게 해서 충돌을 줄인다.

즉 한 문장으로 말하면, `같은 데이터를 여러 시점의 버전으로 관리해서 읽기 성능과 동시성을 확보하는 방식`이다.

## 왜 필요한가

- 조회가 많은 서비스에서 읽기 성능이 좋아진다.
- 읽기 작업이 쓰기 락에 덜 막힌다.
- 동시 요청이 많은 환경에서 전체 처리량을 높이기 좋다.

면접에서는 "모든 걸 락으로 해결하면 안전하지만 너무 느려질 수 있어서 나온 방식"이라고 풀어도 좋다.

## 어떻게 동작하나

예를 들어 어떤 행의 값이 `100`이었는데 다른 트랜잭션이 `120`으로 수정했다고 해 보자.  
MVCC 환경에서는 수정 전 버전과 수정 후 버전을 바로 하나로 덮지 않고 관리한다.

읽는 트랜잭션은 자신의 시작 시점 기준으로 보이는 버전을 읽는다.  
그래서 어떤 트랜잭션은 `100`을 보고, 더 뒤에 시작한 트랜잭션은 `120`을 볼 수 있다.

이때 실제 구현은 DB마다 다르지만 보통 다음 개념이 같이 나온다.

- Undo Log 또는 이전 버전 정보
- 트랜잭션 ID
- 스냅샷 읽기(Read View)

## 락이 아예 필요 없다는 뜻이냐

그건 아니다.  
MVCC는 읽기와 쓰기 충돌을 줄이는 데 강하지만, 쓰기와 쓰기 충돌까지 완전히 없애지는 못한다.

예를 들어 같은 행을 동시에 수정하려고 하면 여전히 락이나 충돌 제어가 필요하다.  
그래서 MVCC를 설명할 때 "락이 필요 없다"라고 말하면 틀리기 쉽다.

정확하게는 `읽기 중심 충돌을 줄여 주지만, 일관성을 위해 필요한 락은 여전히 존재한다`가 맞다.

## 격리 수준과는 어떤 관계냐

MVCC는 격리 수준을 구현하는 데 자주 쓰인다.  
다만 MVCC가 있다고 해서 모든 DB가 같은 방식으로 동작하는 건 아니다.

- `READ COMMITTED`: 보통 커밋된 최신 버전을 기준으로 읽는다.
- `REPEATABLE READ`: 트랜잭션 시작 시점의 스냅샷을 더 강하게 유지한다.

여기서 세부 동작은 MySQL, PostgreSQL 같은 DBMS마다 다를 수 있으니, 면접에서는 "대표적인 방향"으로 설명하는 게 안전하다.

## 실무에서 왜 중요하냐

실무에서는 단순히 정의만 아는 것보다, "왜 조회 성능이 괜찮은지"를 설명할 때 MVCC가 자주 나온다.

- 긴 조회 트랜잭션이 많은 서비스
- 읽기 트래픽이 매우 높은 서비스
- 락 대기 시간을 줄여야 하는 환경

또 트랜잭션 격리 수준, 팬텀 리드, Undo 공간 증가 같은 주제와도 자연스럽게 이어진다.

## 면접 답변 예시

> MVCC는 Multi-Version Concurrency Control의 약자로, 데이터를 여러 버전으로 관리해서 읽기와 쓰기 충돌을 줄이는 동시성 제어 방식입니다. 핵심은 읽는 트랜잭션이 자신의 시점에 맞는 스냅샷 데이터를 보게 해서, 조회가 쓰기 락에 덜 막히도록 만드는 것입니다. 다만 락이 완전히 없어지는 건 아니고, 쓰기 충돌 제어나 일관성 보장을 위해 필요한 락은 여전히 있습니다. 그래서 저는 MVCC를 설명할 때 읽기 성능과 동시성을 높이기 위한 전략이라는 점, 그리고 격리 수준 구현과 연결된다는 점을 같이 말하는 편입니다.

## 장점

- 데이터베이스 동시성 제어를 원리 중심으로 설명하기 좋다.
- 락, 격리 수준, 스냅샷 읽기 같은 후속 질문으로 확장하기 쉽다.
- "왜 읽기 성능이 잘 나오냐"를 기술적으로 풀어내기 좋다.

## 단점

- DBMS별 차이를 무시하고 하나의 정답처럼 말하면 위험하다.
- 락이 없다고 오해하면 바로 허점이 드러난다.
- 버전 관리 비용이나 Undo 공간 증가 같은 현실적 비용을 빼면 설명이 평면적이 된다.

## 주의사항 / 실무 팁

- 면접에서는 `여러 버전`, `스냅샷 읽기`, `읽기-쓰기 충돌 감소` 세 키워드로 시작하면 안정적이다.
- "MVCC = 무락(no lock)"으로 말하지 않는 게 중요하다.
- DB마다 구현 차이가 있다는 점을 한 문장 넣으면 과한 일반화를 피할 수 있다.
- 격리 수준 질문이 이어질 수 있으니 `READ COMMITTED`, `REPEATABLE READ` 정도는 연결해서 준비해 두는 게 좋다.
