# 분산 락 정리

## 핵심 요약

- `분산 락`은 여러 서버 인스턴스가 동시에 같은 자원에 접근할 때 충돌을 줄이기 위해, 네트워크 너머에서 공유하는 락이다.
- 면접에서는 단순히 "한 번에 하나만 실행하게 한다"가 아니라, 왜 DB 락만으로 안 되는 상황이 있는지부터 설명하면 좋다.
- 스케줄러 중복 실행, 재고 차감, 배치 작업 중복 처리처럼 여러 프로세스가 같은 일을 동시에 만질 수 있을 때 자주 등장한다.
- 다만 구현이 까다롭고, 락 획득 실패나 만료, 장애 복구까지 설계하지 않으면 오히려 더 위험할 수 있다.

## 왜 분산 락이 필요하냐

서버가 한 대뿐이면 메모리 락이나 JVM 내부 락만으로도 어느 정도 제어가 된다.  
하지만 서버가 여러 대면 각 인스턴스 메모리는 서로 공유되지 않는다.

예를 들어 인스턴스 A와 B가 동시에 같은 쿠폰 발급 로직을 실행하면, 각자 자기 메모리 안에서는 락을 잘 걸어도 서로를 막지 못한다.  
이럴 때 공통으로 바라볼 수 있는 외부 저장소를 기준으로 락을 잡는 방식이 필요해지는데, 그게 분산 락이다.

## 어디에 자주 쓰이나

### 스케줄러 중복 실행 방지

배치 잡이 여러 인스턴스에서 동시에 돌아가면 중복 적재나 중복 알림이 생길 수 있다.  
이때 한 인스턴스만 작업을 수행하게 하려고 분산 락을 둔다.

### 재고나 쿠폰 같은 경쟁 자원 보호

동시에 많이 들어오는 요청에서 같은 자원을 여러 번 처리하지 않도록 임계 구역을 좁게 잡아 보호할 수 있다.

### 순서가 중요한 작업

같은 사용자나 같은 주문에 대한 후속 처리가 동시에 들어오면 꼬일 수 있는데, 이를 직렬화하려고 쓰기도 한다.

## 보통 어떻게 구현하나

실무에서 많이 거론되는 건 `Redis` 기반 방식이다.  
특정 키를 `SET NX EX` 같은 방식으로 저장해서 "없을 때만 락 획득"하게 만드는 식이다.

핵심은 아래 세 가지다.

- 락 키가 유일해야 한다.
- 만료 시간이 있어야 한다.
- 해제할 때는 내가 잡은 락만 풀어야 한다.

특히 해제 단계에서 다른 인스턴스가 새로 잡은 락을 실수로 지우면 큰 문제가 생긴다.  
그래서 보통 랜덤 토큰을 값으로 저장해 두고, 해제 시에도 그 값이 맞는지 확인한다.

## DB 락과 뭐가 다르냐

DB의 비관적 락, 낙관적 락은 주로 데이터베이스 트랜잭션 범위 안에서 동시성을 제어한다.  
반면 분산 락은 여러 애플리케이션 인스턴스 사이의 작업 단위를 제어하려는 목적이 더 강하다.

즉, DB 레코드 보호가 목적이면 DB 락이 더 적절할 수 있고, 여러 서버에서 "같은 작업을 동시에 시작하지 않게" 막고 싶다면 분산 락이 더 잘 맞는다.

면접에서는 이 차이를 구분해서 말하는 게 중요하다.

## 위험한 포인트

### 락 만료 시간이 너무 짧은 경우

작업이 끝나기 전에 락이 풀리면 다른 인스턴스가 같은 작업을 다시 시작할 수 있다.

### 락 만료 시간이 너무 긴 경우

작업 실패 후에도 오래 잠겨 있으면 처리 지연이 커지고, 장애 복구가 느려진다.

### 락만 믿고 비즈니스 안전장치를 안 두는 경우

분산 락은 좋은 보조 수단이지만 완벽한 진실 원천처럼 다루면 위험하다.  
중복 요청 방지 키, 멱등성 설계, 재시도 정책 같은 장치가 함께 가야 한다.

## 면접 답변 예시

> 분산 락은 여러 서버 인스턴스가 동시에 같은 자원이나 같은 작업을 처리하지 않도록 외부 저장소를 기준으로 거는 락입니다. 단일 서버에서는 프로세스 내부 락으로 충분할 수 있지만, 서버가 여러 대면 각 인스턴스 메모리가 분리돼 있어서 공통으로 조율할 수단이 필요합니다. 예를 들어 스케줄러 중복 실행 방지나 재고, 쿠폰 처리 같은 경쟁 상황에서 많이 사용합니다. Redis 기반으로 구현하는 경우가 많고, 락 키의 유일성, 적절한 만료 시간, 내가 획득한 락만 해제하는 검증이 중요합니다. 또 분산 락만으로 모든 동시성 문제가 해결되는 건 아니어서 멱등성이나 재시도 설계도 같이 가져가야 합니다.

## 장점

- 여러 인스턴스 환경에서 작업 중복 실행을 줄이기 좋다.
- 스케줄러, 배치, 경쟁 자원 처리 같은 실무 사례와 연결해 설명하기 좋다.
- DB 락과의 차이를 말하면 동시성 제어 관점을 더 넓게 보여 줄 수 있다.

## 단점

- 구현과 운영이 생각보다 까다롭다.
- 락 만료와 해제 타이밍을 잘못 잡으면 오동작이 생길 수 있다.
- 네트워크와 외부 저장소에 의존하므로 장애 시나리오를 반드시 고려해야 한다.

## 주의사항 / 실무 팁

- 면접에서는 "여러 서버라서 프로세스 내부 락이 안 통한다"는 전제를 먼저 말하면 흐름이 좋다.
- "분산 락 = 만능"처럼 말하지 말고, 멱등성과 함께 가야 안전하다고 덧붙이는 편이 낫다.
- Redis 기반 예시를 말할 때는 `SET NX EX`와 락 해제 시 토큰 검증 정도만 알아도 충분히 실무 감각이 난다.
- 락 범위를 너무 크게 잡으면 처리량이 급격히 떨어질 수 있으니, 임계 구역은 가능한 짧게 유지하는 게 좋다.
