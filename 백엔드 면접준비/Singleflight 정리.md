# Singleflight 정리

## 핵심 요약
- `Singleflight`는 같은 키에 대한 중복 요청이 동시에 몰릴 때 실제 작업은 한 번만 수행하고 결과를 여러 요청이 공유하게 만드는 방식이다.
- 캐시 미스 순간 DB나 외부 API로 동일한 요청이 폭발하는 문제를 줄일 때 자주 쓴다.
- 면접에서는 "동일한 작업을 합쳐서 중복 실행을 막는 in-flight 요청 병합 전략"이라고 설명하면 핵심이 잘 잡힌다.
- 캐시 자체를 대체하는 개념은 아니고, 캐시 미스나 재계산 구간을 보호하는 보조 장치에 가깝다.

## 개념 설명
인기 있는 데이터가 캐시에서 만료됐다고 가정해 보자. 그 순간 수백 개 요청이 동시에 들어오면 모두가 캐시 미스를 보고 DB나 외부 API를 때릴 수 있다. 이것이 심해지면 원래는 캐시 때문에 줄였어야 할 부하가 특정 순간에 다시 폭발한다.

`Singleflight`는 이런 상황에서 같은 키에 대한 첫 요청만 실제 조회를 수행하게 하고, 뒤에 들어온 요청들은 그 작업이 끝날 때까지 기다렸다가 같은 결과를 받아가게 만든다. 즉 "같은 일을 동시에 여러 번 하지 않게 막는 것"이 핵심이다.

이 방식은 특히 캐시 스탬피드 완화에 유용하다. 캐시가 비어 있거나 재생성 중일 때, 요청마다 원본 시스템을 다시 치지 않고 하나의 대표 요청만 원본을 조회하게 하면 DB나 외부 API 부하를 크게 줄일 수 있다. 다만 대표 요청이 느리거나 실패하면 대기 중인 요청들이 함께 영향을 받으므로 timeout과 에러 처리 전략을 같이 둬야 한다.

## 예시
사용자 프로필을 캐시에 저장하는 API를 생각해 보자.

- `user:123` 캐시가 막 만료됐다
- 같은 시점에 프로필 조회 요청 200개가 들어온다
- `Singleflight`가 없으면 200개가 모두 DB를 조회할 수 있다
- `Singleflight`가 있으면 첫 요청만 DB를 조회하고, 나머지는 그 결과를 같이 받는다

이렇게 하면 짧은 순간의 폭발적 중복 부하를 막을 수 있다.

## 면접 답변 예시
- 질문: "`Singleflight`는 언제 쓰나요?"  
  답변: "같은 키에 대한 요청이 동시에 몰릴 때 동일한 조회나 계산을 한 번만 수행하도록 묶고 싶을 때 씁니다. 대표적으로 캐시 미스 순간 DB나 외부 API 호출이 중복되는 문제를 줄이는 데 유용합니다."
- 질문: "캐시와 어떤 관계가 있나요?"  
  답변: "캐시를 대신하는 개념은 아니고, 캐시가 비었거나 재생성되는 순간 중복 원본 조회를 줄이는 보조 전략입니다. 그래서 캐시 스탬피드 대응과 같이 설명하는 경우가 많습니다."

## 장점
- 짧은 순간에 몰리는 중복 부하를 효과적으로 줄일 수 있다
- 캐시 스탬피드, 인기 키 재계산, 외부 API 비용 문제와 연결해서 설명하기 좋다
- 락보다는 목적이 선명해서 "같은 작업 병합"이라는 관점으로 말하기 쉽다

## 단점
- 대표 요청이 느리거나 실패하면 기다리던 요청도 함께 지연되거나 실패할 수 있다
- 키 기준을 잘못 잡으면 서로 다른 요청까지 과하게 묶일 수 있다
- 프로세스 로컬 방식이면 멀티 인스턴스 환경에서는 효과가 제한될 수 있다

## 주의사항 / 실무 팁
- `Singleflight`는 중복 제거 대상이 되는 키 설계가 가장 중요하다.
- 대기 시간이 길어질 수 있으므로 timeout, fallback, 실패 전파 정책을 함께 정해야 한다.
- 단일 인스턴스에서는 간단하지만, 여러 인스턴스가 있으면 분산 락이나 공유 캐시 전략과 함께 검토해야 한다.
- 면접에서는 `캐시 스탬피드`, `Bulkhead Pattern`, `Rate Limiting`과 구분해서 설명하면 더 깔끔하다.
