# 멱등성(Idempotency) 정리

## 핵심 요약

- 멱등성은 같은 요청을 여러 번 보내도 서버 상태가 한 번 처리한 것과 같게 유지되는 성질이다.
- 면접에서는 "같은 요청이 재시도돼도 결과가 꼬이지 않게 만드는 방법" 정도로 설명하면 이해가 쉽다.
- 네트워크 타임아웃, 중복 클릭, 메시지 재전달 같은 상황에서 특히 중요하다.
- HTTP 메서드의 멱등성과 비즈니스 로직의 멱등성은 완전히 같은 얘기가 아니어서 구분해서 말하는 게 좋다.

## 멱등성이란?

멱등성은 같은 연산을 여러 번 수행해도 최종 결과가 같게 유지되는 성질이다.

백엔드에서는 보통 이런 상황을 떠올리면 된다.

- 결제 요청 버튼이 두 번 눌렸다.
- 클라이언트가 응답을 못 받아서 같은 요청을 재시도했다.
- 메시지 큐가 같은 이벤트를 한 번 더 전달했다.

이때 서버가 요청을 받은 횟수만큼 계속 처리해 버리면 중복 결제, 중복 주문, 중복 포인트 적립 같은 문제가 생긴다.  
그래서 실무에서는 "요청이 중복으로 들어와도 한 번만 처리한 것처럼 보이게" 설계하는데, 이게 멱등성을 챙기는 이유다.

## 왜 중요한가?

분산 시스템에서는 요청이 정확히 한 번만 전달된다고 믿기 어렵다.  
네트워크는 실패할 수 있고, 클라이언트는 재시도할 수 있고, 브로커는 같은 메시지를 다시 전달할 수 있다.

즉 문제는 "중복 요청이 들어오느냐"가 아니라 "중복 요청이 들어오는 걸 전제로 어떻게 안전하게 처리하느냐"에 가깝다.

면접에서는 이렇게 설명하면 자연스럽다.

> 멱등성은 재시도 가능한 시스템을 만들기 위한 핵심 조건입니다. 요청이 실패했을 때 안심하고 다시 보내려면, 같은 요청이 여러 번 들어와도 상태가 깨지지 않아야 합니다.

## HTTP 메서드에서의 멱등성

HTTP 스펙에서도 멱등성 얘기가 자주 나온다.

### GET

같은 조회를 여러 번 해도 서버 상태를 바꾸지 않으므로 보통 멱등적이라고 본다.

### PUT

리소스를 "이 값으로 덮어쓴다"는 의미라서 같은 요청을 여러 번 보내도 최종 상태는 같다.  
예를 들어 사용자 닉네임을 `"bocelog"`로 바꾸는 PUT 요청을 세 번 보내도 결과는 동일하다.

### DELETE

첫 번째 요청에서 삭제되고, 그 뒤 요청은 이미 없는 상태일 수 있지만 최종 상태는 여전히 "삭제됨"이다.  
그래서 일반적으로 멱등적이라고 본다.

### POST

POST는 보통 멱등적이지 않다.  
주문 생성, 결제 생성처럼 호출할 때마다 새로운 리소스나 새로운 부작용이 생길 수 있기 때문이다.

다만 여기서 많이 헷갈리는 포인트가 있다.  
HTTP 메서드로서 POST는 기본적으로 비멱등적이지만, 서버 구현을 잘하면 "특정 POST API를 멱등하게" 만들 수는 있다.

## HTTP 멱등성과 비즈니스 멱등성은 다르다

이 부분을 짚어 주면 답변이 한 단계 좋아진다.

- HTTP 멱등성: 메서드 의미 차원에서 같은 요청을 반복했을 때의 성질
- 비즈니스 멱등성: 실제 서비스 로직에서 중복 처리 방지까지 포함한 성질

예를 들어 `POST /payments`는 원래 비멱등적으로 보이지만, `Idempotency-Key`를 받아서 같은 키로 들어온 요청은 같은 결제 결과를 반환하게 만들면 비즈니스적으로는 멱등하게 동작시킬 수 있다.

## 멱등성을 만드는 대표적인 방법

## 1. 고유 요청 키(Idempotency Key) 사용

클라이언트가 요청마다 고유한 키를 보내고, 서버는 이 키를 저장해 둔다.  
같은 키로 다시 요청이 오면 새로 처리하지 않고 기존 결과를 반환한다.

예:

- 첫 요청: `Idempotency-Key: pay-20260519-001`
- 서버: 결제 처리 후 결과 저장
- 재시도 요청: 같은 키 전달
- 서버: 중복 결제 대신 기존 응답 반환

결제, 주문 생성, 외부 API 연동에서 자주 쓰는 방식이다.

## 2. 유니크 제약 조건 사용

DB에서 중복되면 안 되는 값에 유니크 인덱스를 걸어 중복 생성을 막는 방식이다.

예:

- `order_id`
- `payment_request_id`
- `event_id`

애플리케이션 단에서 한 번 체크하고 끝내는 것보다, 최종적으로 DB 제약 조건까지 두는 편이 더 안전하다.  
실무에서는 "애플리케이션 체크 + DB 유니크 제약"을 같이 가는 경우가 많다.

## 3. 상태 전이 조건으로 막기

이미 처리된 상태라면 다시 처리하지 않도록 조건을 거는 방식이다.

예를 들어 주문 상태가:

- `CREATED`
- `PAID`
- `CANCELED`

처럼 관리된다면, 이미 `PAID`인 주문에 다시 결제 완료 처리를 하지 않도록 막을 수 있다.

이 방식은 "현재 상태에서 허용되는 전이만 수행한다"는 관점이라서, 워크플로우가 분명한 시스템에서 특히 유용하다.

## 4. 메시지 소비 이력 저장

이벤트 기반 시스템에서는 같은 메시지가 다시 올 수 있다.  
그래서 메시지 ID를 저장해 두고 이미 처리한 메시지면 건너뛴다.

예:

- Kafka, RabbitMQ, SQS 소비자
- 웹훅 재전송 처리
- 배치 재실행

이때도 핵심은 "중복 전달은 정상 상황"이라고 보고 방어하는 것이다.

## 예시로 이해하기

### 결제 API

가장 흔한 예시다.  
사용자가 결제 버튼을 눌렀는데 응답이 늦어서 다시 눌렀다고 해 보자.

멱등성이 없으면:

- 카드가 두 번 승인될 수 있다.
- 주문이 두 건 생성될 수 있다.

멱등성이 있으면:

- 같은 요청 키는 한 번만 처리된다.
- 이후 재시도는 기존 결제 결과를 돌려준다.

### 회원 포인트 적립

`event_id=signup_bonus_123` 같은 값을 기준으로 이미 적립한 이벤트인지 확인하면 중복 적립을 막을 수 있다.

### 웹훅 처리

외부 서비스가 타임아웃 때문에 웹훅을 여러 번 보낼 수 있다.  
이때 단순히 "올 때마다 처리"하면 재고, 정산, 알림이 중복될 수 있다.  
웹훅 이벤트 ID를 기준으로 처리 이력을 저장해 두면 안정적이다.

## 면접 답변 예시

멱등성은 같은 요청을 여러 번 수행해도 최종 상태가 같게 유지되는 성질입니다. 백엔드에서는 네트워크 재시도나 중복 클릭, 메시지 재전달 때문에 같은 요청이 다시 들어오는 상황이 자주 있어서 중요합니다. 예를 들어 결제 API가 멱등하지 않으면 같은 결제가 두 번 처리될 수 있습니다. 그래서 실무에서는 `Idempotency-Key`를 두거나, DB 유니크 제약 조건을 걸거나, 이미 처리한 이벤트 ID를 저장해서 중복 처리를 막습니다. 참고로 HTTP에서 GET, PUT, DELETE는 일반적으로 멱등적이라고 보지만, POST는 기본적으로 비멱등적입니다. 다만 POST도 서버 구현에 따라 비즈니스적으로 멱등하게 만들 수 있습니다.

## 장점

- 재시도에 강한 시스템을 만들 수 있다.
- 중복 결제, 중복 주문, 중복 적립 같은 치명적인 버그를 줄일 수 있다.
- 분산 시스템이나 이벤트 기반 구조에서 안정성이 높아진다.
- 장애 복구나 배치 재실행 시에도 운영 부담이 줄어든다.

## 단점

- 요청 키 저장, 중복 체크, 상태 관리 등 추가 설계가 필요하다.
- 저장소를 어떻게 관리할지, 키를 얼마나 오래 보관할지 같은 운영 고민이 생긴다.
- 잘못 설계하면 정상 요청까지 중복으로 오판할 수 있다.
- 단순 CRUD보다 구현 복잡도가 올라간다.

## 주의사항 / 실무 팁

- 멱등성은 "응답이 완전히 똑같다"보다 "최종 상태가 안전하게 유지된다"에 더 가깝다.
- 애플리케이션 코드만 믿지 말고 DB 유니크 제약이나 상태 조건 같은 최종 방어선도 두는 게 좋다.
- `Idempotency-Key`는 충돌하지 않도록 충분히 고유해야 하고, TTL 정책도 함께 정해야 한다.
- 분산 환경에서는 캐시 한 곳만 보고 중복 체크하면 레이스 컨디션이 날 수 있어서 저장소 일관성을 같이 봐야 한다.
- HTTP 메서드의 멱등성과 실제 비즈니스 멱등성을 섞어서 말하면 답변이 흐려진다. 면접에서는 둘을 분리해서 설명하는 편이 좋다.
- "정확히 한 번 처리"를 보장한다고 말하기보다, 보통은 "중복 요청이 와도 한 번 처리한 효과를 내도록 설계한다"라고 표현하는 게 더 실무적이다.
