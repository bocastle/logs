# 로드 밸런싱 알고리즘 정리

## 핵심 요약

- 로드 밸런싱은 들어오는 요청을 여러 서버에 고르게 나눠서 한쪽 서버만 과부하가 걸리지 않게 만드는 방식이다.
- 알고리즘마다 기준이 다르다. 단순히 순서대로 보내는 방식도 있고, 현재 연결 수나 응답 시간처럼 "지금 덜 바쁜 서버"를 보는 방식도 있다.
- 면접에서는 알고리즘 이름만 외우기보다 "트래픽 특성에 따라 왜 이 방식을 고르는지"까지 설명할 수 있으면 훨씬 좋다.
- 실무에서는 로드 밸런싱 알고리즘 하나만 보지 않고, 세션 유지, 서버 스펙 차이, 헬스 체크, 캐시 특성까지 함께 본다.

## 로드 밸런싱이 왜 필요한가

서버가 한 대일 때는 요청이 몰리면 CPU, 메모리, DB 커넥션 같은 자원이 금방 한계에 닿는다.  
그래서 보통 같은 역할을 하는 서버를 여러 대 두고, 앞단의 로드 밸런서가 요청을 나눠서 보내게 된다.

이때 중요한 게 "어떻게 나눌 것인가"다.  
아무 기준 없이 보내면 특정 서버에 요청이 몰릴 수 있고, 반대로 너무 복잡한 기준을 쓰면 운영이 어려워질 수 있다.

결국 로드 밸런싱 알고리즘은 다음 질문에 대한 답이라고 보면 된다.

- 서버들을 어떤 기준으로 선택할까?
- 모든 서버를 똑같이 볼까, 아니면 성능 차이를 반영할까?
- 짧은 요청이 많은가, 오래 붙는 연결이 많은가?

## 대표 알고리즘

## 1. Round Robin

가장 기본적인 방식이다.  
요청이 들어오면 서버 1, 서버 2, 서버 3 순서로 돌아가면서 하나씩 배분한다.

예를 들어 서버가 3대라면:

1. 첫 번째 요청은 A 서버
2. 두 번째 요청은 B 서버
3. 세 번째 요청은 C 서버
4. 네 번째 요청은 다시 A 서버

### 언제 좋을까

- 서버 스펙이 비슷할 때
- 요청 처리 시간이 크게 다르지 않을 때
- 단순하고 예측 가능한 분산이 필요할 때

### 한계

겉보기에는 공평하지만, 실제 부하가 공평하다는 뜻은 아니다.  
어떤 요청은 10ms 만에 끝나고 어떤 요청은 10초 동안 연결을 잡고 있을 수 있다. 이런 상황에서는 단순 Round Robin만으로는 실제 부하를 잘 반영하지 못한다.

## 2. Weighted Round Robin

Round Robin에 가중치를 넣은 방식이다.  
성능이 좋은 서버에는 더 많은 요청을 보내고, 성능이 낮은 서버에는 덜 보낸다.

예를 들어:

- A 서버 가중치 3
- B 서버 가중치 2
- C 서버 가중치 1

이면 A에 가장 많이, C에 가장 적게 요청이 간다.

### 언제 좋을까

- 서버 사양이 서로 다를 때
- 점진적으로 서버를 교체하는 환경일 때
- 특정 서버에 더 많은 트래픽을 받아도 되는 구조일 때

### 한계

가중치 설정이 고정값이면 순간적인 부하 변화까지 반영하진 못한다.  
즉, "원래 더 좋은 서버"라는 사실은 반영하지만 "지금 더 바쁜 서버"인지는 모를 수 있다.

## 3. Least Connections

현재 연결 수가 가장 적은 서버로 요청을 보내는 방식이다.  
즉, 지금 가장 한가해 보이는 서버를 고른다고 이해하면 된다.

이 방식은 요청마다 처리 시간이 꽤 다를 때 유리하다.  
예를 들어 어떤 사용자는 짧게 API 한 번만 호출하고 끝나지만, 어떤 사용자는 오래 유지되는 연결을 많이 쓰는 경우가 있다. 이런 경우 단순 Round Robin보다 현재 연결 수를 보는 편이 더 현실적이다.

### 언제 좋을까

- 연결 유지 시간이 제각각일 때
- WebSocket, SSE처럼 오래 붙는 연결이 있을 때
- 요청 수보다 현재 활성 연결 수가 더 중요한 서비스일 때

### 한계

연결 수가 적다고 해서 반드시 CPU나 메모리 사용량도 적은 건 아니다.  
연결은 적어도 아주 무거운 작업을 수행 중일 수 있기 때문이다.

## 4. Weighted Least Connections

Least Connections에 가중치를 더한 방식이다.  
서버 성능 차이와 현재 연결 수를 같이 보겠다는 접근이다.

예를 들어 사양이 좋은 서버는 연결이 조금 더 많아도 감당할 수 있으니, 단순 연결 수만 보는 것보다 더 현실적인 분산이 가능하다.

### 언제 좋을까

- 서버 스펙이 다르고
- 연결 유지 시간도 일정하지 않을 때

실무에서는 꽤 설득력 있는 선택지다.

## 5. IP Hash

클라이언트 IP를 기준으로 해시 값을 만들고, 그 결과에 따라 항상 비슷한 서버로 보내는 방식이다.

이 방식의 핵심은 "같은 사용자는 웬만하면 같은 서버로 간다"는 점이다.

### 언제 좋을까

- 세션을 서버 메모리에 들고 있는 구조일 때
- 사용자별 캐시 적중률이 중요할 때
- 어느 정도의 sticky session 효과가 필요할 때

### 한계

- NAT 환경에서는 많은 사용자가 같은 IP로 보일 수 있다.
- 특정 IP 대역에 트래픽이 몰리면 분산이 깨질 수 있다.
- 서버 수가 바뀌면 매핑 결과가 크게 흔들릴 수 있다.

## 6. 응답 시간 기반 / 리소스 기반 방식

더 고급 설정에서는 단순 연결 수 대신 응답 시간, CPU 사용률, 큐 길이 같은 지표를 반영하기도 한다.

이 방식은 이론적으로는 더 똑똑하지만, 운영 난이도가 올라간다.  
메트릭 수집 주기, 지표 신뢰도, 반응 속도가 맞지 않으면 오히려 분산이 출렁거릴 수 있다.

## 알고리즘을 어떻게 선택하면 좋을까

면접에서는 "정답 하나"처럼 말하기보다 트래픽 특성과 시스템 구조를 기준으로 고른다고 설명하는 편이 좋다.

예를 들면 이렇게 정리할 수 있다.

- 서버 스펙이 비슷하고 요청도 짧고 균일하다면: Round Robin
- 서버 스펙이 다르면: Weighted Round Robin
- 요청 처리 시간 차이가 크고 오래 붙는 연결이 많다면: Least Connections
- 세션 고정 성격이 필요하면: IP Hash 또는 Sticky Session 계열

## 예시로 이해하기

채팅 서비스나 실시간 알림 서비스처럼 WebSocket 연결이 오래 유지되는 환경을 떠올려보면 좋다.  
이때 Round Robin만 쓰면 앞에서 먼저 연결을 많이 받은 서버가 계속 무거워질 수 있다. 반면 Least Connections는 현재 연결 수를 보고 덜 바쁜 서버로 보내기 때문에 더 자연스럽다.

반대로 짧은 REST API 호출 위주 서비스라면 연결이 금방 끝나므로 Round Robin도 충분히 괜찮을 수 있다.

## Sticky Session과의 관계

면접에서 자주 같이 나오는 포인트다.  
로드 밸런싱 알고리즘과 sticky session은 비슷해 보이지만 완전히 같은 개념은 아니다.

- 로드 밸런싱 알고리즘: 어떤 서버를 선택할지에 대한 규칙
- Sticky Session: 한 번 연결된 사용자를 같은 서버에 붙잡아 두는 전략

예전 구조에서는 세션을 각 서버 메모리에 저장하는 경우가 많아서 sticky session이 자주 쓰였다.  
하지만 서버가 죽으면 세션이 날아가고, 분산도 고르지 않아질 수 있다. 그래서 요즘은 세션을 Redis 같은 외부 저장소로 빼고, 로드 밸런서는 좀 더 자유롭게 분산시키는 구조를 많이 쓴다.

## 면접 답변 예시

로드 밸런싱 알고리즘은 여러 서버에 요청을 어떤 기준으로 분산할지 정하는 방식입니다. 가장 기본은 Round Robin으로, 요청을 순서대로 서버에 배분합니다. 서버 성능이 다르면 Weighted Round Robin을 쓸 수 있고, 현재 활성 연결 수를 기준으로 더 한가한 서버에 보내고 싶다면 Least Connections가 적합합니다. 만약 WebSocket처럼 오래 유지되는 연결이 많다면 단순 Round Robin보다 Least Connections 계열이 더 현실적입니다. 또 세션을 같은 서버에 유지해야 하는 구조라면 IP Hash나 Sticky Session 전략도 함께 고려할 수 있습니다. 실무에서는 알고리즘 이름만 고르기보다 요청 특성, 서버 스펙, 세션 저장 방식, 장애 대응까지 함께 보고 결정하는 게 중요합니다.

## 장점

- 트래픽을 여러 서버에 분산해 확장성과 가용성을 높일 수 있다.
- 서비스 특성에 맞는 알고리즘을 쓰면 자원 활용 효율이 좋아진다.
- 장애가 난 서버를 헬스 체크로 제외하면 안정성이 올라간다.

## 단점

- 알고리즘이 복잡해질수록 운영과 튜닝 난이도가 올라간다.
- 잘못된 분산 전략은 오히려 특정 서버에 부하를 몰 수 있다.
- 세션, 캐시, 연결 유지 방식과 맞물리면 생각보다 단순하지 않다.

## 주의사항 / 실무 팁

- 서버 수가 같다고 해서 무조건 Round Robin이 정답은 아니다. 요청 처리 시간 분포를 먼저 봐야 한다.
- 세션을 서버 메모리에 저장하는 구조라면 로드 밸런싱 정책과 세션 유지 전략을 같이 봐야 한다.
- WebSocket, SSE, 스트리밍처럼 연결이 오래 지속되는 서비스는 연결 수 기반 정책이 더 잘 맞는 경우가 많다.
- CPU 사용률, 응답 시간 기반 정책은 좋아 보이지만 메트릭 수집 지연 때문에 오히려 흔들릴 수 있다.
- 면접에서는 "우리 서비스가 어떤 트래픽인지에 따라 알고리즘을 고른다"는 관점을 넣어 말하면 답변이 훨씬 탄탄해진다.
