# API cursor pagination에서 중복 누락을 막는 법 정리

## 핵심 요약

- 면접에서 "API cursor pagination에서 중복 누락을 막는 법" 질문을 받으면 핵심은 "호출자가 실패를 예측 가능하게 처리할 수 있는 계약인가?"이다.
- 정상 흐름만 설명하면 얕게 들리므로 timeout, 재시도, 중복 처리까지 이어 가는 편이 좋다.
- 기술 선택의 장점과 함께 비용, 복잡도, 되돌림 기준을 같이 정리해야 한다.

## 개념 설명

API 운영 주제는 클라이언트와 서버 사이의 약속을 어디까지 명확히 할지 묻는 문제다.

timeout, retry, rate limit, pagination, 호환성은 각각 따로 보이지만 결국 장애 전파와 사용자 경험을 줄이는 기준이다.

"API cursor pagination에서 중복 누락을 막는 법" 관련 질문을 받으면 용어 정의에서 멈추기보다 적용 조건과 실패 시나리오를 같이 잡아야 한다.

## 예시

```text
request -> gateway -> service -> datastore
             |          |
             |          +-> timeout / retry / metric
             +-> auth / rate limit / trace id
```

코드는 짧지만 면접에서는 이 흐름이 실패했을 때 어떤 상태가 남는지까지 이어서 설명하는 것이 좋다.

## 면접 답변 예시

> "API cursor pagination에서 중복 누락을 막는 법" 질문이라면 저는 호출자가 예측 가능한 오류와 timeout을 받도록 계약을 정하고 retry 가능 여부를 명확히 표시하겠다고 답하겠습니다. 단순 구현 방법보다 어떤 장애를 줄이는지, 문제가 생겼을 때 어떤 지표로 알아차릴지, 변경을 되돌릴 기준은 무엇인지까지 말하는 것이 중요합니다.

## 장점

- 호출자가 실패를 예측 가능하게 처리할 수 있다.
- 장애 전파를 timeout과 rate limit으로 제한할 수 있다.
- 버전과 호환성 정책을 문서화하기 쉽다.

## 단점

- 호환성을 지키려면 오래된 계약을 오래 유지해야 한다.
- 제한 정책이 과하면 정상 사용자도 영향을 받을 수 있다.
- 재시도 정책이 잘못되면 트래픽 폭증을 만들 수 있다.

## 주의사항 / 실무 팁

- 재시도 가능 여부와 중복 처리 기준을 항상 같이 말한다.
- 로그, 메트릭, trace 중 어떤 신호로 문제를 발견할지 정한다.
- 운영 변경은 작은 범위로 적용하고 되돌릴 조건을 먼저 적어 둔다.
