# Negative Cache 정리

## 핵심 요약
- `Negative Cache`는 "없는 결과"나 "실패 결과"도 일정 시간 캐시에 저장하는 전략이다.
- 핵심 목적은 반복적으로 같은 실패를 조회하는 비용을 줄이고, 불필요한 DB나 외부 API 호출을 막는 것이다.
- 면접에서는 "존재하지 않는 데이터까지 캐싱해서 시스템 낭비를 줄이는 방식"이라고 설명하면 이해가 빠르다.
- 장점만 말하지 말고 TTL 설정 실수, 데이터 생성 직후의 stale 문제, 실패 종류별 분리 필요성도 함께 짚어야 한다.

## 개념 설명
보통 캐시를 이야기할 때는 "조회 성공 결과를 저장한다"는 관점이 먼저 떠오른다. 하지만 실제 서비스에서는 존재하지 않는 사용자 ID, 이미 삭제된 게시물, 외부 API의 일시적 404 같은 실패 조회도 반복해서 들어올 수 있다.

이때 매번 원본 저장소까지 내려가서 "없다"는 사실을 다시 확인하면 낭비가 커진다. `Negative Cache`는 이런 "없음" 또는 "실패" 결과 자체를 짧게 저장해서 같은 요청이 반복될 때 빠르게 응답하도록 만든다.

예를 들어 인기 없는 잘못된 키에 대한 조회가 대량으로 들어오면, 캐시가 없다면 DB에 쓸데없는 부하가 걸릴 수 있다. 반대로 `Negative Cache`를 두면 짧은 TTL 동안은 캐시가 바로 "없음"을 응답한다.

다만 모든 실패를 똑같이 캐시하면 위험하다. 영구적으로 없는 데이터와 일시적 장애를 구분해야 하고, TTL도 성공 캐시보다 더 짧게 잡는 경우가 많다.

## 예시
예를 들어 상품 상세 API에서 `productId=999999` 요청이 반복적으로 들어온다고 해 보자.

- DB를 조회해도 해당 상품이 없다
- 이 결과를 매번 DB에서 다시 확인하면 같은 낭비가 계속 발생한다
- 그래서 캐시에 `product:999999 -> NOT_FOUND` 같은 형태로 짧은 TTL을 둔다
- 이후 같은 요청이 오면 DB를 다시 조회하지 않고 바로 404를 반환한다

또 다른 예로 외부 사용자 프로필 API가 특정 ID에 대해 반복해서 "없음"을 반환한다면, 이 결과를 잠깐 캐싱해서 외부 호출 수를 줄일 수 있다.

## 면접 답변 예시
- 질문: "`Negative Cache`는 무엇인가요?"  
  답변: "존재하지 않는 데이터나 실패 결과도 일정 시간 캐시에 저장하는 전략입니다. 반복되는 실패 조회가 원본 저장소에 계속 부담을 주지 않도록 막는 목적이 있습니다."
- 질문: "왜 필요한가요?"  
  답변: "없는 데이터에 대한 요청도 실제 운영에서는 반복될 수 있기 때문입니다. 이런 요청마다 DB나 외부 API를 다시 호출하면 낭비가 커져서, 짧은 TTL로 실패 결과를 캐싱하면 부하를 줄일 수 있습니다."
- 질문: "주의할 점은 무엇인가요?"  
  답변: "실패 원인을 구분해야 합니다. 영구적인 404와 일시적 장애를 같은 방식으로 캐시하면 잘못된 응답을 오래 유지할 수 있습니다."

## 장점
- 존재하지 않는 데이터 조회로 인한 불필요한 원본 부하를 줄일 수 있다
- 잘못된 키나 반복 공격성 트래픽에 대한 방어 효과가 있다
- 외부 API 호출 수와 응답 지연을 줄이는 데 도움이 된다

## 단점
- TTL을 잘못 잡으면 실제로 데이터가 생긴 뒤에도 한동안 "없음"으로 응답할 수 있다
- 실패 원인을 구분하지 않으면 장애 상황을 더 오래 숨길 수 있다
- 성공 캐시보다 정책 설계가 더 섬세해야 해서 운영 복잡도가 올라간다

## 주의사항 / 실무 팁
- `404 Not Found`와 `500 Internal Server Error`를 같은 negative cache 대상으로 보면 안 된다.
- 보통은 성공 캐시보다 짧은 TTL을 두고, 데이터 생성 이벤트가 있으면 관련 negative cache를 함께 날리는 전략이 필요하다.
- 면접에서는 "`없는 것도 캐싱한다`는 발상이 포인트"라고 짚어 주면 인상적으로 들린다.
- 실무에서는 캐시 키 설계, TTL, 무효화 타이밍을 함께 설명해야 답변이 더 탄탄해진다.
