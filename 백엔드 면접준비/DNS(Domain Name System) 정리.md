# DNS(Domain Name System) 정리

## 핵심 요약

- DNS는 **도메인 이름을 IP 주소로 변환**하고, 그 매핑을 **관리/질의**하기 위한 시스템(및 프로토콜)입니다.
- DNS는 **계층 구조의 네임 서버**(로컬/루트/TLD/권한)를 통해 질의를 단계적으로 해결합니다.

## DNS란 무엇인가요?

IP 주소는 변환될 수 있으며, 기억하기 어렵기 때문에 대부분의 웹 서비스는 도메인 주소를 사용합니다. **DNS(Domain Name System)** 는 도메인 주소에 대응되는 원격 호스트 IP 주소를 관리하고 질의할 수 있는 시스템 혹은 이를 이용하기 위한 프로토콜을 의미합니다. DNS 시스템 내에서 도메인 주소를 관리하는 서버들을 **네임 서버(Name Server)** 라고 부르며, **계층적인 형태**로 존재합니다.

## DNS 질의 과정을 설명해 주세요.

DNS를 이용해 IP 주소를 찾는 과정에는 **로컬 네임 서버**, **루트 네임 서버**, **TLD 네임 서버**, **권한 네임 서버**가 등장합니다.

### 등장하는 네임 서버

- **로컬 네임 서버(Local Name Server)** 는 통신사 DNS, 구글 DNS 처럼 클라이언트와 가장 가까이 존재하는 네임 서버입니다.
- **루트 네임 서버(Root Name Server)** 는 루트 도메인을 관리하는 서버이며, TLD 네임 서버의 주소를 알고 있습니다.
- **TLD 네임 서버(Top-level Domain Server)** 는 `.com`, `.kr`, `.net`을 관리하는 네임 서버입니다.
- **권한 네임 서버(Authoriative Name Server)** 는 네트워크를 운영하는 기관에서 보유하고 있는 도메인을 관리하는 네임 서버입니다.
  - 예: `maeil-mail.kr`을 관리하고, `api.maeil-mail.kr`, `mail.maeil-mail.kr` 의 정보를 관리하는 네임 서버

### 예시: api.maeil-mail.kr의 IP 주소를 찾는 과정

1. 클라이언트는 **로컬 네임 서버**에 `api.maeil-mail.kr`의 IP 주소를 질의합니다.
2. 로컬 네임 서버에서 값을 찾지 못했다면, 로컬 네임 서버는 **루트 네임 서버**에게 IP 주소를 물어봅니다.
3. 요청을 받은 루트 네임 서버는 `.kr` **TLD 네임 서버 주소**를 응답합니다.
4. 로컬 네임 서버는 `.kr` **TLD 네임 서버**에 다시 요청을 보내고, `maeil-mail.kr` **권한 네임 서버 주소**를 응답받습니다.
5. 최종적으로 로컬 네임 서버는 **권한 네임 서버**에서 `api.maeil-mail.kr` IP 주소를 받아서 클라이언트에게 응답합니다.

## 장점

- 사람이 읽기 쉬운 도메인 이름으로도 서비스를 접근할 수 있게 해줍니다.
- 계층적인 네임 서버 구조로 도메인/IP 매핑을 체계적으로 관리할 수 있습니다.

## 단점

- 로컬 네임 서버에 캐시가 없으면 상위 네임 서버로 단계적으로 질의가 진행되어 지연이 발생할 수 있습니다.

## 주의사항 및 실무 팁

- 로컬 네임 서버가 값을 찾지 못하면 루트 → TLD → 권한 네임 서버로 이어지는 질의가 발생하므로, **어느 단계에서 응답이 늦거나 실패하는지**를 추적하는 것이 중요합니다.
- 권한 네임 서버는 특정 도메인(예: `maeil-mail.kr`)과 그 하위 도메인(예: `api.maeil-mail.kr`)의 정보를 관리하므로, 장애 시 **해당 도메인 계열 서비스 전반**에 영향이 생길 수 있습니다.
