# 데드락 정리

## 핵심 요약

- 데드락은 여러 트랜잭션이 서로가 가진 자원을 기다리면서 아무도 앞으로 못 가는 상태다.
- DB 락을 다루는 환경에서는 피하기보다 "줄이고, 빨리 감지하고, 재시도 가능하게" 설계하는 게 현실적이다.
- 면접에서는 원인, 예시, 대응 방식을 같이 말해야 답이 단단해진다.
- 데드락과 단순 락 대기는 다르다는 점을 구분해두면 좋다.

## 데드락이란

가장 흔한 예시는 이렇다.

1. 트랜잭션 A가 자원 1을 점유한다.
2. 트랜잭션 B가 자원 2를 점유한다.
3. A는 자원 2를 기다린다.
4. B는 자원 1을 기다린다.

이러면 둘 다 상대가 놓아주길 기다리기만 하고 끝나지 않는다. 이 상태가 데드락이다.

면접에서는 이렇게 설명하면 된다.

> 데드락은 둘 이상의 트랜잭션이 서로가 점유한 락을 기다리면서 무한 대기 상태에 빠지는 문제입니다. 단순히 느린 게 아니라, 구조적으로 서로 막고 있어서 한쪽을 중단하지 않으면 풀리지 않습니다.

## 왜 생기나

### 자원 획득 순서가 제각각일 때

어떤 요청은 A 테이블부터 잠그고, 다른 요청은 B 테이블부터 잠그면 데드락 가능성이 올라간다.

### 트랜잭션이 길 때

트랜잭션 안에서 외부 API 호출, 긴 계산, 불필요한 조회까지 해버리면 락을 오래 쥐고 있게 된다.  
그러면 충돌 확률이 커진다.

### 동시에 같은 데이터를 갱신할 때

재고 차감, 계좌 이체, 정산 같은 시나리오에서 자주 나온다.

## 단순 락 대기와 뭐가 다르나

락 대기는 기다리면 풀릴 수 있다.  
예를 들어 한 트랜잭션이 끝나면 다른 트랜잭션이 이어서 실행될 수 있다.

반면 데드락은 기다리는 구조 자체가 원형으로 꼬여 있어서 그냥 기다려서는 안 풀린다.  
보통 DB가 데드락을 감지하면 둘 중 하나를 강제로 롤백한다.

## 예시로 이해하기

계좌 이체를 생각해보자.

- A 요청: 계좌 100의 잔액을 먼저 잠그고, 그다음 계좌 200을 잠근다.
- B 요청: 계좌 200의 잔액을 먼저 잠그고, 그다음 계좌 100을 잠근다.

두 요청이 동시에 들어오면 서로 상대 계좌 락이 풀리기만 기다리게 될 수 있다.

실무에서 이 문제는 꼭 "DB 두 개"일 때만 생기는 게 아니다.  
같은 테이블 안의 두 행을 다른 순서로 잡아도 충분히 발생한다.

## 어떻게 예방하나

## 1. 락 획득 순서를 일관되게 맞춘다

가장 대표적인 방법이다.  
예를 들어 계좌 ID가 작은 것부터 먼저 잠그기로 통일하면, 서로 반대 순서로 잡는 상황을 줄일 수 있다.

면접에서도 이 포인트를 말하면 실무 감각이 있다는 인상을 준다.

## 2. 트랜잭션을 짧게 유지한다

트랜잭션 안에서 오래 걸리는 작업을 줄이면 락 점유 시간이 줄어든다.

- 외부 API 호출은 가능하면 트랜잭션 바깥으로 뺀다.
- 불필요한 조회나 복잡한 비즈니스 로직을 최소화한다.
- 사용자 입력 대기 같은 작업은 절대 트랜잭션 안에 두지 않는다.

## 3. 작은 범위만 잠근다

필요한 데이터만 정확히 잠그고, 넓은 범위를 한 번에 잠그지 않도록 설계하는 게 좋다.

## 4. 재시도 전략을 준비한다

DB가 데드락을 감지하면 보통 한 트랜잭션을 실패시킨다.  
그래서 애플리케이션은 이 실패를 "일시적 충돌"로 보고 재시도할 수 있어야 한다.

물론 무한 재시도는 위험하고, 횟수 제한과 백오프를 두는 편이 낫다.

## 어떻게 발견하나

### DB 에러 로그

MySQL, PostgreSQL 같은 DB는 데드락 발생 시 관련 에러를 남긴다.  
운영에서 간헐적으로 특정 갱신 API만 실패한다면 로그를 먼저 봐야 한다.

### 실패 패턴 분석

동시에 요청이 몰릴 때만 간헐적으로 롤백된다면 데드락 가능성을 의심할 수 있다.

### 재현 시나리오 작성

같은 자원을 다른 순서로 잠그는 테스트를 만들면 문제를 더 명확히 볼 수 있다.

## 면접 답변 예시

> 데드락은 여러 트랜잭션이 서로가 가진 락을 기다리면서 무한 대기에 빠지는 상황입니다. 예를 들어 두 요청이 같은 두 자원을 반대 순서로 잠그면 발생할 수 있습니다. 저는 예방 측면에서는 락 획득 순서를 일관되게 맞추고 트랜잭션을 짧게 유지하는 게 중요하다고 생각합니다. 그래도 완전히 없어지지 않을 수 있어서, 운영에서는 DB 에러 로그를 보고 감지하고 애플리케이션에서 제한된 재시도 전략도 준비하는 편이 현실적입니다.

## 장점

- 락과 트랜잭션을 깊이 이해하고 있다는 점을 보여주기 좋다.
- 단순 이론이 아니라 실무 장애 대응 관점까지 연결해서 말할 수 있다.
- 동시성 제어 주제와 함께 묶어 답변하기 좋다.

## 단점

- 재현이 쉽지 않아 원인 파악에 시간이 걸릴 수 있다.
- 예방책이 있어도 완전히 0으로 만들기 어렵다.
- 대응을 잘못하면 재시도 폭주나 긴 트랜잭션 같은 2차 문제가 생길 수 있다.

## 주의사항 / 실무 팁

- 데드락을 "DB가 느린 현상" 정도로 말하면 부족하다. 상호 대기 구조라는 점을 짚어야 한다.
- 락 순서 통일은 생각보다 효과가 크다. 같은 비즈니스 로직끼리 조회/수정 순서를 맞춰두자.
- 트랜잭션 안에서 외부 연동을 하지 않는 습관이 중요하다.
- 재시도 로직을 넣을 때는 멱등성과 함께 검토해야 안전하다.
