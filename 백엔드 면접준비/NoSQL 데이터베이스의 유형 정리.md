# NoSQL 데이터베이스의 유형 정리

## 핵심 요약

- NoSQL 데이터베이스의 유형: **키-값**, **문서 지향**, **열 지향**, **그래프**, **시계열**
- 각 유형은 데이터 모델과 강점이 달라 **용도(세션/캐시, 콘텐츠/프로필, 대규모 분석, 관계 분석, 시계열 메트릭 등)** 에 따라 선택합니다.
- 실시간 채팅 앱 예시로는 **Redis(Pub/Sub)** + **MongoDB(영구 저장, 샤딩 기반 수평 확장)** 구성이 제시됩니다.

## NoSQL 데이터베이스의 유형

### 키-값 데이터베이스(Key-value Database)

키를 고유한 식별자로 사용하는 **키-값 쌍** 형태로 데이터를 저장합니다.

- 특징: 구조가 단순하고, 빠른 읽기 및 쓰기 성능 제공
- 대표 예시: Redis, Amazon DynamoDB
- 활용 예: 세션 저장, 캐시, 실시간 순위

### 문서 지향 데이터베이스(Document-oriented Database)

JSON, BSON, XML 등의 형식으로 데이터를 저장합니다.

- 특징: 유연한 스키마, 복잡한 데이터 구조를 쉽게 표현
- 대표 예시: MongoDB, CouchDB
- 활용 예: 콘텐츠 관리 시스템, 사용자 프로필 저장

### 열 지향 데이터베이스(Column Family Database)

데이터를 **열 단위**로 저장합니다.

- 특징: 대량 데이터 처리에 적합, 행마다 각기 다른 수의 열과 여러 데이터 유형을 가질 수 있음
- 대표 예시: Apache Cassandra, HBase
- 활용 예: 대규모 데이터 분석, 로그 수집

### 그래프 데이터베이스(Graph Database)

노드(Node), 엣지(Edge) 구조로 구성된 그래프로 데이터를 저장합니다.

- 특징: 복잡한 관계 표현에 사용, 레이블(그룹화된 노드)로 쿼리를 쉽게 작성/효율적으로 실행
- 대표 예시: Neo4j, Amazon Neptune
- 활용 예: 소셜 네트워크 분석, 추천 시스템

### 시계열 데이터베이스(Time Series Database)

시간에 따라 변화하는 데이터를 저장합니다.

- 특징: 타임스탬프가 있는 메트릭/이벤트 처리에 사용, 시간 경과에 따른 변화를 측정하는 데 최적화
- 대표 예시: InfluxDB, Prometheus, TimescaleDB
- 활용 예: IoT 데이터 수집, 금융 데이터 분석

## 실시간 채팅 앱에 적합한 NoSQL 구성 예시

> 개인의 경험과 지식에 따라 다르게 답변할 수 있으며, 아래는 하나의 예시입니다.

실시간 채팅 앱에서는 메시지를 빠르게 주고받는 처리 속도와, 유연하고 수평 확장이 가능한 저장 구조가 중요하다고 봅니다.

- 실시간 메시지 전송: **Redis의 Pub/Sub 기능**  
  낮은 지연 시간으로 사용자 간 메시지를 브로드캐스트할 수 있기 때문입니다.

- 메시지 영구 저장: **MongoDB**  
  Redis는 영구 저장보다 캐시/메시지 브로커 역할에 더 적합하므로, 실제 메시지는 MongoDB에 저장합니다.  
  MongoDB는 문서 지향 DB로서 채팅 메시지/사용자 정보를 JSON 형식으로 유연하게 저장할 수 있고, **샤딩(Sharding)** 기능으로 수평 확장이 가능해 사용자 수/메시지 양 증가에도 안정적으로 확장할 수 있다고 봅니다.

## 장점

- (유형별로) 키-값은 구조가 단순하고 읽기/쓰기가 빠릅니다.
- 문서 지향은 유연한 스키마로 복잡한 데이터 구조를 표현하기 쉽습니다.
- 열 지향은 대량의 데이터를 처리하는 데 적합합니다.
- 그래프는 복잡한 관계를 표현하고 효율적으로 질의하기 좋습니다.
- 시계열은 시간 기반 메트릭/이벤트 처리에 최적화되어 있습니다.

## 단점

- 원문에서는 별도의 단점이 직접적으로 언급되지는 않았습니다.

## 주의사항 / 실무 팁

- NoSQL은 한 가지로 통일하기보다, 요구사항에 따라 **역할을 분리**해 조합할 수 있습니다.  
  (예: 실시간 전송은 Redis, 영구 저장은 MongoDB)
- “실시간성(낮은 지연)”과 “영구 저장/확장” 요구를 분리해, 각 저장소의 강점에 맞게 책임을 나누면 설계가 명확해집니다.
