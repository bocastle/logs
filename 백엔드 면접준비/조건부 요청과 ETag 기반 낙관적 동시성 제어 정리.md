# 조건부 요청과 ETag 기반 낙관적 동시성 제어 정리

## 핵심 요약
- `ETag`는 리소스 버전을 식별하는 값이고, 조건부 요청은 이 값을 기준으로 요청의 유효성을 검증하는 방식이다.
- 이를 동시성 제어에 활용하면 "내가 읽은 버전이 아직 최신일 때만 수정하라"는 조건을 서버에 전달할 수 있다.
- 면접에서는 "락을 오래 잡지 않고, 버전 비교로 충돌을 감지하는 낙관적 방식"이라고 설명하면 핵심이 전달된다.
- 충돌 빈도가 낮은 시스템에 잘 맞지만, 충돌이 나면 재시도나 사용자 재확인이 필요하다는 점까지 같이 말해야 답변이 균형 잡힌다.

## 개념 설명
여러 사용자가 같은 데이터를 수정할 수 있는 시스템에서는 "나중에 저장한 사람이 이전 사람의 변경을 덮어써 버리는 문제"가 자주 생긴다. 대표적으로 게시글 수정, 설정 변경, 관리자 콘솔 편집에서 이런 문제가 발생한다.

이때 비관적 락처럼 미리 잠그는 방식도 있지만, 모든 수정에 락을 걸면 대기 시간이 늘고 확장성도 떨어질 수 있다. 그래서 충돌이 자주 나지 않는다고 가정하고, 수정 시점에만 버전을 비교하는 낙관적 동시성 제어를 많이 쓴다.

`ETag`는 원래 캐시 검증에 자주 쓰이지만, 같은 원리로 동시성 제어에도 쓸 수 있다. 클라이언트가 데이터를 조회할 때 `ETag`를 함께 받고, 수정 요청 시 `If-Match`에 그 값을 넣어 보낸다. 서버는 현재 리소스의 `ETag`와 비교해서 같을 때만 수정하고, 다르면 충돌로 판단해 거절한다.

즉, 핵심은 "조회 시 본 버전과 수정 시점의 최신 버전이 같은지 확인해 안전하게 갱신하는 것"이다.

## 예시
예를 들어 두 명의 관리자가 같은 상품 설명을 수정한다고 해 보자.

- A와 B가 같은 상품을 조회했고, 둘 다 `ETag=v12`를 받았다
- A가 먼저 수정 요청을 보내고 서버가 저장한 뒤 `ETag=v13`이 된다
- B는 여전히 `If-Match: v12`로 수정 요청을 보낸다
- 서버는 현재 버전이 `v13`이므로 요청을 거절한다

이렇게 하면 B의 수정이 A의 최신 변경을 조용히 덮어쓰는 일을 막을 수 있다. 이후 B는 최신 데이터를 다시 읽고 수정 내용을 재적용해야 한다.

## 면접 답변 예시
- 질문: "`ETag`를 동시성 제어에 어떻게 활용할 수 있나요?"  
  답변: "조회 시 받은 `ETag`를 수정 요청의 `If-Match` 조건으로 보내고, 서버가 현재 버전과 비교해 같을 때만 반영하도록 하면 낙관적 동시성 제어를 구현할 수 있습니다."
- 질문: "왜 낙관적이라고 하나요?"  
  답변: "수정 전에 락을 걸지 않고 충돌이 드물다고 가정한 뒤, 실제 저장 시점에만 버전을 비교해 충돌을 감지하기 때문입니다."
- 질문: "주의할 점은 무엇인가요?"  
  답변: "충돌이 발생했을 때 단순 실패로 끝내지 말고, 재조회나 사용자 재확인 흐름을 준비해야 합니다. 또 `ETag`가 실제 리소스 변경을 정확히 반영하도록 관리돼야 합니다."

## 장점
- 락을 오래 유지하지 않아 확장성에 유리하다
- 덮어쓰기 같은 Lost Update 문제를 줄일 수 있다
- HTTP 표준 메커니즘을 활용해 API 의미를 비교적 명확하게 가져갈 수 있다

## 단점
- 충돌이 잦은 환경에서는 실패와 재시도가 많아질 수 있다
- 클라이언트가 버전 정보를 정확히 들고 다녀야 한다
- `ETag` 생성 기준이 부정확하면 충돌 감지가 흔들릴 수 있다

## 주의사항 / 실무 팁
- 면접에서는 캐시 검증용 `If-None-Match`와 수정 보호용 `If-Match`를 구분해서 말하면 좋다.
- "Lost Update 방지"라는 표현을 같이 쓰면 왜 필요한지 더 명확해진다.
- 충돌 시 HTTP 상태 코드와 사용자 경험을 어떻게 설계할지도 함께 언급하면 실무 감각이 드러난다.
- DB의 버전 컬럼 기반 낙관적 락과 같은 개념을 HTTP 계층으로 확장한 것이라고 연결해 설명하면 이해가 쉽다.
