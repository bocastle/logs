# 인덱스 스킵 스캔 정리

## 핵심 요약
- `인덱스 스킵 스캔(Index Skip Scan)`은 복합 인덱스의 선두 컬럼 조건이 없더라도, 뒤쪽 컬럼 조건을 활용하기 위해 인덱스를 여러 구간으로 나눠 탐색하는 최적화 방식이다.
- 즉 "복합 인덱스는 첫 컬럼부터 써야만 한다"는 원칙의 예외적 활용 방법으로 이해하면 된다.
- 면접에서는 인덱스를 새로 만들지 않고도 기존 복합 인덱스를 일부 활용할 수 있는 실행 전략이라고 설명하면 좋다.
- 다만 선두 컬럼의 값 종류가 너무 많으면 효율이 떨어질 수 있다.

## 개념 설명
복합 인덱스가 `(성별, 나이)` 순서로 있다고 해보자. 일반적으로 `나이 = 30` 조건만 있으면 선두 컬럼인 `성별` 조건이 없기 때문에 인덱스를 잘 못 쓴다고 배운다.

그런데 DB 옵티마이저가 선두 컬럼의 distinct 값이 많지 않다고 판단하면, `성별 = 남` 구간에서 `나이 = 30`을 찾고, 다시 `성별 = 여` 구간에서 `나이 = 30`을 찾는 식으로 인덱스를 건너뛰며 탐색할 수 있다. 이게 `인덱스 스킵 스캔`이다.

핵심은 선두 컬럼을 완전히 무시하는 것이 아니라, 가능한 값들을 기준으로 여러 번 짧게 들어갔다 나오는 전략이라는 점이다. 그래서 면접에서는 "복합 인덱스의 일부 우회 활용"이라고 표현하면 이해가 쉽다.

## 예시
인덱스가 `(region, created_at)` 순서로 있고, `region` 값은 `KR`, `US`, `JP` 정도만 있다고 해보자.

- 쿼리 조건: `created_at >= '2026-06-01'`
- 일반 원칙만 보면 선두 컬럼 조건이 없어 인덱스 활용이 애매하다
- 하지만 `region`의 종류가 적다면
  - `region = KR` 범위에서 날짜 조건 탐색
  - `region = US` 범위에서 날짜 조건 탐색
  - `region = JP` 범위에서 날짜 조건 탐색

처럼 여러 구간을 나눠 읽는 편이 전체 테이블 스캔보다 유리할 수 있다.

## 면접 답변 예시
- 질문: "`인덱스 스킵 스캔`은 무엇인가요?"  
  답변: "`인덱스 스킵 스캔`은 복합 인덱스의 선두 컬럼 조건이 없을 때도, 선두 컬럼의 값 종류가 적다면 그 값을 기준으로 인덱스를 여러 번 나눠 탐색해서 뒤쪽 컬럼 조건을 활용하는 실행 전략입니다."
- 질문: "그럼 복합 인덱스 순서는 중요하지 않은 건가요?"  
  답변: "아닙니다. 기본 원칙은 여전히 선두 컬럼부터 잘 타는 순서가 중요합니다. `인덱스 스킵 스캔`은 예외적으로 우회 활용하는 전략일 뿐이고, 항상 효율적인 것은 아닙니다."

## 장점
- 기존 복합 인덱스를 새 인덱스 없이도 일부 활용할 수 있다
- 선두 컬럼 종류가 적을 때 테이블 풀 스캔보다 유리할 수 있다
- 실행 계획을 해석할 때 옵티마이저가 왜 기존 인덱스를 선택했는지 설명하기 좋다

## 단점
- 선두 컬럼 cardinality가 높으면 반복 탐색 비용이 커진다
- DB 엔진과 버전에 따라 지원 수준이나 동작 방식이 다를 수 있다
- 잘못 기대하면 "인덱스가 있으니 무조건 빠르다"는 오해로 이어질 수 있다

## 주의사항 / 실무 팁
- 핵심 판단 기준은 선두 컬럼의 distinct 수와 실제 데이터 분포다.
- 실행 계획에서 skip scan 여부를 확인하지 않고 추측만으로 튜닝하면 위험하다.
- 면접에서는 "복합 인덱스는 선두 컬럼이 중요하다"는 기본 원칙을 먼저 말한 뒤, 예외 전략으로 `인덱스 스킵 스캔`을 붙이면 답변이 안정적이다.
- 실무에서는 이 전략에 기대기보다 조회 패턴이 반복되면 컬럼 순서를 재검토하거나 별도 인덱스를 두는 편이 더 명확할 때가 많다.
