# API 오류 응답을 Problem Details로 표준화하는 법 정리

## 핵심 요약

- Problem Details는 API 오류 응답의 모양을 `type`, `title`, `status`, `detail`, `instance` 같은 필드로 맞추는 방식이다.
- 면접에서는 "에러를 예쁘게 포장하는 것"이 아니라 클라이언트와 서버가 실패를 같은 언어로 이해하게 만드는 계약이라고 설명하면 좋다.
- 중요한 점은 모든 예외를 같은 500으로 뭉개지 않고, 사용자에게 보여줄 메시지와 운영자가 추적할 정보를 분리하는 것이다.

## 개념 설명

API 오류 응답은 생각보다 금방 지저분해진다. 어떤 API는 `{ message }`를 주고, 어떤 API는 `{ errorCode, reason }`을 주고, 어떤 API는 HTML 에러 페이지를 돌려주면 클라이언트는 실패 처리를 매번 다르게 해야 한다.

Problem Details는 이런 오류 응답을 일정한 형태로 맞추기 위한 형식이다. 예를 들어 잘못된 요청이면 `status`는 400, `title`은 짧은 오류 이름, `detail`은 사람이 이해할 수 있는 설명, `type`은 문서로 연결 가능한 식별자처럼 쓴다.

면접에서는 "오류 응답도 API 계약의 일부"라고 말하면 좋다. 성공 응답만 문서화하고 실패 응답을 대충 두면, 프론트엔드도 사용자 메시지를 안정적으로 만들기 어렵고 백엔드도 장애 분석에 필요한 정보를 놓치기 쉽다.

## 예시

```http
HTTP/1.1 409 Conflict
Content-Type: application/problem+json

{
  "type": "https://api.example.com/problems/order-already-paid",
  "title": "Order already paid",
  "status": 409,
  "detail": "이미 결제된 주문입니다.",
  "instance": "/orders/123/payments/456",
  "code": "ORDER_ALREADY_PAID"
}
```

여기서 `title`과 `status`는 클라이언트가 분기하기 좋고, `detail`은 사용자 안내에 가깝다. `code` 같은 확장 필드는 팀 내부 규칙에 따라 추가할 수 있다.

## 면접 답변 예시

> API 오류 응답을 Problem Details로 표준화한다는 건 실패 응답도 성공 응답처럼 계약으로 관리하겠다는 뜻입니다. 예를 들어 400, 409, 422 같은 상태 코드를 명확히 쓰고, `type`, `title`, `detail`, `instance`로 오류의 종류와 발생 위치를 표현합니다. 이렇게 하면 프론트엔드는 오류별 메시지와 복구 흐름을 안정적으로 만들 수 있고, 백엔드는 로그의 trace id나 내부 code와 연결해서 장애 분석을 하기 쉬워집니다. 다만 내부 예외 메시지나 민감한 값이 그대로 노출되지 않도록 사용자용 정보와 운영용 정보를 분리해야 합니다.

## 장점

- 클라이언트가 오류 응답을 일관된 방식으로 처리할 수 있다.
- API 문서에서 실패 케이스를 명확히 설명하기 쉽다.
- 로그, trace id, 오류 code와 연결하면 장애 분석 흐름이 좋아진다.

## 단점

- 초기에 오류 code와 문서 규칙을 정하는 비용이 든다.
- 모든 예외를 억지로 같은 형식에 넣으려 하면 오히려 의미가 흐려질 수 있다.
- 기존 클라이언트가 있는 서비스에서는 응답 형식 변경이 호환성 문제가 될 수 있다.

## 주의사항 / 실무 팁

- `detail`에 stack trace, SQL, 개인정보 같은 내부 정보를 넣지 않는다.
- HTTP status와 내부 error code의 역할을 나눠야 한다. status는 프로토콜 관점, code는 비즈니스 분기 관점이다.
- 같은 오류라도 사용자가 해결할 수 있는 오류인지, 운영자가 봐야 하는 오류인지 구분해서 메시지를 설계한다.
