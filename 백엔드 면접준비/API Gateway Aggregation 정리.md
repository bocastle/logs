# API Gateway Aggregation 정리

## 핵심 요약
- `API Gateway Aggregation`은 클라이언트가 여러 백엔드 API를 각각 호출하지 않고, 게이트웨이가 필요한 응답을 모아서 한 번에 내려주는 패턴이다.
- 모바일 앱이나 초기 렌더링처럼 네트워크 왕복 비용이 민감한 환경에서 특히 유용하다.
- 면접에서는 단순 프록시가 아니라 "응답 조합 책임을 게이트웨이 계층으로 올리는 설계"라고 설명하면 좋다.
- 편리하지만 게이트웨이가 비대해지면 병목과 결합도가 커질 수 있다.

## 개념 설명
클라이언트가 화면 하나를 그리기 위해 사용자 정보, 주문 목록, 추천 상품, 쿠폰 정보를 각각 다른 서비스에서 가져와야 한다고 해보자. 이때 클라이언트가 직접 여러 요청을 날리면 네트워크 왕복 수가 늘고, 조합 로직도 프론트엔드에 흩어질 수 있다.

`API Gateway Aggregation`은 이 문제를 줄이기 위해 게이트웨이가 여러 서비스 호출을 대신 수행하고, 필요한 필드만 묶어서 응답을 구성하는 방식이다. 클라이언트는 한 번만 호출하고, 내부 복잡성은 서버 쪽에서 흡수한다.

핵심은 "클라이언트를 단순하게 만들고 호출 횟수를 줄인다"는 점이다. 대신 게이트웨이가 오케스트레이션 책임을 가지게 되므로 어디까지 조합할지 경계 설정이 중요하다.

## 예시
마이페이지 첫 화면을 생각해보자.

- 사용자 프로필 서비스
- 주문 서비스
- 포인트 서비스
- 알림 서비스

이 네 서비스를 클라이언트가 각각 호출하면 로딩 관리도 복잡해진다. 반면 `/mypage/summary` 같은 게이트웨이 엔드포인트를 두면

- 내부에서는 여러 서비스에 병렬 호출
- 실패 허용 범위가 있는 데이터는 부분 응답 처리
- 클라이언트에는 화면에 바로 필요한 형태로 조합된 JSON 반환

같은 흐름으로 정리할 수 있다.

## 면접 답변 예시
- 질문: "`API Gateway Aggregation`은 왜 쓰나요?"  
  답변: "클라이언트가 여러 마이크로서비스를 직접 호출하면 네트워크 왕복 수와 조합 복잡도가 커지기 때문에, 게이트웨이에서 여러 응답을 모아 한 번에 내려주기 위해 사용합니다. 특히 모바일 환경이나 초기 화면 로딩 최적화에 유리합니다."
- 질문: "항상 좋은 패턴인가요?"  
  답변: "아닙니다. 게이트웨이에 너무 많은 비즈니스 조합 로직이 쌓이면 서비스 경계가 흐려지고 병목이 될 수 있습니다. 그래서 화면 조합 목적의 얇은 집계인지, 도메인 로직까지 가져가는지 구분해야 합니다."

## 장점
- 클라이언트 요청 수를 줄여 초기 로딩 성능을 개선하기 좋다
- 여러 서비스 응답 형식을 서버에서 일관되게 정리할 수 있다
- 프론트엔드가 내부 서비스 구조를 덜 알아도 되어 결합도를 낮출 수 있다

## 단점
- 게이트웨이에 책임이 몰리면 비대해지기 쉽다
- 내부 서비스 중 하나라도 느리면 집계 응답 전체가 지연될 수 있다
- 캐시, 타임아웃, 부분 실패 처리 같은 운영 복잡도가 커진다

## 주의사항 / 실무 팁
- 게이트웨이는 조합 계층이지 새로운 거대 비즈니스 서비스가 되지 않도록 경계를 잡아야 한다.
- 내부 호출은 병렬화하되, 각 서비스별 타임아웃과 fallback 정책을 분리하는 편이 안전하다.
- 면접에서는 `BFF`, `API Gateway`, `GraphQL`과 차이를 같이 설명하면 좋다. 특히 `BFF`는 클라이언트별 최적화 관점이고, aggregation은 응답 집계 자체에 초점이 있다.
- 실무에서는 부분 실패 시 어떤 필드를 비워도 되는지, 전체 실패로 볼지 기준을 먼저 정해 두는 것이 중요하다.
