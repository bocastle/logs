# 캐시 무효화 전략 정리

## 핵심 요약
- 캐시는 성능을 높이지만, 값이 바뀐 뒤에도 예전 데이터를 보여 줄 수 있어서 무효화 전략이 중요하다.
- 면접에서는 "캐시는 저장보다 무효화가 더 어렵다"는 말을 자주 하지만, 왜 어려운지 구조적으로 설명하는 것이 핵심이다.
- 대표 전략은 `TTL 기반 만료`, `쓰기 시점 즉시 삭제/갱신`, `이벤트 기반 무효화`, `버전 키` 같은 방식이다.
- 답변할 때는 "정확성 우선인지, 성능 우선인지, 얼마나 오래 stale 데이터를 허용할지"를 같이 말해야 실무 감각이 드러난다.

## 개념 설명
캐시는 DB나 원본 시스템보다 빠르게 데이터를 주기 위해 중간에 복사본을 두는 방식이다. 문제는 원본 데이터가 변경됐을 때 캐시가 그 사실을 즉시 모를 수 있다는 점이다. 이 상태에서 사용자는 이미 바뀐 데이터가 아니라 예전 데이터를 읽게 된다.

그래서 캐시를 도입할 때는 "어떻게 저장할까"보다 "언제, 어떤 조건에서, 어떤 범위까지 지울까"를 먼저 생각해야 한다. 가장 단순한 방식은 `TTL`을 두고 일정 시간이 지나면 자동 만료되게 하는 것이다. 구현은 쉽지만, 만료 전까지는 오래된 데이터를 보여 줄 수 있다.

좀 더 강한 정합성이 필요하면 쓰기 시점에 관련 캐시 키를 즉시 삭제하거나 새 값으로 갱신한다. 이 방식은 최신성은 좋아지지만, 어떤 키들이 영향을 받는지 추적해야 해서 복잡해진다. 특히 목록, 상세, 통계처럼 하나의 변경이 여러 캐시 엔트리에 영향을 줄 때 어렵다.

규모가 커지면 이벤트 기반 무효화도 많이 쓴다. 예를 들어 주문 상태가 바뀌면 이벤트를 발행하고, 이를 구독한 캐시 계층이 관련 키를 지우는 식이다. 유연하지만 전파 지연, 중복 이벤트, 누락 처리 같은 운영 이슈가 생긴다. 결국 캐시 무효화 전략은 "최신성, 단순성, 운영 복잡도" 사이의 균형 문제다.

## 예시
상품 상세와 상품 목록을 캐싱하는 쇼핑몰을 생각해 보자.

- 상품 이름이 바뀌면 상세 캐시만 지우는 것으로 끝나지 않을 수 있다
- 목록 페이지, 추천 영역, 검색 결과에도 같은 상품명이 노출될 수 있다
- 단순 TTL만 쓰면 일정 시간 동안 예전 이름이 보일 수 있다
- 쓰기 시점 삭제를 쓰면 더 정확하지만 어떤 키를 함께 지울지 설계가 필요하다

이 사례를 말하면 캐시 무효화가 어려운 이유를 면접관이 바로 이해하기 쉽다.

## 면접 답변 예시
- 질문: "캐시 무효화가 왜 어렵나요?"  
  답변: "원본 데이터 하나의 변경이 여러 캐시 키에 영향을 줄 수 있기 때문입니다. TTL만 쓰면 stale 데이터를 허용해야 하고, 즉시 무효화는 정확하지만 어떤 키를 지울지 추적하는 비용이 큽니다. 그래서 최신성 요구 수준에 맞춰 전략을 선택해야 합니다."
- 질문: "어떤 전략을 주로 쓰나요?"  
  답변: "읽기 부하가 크고 약간의 stale 데이터를 허용할 수 있으면 TTL 기반을 쓰고, 정합성이 더 중요하면 쓰기 시점 삭제나 갱신을 씁니다. 시스템이 커지면 이벤트 기반 무효화나 버전 키를 섞어서 관리하기도 합니다."

## 장점
- 캐시를 단순 성능 기술이 아니라 데이터 일관성 관점에서 설명할 수 있다
- TTL, 이벤트, 쓰기 연계 전략을 비교하며 설계 판단력을 보여 주기 좋다
- `Cache Aside Pattern`, `캐시 스탬피드`, `Read Replica` 같은 기존 주제와 자연스럽게 연결된다

## 단점
- 시스템마다 요구하는 최신성 수준이 달라 정답처럼 말하기 어렵다
- 단순 삭제와 갱신, 동기와 비동기 전파를 섞어 설명하면 구조가 흐려질 수 있다
- "캐시를 쓰면 무조건 빨라진다"는 식으로 말하면 운영 복잡도를 놓치게 된다

## 주의사항 / 실무 팁
- 캐시 설계에서 먼저 물어봐야 할 것은 "얼마나 stale 데이터를 허용할 수 있나"다.
- 목록, 상세, 집계처럼 파생 뷰가 많은 데이터일수록 키 설계를 단순하게 유지하는 것이 중요하다.
- 면접에서는 TTL만 말하고 끝내지 말고, 쓰기 시점 무효화나 이벤트 기반 전파까지 한 단계 더 설명하면 좋다.
- 실무에서는 무효화 실패 시 fallback으로 TTL을 같이 두는 경우가 많다.
