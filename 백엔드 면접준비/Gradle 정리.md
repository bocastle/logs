# Gradle 정리

## 핵심 요약(짧게)

- Gradle은 Java/Kotlin/Scala 등 JVM 생태계에서 자주 쓰는 **빌드 자동화 도구**입니다.
- **증분 빌드**, **빌드 캐시**, **데몬(daemon)** 등을 활용해 **빌드 속도를 최적화**할 수 있습니다.
- **멀티 프로젝트** 관리에 유리하며, **플러그인/커스텀 태스크**로 확장성이 높습니다.
- 빌드 스크립트는 **Groovy DSL** 또는 **Kotlin DSL**로 작성합니다.

## Gradle이란?

Gradle은 기존의 Ant/Maven의 단점을 보완한 빌드 자동화 도구입니다.  
증분 빌드, 빌드 캐시, 데몬 프로세스를 활용해 빌드 속도를 최적화하고, 멀티 프로젝트를 쉽게 관리할 수 있도록 설계되었습니다. 또한 다양한 플러그인과 커스텀 태스크를 통해 유연하게 확장할 수 있습니다.

## 빌드 자동화 도구를 왜 사용할까요?

1. 컴파일, 테스트, 패키징, 배포 같은 반복 작업을 자동화해 **개발 생산성**을 높일 수 있습니다.
2. **일관된 빌드 환경**을 제공해 어디서나 동일한 빌드 결과를 보장할 수 있습니다.
3. 증분 빌드, 빌드 캐시, 병렬 처리 등으로 **빌드 속도를 최적화**할 수 있습니다.
4. 테스트 누락 등 수동 작업에서 발생할 수 있는 **휴먼 에러를 방지**할 수 있습니다.
5. 외부 라이브러리를 자동 관리해 **의존성 버전 충돌을 줄일 수** 있습니다.
6. CI/CD와 연동해 빌드 → 패키징 → 배포를 **연속적으로 처리**할 수 있습니다.

## Maven과 Gradle의 차이

가장 큰 차이점은 **빌드 스크립트 작성 방식**과 **빌드 속도**입니다.

| 항목          | Maven                  | Gradle                               |
| ------------- | ---------------------- | ------------------------------------ |
| 빌드 스크립트 | XML (`pom.xml`)        | Groovy/Kotlin DSL                    |
| 빌드 속도     | 느림(전체 빌드 중심)   | 빠름(증분 빌드/빌드 캐시)            |
| 의존성 관리   | 기본적인 의존성 관리   | 동적 버전 관리, 의존성 캐싱 최적화   |
| 확장성        | 한정적인 플러그인 기능 | 다양한 플러그인, 커스텀 태스크       |
| Android 지원  | 공식적으로 지원 X      | Android 공식 빌드 도구               |
| 멀티 프로젝트 | 상속 방식, 설정 복잡   | 설정 주입 방식, 멀티 프로젝트 최적화 |

## Dependency Configuration이란?

Dependency Configuration은 애플리케이션에 필요한 **의존성의 사용 범위(scope)** 를 정의하는 설정입니다.  
사용 범위를 명확히 구분하는 이유는 **빌드 성능 개선**과 **불필요한 의존성 제거**로 빌드 결과물 크기를 최적화하기 위함입니다.

### 주요 Configuration 종류

- `implementation`  
  컴파일/런타임에 모두 필요한 의존성. **해당 모듈 내부에서만** 필요할 때 주로 사용합니다.
- `api`  
  `implementation`과 비슷하지만, **다른 모듈에서도 의존성을 “노출”** 해야 할 때 사용합니다.  
  예: `a -> b -> c` 의존성이 있을 때, `a`에서 `c`를 사용하려면 `b`가 `c`를 `api`로 추가해야 합니다.
- `compileOnly`  
  **컴파일 시점에만** 필요한 의존성(예: Lombok).
- `annotationProcessor`  
  컴파일 시점에 실행되는 **어노테이션 프로세서**(예: MapStruct, Lombok 등).
- `runtimeOnly`  
  **런타임에만** 필요한 의존성(예: DB 드라이버).
- `testImplementation`, `testCompileOnly`, `testRuntimeOnly`  
  **테스트 코드에서만** 사용하는 의존성.

### 설정 예시 (`build.gradle`)

```gradle
dependencies {
  implementation("org.springframework.boot:spring-boot-starter-web")

  // 다른 모듈에서도 이 의존성을 사용할 수 있게 노출
  api("com.fasterxml.jackson.core:jackson-databind:2.17.0")

  // 컴파일 시점에만 필요
  compileOnly("org.projectlombok:lombok:1.18.34")
  annotationProcessor("org.projectlombok:lombok:1.18.34")

  // 런타임에만 필요 (예: DB 드라이버)
  runtimeOnly("com.mysql:mysql-connector-j:8.4.0")

  // 테스트 전용
  testImplementation("org.springframework.boot:spring-boot-starter-test")
}
```

## 장점

- 증분 빌드/캐시/데몬으로 **빌드 속도 최적화**에 유리합니다.
- Groovy/Kotlin DSL로 **유연하고 가독성 좋은 빌드 스크립트**를 작성할 수 있습니다.
- 플러그인/태스크로 **확장성**이 높고, 멀티 프로젝트 관리에 최적화되어 있습니다.

## 단점

- 원문에서는 별도의 단점이 직접적으로 언급되지는 않았습니다.

## 주의사항 / 실무 팁

- `api`를 남용하면 의존성이 외부로 “새어” 나가 모듈 경계가 흐려질 수 있으니, 기본은 `implementation`을 우선 고려하고 필요한 경우에만 `api`를 사용하세요.
- 의존성 범위를 잘못 지정하면(예: 런타임에 필요한 라이브러리를 `compileOnly`로 둠) 실행 시점 오류가 날 수 있으니, **컴파일/런타임/테스트** 범위를 기준으로 점검하세요.
- 빌드 속도가 이슈라면 **증분 빌드/캐시**가 제대로 적용되는지(캐시 히트율, CI 환경 설정 등)를 먼저 확인하는 것이 좋습니다.
