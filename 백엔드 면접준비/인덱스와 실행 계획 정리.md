# 인덱스와 실행 계획 정리

## 핵심 요약

- 인덱스는 데이터를 더 빨리 찾기 위한 별도 자료구조다.
- 실행 계획은 DB가 SQL을 어떤 순서와 방식으로 처리할지 보여주는 계획표다.
- 인덱스가 있다고 항상 좋은 것은 아니고, 조회 패턴에 맞아야 효과가 있다.
- 면접에서는 "인덱스를 추가했다"에서 끝내지 말고, 실행 계획으로 실제로 바뀐 점을 확인했다고 말하는 게 좋다.

## 인덱스란?

인덱스는 책의 맨 뒤 색인과 비슷하다. 책 전체를 처음부터 끝까지 읽지 않고도 원하는 단어가 어느 페이지에 있는지 빨리 찾게 도와준다.

DB도 비슷하다. 테이블 데이터를 매번 처음부터 끝까지 훑는 대신, 자주 조회하는 컬럼 기준으로 정리된 구조를 따로 두고 그걸 이용해 빠르게 접근한다.

보통 관계형 DB에서는 B-Tree 기반 인덱스를 많이 쓴다. 그래서 동등 비교뿐 아니라 범위 조회에도 자주 활용된다.

## 실행 계획이란?

실행 계획은 DB 옵티마이저가 SQL을 실행하기로 결정한 방식이다.

예를 들어 다음 같은 내용을 보여준다.

- 어떤 테이블부터 읽는지
- 인덱스를 타는지, 전체 스캔을 하는지
- 조인을 어떤 방식으로 하는지
- 대략 몇 건을 읽을 것으로 예상하는지

즉 "이 쿼리가 왜 느린지"를 감으로 추측하는 대신, DB가 실제로 어떤 선택을 했는지 확인하는 도구라고 보면 된다.

## 인덱스가 왜 필요한가?

테이블이 작을 때는 전체 스캔도 큰 문제가 아닐 수 있다. 하지만 데이터가 커지면 조건에 맞는 일부 행만 찾기 위해 전체를 다 읽는 비용이 커진다.

예를 들어 `WHERE email = ?` 같은 조회가 자주 일어나는데 `email`에 인덱스가 없다면, 사용자 수가 많아질수록 매번 더 많은 행을 확인하게 된다.

반대로 적절한 인덱스가 있으면 필요한 위치를 더 빨리 찾아가서 읽는 범위를 줄일 수 있다.

## 인덱스를 만들 때 자주 보는 포인트

### 1. 조회 조건이 자주 걸리는 컬럼인가?

조회, 조인, 정렬, 범위 검색에 반복적으로 쓰이는 컬럼이 우선 대상이다.

### 2. 선택도가 어느 정도 있는가?

컬럼 값이 너무 단순해서 많은 행이 같은 값을 가진다면 인덱스 효율이 낮을 수 있다. 예를 들어 거의 대부분이 `ACTIVE`인 상태 컬럼 하나만 인덱싱하면 기대만큼 효과가 안 날 수 있다.

### 3. 복합 인덱스 순서가 맞는가?

복합 인덱스는 컬럼 순서가 중요하다. 예를 들어 `(user_id, created_at)` 인덱스가 있으면 `user_id` 조건이 있는 조회에는 잘 맞지만, `created_at`만 단독으로 찾는 쿼리에는 기대만큼 못 쓸 수 있다.

이 부분은 면접에서 자주 물어보는 포인트다.

## 인덱스의 비용

인덱스는 조회를 빠르게 만들 수 있지만 공짜는 아니다.

- 저장 공간을 더 사용한다.
- `INSERT`, `UPDATE`, `DELETE` 때 인덱스도 같이 갱신해야 해서 쓰기 비용이 늘어난다.
- 인덱스가 너무 많으면 오히려 전체 운영 비용이 커질 수 있다.

그래서 "느린 쿼리가 있으니 일단 인덱스부터 추가한다"는 접근은 위험하다.

## 실행 계획으로 무엇을 확인하나?

실행 계획을 볼 때 초보자가 가장 먼저 보면 좋은 건 다음이다.

- `Full Table Scan`인지, 인덱스를 타는지
- 예상 읽기 건수가 지나치게 큰지
- 조인 순서가 비효율적이지 않은지
- 정렬이나 임시 테이블 비용이 큰지

핵심은 인덱스를 추가한 뒤 실제로 읽는 범위가 줄었는지 확인하는 것이다. 단순히 인덱스 생성만으로 끝내면 안 된다.

## 실무 예시

예를 들어 주문 테이블에서 특정 사용자의 최근 주문 목록을 자주 조회한다고 해보자.

```sql
SELECT *
FROM orders
WHERE user_id = 10
ORDER BY created_at DESC
LIMIT 20;
```

이 경우 `(user_id, created_at)` 같은 복합 인덱스를 고려할 수 있다. 그러면 특정 사용자의 데이터 범위 안에서 정렬과 조회를 더 효율적으로 처리할 가능성이 높아진다.

반대로 `created_at`만 인덱싱하거나, 컬럼 순서를 반대로 잡으면 기대한 만큼 효율이 안 나올 수 있다. 이 차이는 결국 실행 계획에서 확인해야 한다.

## 면접 답변 예시

인덱스는 테이블 데이터를 더 빨리 찾기 위한 별도 자료구조이고, 실행 계획은 DB가 SQL을 실제로 어떤 방식으로 처리할지 보여주는 정보라고 이해하고 있습니다. 실무에서는 느린 조회가 있다고 바로 인덱스를 추가하기보다, 어떤 조건과 정렬이 자주 쓰이는지 먼저 보고 후보 인덱스를 잡은 뒤 실행 계획으로 Full Scan이 줄었는지와 읽는 건수가 실제로 개선됐는지를 확인하는 식으로 접근합니다. 또 인덱스는 조회 성능에는 도움이 되지만 쓰기 비용과 저장 공간이 늘기 때문에, 자주 쓰는 패턴 중심으로 필요한 만큼만 두는 게 중요합니다.

## 장점

- 대용량 데이터에서 조회 범위를 줄여 성능 개선에 도움이 된다.
- 조인, 정렬, 범위 조회를 더 효율적으로 처리할 수 있다.
- 실행 계획과 함께 보면 병목 원인을 더 구체적으로 파악할 수 있다.
- 감이 아니라 근거 기반으로 튜닝 방향을 잡기 좋다.

## 단점

- 저장 공간을 추가로 사용한다.
- 쓰기 작업 시 인덱스 유지 비용이 늘어난다.
- 잘못 만든 인덱스는 효과가 적거나 오히려 운영 부담만 늘린다.
- 실행 계획은 DB 종류와 데이터 분포에 따라 해석 난도가 있을 수 있다.

## 주의사항 / 실무 팁

- 인덱스는 "컬럼 단위"가 아니라 "쿼리 패턴 단위"로 본다.
- 복합 인덱스는 컬럼 순서가 중요하니 조회 조건과 정렬 순서를 함께 본다.
- 인덱스를 추가한 뒤에는 반드시 실행 계획과 실제 응답 시간을 같이 확인한다.
- 데이터가 적은 개발 환경 결과만 믿지 말고, 운영과 비슷한 규모에서 판단해야 한다.
- 느린 원인이 인덱스 부재가 아니라 잘못된 조인, 과한 조회 범위, 불필요한 `SELECT *`일 수도 있다는 점을 같이 봐야 한다.
