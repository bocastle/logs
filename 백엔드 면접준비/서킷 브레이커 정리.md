# 서킷 브레이커 정리

## 핵심 요약

- 서킷 브레이커는 장애가 난 외부 시스템을 계속 호출하지 않도록 막아 주는 패턴이다.
- 목적은 실패를 빨리 끊고, 전체 서비스로 장애가 번지는 것을 줄이는 데 있다.
- 보통 `Closed`, `Open`, `Half-Open` 세 상태로 설명한다.
- 면접에서는 "왜 필요한가", "상태 전이", "타임아웃/재시도와 어떤 관계인가"를 같이 말하면 좋다.

## 서킷 브레이커가 왜 필요한가

백엔드 서비스는 DB, 캐시, 결제사, 메시지 브로커, 다른 내부 API처럼 외부 의존성을 많이 가진다. 문제는 한 곳이 느려지거나 실패하기 시작하면, 호출하는 쪽도 스레드나 커넥션을 계속 붙잡게 되고 결국 전체 서비스가 같이 흔들릴 수 있다는 점이다.

예를 들어 결제 API가 이미 장애 상태인데도 매 요청마다 몇 초씩 기다렸다가 실패하면, 사용자 입장에서는 응답이 느리고 서버 입장에서는 자원이 계속 소모된다.

서킷 브레이커는 이 상황에서 "지금은 계속 시도해도 이득이 없다"라고 판단하면 일정 시간 동안 호출 자체를 빠르게 차단한다.

## 이름이 왜 서킷 브레이커인가

전기 차단기와 비슷하게 생각하면 된다. 과부하가 걸렸을 때 회로를 끊어서 더 큰 문제를 막는 개념이다. 소프트웨어에서도 장애가 난 의존성 쪽으로 계속 요청을 보내지 않고, 일단 회로를 열어 피해 확산을 줄인다.

## 상태별 동작

### 1. Closed

정상 상태다. 요청을 그대로 외부 시스템으로 보낸다. 다만 실패율, 응답 시간, 타임아웃 횟수 같은 지표를 계속 본다.

### 2. Open

실패가 임계치를 넘으면 회로를 연다. 이 상태에서는 외부 호출을 실제로 보내지 않고 바로 실패시키거나, 대체 응답을 돌려준다.

예를 들어 "현재 결제사 응답이 불안정합니다. 잠시 후 다시 시도해 주세요." 같은 메시지나, 캐시된 값, 기본값을 반환할 수 있다.

### 3. Half-Open

일정 시간이 지나면 회복 여부를 확인하기 위해 일부 요청만 시험적으로 통과시킨다. 성공하면 다시 `Closed`로 돌아가고, 여전히 실패하면 다시 `Open`으로 간다.

## 재시도와의 차이

재시도는 실패했을 때 한 번 더 시도해 보는 전략이다. 일시적인 네트워크 오류에는 도움이 될 수 있다.

하지만 이미 상대 시스템이 죽어 있거나 과부하 상태라면 재시도는 오히려 트래픽을 더 늘릴 수 있다. 그래서 실무에서는 재시도만 쓰기보다, 타임아웃과 서킷 브레이커를 같이 둬서 "언제까지 시도할지"와 "언제 아예 끊을지"를 분리해서 설계한다.

정리하면 이렇다.

- 타임아웃: 너무 오래 기다리지 않게 함
- 재시도: 잠깐의 실패를 한 번 더 시도
- 서킷 브레이커: 실패가 누적되면 잠시 호출 자체를 막음

## 어디에 적용하면 좋나

- 외부 결제 API
- 문자, 메일 발송 API
- 느리거나 불안정한 내부 마이크로서비스 호출
- 캐시 서버 장애 시 원본 조회로 쏠림이 커지는 구간

특히 사용자 요청 흐름 안에 있는 외부 API는 장애 전파를 막는 관점에서 우선순위가 높다.

## 실무에서 같이 보는 포인트

### 1. fallback 전략

호출을 막았을 때 무엇을 대신 줄지 정해야 한다. 단순 에러 반환인지, 캐시된 데이터인지, 비핵심 기능만 비활성화할지 서비스 특성에 따라 다르다.

### 2. 임계치 설정

실패율 몇 퍼센트, 몇 초 동안 몇 건 이상 실패하면 열 것인지 기준이 필요하다. 너무 민감하면 멀쩡한 시스템도 쉽게 열리고, 너무 둔하면 이미 장애가 퍼진 뒤에야 반응한다.

### 3. 모니터링

서킷 브레이커는 넣고 끝이 아니다. 열림 횟수, 반열림 상태, fallback 비율을 관찰해야 실제 운영에 도움이 된다.

## 면접 답변 예시

서킷 브레이커는 장애가 난 외부 시스템을 계속 호출해서 전체 서비스가 같이 느려지거나 장애가 번지는 걸 막기 위한 패턴입니다. 보통 정상 호출 상태인 `Closed`, 실패가 누적돼 호출을 차단하는 `Open`, 일부 요청으로 복구 여부를 확인하는 `Half-Open` 상태로 설명합니다. 실무에서는 재시도와 함께 쓰지만 역할은 다릅니다. 재시도는 잠깐의 실패를 다시 시도하는 전략이고, 서킷 브레이커는 실패가 누적되면 일정 시간 호출 자체를 끊어서 시스템을 보호하는 쪽에 가깝습니다.

## 장점

- 장애 전파를 줄이고 자원 고갈을 막는 데 도움이 된다.
- 실패를 빨리 반환해서 전체 응답 시간을 통제하기 좋아진다.
- 외부 의존성이 많은 시스템에서 복원력 설계의 핵심 도구가 된다.

## 단점

- 임계치와 대기 시간을 잘못 잡으면 정상 요청까지 과하게 차단할 수 있다.
- fallback 설계를 허술하게 하면 사용자 경험이 오히려 더 나빠질 수 있다.
- 운영 지표를 같이 보지 않으면 왜 열렸는지 파악하기 어려울 수 있다.

## 주의사항 / 실무 팁

- 서킷 브레이커만 믿지 말고 타임아웃, 재시도, 벌크헤드 같은 전략과 함께 봐야 한다.
- 핵심 결제나 주문 흐름에서는 fallback이 비즈니스적으로 허용되는지 먼저 따져야 한다.
- 장애 복구 후에도 바로 전체 트래픽을 열지 않고 `Half-Open`처럼 점진적으로 확인하는 접근이 안전하다.
- 라이브러리를 도입하더라도 상태 변화와 임계치 기준을 팀이 설명할 수 있어야 면접과 실무 둘 다 강하다.
