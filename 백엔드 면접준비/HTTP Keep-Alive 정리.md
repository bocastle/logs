# HTTP Keep-Alive 정리

## 핵심 요약

- HTTP Keep-Alive는 하나의 TCP 연결을 **여러 HTTP 요청/응답에 재사용**하는 방식입니다.
- 연결을 매번 새로 맺지 않아도 되어 **핸드셰이크 비용/지연 시간**을 줄이고 성능을 개선할 수 있습니다.
- HTTP/1.1에서는 기본적으로 지속 연결(persistent connection)이 사용되며, 서버/프록시는 **idle timeout** 등으로 연결을 닫을 수 있습니다.

## HTTP Keep-Alive란?

HTTP Keep-Alive는 클라이언트와 서버가 TCP 연결을 한 번 맺은 뒤, 그 연결을 유지하여 여러 번의 HTTP 요청/응답을 처리하는 방식입니다.  
즉, 매 요청마다 TCP 연결을 새로 맺는 대신, 기존 연결을 재사용해 통신 비용을 줄입니다.

## 왜 필요한가요?

HTTP 요청을 처리하려면 TCP 연결 수립(예: 3-way handshake)이 필요합니다.  
매번 연결을 새로 만들면 핸드셰이크/소켓 생성/종료에 비용이 들고, 요청 수가 많을수록 지연이 커질 수 있습니다.

Keep-Alive를 사용하면:

- TCP 연결 수립/해제 비용 감소
- 네트워크 지연 감소
- 서버/클라이언트 자원 사용 최적화  
  같은 효과를 기대할 수 있습니다.

## 동작 방식

### 기본 개념

1. 클라이언트가 서버와 TCP 연결을 수립
2. 동일 연결로 여러 HTTP 요청/응답을 주고받음
3. 일정 시간 유휴(idle) 상태가 지속되면 서버/클라이언트가 연결을 종료할 수 있음

## 헤더 예시

### HTTP/1.0에서의 Keep-Alive

HTTP/1.0은 기본적으로 연결을 매 요청마다 닫는 방식이었기 때문에, Keep-Alive를 쓰려면 명시가 필요했습니다.

```http
Connection: keep-alive
Keep-Alive: timeout=5, max=1000
```

### HTTP/1.1에서의 지속 연결

HTTP/1.1은 기본적으로 persistent connection을 사용하므로, 연결을 닫고 싶으면 `Connection: close`를 보냅니다.

```http
Connection: close
```

## 장점

- TCP 연결 재사용으로 **성능(지연/처리량)** 개선에 유리합니다.
- 핸드셰이크/소켓 생성 비용이 줄어 **서버 부하**를 낮출 수 있습니다.
- 다수의 리소스를 가져오는 웹 페이지에서 네트워크 효율이 좋아질 수 있습니다.

## 단점

- 연결을 오래 유지하면 서버는 그만큼 소켓/커넥션을 관리해야 하므로, 트래픽 패턴에 따라 **동시 연결 관리 비용**이 늘 수 있습니다.
- idle timeout 설정이 부적절하면(너무 길거나 짧으면) 리소스 낭비 또는 연결 재수립이 잦아질 수 있습니다.

## 주의사항 / 실무 팁

- 서버/프록시/로드밸런서에는 보통 Keep-Alive 관련 타임아웃이 있습니다.  
  (예: 서버 keep-alive timeout, LB idle timeout 등) 운영 환경 기준으로 함께 맞추는 것이 중요합니다.
- Keep-Alive는 “무조건 켜면 좋다”기보다, 서비스 트래픽 특성(짧고 빈번한 요청 vs 긴 연결 다수)에 맞춰 타임아웃/최대 요청 수를 튜닝하는 편이 좋습니다.
- HTTP/2 이상에서는 하나의 연결에서 멀티플렉싱이 가능해, Keep-Alive와는 다른 차원의 최적화가 함께 일어납니다(지속 연결 + 스트림).
