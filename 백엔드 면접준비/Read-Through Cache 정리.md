# Read-Through Cache 정리

## 핵심 요약
- `Read-Through Cache`는 애플리케이션이 직접 DB를 조회하지 않고, 캐시 계층에 조회를 맡기는 방식이다.
- 캐시에 데이터가 없으면 캐시가 원본 저장소를 대신 조회해 값을 채운 뒤 반환한다.
- 면접에서는 "애플리케이션이 캐시만 바라보고, miss 시 캐시가 원본을 읽어 오는 구조"라고 설명하면 핵심이 잘 잡힌다.
- `Cache Aside`와 차이를 구분해 말하는 것이 중요하고, 캐시 계층 복잡도와 장애 전파 가능성도 같이 짚어야 한다.

## 개념 설명
캐시 전략을 이야기할 때 가장 많이 비교되는 것이 `Cache Aside`와 `Read-Through Cache`다. `Cache Aside`에서는 애플리케이션이 직접 캐시를 조회하고, miss가 나면 DB를 읽어서 다시 캐시에 넣는다.

반면 `Read-Through Cache`에서는 애플리케이션이 DB를 직접 신경 쓰지 않는다. 애플리케이션은 캐시에만 조회 요청을 보내고, 캐시가 miss를 감지하면 내부적으로 원본 저장소를 읽어와 값을 적재한 뒤 응답한다.

즉, 조회 경로를 캐시 계층이 더 많이 책임지는 구조다. 이 방식은 애플리케이션 코드에서 캐시 처리 로직을 줄일 수 있지만, 캐시가 단순 저장소가 아니라 데이터 접근 책임까지 가지게 된다.

그래서 설계 관점에서는 "코드 단순화"와 "인프라 계층 복잡도 증가" 사이의 트레이드오프로 보는 것이 좋다.

## 예시
예를 들어 상품 상세 조회 API가 있다고 해 보자.

- 애플리케이션은 `product:123`을 캐시에 요청한다
- 캐시에 값이 있으면 바로 반환한다
- 값이 없으면 캐시가 DB에서 `product:123`을 읽어 온다
- 읽어 온 결과를 캐시에 채운 뒤 애플리케이션에 반환한다

이 구조에서는 애플리케이션 코드가 miss 처리와 캐시 적재를 직접 하지 않는다. 대신 캐시 시스템이 원본 저장소 접근 권한과 로딩 로직을 가져야 한다.

## 면접 답변 예시
- 질문: "`Read-Through Cache`는 무엇인가요?"  
  답변: "애플리케이션이 캐시만 조회하고, 캐시 miss가 나면 캐시 계층이 원본 저장소를 대신 조회해 값을 채우는 방식입니다."
- 질문: "`Cache Aside`와 차이는 무엇인가요?"  
  답변: "`Cache Aside`는 애플리케이션이 miss를 처리하지만, `Read-Through Cache`는 그 책임을 캐시 계층이 가져갑니다. 그래서 애플리케이션 코드는 단순해질 수 있습니다."
- 질문: "주의할 점은 무엇인가요?"  
  답변: "캐시 계층이 더 똑똑해지는 만큼 운영 복잡도가 올라가고, 캐시 장애가 조회 경로 전체에 영향을 줄 수 있습니다."

## 장점
- 애플리케이션 코드에서 캐시 miss 처리 로직을 줄일 수 있다
- 조회 경로를 캐시 계층에 일관되게 모을 수 있다
- 캐시 정책을 중앙에서 관리하기 쉬운 편이다

## 단점
- 캐시 계층이 원본 조회 책임까지 가지므로 구조가 복잡해진다
- 캐시 시스템에 대한 의존성이 커진다
- 일반적인 애플리케이션 레벨 구현보다 도입 선택지가 제한될 수 있다

## 주의사항 / 실무 팁
- 면접에서 가장 많이 묻는 포인트는 `Cache Aside`와의 차이다. 이 구분을 먼저 또렷하게 말하는 게 좋다.
- `Read-Through Cache`는 캐시 제품이나 플랫폼 지원 여부에 영향을 받는 경우가 많다.
- 캐시 miss 폭증 시 원본 저장소 부하가 어떻게 전달되는지도 함께 생각해야 한다.
- 실무에서는 읽기 경로 단순화가 장점이지만, 장애 분석 시 책임 경계가 캐시 쪽으로 이동한다는 점을 반드시 염두에 둬야 한다.
