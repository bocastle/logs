# 샤딩 정리

## 핵심 요약

- `샤딩`은 하나의 큰 데이터베이스를 여러 조각으로 나눠 여러 서버에 분산 저장하는 방식이다.
- 데이터 양이나 트래픽이 한 DB 인스턴스로 감당하기 어려울 때 확장 전략으로 검토한다.
- 읽기 부하 분산인 `Read Replica`와 달리, 샤딩은 데이터 자체를 나눠 저장한다는 점이 핵심이다.
- 면접에서는 "왜 필요한지", "샤드 키를 어떻게 정하는지", "무엇이 어려워지는지"를 같이 답하는 편이 좋다.

## 샤딩이란 무엇인가

서비스가 커지면 데이터 양도 늘고, 특정 테이블에 읽기와 쓰기가 몰리기 시작한다.  
이때 서버 스펙을 계속 올리는 방식만으로는 한계가 올 수 있다.

샤딩은 이 문제를 해결하기 위해 데이터를 여러 DB에 나눠 담는 방식이다.  
예를 들어 사용자 ID 범위나 해시 값을 기준으로 사용자 데이터를 서로 다른 DB에 분산 저장할 수 있다.

쉽게 말하면 "DB를 복사해서 같은 데이터를 여러 군데 두는 것"이 아니라, "전체 데이터를 조각내서 나눠 담는 것"이다.

## 왜 샤딩을 하는가

샤딩을 고려하는 이유는 대체로 확장성과 부하 분산 때문이다.

- 한 서버에 데이터가 너무 커져서 저장과 조회가 버거울 때
- 쓰기 트래픽이 많아 단일 DB가 병목이 될 때
- 특정 테이블의 인덱스 크기나 관리 비용이 과도해질 때
- 서비스 규모상 수평 확장이 필요할 때

즉 단일 DB 구조가 기능적으로는 맞지만, 물리적으로 감당이 안 되는 상황에서 나오는 선택지다.

## 샤드 키가 중요하다

샤딩에서 가장 중요한 설계 포인트는 `샤드 키`다.  
어떤 기준으로 데이터를 나눌지에 따라 성능과 운영 난도가 크게 달라진다.

샤드 키는 보통 다음을 고려해서 정한다.

- 조회가 자주 묶이는 데이터인가
- 한 샤드로 쏠림이 생기지 않는가
- 범위 조회가 많은가, 랜덤 조회가 많은가
- 재분배가 필요해질 때 이동 비용이 큰가

예를 들어 `user_id` 기준으로 샤딩하면 특정 사용자의 데이터는 한 샤드에 모이기 쉬워 조회 경로가 명확하다.  
반면 이벤트 시간만 기준으로 나누면 특정 시점에 쓰기 쏠림이 생길 수 있다.

## 샤딩 방식 예시

대표적으로 많이 이야기하는 방식은 두 가지다.

### 범위 기반 샤딩

예를 들어 1번부터 100만 번 사용자는 A 샤드, 그 다음은 B 샤드처럼 나누는 방식이다.

장점은 이해하기 쉽다는 점이다.  
단점은 특정 구간에 데이터가 몰리면 불균형이 생기기 쉽다는 점이다.

### 해시 기반 샤딩

`user_id % N` 같은 규칙으로 샤드를 정하는 방식이다.

장점은 상대적으로 고르게 분산시키기 쉽다는 점이다.  
단점은 샤드 수가 바뀌면 재배치 부담이 커질 수 있다는 점이다.

면접에서는 굳이 모든 방식을 길게 설명하기보다, "분산 균형과 조회 패턴 사이의 trade-off"를 짚으면 충분하다.

## 샤딩을 하면 어려워지는 것들

샤딩은 확장성에는 도움이 되지만 개발과 운영은 확실히 복잡해진다.

- 조인이 어려워지거나 비싸진다.
- 트랜잭션이 샤드를 넘나들면 복잡도가 급격히 올라간다.
- 집계 쿼리가 어려워진다.
- 운영 중 리샤딩이 큰 작업이 된다.
- 장애 분석과 모니터링도 더 복잡해진다.

특히 "샤드 간 트랜잭션"은 면접에서 자주 나오는 포인트다.  
단일 DB에서는 익숙한 작업이 샤딩 환경에서는 분산 시스템 문제로 바뀔 수 있다.

## Read Replica와의 차이

이 질문은 같이 나올 가능성이 높다.

`Read Replica`는 같은 데이터를 복제해서 읽기 부하를 분산하는 방식이다.  
반면 `샤딩`은 데이터 자체를 분할해서 저장하는 방식이다.

즉 Replica는 읽기 확장에 강하고, 샤딩은 저장량과 쓰기 부하까지 포함한 수평 확장 전략에 가깝다.

## 면접 답변 예시

> 샤딩은 하나의 큰 데이터베이스를 여러 조각으로 나눠 서로 다른 DB 서버에 분산 저장하는 방식입니다. 서비스 규모가 커져서 단일 DB가 데이터 양이나 쓰기 트래픽을 감당하기 어려울 때 검토하는 수평 확장 전략입니다. 여기서 가장 중요한 것은 샤드 키 설계인데, 조회 패턴과 데이터 쏠림 가능성을 함께 봐야 합니다. 예를 들어 user_id 기준 샤딩은 특정 사용자 데이터를 한 샤드에 모으기 좋지만, 샤드 수 변경이나 재분배는 별도 고민이 필요합니다. 장점은 확장성과 부하 분산이지만, 조인과 트랜잭션, 집계, 운영 복잡도가 크게 올라간다는 점은 분명한 trade-off라고 생각합니다.

## 장점

- 데이터 양과 쓰기 트래픽을 수평으로 확장할 수 있다.
- 특정 서버 병목을 줄이는 데 도움이 된다.
- 대규모 서비스에서 물리적 한계를 넘는 전략이 될 수 있다.

## 단점

- 설계와 운영 난도가 높다.
- 샤드 간 조인, 트랜잭션, 집계가 어려워진다.
- 리샤딩 비용이 크고 위험하다.

## 주의사항 / 실무 팁

- 샤딩은 먼저 쓰는 기술이 아니라, 단일 DB 최적화 이후에도 한계가 있을 때 검토하는 경우가 많다.
- 샤드 키를 잘못 고르면 확장성보다 데이터 쏠림 문제가 더 커질 수 있다.
- 면접에서는 `Read Replica와의 차이`, `샤드 키`, `운영 복잡도`를 같이 말하면 답변 밀도가 높아진다.
- "확장성 확보"만 강조하지 말고, 분산 시스템으로 가면서 생기는 비용도 꼭 같이 언급하는 편이 좋다.
