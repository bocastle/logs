# 논리 삭제(Soft Delete)와 물리 삭제(Hard Delete) 정리

## 핵심 요약

- **물리 삭제(Hard Delete)**: `DELETE`로 **데이터를 실제로 제거**합니다.
- **논리 삭제(Soft Delete)**: `UPDATE`로 삭제 여부 컬럼(예: `deleted_at`)을 갱신해 **삭제 표시만** 합니다.
- 상황에 따라 장단점이 있으므로, 요구사항(복구/감사/성능/정책)에 맞게 선택합니다.

## 개념과 차이점

데이터베이스에서 데이터를 삭제하는 방법은 크게 **물리 삭제(Hard Delete)** 와 **논리 삭제(Soft Delete)** 가 존재합니다.

- 물리 삭제는 `DELETE` 명령어를 통해 직접 데이터를 삭제하는 방식입니다.
- 논리 삭제는 `UPDATE` 명령을 사용하여 삭제를 여부를 나타내는 컬럼을 수정하는 방식을 의미합니다.

즉, **물리 삭제는 실제로 데이터를 삭제**하는 반면 **논리 삭제는 데이터가 삭제되었음을 표시만** 한다는 점에서 차이가 있습니다.

## 예시 SQL

### 물리 삭제 처리

```sql
delete from member where id = 1;
```

### 논리 삭제 처리와 조회

```sql
update member set deleted_at = curdate() where id = 1;
select * from member where deleted_at is null;
```

## 어떤 방식을 선택하는 것이 가장 좋은가요?

방식마다 장단점이 존재하기 때문에 상황에 맞는 선택을 하는 것이 옳다고 생각합니다.

## 장점

### 물리 삭제의 장점

- 실제로 데이터를 삭제하기 때문에 **저장 공간을 새로 확보**할 수 있습니다.
- 테이블의 크기를 줄여 **검색 속도 향상**을 기대할 수 있습니다.

### 논리 삭제의 장점

- 데이터를 삭제하지 않기 때문에 **데이터 복구에 용이**합니다.
- 삭제된 데이터가 **비즈니스 의사결정에 사용**될 수 있습니다.

## 단점

### 물리 삭제의 단점

- 데이터를 다시 **복구하기 어렵습니다.**
- 삭제된 데이터가 **비즈니스 의사결정에 사용되기 어렵습니다.**

### 논리 삭제의 단점

- 테이블에 데이터가 많아져 **성능에 악영향**을 줄 수 있습니다.
- 논리 삭제된 데이터를 제외하지 않고 조회하는 **실수가 발생**할 수 있습니다.

## 주의사항 / 실무 팁

- 논리 삭제를 쓴다면 조회 시 `deleted_at is null` 조건을 **일관되게 적용**할 수 있도록(예: 뷰/쿼리 빌더/ORM 필터) 규칙을 정해두는 편이 안전합니다.
- 논리 삭제 컬럼(`deleted_at`)에 인덱스를 두거나, 자주 쓰는 조회 조건에 맞춘 인덱싱을 고려하면 성능 저하를 줄이는 데 도움이 됩니다.
- “복구 가능해야 하는 기간”이 있다면 **보관 기간(Retention) 정책**을 두고, 필요 시 일정 기간 이후 물리 삭제로 정리하는 전략도 고려할 수 있습니다.
