# API 버저닝 전략 정리

## 핵심 요약
- `API 버저닝`은 클라이언트와 서버가 동시에 진화할 수 있도록 변경 호환성을 관리하는 방법이다.
- 대표 방식으로 URL 버전(`/v1`), 헤더 버전, 쿼리 파라미터 버전, 미디어 타입 버전이 있다.
- 면접에서는 "기존 클라이언트를 깨지 않으면서 API를 변경하기 위한 계약 관리 전략"이라고 설명하면 핵심이 잘 잡힌다.
- 중요한 것은 단순히 버전 표기를 어디에 두느냐보다, 어떤 변경을 버전 분리 대상으로 볼지 기준을 세우는 것이다.

## 개념 설명
서비스가 커지면 API는 계속 바뀐다. 필드 이름이 바뀌기도 하고, 응답 구조가 달라지기도 하고, 인증 방식이나 에러 모델이 바뀌기도 한다. 문제는 서버는 보통 한 번 배포하면 끝이지만, 클라이언트는 즉시 모두 업데이트되지 않는다는 점이다. 특히 모바일 앱이나 외부 파트너 연동은 구버전이 오래 남아 있을 수 있다.

그래서 API 버저닝은 "변경을 아예 하지 말자"가 아니라, 변경이 생겼을 때 호환성을 어떻게 통제할지에 대한 운영 원칙이다. 보통 하위 호환을 유지할 수 있는 변경은 같은 버전에서 처리하고, 기존 클라이언트를 깨뜨릴 수 있는 변경은 새 버전으로 분리한다.

대표 전략은 다음과 같다.

- URL 버전: `/api/v1/orders`
- 헤더 버전: `X-API-Version: 2`
- Accept 헤더 기반 미디어 타입 버전
- 쿼리 파라미터 버전: `?version=2`

실무에서는 URL 버전이 가장 직관적이라 자주 쓰이지만, 어떤 방식이든 더 중요한 것은 "폐기 시점 공지, 마이그레이션 기간, 문서화, 모니터링"까지 함께 가야 한다는 점이다.

## 예시
주문 조회 API가 기존에는 아래처럼 단순했다고 해 보자.

- `status`: `"PAID"`
- `price`: 정수 하나

그런데 글로벌 확장으로 통화 정보와 상세 상태 모델이 필요해졌다.

- `status`를 더 세분화해야 한다
- `price`를 `{ amount, currency }` 구조로 바꾸고 싶다

이 변경은 기존 앱을 깨뜨릴 가능성이 높다. 이런 경우 `v1`은 유지하고, 새 구조는 `v2`로 제공하는 방식이 일반적이다. 동시에 서버는 트래픽 비중을 보면서 구버전 사용량을 관찰하고, 충분한 공지 후에 점진적으로 종료한다.

## 면접 답변 예시
- 질문: "API 버저닝은 왜 필요한가요?"  
  답변: "서버 API는 계속 진화하지만 모든 클라이언트가 동시에 업데이트되지는 않기 때문입니다. 그래서 기존 클라이언트를 깨지 않으면서 변경을 도입하려면 버전 단위로 계약을 관리해야 합니다."
- 질문: "어떤 방식이 가장 많이 쓰이나요?"  
  답변: "실무에서는 URL 버전이 가장 직관적이라 많이 씁니다. 다만 중요한 건 표기 위치보다, 어떤 변경을 breaking change로 보고 언제 새 버전을 만들지 기준을 명확히 두는 것입니다."

## 장점
- 하위 호환성 관리 기준을 명확히 잡을 수 있다
- 외부 파트너나 모바일 앱처럼 배포 통제가 어려운 환경에서 특히 유리하다
- API 설계, 문서화, 운영 정책까지 연결해서 답변하기 좋다

## 단점
- 버전이 늘어날수록 유지보수 비용이 커진다
- 서버 코드와 문서가 중복되기 쉽다
- 버전 정책이 느슨하면 불필요한 버전 난립이 생길 수 있다

## 주의사항 / 실무 팁
- 사소한 필드 추가까지 무조건 새 버전을 만드는 것은 과하다. 하위 호환 가능 여부를 기준으로 판단해야 한다.
- 버전 분리만 해 두고 종료 정책이 없으면 오래된 버전이 영구 유지되어 운영 부담이 커진다.
- API Gateway, 문서 포털, 로그 대시보드에서 버전별 사용량을 추적할 수 있어야 실제 폐기가 가능하다.
- 면접에서는 "`버전 표기 방식`보다 `호환성 기준과 폐기 정책`이 더 중요하다"는 관점을 같이 말하면 실무 감각이 좋아 보인다.
