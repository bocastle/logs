# 파일 업로드 바이러스 스캔 흐름을 분리하는 법 정리

## 핵심 요약

- 업로드 저장 성공과 사용자에게 공개 가능한 상태를 분리해 검증 전 파일이 다운로드되지 않게 한다.
- Quarantine 저장소는 공개 bucket과 권한·도메인을 나누고 application도 판정 전 object를 제공하지 않는다.
- Scanner timeout과 오류는 clean으로 처리하지 않고 재시도·수동 격리 정책을 적용한다.

## 개념 설명

파일 업로드가 object storage에 저장됐다는 사실은 안전하게 공개해도 된다는 뜻이 아니다. 업로드 API는 먼저 비공개 quarantine에 저장하고 상태를 `SCANNING`으로 기록한 뒤 비동기 scanner가 판정하도록 한다.

Clean 판정 뒤에만 공개 bucket으로 복사하거나 다운로드 권한을 활성화한다. Infected는 격리·삭제 정책에 따라 처리하고 감사 기록을 남긴다. Scanner 장애와 timeout은 clean과 구분해 fail closed하고, 압축 폭탄·암호화 archive처럼 스캔할 수 없는 파일의 정책도 정한다.

## 예시

```text
upload -> quarantine/object-id
scan job -> clean -> publish
         -> infected -> reject + audit log
client sees: UPLOADED -> SCANNING -> AVAILABLE or REJECTED
```

사용자에게는 업로드 완료와 검사 완료를 다른 상태로 보여 준다. Object key를 추측해도 quarantine에서 직접 다운로드할 수 없어야 하며 presigned URL도 공개 가능 상태에서만 발급한다.

## 면접 답변 예시

> 파일 업로드는 저장 완료와 공개 가능 상태를 분리하겠습니다. 원본은 공개 권한이 없는 quarantine에 두고 비동기 scanner가 clean으로 판정한 뒤에만 다운로드를 허용합니다. Scanner timeout과 오류는 안전한 결과로 간주하지 않고 재시도하거나 수동 격리합니다. 파일 hash, scanner version과 판정 결과를 감사 로그에 남기고 queue age와 scan latency를 사용자 상태와 함께 관찰합니다.

## 장점

- 스캔 지연과 거절을 비동기 상태로 명확하게 사용자에게 보여 줄 수 있다.
- 악성 파일이 공개 URL과 일반 다운로드 경로에 바로 노출되는 것을 막는다.

## 단점

- Quarantine 접근 제어가 약하면 애플리케이션 상태와 무관하게 검증 전 파일이 노출된다.
- 대용량 파일과 archive는 scanner 시간과 CPU를 오래 점유해 대기열을 막을 수 있다.

## 주의사항 / 실무 팁

- Scan latency뿐 아니라 queue oldest age, scanner 오류율과 판정 불가 수를 알림으로 둔다.
- Presigned download URL은 `AVAILABLE` 상태에서만 발급되는지 권한 테스트를 둔다.
