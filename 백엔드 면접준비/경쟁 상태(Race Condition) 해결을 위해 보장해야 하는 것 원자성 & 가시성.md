# 경쟁 상태(Race Condition) 해결을 위해 보장해야 하는 것: 원자성 & 가시성

## 핵심 요약

- **경쟁 상태(Race Condition)** 는 여러 스레드가 **공유 자원**에 동시에 접근할 때, **실행 순서(interleaving)** 에 따라 결과가 달라지는 현상입니다.
- 경쟁 상태를 해결하려면 **원자성(Atomicity)** 과 **가시성(Visibility)** 이 **모두 보장**되어야 합니다.

## 개요

경쟁 상태(Race Condition)는 두 개 이상의 스레드가 공유 자원에 동시에 접근할 때 스레드 간의 실행 순서에 따라 결과가 달라지는 현상입니다. 이 문제를 해결하려면 **원자성**과 **가시성**을 모두 보장해야 합니다.

### 원자성(Atomicity)

- 공유 자원에 대한 작업의 단위가 더 이상 쪼갤 수 없는 **하나의 연산처럼 동작**하는 성질을 의미합니다.

### 가시성(Visibility)

- 한 스레드에서 변경한 값이 다른 스레드에서 **즉시 확인 가능**한 성질을 의미합니다.

## 원자성을 보장하지 않으면 어떤 문제가 발생하나요?

예를 들어 `i++` 연산은 하나의 문장이지만, CPU가 이를 수행하려면 **세 단계의 instruction**으로 분리됩니다.

```text
1) i 변수의 기존 값을 읽음 (Read)
2) 기존 값에 1을 더함 (Modify)
3) 결과 값을 다시 i 변수에 할당 (Write)
```

연산 사이에 다른 스레드가 개입하면 기대하지 않은 결과가 발생할 수 있습니다.

- 두 개의 스레드가 동시에 `i++` 연산을 수행할 때,
  - `Thread 1`이 `i + 1`을 하기 전에,
  - `Thread 2`가 `i`를 읽어서 `i + 1`을 수행한 후 반영하면,
  - `Thread 2`의 연산은 **무시(유실)** 될 수 있습니다.

## 가시성을 보장하지 않으면 어떤 문제가 발생하나요?

현대의 컴퓨터는 여러 개의 CPU 코어가 있고 각 코어마다 **CPU 캐시**가 존재합니다.

- 한 스레드에서 공유 자원을 변경할 경우
  - 메인 메모리에서 CPU 캐시로 값을 읽어들인 후,
  - 변경된 값을 자신의 CPU 캐시에 반영합니다.
- 하지만 변경된 값이 **메인 메모리에 언제 반영될지 알 수 없기 때문에**
  - 다른 스레드가 공유 자원을 읽을 때 변경 사항을 **즉시 확인하지 못할 수 있습니다**.

## Java에서 원자성과 가시성을 보장하는 방법

### 원자성과 가시성을 모두 보장하려면

다음과 같은 방법으로 동기화할 수 있습니다.

```text
- synchronized 키워드
- CAS(Compare-And-Swap) 알고리즘을 사용하는 Atomic 클래스
- ReentrantLock 같은 lock 클래스
- Concurrent Collections
```

### 가시성만 보장하려면

`volatile` 키워드를 사용하여 CPU 캐시를 사용하지 않고 메인 메모리에서 공유 자원을 직접 읽거나 쓸 수 있습니다.

```text
- volatile 사용 시 주의
  - 하나의 스레드에서만 쓰기 작업을 수행하고,
  - 나머지 스레드는 읽기 작업만 수행해야 합니다.
```

## 장점

- 원자성과 가시성을 적절히 보장하면, 공유 자원 접근에서 발생하는 **경쟁 상태를 안정적으로 예방/완화**할 수 있습니다.

## 단점

- 동기화(`synchronized`, lock 등)는 상황에 따라 **성능 비용**(경합, 컨텍스트 스위칭 등)이 발생할 수 있습니다.
- `volatile`은 **가시성만** 보장하므로, 복합 연산(예: read-modify-write)에서는 단독으로 경쟁 상태를 해결하지 못합니다.

## 주의사항 및 실무 팁

- 경쟁 상태는 단순히 “여러 스레드가 동시에 접근한다”가 아니라, **타이밍/순서에 따라 결과가 달라질 때** 문제가 됩니다.
- 복합 연산(예: `i++`)은 원자적이지 않으므로, 단순히 변수를 공유하는 것만으로 안전하지 않습니다.
- `volatile`은 “항상 안전한 동시성 도구”가 아니라, **단일 writer + 다수 reader** 같은 조건에서 적합합니다.
- 여러 스레드가 수정하는 공유 상태는 `synchronized`, `Lock`, `Atomic*`, concurrent 컬렉션 등 **원자성과 가시성을 함께** 만족하는 수단을 우선 고려합니다.
