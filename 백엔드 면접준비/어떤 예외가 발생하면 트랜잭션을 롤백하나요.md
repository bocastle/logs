# 어떤 예외가 발생하면 트랜잭션을 롤백하나요?

## 핵심 요약

- 트랜잭션 롤백 기준은 **개발 환경/프레임워크**에 따라 달라집니다.
- **Spring 기본 정책**: `Unchecked Exception(RuntimeException, Error)` 발생 시 롤백, `Checked Exception`은 기본적으로 롤백하지 않음.
- Spring에서는 `@Transactional`의 `rollbackFor`, `noRollbackFor`로 기본 정책을 **커스터마이징**할 수 있습니다.
- Java(EE) 환경에서는 **컨테이너 관리 트랜잭션(CMT)** 과 **프로그램 방식 트랜잭션 제어**에 따라 동작이 달라질 수 있습니다.

## Spring 트랜잭션 예외 처리 동작

### Checked Exception

- 기본적으로 **Checked Exception이 발생하더라도 롤백하지 않습니다.**
- Checked Exception은 컴파일 시점에 예외 처리를 강제하는 예외로, 개발자가 예외 발생 가능성을 예상하고 적절히 처리할 수 있는 **정상적인 예외 상황**이라고 가정하기 때문입니다.

### Unchecked Exception (RuntimeException, Error)

- Spring은 기본적으로 **Unchecked Exception 또는 Error가 발생하면 트랜잭션을 롤백**합니다.
- Unchecked Exception은 보통 프로그래머 실수나 시스템 문제로 인해 회복하기 어려운 상황(`NullPointerException`, `IllegalArgumentException` 등)이라고 가정합니다.
- Spring은 JDBC, JPA, Hibernate 등 하위 데이터 액세스 계층에서 발생하는 다양한 예외를 공통의 Unchecked Exception인 `DataAccessException` 계층으로 변환하여 처리합니다. 이를 통해 **일관된 예외 처리 전략**을 수립할 수 있습니다.

### `@Transactional`로 롤백 정책 조정

기본 동작과 별개로 `@Transactional`의 `rollbackFor`나 `noRollbackFor` 속성을 사용해 다음을 설정할 수 있습니다.

- 특정 **Checked Exception**에 대해서도 **롤백 유도**
- 특정 **Unchecked Exception**에 대해 **롤백하지 않도록** 설정

> 설정 예시는 아래처럼 “속성명”을 코드 블록으로 정리합니다.

```java
@Transactional(
  rollbackFor = { /* 특정 Checked Exception */ },
  noRollbackFor = { /* 특정 Unchecked Exception */ }
)
```

## Java(EE) 환경에서의 트랜잭션 롤백 동작

자바(EE) 환경에서는 컨테이너가 관리하는 트랜잭션(CMT)과 개발자가 직접 관리하는 프로그램 방식 트랜잭션 제어가 모두 존재하며, 이들 환경은 Spring과 **다르게 동작할 수 있습니다.**

### 컨테이너 관리 트랜잭션(CMT) 환경 (EJB, CDI 등)

- 기본적으로 **Unchecked Exception**이 발생하면 컨테이너가 자동으로 트랜잭션을 롤백합니다.
- **Checked Exception**은 기본적으로 롤백을 트리거하지 않습니다.
- 필요 시 `@ApplicationException(rollback=true)` 등의 어노테이션으로 롤백을 강제할 수 있습니다.

```java
@ApplicationException(rollback = true)
public class MyCheckedException extends Exception {}
```

### 프로그램 방식의 트랜잭션 제어

- 개발자가 트랜잭션의 시작/커밋/롤백을 직접 관리합니다.
- 이 경우 예외 종류와 관계없이, 상황에 맞게 **명시적으로 `rollback()`을 호출**해 트랜잭션을 종료합니다.

```java
try {
  // tx.begin();
  // ...
  // tx.commit();
} catch (Exception e) {
  // tx.rollback();
  throw e;
}
```

## 장점

### (Spring 기본 정책의) 장점

- “회복하기 어려운 예외(대개 Unchecked)” 발생 시 자동 롤백으로 **안전한 기본값**을 제공합니다.
- 데이터 액세스 계층 예외를 `DataAccessException`으로 통일해 **일관된 예외 처리**를 구성하기 쉽습니다.

## 단점

### (기본 정책 고정 시) 단점

- 실제 도메인 요구사항에 따라 “Checked Exception도 롤백해야 하는 경우”가 있는데, 기본 정책만 따르면 **의도와 다른 결과**가 날 수 있습니다.
- 반대로 특정 Unchecked Exception은 롤백하지 않아야 하는 경우가 있을 수 있어, 상황에 맞게 설정을 점검해야 합니다.

## 주의사항 / 실무 팁

- “예외가 Checked냐/Unchecked냐”만 보지 말고, **비즈니스적으로 롤백이 필요한지**를 기준으로 `rollbackFor`, `noRollbackFor`를 설계하세요.
- 하위 데이터 접근 계층(JDBC/JPA/Hibernate 등) 예외가 Spring에서 `DataAccessException`으로 감싸질 수 있으므로, **최종적으로 올라오는 예외 타입**을 로깅/디버깅으로 확인해 두면 설정 실수를 줄일 수 있습니다.
- Java(EE)처럼 컨테이너/트랜잭션 관리 방식이 다른 환경에서는 “Spring에서의 상식”이 그대로 적용되지 않을 수 있으니, **사용 중인 트랜잭션 모델(CMT vs 프로그램 방식)** 을 먼저 확정한 뒤 정책을 정리하세요.
