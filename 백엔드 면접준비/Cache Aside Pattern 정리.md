# Cache Aside Pattern 정리

## 핵심 요약

- `Cache Aside Pattern`은 애플리케이션이 먼저 캐시를 조회하고, 없으면 DB에서 읽어 온 뒤 캐시에 다시 넣는 방식이다.
- 구현이 단순하고 가장 널리 쓰이는 캐시 패턴이지만, 데이터 최신성과 캐시 무효화가 항상 숙제다.
- 읽기 성능을 높이는 데 효과적이지만, 캐시 미스가 몰리면 DB 부하가 다시 커질 수 있다.
- 면접에서는 "조회 흐름", "쓰기 시 캐시 정리 방식", "어떤 문제가 생길 수 있는지"를 같이 설명하면 좋다.

## Cache Aside Pattern이란 무엇인가

가장 기본적인 캐시 전략은 "필요할 때만 캐시에 올리는 것"이다.  
이 방식을 보통 `Cache Aside Pattern`이라고 부른다.

흐름은 단순하다.

1. 애플리케이션이 캐시를 먼저 조회한다.
2. 캐시에 값이 있으면 바로 반환한다.
3. 캐시에 값이 없으면 DB를 조회한다.
4. 조회한 결과를 캐시에 저장한 뒤 반환한다.

즉 캐시가 데이터를 스스로 가져오는 게 아니라, 애플리케이션이 DB와 캐시 사이를 직접 조율하는 방식이다.

## 왜 많이 쓰는가

가장 큰 이유는 단순함과 유연성이다.

- 자주 조회되는 데이터를 빠르게 응답할 수 있다.
- 캐시에 올릴 대상을 애플리케이션 레벨에서 통제하기 쉽다.
- 기존 DB 구조를 크게 바꾸지 않고 도입할 수 있다.

특히 읽기 요청이 훨씬 많은 서비스에서 효과가 크다.  
상품 상세, 사용자 프로필, 게시글 조회 같은 패턴이 대표적이다.

## 동작 예시

예를 들어 상품 상세 조회 API가 있다고 해 보자.

- `GET /products/1` 요청이 들어온다.
- Redis에서 `product:1` 키를 먼저 찾는다.
- 있으면 Redis 값을 반환한다.
- 없으면 DB에서 조회하고, 일정 TTL과 함께 Redis에 저장한 뒤 응답한다.

이 흐름은 단순해서 인터뷰에서 말하기도 좋다.  
핵심은 "캐시 히트면 빠르고, 미스면 DB를 거쳐 캐시를 채운다"는 점이다.

## 쓰기 시에는 어떻게 하나

읽기보다 더 중요한 질문은 쓰기다.  
데이터가 바뀌었는데 캐시가 예전 값을 갖고 있으면 정합성 문제가 생기기 때문이다.

보통은 다음 둘 중 하나를 많이 쓴다.

- DB를 먼저 수정하고 관련 캐시를 삭제한다.
- DB를 수정한 뒤 캐시 값을 갱신한다.

실무에서는 "업데이트 후 캐시 삭제"가 비교적 단순해서 많이 쓰인다.  
다음 조회에서 다시 DB를 읽고 최신 값을 캐시에 채우는 방식이다.

## 자주 나오는 문제

### 캐시 미스 폭증

인기 데이터가 만료되는 순간 동시에 요청이 몰리면 DB로 트래픽이 한꺼번에 떨어질 수 있다.  
이 문제는 `캐시 스탬피드`와 연결된다.

### 오래된 데이터

TTL이 길거나 무효화가 늦으면 사용자는 stale 데이터를 볼 수 있다.

### 캐시 무효화 복잡도

연관된 키가 많아질수록 어떤 캐시를 같이 지워야 하는지 관리가 어려워진다.

즉 Cache Aside는 단순하지만, 운영에서는 "언제 비우고 얼마나 유지할지"가 핵심이다.

## Write Through와의 차이

`Write Through`는 데이터를 쓸 때 DB와 캐시에 동시에 반영하는 방식이다.  
반면 `Cache Aside`는 읽을 때 캐시를 채우고, 쓸 때는 보통 무효화 중심으로 처리한다.

그래서 Cache Aside는 읽기 최적화에 유연하고, Write Through는 정합성을 더 직접적으로 관리하기 쉽다.  
대신 Write Through는 모든 쓰기에 캐시 관여가 들어간다는 부담이 있다.

## 면접 답변 예시

> Cache Aside Pattern은 애플리케이션이 먼저 캐시를 조회하고, 캐시에 값이 없을 때만 DB를 읽은 뒤 그 결과를 다시 캐시에 저장하는 패턴입니다. 읽기 트래픽이 많은 서비스에서 가장 널리 쓰이는 방식인데, 구현이 단순하고 어떤 데이터를 캐싱할지 애플리케이션에서 제어하기 쉽다는 장점이 있습니다. 다만 데이터 변경 시 캐시 무효화를 잘못하면 오래된 값을 보여줄 수 있고, 인기 키가 한꺼번에 만료되면 DB에 부하가 몰릴 수 있습니다. 그래서 TTL, 무효화 전략, 캐시 스탬피드 대응까지 같이 설계해야 실무적으로 안정적입니다.

## 장점

- 구현이 직관적이고 도입이 쉽다.
- 읽기 성능 개선 효과가 크다.
- 어떤 데이터를 캐싱할지 유연하게 제어할 수 있다.

## 단점

- 캐시 무효화가 어렵다.
- 오래된 데이터를 노출할 수 있다.
- 캐시 미스 집중 시 DB 부하가 튈 수 있다.

## 주의사항 / 실무 팁

- 쓰기 후 캐시 삭제를 기본 전략으로 두면 설계가 단순해지는 경우가 많다.
- TTL만 믿지 말고 데이터 변경 이벤트와 무효화 정책을 같이 봐야 한다.
- 인기 키에는 락, 선갱신, TTL 분산 같은 스탬피드 대응을 검토할 수 있다.
- 면접에서는 `조회 흐름`, `쓰기 후 무효화`, `stale data 위험`을 묶어서 설명하면 답변이 깔끔하다.
