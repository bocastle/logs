# 파티션 프루닝 정리

## 핵심 요약
- `파티션 프루닝`은 조건에 맞지 않는 파티션을 아예 읽지 않고 필요한 파티션만 조회하는 최적화다.
- 핵심은 "파티셔닝을 했다"가 아니라, 쿼리 조건 덕분에 불필요한 파티션 스캔을 줄일 수 있느냐이다.
- 면접에서는 대용량 테이블 조회 성능 개선과, 파티션 키 설계가 왜 중요한지 함께 설명하면 좋다.
- 파티션을 나눴다고 자동으로 성능이 좋아지는 것은 아니며, 조건식이 파티션 키를 제대로 타야 효과가 난다.

## 개념 설명
대용량 로그나 주문 이력처럼 데이터가 아주 많은 테이블은 날짜나 범위 기준으로 파티션을 나누는 경우가 많다. 이렇게 하면 전체 데이터를 한 덩어리로 관리하지 않아도 돼 운영이 편해진다.

그런데 진짜 성능 이점은 `파티션 프루닝`이 잘 일어날 때 나온다. 예를 들어 월별 파티션 테이블에서 `2026-06` 데이터만 조회한다면, DB는 모든 월 파티션을 다 뒤지는 대신 6월 파티션만 읽을 수 있다. 이처럼 조건과 무관한 파티션을 쿼리 계획에서 제거하는 것이 파티션 프루닝이다.

즉, 파티셔닝은 구조이고, 파티션 프루닝은 그 구조를 실제 쿼리 최적화에 활용하는 단계라고 볼 수 있다. 면접에서는 이 둘을 분리해서 설명해야 답변이 선명해진다.

## 예시
주문 테이블이 `created_date` 기준 월 단위 파티션으로 나뉘어 있다고 가정해보자.

- 좋은 예
  - 조건: `WHERE created_date >= '2026-06-01' AND created_date < '2026-07-01'`
  - 결과: 6월 파티션만 읽을 가능성이 높다
- 아쉬운 예
  - 조건: `WHERE DATE_FORMAT(created_date, '%Y-%m') = '2026-06'`
  - 결과: 함수 적용 때문에 파티션 키를 직접 활용하지 못해 프루닝이 약해질 수 있다

즉, 같은 의미의 조회라도 조건식을 어떻게 쓰느냐에 따라 읽는 데이터 범위가 크게 달라질 수 있다.

## 면접 답변 예시
- 질문: "파티션 프루닝은 무엇인가요?"  
  답변: "파티션 테이블에서 쿼리 조건에 맞지 않는 파티션을 실행 계획에서 제외하고, 필요한 파티션만 읽는 최적화입니다. 그래서 대용량 테이블에서도 스캔 범위를 줄여 조회 성능을 높일 수 있습니다."
- 질문: "파티셔닝만 하면 자동으로 빨라지나요?"  
  답변: "그렇지는 않습니다. 쿼리 조건이 파티션 키를 잘 활용해야 프루닝이 일어나고, 그렇지 않으면 여러 파티션을 다 읽게 돼 기대한 성능 이점을 못 얻을 수 있습니다."

## 장점
- 대용량 테이블 조회 시 스캔 범위를 크게 줄일 수 있다
- 운영상 보관 주기 관리와 성능 최적화를 함께 가져가기 좋다
- 날짜 기반 로그, 주문, 이벤트 데이터처럼 범위 조회가 많은 경우 특히 효과적이다

## 단점
- 파티션 키 설계를 잘못하면 성능 이점이 거의 없다
- 쿼리 작성 방식이 조금만 달라져도 프루닝 효과가 깨질 수 있다
- 파티션 수가 과도하게 많아지면 관리 복잡도가 올라간다

## 주의사항 / 실무 팁
- 파티션 키는 자주 쓰는 조회 조건과 맞물리게 설계해야 한다.
- 함수로 감싼 조건이나 암묵적 형변환은 프루닝을 방해할 수 있어서 주의해야 한다.
- 면접에서는 "파티셔닝"과 "인덱싱"을 혼동하지 말고, 각각 스캔 범위 축소 방식이 다르다고 설명하면 좋다.
- 실무에서는 실행 계획으로 실제 프루닝 여부를 확인하는 습관이 중요하다.
