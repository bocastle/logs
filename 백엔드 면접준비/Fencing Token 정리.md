# Fencing Token 정리

## 핵심 요약
- `Fencing Token`은 분산 락이나 리더 선출 환경에서 오래된 작업 주체를 구분하기 위한 단조 증가 값이다.
- 핵심 목적은 "락을 잡았다고 믿는 주체가 실제로는 더 이상 유효하지 않을 때" 잘못된 쓰기를 막는 것이다.
- 면접에서는 "분산 락만으로는 부족할 때, 더 최신 권한인지 검증하는 안전장치"라고 설명하면 전달이 쉽다.
- `lease`, `Split Brain`, 중복 실행 방지와 함께 연결해서 말하면 답변이 더 실무적으로 들린다.

## 개념 설명
분산 시스템에서 락을 잡았다고 해서 항상 안전한 것은 아니다. 예를 들어 A 서버가 락을 획득한 뒤 GC pause나 네트워크 지연으로 한동안 멈췄다고 해 보자. 그 사이 락 만료 시간이 지나고 B 서버가 같은 락을 다시 잡을 수 있다.

문제는 A 서버가 뒤늦게 깨어나 자신이 여전히 작업 권한이 있다고 착각할 수 있다는 점이다. 이때 그냥 "락 획득 여부"만 믿고 쓰기를 수행하면 오래된 주체가 최신 데이터를 덮어쓸 수 있다.

`Fencing Token`은 이런 상황을 막기 위해 락 획득 시점마다 증가하는 번호를 함께 발급한다. 그리고 실제 저장소나 다운스트림 시스템은 "더 큰 토큰만 허용"하도록 검증한다. 그러면 늦게 도착한 오래된 요청은 자동으로 거절된다.

즉, `Fencing Token`의 본질은 락 자체가 아니라 "권한의 최신성 검증"이다.

## 예시
예를 들어 분산 락으로 파일 업로드 메타데이터를 갱신하는 서비스가 있다고 해 보자.

- A 인스턴스가 락을 잡고 토큰 `41`을 받는다
- A 인스턴스가 멈춘 사이 락이 만료된다
- B 인스턴스가 새 락을 잡고 토큰 `42`를 받는다
- B가 먼저 DB에 반영한다
- 뒤늦게 A가 깨어나 예전 작업 결과를 저장하려고 한다

이때 DB나 스토리지 계층이 "현재 토큰보다 작은 요청은 거절"하도록 구현돼 있으면 A의 쓰기는 실패한다. 반대로 이런 검증이 없으면 A의 늦은 쓰기가 최신 상태를 덮어쓸 수 있다.

## 면접 답변 예시
- 질문: "`Fencing Token`은 무엇인가요?"  
  답변: "분산 락 환경에서 락 획득 주체가 정말 최신 권한을 가진 주체인지 확인하기 위해 함께 발급하는 증가하는 토큰입니다. 다운스트림 시스템이 이 토큰을 검증해서 오래된 요청을 막습니다."
- 질문: "분산 락만 있으면 안 되나요?"  
  답변: "락이 만료된 뒤에도 기존 작업자가 늦게 살아나서 쓰기를 시도할 수 있기 때문입니다. 그래서 락 획득 여부만 믿지 말고, 최신 권한인지 토큰으로 한 번 더 검증해야 안전합니다."
- 질문: "언제 필요한가요?"  
  답변: "락 기반 스케줄러, 리더 기반 쓰기 조정, 외부 스토리지 갱신처럼 중복 실행이나 늦은 쓰기가 큰 문제를 만드는 환경에서 특히 중요합니다."

## 장점
- 분산 락만으로 막기 어려운 늦은 요청 문제를 줄일 수 있다
- `Split Brain`, 네트워크 지연, GC pause 같은 현실적인 장애 상황을 더 안전하게 다룰 수 있다
- 저장 계층에서 최신성 검증 기준이 명확해져 운영 중 원인 분석이 쉬워진다

## 단점
- 락 서비스만이 아니라 실제 쓰기를 받는 저장소도 토큰 검증 로직을 알아야 한다
- 단순 락 도입보다 설계가 한 단계 복잡해진다
- 모든 외부 시스템이 토큰 비교를 지원하는 것은 아니라 적용 범위가 제한될 수 있다

## 주의사항 / 실무 팁
- `Fencing Token`은 락의 대체재가 아니라 보완 장치다. 둘 중 하나만으로 충분하다고 보면 위험하다.
- 토큰은 반드시 단조 증가해야 하고, 검증하는 쪽이 그 규칙을 신뢰할 수 있어야 한다.
- 면접에서는 "`락을 잡았는가`와 `가장 최신 권한인가`는 다른 문제"라고 짚어 주면 이해도가 높아 보인다.
- 실무에서는 Redis 락만으로 안전하다고 단정하지 말고, 실제 쓰기 대상에서 늦은 요청을 어떻게 막을지까지 설계해야 한다.
