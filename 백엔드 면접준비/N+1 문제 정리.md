# N+1 문제 정리

## 핵심 요약

- N+1 문제는 "목록 1번 조회 후 연관 데이터 때문에 추가 조회가 N번 더 발생하는 상황"이다.
- ORM을 쓸 때 특히 자주 나오고, JPA 면접에서도 거의 단골 질문이다.
- 무조건 조인을 많이 쓰는 게 답은 아니고, 데이터 양과 페이징 여부까지 같이 봐야 한다.
- 면접에서는 "왜 생기고, 어떻게 찾고, 어떤 방식으로 완화하는지"를 순서대로 말하면 깔끔하다.

## N+1 문제가 뭐냐

예를 들어 게시글 10개를 조회했다고 해보자.  
겉으로는 게시글 목록을 한 번 조회한 것 같지만, 각 게시글 작성자 정보를 보는 순간 작성자 조회가 10번 더 나갈 수 있다.

즉, 처음 1번의 조회 뒤에 연관 데이터를 가져오기 위한 쿼리가 N번 추가되는 구조라서 N+1 문제라고 부른다.

면접에서는 이렇게 짧게 말해도 충분하다.

> N+1 문제는 연관 엔티티를 조회하는 과정에서 예상보다 훨씬 많은 쿼리가 발생하는 문제입니다. 처음 쿼리 1번에 더해, 조회한 개수만큼 추가 쿼리가 붙어서 성능 저하가 생깁니다.

## 왜 생기나

핵심 원인은 "객체 그래프를 편하게 다루는 방식"과 "DB 조회 방식"이 다르기 때문이다.

ORM은 객체끼리 연결된 것처럼 보이게 해주지만, 실제 DB는 필요한 시점마다 쿼리를 날려서 데이터를 가져온다.  
특히 지연 로딩 상태에서 반복문 안에서 연관 필드를 접근하면 그때마다 추가 조회가 발생하기 쉽다.

예를 들어 이런 흐름이다.

1. 게시글 목록을 조회한다.
2. 애플리케이션 코드에서 게시글마다 작성자 이름을 꺼낸다.
3. 작성자 정보가 아직 없으면 각 게시글마다 추가 쿼리가 실행된다.

## 언제 잘 터지나

### 목록 화면

게시글 목록, 주문 목록, 댓글 목록처럼 여러 건을 한 번에 보여주는 화면에서 자주 발생한다.

### 반복문 안에서 연관 객체 접근할 때

`for` 문이나 스트림에서 `post.getUser().getName()` 같은 코드를 무심코 넣으면 바로 발생할 수 있다.

### API 응답 직렬화 과정

컨트롤러에서 엔티티를 그대로 반환하면 JSON 변환 과정에서 연관 객체를 건드리면서 추가 쿼리가 나갈 수도 있다.

## 어떻게 찾나

### SQL 로그 확인

가장 단순하고 확실한 방법이다.  
목록 조회 한 번만 생각했는데 비슷한 `select` 문이 여러 번 반복되면 의심해볼 수 있다.

### 성능 테스트나 APM

응답 시간은 큰데 코드상으론 단순 조회처럼 보일 때, 실제 쿼리 수를 보면 원인이 드러나는 경우가 많다.

### 코드 리뷰 포인트

- 반복문 안에서 연관 객체를 읽는지
- DTO로 바로 조회하지 않고 엔티티를 그대로 순회하는지
- 화면 요구사항에 비해 조회 전략이 너무 느슨한지

## 해결 방법

## 1. Fetch Join 사용

한 번의 조인 쿼리로 필요한 연관 데이터를 같이 가져오는 방식이다.

JPA에서는 `join fetch`를 써서 많이 해결한다.  
목록과 연관 데이터가 함께 꼭 필요할 때 직관적인 방법이다.

다만 컬렉션 페치 조인은 데이터가 불어나거나 페이징과 충돌할 수 있어서 무조건 만능은 아니다.

## 2. DTO 직접 조회

처음부터 화면이나 API 응답에 필요한 필드만 골라서 조회하는 방식이다.

실무에서는 읽기 전용 화면에서 꽤 자주 쓴다.  
엔티티 그래프를 따라가다가 쿼리가 터지는 일을 줄일 수 있고, 응답 형태도 명확해진다.

## 3. 배치 조회 설정

ORM이 연관 데이터를 하나씩 가져오지 않고 묶어서 가져오도록 설정하는 방법이다.

예를 들어 Hibernate의 배치 사이즈를 잡아두면 `N`번을 완전히 1번으로 줄이지는 못해도, 여러 번의 조회를 몇 번 안 되는 묶음 조회로 줄일 수 있다.

## 4. 조회와 쓰기 모델 분리

복잡한 화면은 엔티티 중심이 아니라 조회 전용 쿼리와 DTO로 다루고, 쓰기 로직은 엔티티 중심으로 유지하는 방식도 실무에서 많이 쓴다.

이렇게 하면 N+1 문제를 구조적으로 덜 만들 수 있다.

## 예시로 이해하기

주문 20건을 조회하는 API가 있다고 해보자.

- 주문 목록 조회 1번
- 각 주문의 회원 정보 조회 20번
- 각 주문의 배송 정보 조회 20번

이러면 총 41번의 조회가 일어날 수 있다.

반면 주문, 회원, 배송을 적절히 같이 조회하면 훨씬 적은 쿼리로 같은 데이터를 만들 수 있다.  
그래서 N+1 문제는 데이터 양이 커질수록 더 아프게 드러난다.

## 면접 답변 예시

> N+1 문제는 ORM에서 연관 데이터를 조회할 때 자주 생기는 성능 문제입니다. 예를 들어 목록을 한 번 가져온 뒤 각 항목의 연관 엔티티를 접근하면서 추가 쿼리가 N번 더 실행되는 구조입니다. 저는 보통 SQL 로그로 먼저 확인하고, 상황에 따라 fetch join, DTO 조회, 배치 조회 같은 방법으로 완화합니다. 다만 fetch join은 페이징이나 중복 데이터 이슈가 있을 수 있어서 화면 요구사항에 맞춰 선택하는 게 중요합니다.

## 장점

- 개념을 이해해두면 ORM 성능 문제를 빨리 감지할 수 있다.
- 단순히 "느리다"가 아니라 쿼리 수 관점에서 문제를 설명할 수 있다.
- 면접에서 JPA, ORM, API 성능 이야기를 연결해서 답하기 좋다.

## 단점

- 해결책마다 트레이드오프가 있다.
- 무리하게 조인을 늘리면 쿼리가 복잡해질 수 있다.
- 엔티티 구조를 편하게 설계한 대가로 조회 전략 고민이 필요해진다.

## 주의사항 / 실무 팁

- `EAGER`로 바꾼다고 자동 해결된다고 생각하면 위험하다. 상황에 따라 더 큰 쿼리를 만들 수도 있다.
- 컬렉션 fetch join과 페이징은 특히 조심해서 봐야 한다.
- 엔티티를 API 응답으로 바로 내보내는 방식은 N+1과 직렬화 문제를 같이 부를 수 있다.
- 성능 이슈는 감으로 판단하지 말고 실제 SQL 로그와 실행 계획, 응답 시간을 같이 봐야 한다.
