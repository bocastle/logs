# CQRS 정리

## 핵심 요약

- `CQRS(Command Query Responsibility Segregation)`는 데이터를 변경하는 명령과 조회하는 책임을 분리하는 설계 방식이다.
- 핵심은 `쓰기 모델과 읽기 모델을 같은 방식으로 다루지 않아도 된다`는 점이다.
- 면접에서는 CRUD를 무조건 버리는 패턴처럼 말하지 말고, `복잡한 도메인이나 조회/쓰기 요구가 크게 다른 시스템에서 선택적으로 쓰는 구조`라고 설명하는 편이 정확하다.
- 이벤트 소싱과 자주 함께 언급되지만, CQRS와 이벤트 소싱은 같은 개념이 아니라는 점도 같이 짚어 주면 좋다.

## CQRS가 뭐냐

일반적인 CRUD 구조에서는 생성, 수정, 조회가 같은 모델과 같은 저장 구조를 공유하는 경우가 많다.  
이 방식은 단순한 서비스에는 잘 맞지만, 도메인 규칙이 복잡해지거나 조회 요구가 매우 다양해지면 한 모델에 모든 책임이 몰릴 수 있다.

CQRS는 여기서 명령(Command)과 조회(Query)를 분리한다.

- Command: 상태를 바꾸는 요청
- Query: 상태를 읽는 요청

즉 `쓰기 쪽은 비즈니스 규칙 중심`, `읽기 쪽은 조회 성능과 응답 형태 중심`으로 따로 설계할 수 있게 해 준다.

## 왜 분리하나

- 쓰기 로직은 도메인 규칙과 일관성이 중요하다.
- 읽기 로직은 조인, 집계, 캐싱, 응답 최적화가 중요하다.

둘의 관심사가 다른데 한 구조에 모두 넣으면 서비스가 커질수록 복잡도가 올라간다.  
CQRS는 이 차이를 인정하고 모델을 나눠서 다루는 접근이다.

## 어떻게 구성되나

가장 단순한 형태는 애플리케이션 레벨에서 핸들러를 분리하는 것이다.

- Command Handler: 검증, 도메인 규칙 적용, 저장
- Query Handler: 읽기 전용 조회, DTO 구성, 최적화

더 나아가면 쓰기 DB와 읽기 DB를 분리하거나, 읽기 전용 프로젝션 테이블을 둘 수도 있다.  
하지만 면접에서는 "분리는 단계적일 수 있다"라고 말하는 게 좋다. 처음부터 거대한 분산 구조를 전제할 필요는 없다.

## 이벤트 소싱과는 같은가

같지 않다.

- `CQRS`: 읽기와 쓰기 책임을 분리하는 설계
- `이벤트 소싱`: 상태 변경을 이벤트의 연속으로 저장하는 방식

둘은 같이 쓰일 수 있다.  
예를 들어 Command 쪽에서 이벤트를 만들고, Query 쪽은 그 이벤트를 반영한 읽기 모델을 사용할 수 있다.  
하지만 CQRS를 쓴다고 반드시 이벤트 소싱을 해야 하는 건 아니다.

## 장점은 뭐냐

- 읽기와 쓰기 요구사항이 크게 다를 때 각각 최적화하기 좋다.
- 복잡한 도메인 규칙을 쓰기 모델에 집중시킬 수 있다.
- 조회 모델을 API 응답 형태에 맞춰 단순하게 설계할 수 있다.

예를 들어 주문 시스템처럼 쓰기 규칙은 복잡하고, 관리자 조회 화면은 집계와 검색이 많은 경우에 잘 맞는다.

## 단점은 뭐냐

CQRS는 구조를 분리하는 만큼 복잡도도 늘어난다.

- 모델이 둘 이상 생겨 학습 비용이 올라간다.
- 읽기 모델 동기화가 필요하면 최종 일관성 문제가 생길 수 있다.
- 작은 서비스에서는 오히려 과한 설계가 될 수 있다.

그래서 면접에서 "무조건 좋은 패턴"처럼 말하면 위험하다.  
`복잡성을 감수할 만큼 읽기/쓰기 요구 차이가 큰가`를 보고 선택한다고 답하는 게 더 좋다.

## 실무에서 언제 고려하나

- 주문, 결제, 정산처럼 도메인 규칙이 복잡한 시스템
- 조회 화면과 리포트 요구가 매우 많은 시스템
- 읽기와 쓰기 트래픽 특성이 크게 다른 시스템

반대로 단순 게시판이나 작은 관리 도구라면 CRUD 구조가 더 실용적일 수 있다.

## 면접 답변 예시

> CQRS는 Command Query Responsibility Segregation의 약자로, 데이터를 변경하는 책임과 조회하는 책임을 분리하는 설계 방식입니다. 쓰기 쪽은 도메인 규칙과 일관성을 중심으로 설계하고, 읽기 쪽은 조회 성능과 응답 최적화에 맞춰 별도로 설계할 수 있다는 게 핵심입니다. 다만 구조가 복잡해지고 읽기 모델 동기화 문제가 생길 수 있어서, 저는 복잡한 도메인이나 조회 요구가 큰 시스템에서 선택적으로 쓰는 패턴이라고 설명하는 편입니다. 또 이벤트 소싱과 자주 같이 나오지만 동일한 개념은 아니라고 같이 말합니다.

## 장점

- 아키텍처 선택 이유를 상황 중심으로 설명하기 좋다.
- DDD, 이벤트 기반 구조, 읽기 최적화와 연결해서 답변을 확장하기 쉽다.
- 단순 CRUD와의 차이를 비교하며 사고 과정을 보여 주기 좋다.

## 단점

- 구조를 과하게 크게 설명하면 실무 감각이 없어 보일 수 있다.
- 이벤트 소싱과 혼동하면 개념 구분이 흐려진다.
- 최종 일관성이나 운영 복잡도를 빼면 장점만 말하는 답변이 된다.

## 주의사항 / 실무 팁

- 면접에서는 먼저 `쓰기와 읽기 책임 분리` 한 줄로 정의하는 게 좋다.
- CQRS는 단계적으로 도입할 수 있고, 물리적 DB 분리가 필수는 아니라고 덧붙이면 안전하다.
- "복잡한 시스템에서 유용하지만 작은 서비스에는 과할 수 있다"는 균형 잡힌 관점을 준비해 두면 좋다.
- 이벤트 소싱과의 관계를 구분해서 말하면 후속 질문 대응이 쉬워진다.
