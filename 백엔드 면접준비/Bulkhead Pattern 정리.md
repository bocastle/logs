# Bulkhead Pattern 정리

## 핵심 요약

- `Bulkhead Pattern`은 장애가 한 영역에서 발생해도 전체 시스템으로 번지지 않게 자원과 흐름을 분리하는 패턴이다.
- 핵심은 "격리"다. 요청별로 스레드 풀, 커넥션 풀, 큐를 나눠서 한쪽 과부하가 다른 기능까지 먹지 못하게 만든다.
- 서킷 브레이커가 실패를 빠르게 끊는 장치라면, 벌크헤드는 애초에 영향을 받을 수 있는 범위를 좁히는 장치에 가깝다.
- 면접에서는 "왜 필요한지", "어디를 분리하는지", "대신 어떤 비용이 생기는지"를 같이 말하면 좋다.

## Bulkhead Pattern이란 무엇인가

`bulkhead`는 원래 배에서 쓰는 격벽 개념이다.  
배 한쪽에 물이 들어와도 전체가 바로 가라앉지 않도록 구획을 나누는 구조다.

소프트웨어에서도 비슷하다.  
특정 외부 API나 특정 기능이 과부하에 걸리더라도, 그 영향이 전체 서비스로 퍼지지 않게 자원 사용 구간을 분리하는 방식이 `Bulkhead Pattern`이다.

쉽게 말하면 "중요한 기능끼리 같은 숨통을 쓰지 않게 만드는 것"이라고 설명할 수 있다.

## 왜 필요한가

실무에서는 장애가 실패 자체보다 전파 때문에 더 커지는 경우가 많다.

- 느린 외부 API 호출이 스레드를 오래 점유할 때
- 특정 기능 요청이 폭증해서 공용 큐를 가득 채울 때
- 한 도메인의 장애가 전체 WAS 자원을 잠식할 때

이런 상황에서 자원이 전부 공유되어 있으면, 원래는 멀쩡한 기능도 같이 느려지거나 실패한다.  
벌크헤드는 이 전파를 줄이는 데 목적이 있다.

## 어떻게 적용하나

보통은 자원을 논리적으로 나눠서 적용한다.

- 외부 API A 전용 스레드 풀
- 배치 작업 전용 큐
- 읽기/쓰기 분리된 커넥션 풀
- 핵심 기능과 부가 기능의 실행 자원 분리

예를 들어 결제 서비스와 추천 서비스가 같은 스레드 풀을 쓴다면, 추천 API 지연이 결제 응답까지 늦출 수 있다.  
이때 추천 호출 전용 풀을 따로 두면 추천 영역에서 문제가 생겨도 결제 요청 처리 여지는 남길 수 있다.

## 서킷 브레이커와의 차이

둘 다 장애 대응 패턴이라 같이 묻는 경우가 많다.

`서킷 브레이커`는 실패율이 높아질 때 호출을 빠르게 차단해서 더 큰 지연을 막는 패턴이다.  
반면 `Bulkhead Pattern`은 자원 자체를 분리해서 한쪽 문제가 다른 쪽을 침범하지 못하게 하는 패턴이다.

즉 서킷 브레이커는 "끊는 장치", 벌크헤드는 "나누는 장치"라고 이해하면 비교가 쉽다.

실무에서는 둘 중 하나만 쓰기보다 같이 쓰는 경우도 많다.

## 장점이 큰 상황

- 외부 의존성이 여러 개 있고 중요도가 다를 때
- 특정 기능의 트래픽 변동폭이 클 때
- 장애 전파를 서비스 전체 관점에서 막아야 할 때
- 공용 스레드 풀이나 큐 병목이 자주 문제 될 때

특히 마이크로서비스든 모놀리식이든, "자원 공유 때문에 장애 범위가 커지는 구조"라면 벌크헤드 얘기가 자연스럽다.

## 단점과 trade-off

격리는 좋지만 운영은 단순해지지 않는다.

- 자원 활용률이 낮아질 수 있다.
- 풀 크기나 큐 크기 튜닝이 필요하다.
- 구조가 복잡해져서 운영 포인트가 늘어난다.
- 잘못 나누면 오히려 특정 구간이 더 빨리 포화될 수 있다.

즉 무조건 많이 쪼개는 게 답은 아니다.  
중요도와 트래픽 특성을 보고 분리할 지점을 정해야 한다.

## 면접 답변 예시

> Bulkhead Pattern은 장애가 한 기능에서 시작돼도 전체 시스템으로 번지지 않게 자원 구획을 나누는 패턴입니다. 예를 들어 외부 API 호출마다 스레드 풀이나 큐를 분리하면, 한쪽 API 지연이 전체 요청 처리 자원을 잠식하는 상황을 줄일 수 있습니다. 서킷 브레이커가 실패한 호출을 빠르게 차단하는 장치라면, 벌크헤드는 애초에 영향 범위를 분리하는 장치라고 볼 수 있습니다. 다만 자원 활용률이 떨어지거나 운영 복잡도가 늘 수 있어서, 중요 기능과 부가 기능처럼 격리 가치가 큰 지점에 선택적으로 적용하는 편이 좋습니다.

## 장점

- 장애 전파 범위를 줄이기 좋다.
- 핵심 기능의 생존 가능성을 높일 수 있다.
- 공용 자원 포화를 줄여 전체 안정성을 개선할 수 있다.

## 단점

- 자원 분배와 튜닝이 필요하다.
- 구조와 운영 복잡도가 올라간다.
- 분리 기준을 잘못 잡으면 비효율이 생긴다.

## 주의사항 / 실무 팁

- 모든 기능을 균등하게 나누기보다 중요도 높은 흐름부터 격리하는 편이 현실적이다.
- 스레드 풀, 큐, 커넥션 풀처럼 실제 병목 자원이 무엇인지 먼저 봐야 한다.
- 서킷 브레이커, 타임아웃, 재시도 정책과 함께 설명하면 답변이 더 실무적으로 들린다.
- 면접에서는 "격리로 안정성을 얻지만 자원 효율은 떨어질 수 있다"는 trade-off를 꼭 같이 말하는 편이 좋다.
