# OAuth 2.0 인증 정리

## 핵심 요약
- OAuth 2.0은 "/자원 접근 권한 위임/"을 위한 표준 프레임워크입니다.
- 핵심은 `인가 코드` 흐름에서 클라이언트가 사용자 비밀번호를 직접 다루지 않는 점입니다.
- 면접에서는 `클라이언트 유형`, `리다이렉트 URI`, `scope`, `토큰 갱신`을 함께 설명하면 완성도가 올라갑니다.

## 개념 정리
OAuth는 회원이 제3자 앱에 본인 정보 접근 권한만을 위임할 수 있게 해줍니다. 사용자가 서비스 A에 로그인하면, 서비스 A는 서비스 B 자원에 대해 접근 토큰만 받습니다.

- `Authorization Code`는 웹 앱에서 가장 많이 쓰는 방식입니다.
- `Scope`는 접근 범위를 제어합니다.
- `Refresh Token`은 접근 토큰이 만료됐을 때 새 토큰을 발급받는 수단입니다.

주요 역할자
- Resource Owner(사용자)
- Client(서비스)
- Authorization Server(인가 서버)
- Resource Server(자원 서버)

## 실무 예시
1. 사용자가 카카오로 로그인 버튼을 누름
2. 클라이언트는 인가 서버로 인증 요청
3. 인가 코드 발급 후 클라이언트가 백엔드로 전달
4. 백엔드가 인가 코드를 교환해 Access Token 획득
5. 사용자 정보 API 호출

OAuth 2.1 권고는 불필요한 `implicit flow` 같은 구형 흐름 사용을 지양합니다.

## 면접 답변 예시
"OAuth는 인증(Authentication) 자체보다 권한 위임(Authorization)을 정형화한 표준입니다. 제일 많이 쓰는 건 Authorization Code Flow이며, 프론트가 토큰을 직접 다루지 않도록 백엔드에서 코드 교환을 처리합니다. 이때 `redirect_uri` 검증, `state` 위변조 방지, `scope` 최소화, `refresh token` 회전 전략을 함께 관리해야 실무적으로 안전합니다."

## 장점
- 사용자 비밀번호 노출 없이 제3자 접근 권한 관리가 가능합니다.
- 권한 단위를 scope로 세분화해 보안 통제가 쉽습니다.
- 소셜 로그인, API 연동 등 범용성이 높습니다.

## 단점
- 플로우가 많아 구현 실수가 생기기 쉽습니다.
- 토큰 탈취나 재사용 공격 방지 정책이 필요합니다.
- 상태(state), CSRF, 리디렉션 URI 검증 누락 시 취약해집니다.

## 주의사항 / 실무 팁
- `redirect_uri`는 사전에 등록된 값만 허용하고 정규표현식으로 완화하지 마세요.
- `state`를 통해 CSRF 공격을 막고, code 교환은 서버에서 수행하세요.
- 토큰 저장은 `httpOnly` 쿠키 또는 안전한 서버 저장소를 우선으로 합니다.
