# DB 커넥션 풀 크기 산정 정리

## 핵심 요약
- DB 커넥션 풀 크기는 "크면 좋다"가 아니라 애플리케이션 동시성, DB 처리 능력, 쿼리 특성에 맞게 맞추는 값이다.
- 풀을 너무 작게 잡으면 요청이 대기하고, 너무 크게 잡으면 DB에 과부하를 몰아줄 수 있다.
- 면접에서는 스레드 수, 응답 시간, DB CPU, 대기 시간 지표를 함께 봐야 한다고 설명하면 좋다.
- 실무에서는 고정 숫자보다 관측 지표 기반 튜닝이 중요하다.

## 개념 설명
애플리케이션은 매 요청마다 새 DB 연결을 만드는 대신 커넥션 풀에서 미리 만들어 둔 연결을 재사용한다. 이 덕분에 연결 생성 비용을 줄이고 응답 속도를 안정화할 수 있다.

문제는 풀 크기를 얼마로 둘지다. 흔히 "트래픽이 많으니 크게 잡자"라고 생각하기 쉽지만, 커넥션 수가 늘어난다고 DB 처리량이 무한히 증가하지는 않는다.

오히려 커넥션이 지나치게 많으면 다음 문제가 생길 수 있다.
- DB CPU 컨텍스트 스위칭 증가
- 락 경쟁 증가
- 슬로우 쿼리 동시 실행 확대
- 애플리케이션은 빨리 밀어 넣지만 DB가 감당하지 못하는 병목

즉, 커넥션 풀은 애플리케이션의 병렬성과 DB의 처리 한계를 맞추는 조절 장치라고 보는 편이 정확하다.

## 예시
예를 들어 API 서버 인스턴스 하나에 요청 처리 스레드가 200개 있다고 해서, 커넥션 풀도 200으로 두는 건 위험할 수 있다.

다음처럼 접근하는 편이 실무적이다.
- 평균/최대 동시 요청 수를 본다
- 실제 DB를 치는 요청 비율을 본다
- 쿼리 응답 시간과 트랜잭션 유지 시간을 본다
- DB 서버가 감당 가능한 활성 세션 수를 확인한다

예를 들어 대부분 요청이 캐시 히트이고 DB 접근 비율이 낮다면 풀을 과하게 키울 이유가 없다. 반대로 긴 트랜잭션이나 배치 작업이 많다면 애플리케이션 풀과 별도 워크로드 격리가 필요할 수 있다.

## 면접 답변 예시
- 질문: "DB 커넥션 풀 크기는 어떻게 정하나요?"  
  답변: "고정 공식 하나로 정하기보다 애플리케이션 동시성, 실제 DB 접근 비율, 쿼리 지연 시간, DB CPU와 대기 시간을 같이 보고 결정합니다. 너무 작으면 애플리케이션이 커넥션을 기다리고, 너무 크면 DB에 과도한 동시 부하를 줘서 전체 응답 시간이 나빠질 수 있습니다."
- 질문: "실무에서 어떤 지표를 보나요?"  
  답변: "애플리케이션 쪽에서는 풀 대기 시간, timeout, active/idle 커넥션 수를 보고, DB 쪽에서는 CPU 사용률, 활성 세션 수, 락 대기, 슬로우 쿼리 비율을 같이 봅니다. 지표를 보면서 단계적으로 늘리거나 줄이는 방식이 안전합니다."

## 장점
- 적절한 풀 크기는 연결 생성 비용을 줄이고 응답 시간을 안정화한다
- 애플리케이션과 DB 사이의 동시성 수준을 제어할 수 있다
- 병목이 애플리케이션인지 DB인지 파악하는 기준점이 된다

## 단점
- 잘못 설정하면 timeout, 대기열 증가, DB 과부하가 동시에 생길 수 있다
- 워크로드가 바뀌면 기존 적정값이 빠르게 무의미해질 수 있다
- 단일 숫자만 보고 튜닝하면 캐시, 쿼리 품질, 트랜잭션 길이 같은 더 큰 문제를 놓칠 수 있다

## 주의사항 / 실무 팁
- 커넥션 풀 튜닝 전에 슬로우 쿼리와 긴 트랜잭션부터 먼저 점검하는 편이 효율적이다.
- API 트래픽과 배치 작업이 섞여 있으면 풀을 분리하거나 우선순위를 나누는 전략을 고려한다.
- 면접에서는 "많은 커넥션 = 높은 성능"이 아니라 "DB가 감당 가능한 수준으로 동시성을 조절하는 값"이라고 설명하면 좋다.
