# JWT 검증 정리

## 핵심 요약
- JWT는 토큰 자체에 클레임을 담는 방식이지만, 서명만으로 만능이 아닙니다.
- 면접에서는 `서명 검증`, `만료시간`, `aud/iss` 확인, `블랙리스트/키 회전`까지 같이 말하면 안정적입니다.
- 실무에서는 짧은 만료시간 + 갱신 토큰 조합이 보편적입니다.

## 개념 정리
JWT는 헤더, 페이로드, 서명 세 파트로 구성됩니다.

- 헤더: 알고리즘(`alg`)과 타입
- 페이로드: 사용자 정보, 발급 시각, 만료 시각(`exp`) 등
- 서명: 헤더+페이로드 위조 여부를 확인

검증은 다음을 기본으로 합니다.
- 서명 알고리즘이 허용된 값인지
- 토큰 만료(`exp`)와 발급/유효 기간
- 발급자(`iss`), 대상자(`aud`), 주제(`sub`) 일관성

주의: JWT는 암호화가 기본이 아닙니다. 서명은 위·변조 검증이지, 내용 비밀화가 아닙니다.

## 실무 예시
- API 게이트웨이에서 `Authorization: Bearer` 헤더를 받으면
  - 서명을 먼저 검증
  - 만료를 확인
  - 사용자/권한 클레임을 읽어 인가 처리
- 공개키 회전(Rotation) 시 이전 키와 새 키를 모두 허용 기간 동안 같이 운영

## 면접 답변 예시
"JWT는 클레임 기반 토큰이라 상태 비저장 방식에 잘 맞습니다. 다만 서명만 맞으면 무조건 신뢰하는 것이 아니라 `alg` 허용 목록, `exp`, `nbf`, `iss`, `aud`를 함께 검증해야 합니다. 저는 실무에서 짧은 Access Token + 회전 가능한 Refresh 전략을 사용하고, 키는 KMS로 관리해 일정 주기로 교체합니다. 또한 블랙리스트가 필요한 경우 서버 측 상태 저장으로 즉시 무효화 흐름을 둡니다."

## 장점
- 확장성이 높고 MSA나 분산 환경에서 활용이 편합니다.
- 서버 상태를 줄이고 캐시/확장성이 좋습니다.
- 토큰에 메타정보를 담아 마이크로서비스에서 빠르게 전달 가능합니다.

## 단점
- 토큰 크기가 커질수록 네트워크 비용과 파싱 비용이 증가합니다.
- 즉시 폐기(revocation)가 어렵습니다.
- 비밀키/공개키 관리 실수가 치명적입니다.

## 주의사항 / 실무 팁
- `HS256`만 쓰면 키 유출 시 전체 토큰 보안이 무너질 수 있어 키 분리와 순환이 중요합니다.
- 민감정보는 JWT에 넣지 마세요.
- 토큰 만료 정책을 지나치게 길게 잡으면 유효기간이 긴 탈취 토큰 리스크가 커집니다.
