# Idempotency Key 정리

## 핵심 요약
- `Idempotency Key`는 같은 요청이 여러 번 들어와도 서버가 한 번 처리한 결과를 기준으로 중복 수행을 막기 위한 식별자다.
- 결제, 주문 생성, 쿠폰 발급처럼 "같은 요청이 두 번 처리되면 문제가 되는" 쓰기 작업에서 특히 중요하다.
- 면접에서는 "재시도는 허용하되, 부작용은 한 번만 발생하게 만드는 장치"라고 설명하면 핵심이 잘 전달된다.
- 멱등성과 관련이 깊지만, HTTP 메서드 의미만 믿는 것이 아니라 애플리케이션 레벨에서 중복 요청을 제어한다는 점이 포인트다.

## 개념 설명
실무에서는 네트워크 타임아웃, 모바일 재전송, 사용자의 중복 클릭, 프록시 재시도 때문에 같은 요청이 두 번 이상 서버에 도착하는 일이 생각보다 자주 생긴다. 문제는 조회 요청이 아니라 생성 요청이다. 예를 들어 결제 승인 API가 두 번 처리되면 이중 결제나 중복 주문으로 이어질 수 있다.

이때 클라이언트가 요청마다 고유한 `Idempotency Key`를 함께 보내고, 서버는 이 키를 기준으로 "이미 처리한 요청인지" 판단한다. 처음 들어온 키라면 정상 처리하고 결과를 저장한다. 같은 키로 동일한 요청이 다시 오면 서버는 새로 처리하지 않고, 기존 결과를 그대로 돌려주거나 중복 요청임을 알려 준다.

중요한 점은 단순히 키만 보는 것이 아니라, 보통 다음을 함께 고려해야 한다는 것이다.

- 같은 키로 들어온 요청 본문이 정말 같은가
- 결과를 얼마 동안 저장할 것인가
- 처리 도중 장애가 나면 어떤 상태를 남길 것인가

즉 `Idempotency Key`는 "중복 방지 토큰" 정도가 아니라, 재시도 친화적인 쓰기 API를 설계하기 위한 운영 장치라고 보는 편이 정확하다.

## 예시
결제 API를 호출하는 모바일 앱을 생각해 보자.

- 사용자가 결제 버튼을 눌렀다
- 네트워크가 불안정해서 응답이 늦는다
- 사용자는 결제가 안 된 줄 알고 다시 눌렀다

이때 두 요청이 모두 서버에 도착할 수 있다. 클라이언트가 같은 `Idempotency Key`를 함께 보냈다면 서버는 첫 요청만 실제 결제 처리하고, 두 번째 요청은 기존 결제 결과를 재사용할 수 있다.

이 구조가 없으면 "타임아웃 뒤 재시도"가 곧 "중복 결제 위험"으로 바뀐다. 반대로 이 구조가 있으면 클라이언트는 비교적 안전하게 재시도할 수 있다.

## 면접 답변 예시
- 질문: "`Idempotency Key`는 왜 필요한가요?"  
  답변: "생성이나 결제 같은 쓰기 요청은 네트워크 재시도나 중복 클릭 때문에 같은 요청이 여러 번 들어올 수 있습니다. 이때 요청마다 고유 키를 붙여서 서버가 이미 처리한 요청인지 판단하면, 재시도는 허용하면서도 실제 부작용은 한 번만 발생하게 만들 수 있습니다."
- 질문: "멱등성과는 어떤 관계가 있나요?"  
  답변: "멱등성은 같은 요청을 여러 번 수행해도 결과가 같아야 한다는 성질이고, `Idempotency Key`는 그 성질을 POST 같은 쓰기 API에서 애플리케이션 레벨로 구현하는 대표적인 방법입니다."

## 장점
- 결제, 주문, 예약 같은 민감한 쓰기 API의 안정성을 크게 높일 수 있다
- 클라이언트가 타임아웃 이후 재시도하기 쉬워진다
- 멱등성, 재시도 전략, 장애 복구를 연결해서 설명하기 좋은 주제다

## 단점
- 결과 저장소와 만료 정책을 별도로 설계해야 한다
- 같은 키에 다른 payload가 들어오는 경우 처리 정책이 복잡해질 수 있다
- 서버 간 공유 저장소가 없으면 분산 환경에서 일관되게 동작시키기 어렵다

## 주의사항 / 실무 팁
- 키만 저장하지 말고 요청 해시나 주요 파라미터를 함께 저장해, 같은 키에 다른 요청이 들어오는 상황을 구분하는 것이 좋다.
- 결제처럼 중요한 도메인에서는 처리 완료 결과뿐 아니라 처리 중 상태도 관리해야 중간 장애 시 중복 수행을 줄일 수 있다.
- TTL을 너무 짧게 잡으면 늦은 재시도를 막지 못하고, 너무 길게 잡으면 저장 비용이 커질 수 있다.
- 면접에서는 `멱등성`, `재시도 전략`, `Exactly Once Delivery는 어렵고 애플리케이션 레벨 중복 제어가 현실적이다` 같은 문장과 연결하면 답변이 단단해진다.
