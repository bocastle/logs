# TCC 패턴 정리

## 핵심 요약
- `TCC`는 `Try`(시도), `Confirm`(확정), `Cancel`(보상)로 나눠 분산 트랜잭션의 일관성을 맞추는 패턴이다.
- 2PC 대비 구현이 가볍고 비즈니스 요구에 맞춘 보상 로직 설계가 가능하다.
- 성공/실패 분기를 명확히 모델링해야 하며, 보상 동작의 멱등성은 실서비스에서 필수다.
- 메시지 유실, 타임아웃, 중복 재시도 같은 장애를 고려한 오케스트레이션이 핵심이다.

## 개념 설명
여러 서비스가 하나의 비즈니스 흐름을 수행할 때 부분 실패가 나면 전체를 되돌려야 한다.  
`TCC`는 각 서비스의 작업을 미리 `Try` 단계에서 **예약 가능 상태**로 바꾸고, 최종 결과에 따라 `Confirm` 또는 `Cancel`을 적용해 정합성을 맞춘다.

핵심 용어:
- `Try`: 검증/예약. 리소스를 잠정 점유하되 아직 최종 반영하지 않는다.
- `Confirm`: 전체 성공 확정. 모든 단계의 예약을 실제 반영한다.
- `Cancel`: 일부 실패 시 예약을 되돌리고 부수 효과를 정리한다.
- 오케스트레이터: 흐름 제어자. 각 참여자에게 단계별 호출 타이밍을 관리한다.

## 예시
`주문 -> 재고 감소 예약 -> 결제 승인 예약 -> 결제 완료/취소` 흐름을 가정한다.  
각 서비스는 `Try` 단계에서 실제 재고/잔액을 잠정 차감하고, 최종 승인 시 `Confirm`, 실패 시 `Cancel`로 원상복구한다.

실전에서 많이 쓰는 설계 포인트:
- 예약 ID를 글로벌로 공유
- 상태 테이블(`TRY`, `CONFIRMED`, `CANCELED`)에 대한 상태 머신 관리
- 타임아웃 기반 재처리(worker) 운영

## 면접 답변 예시
- 질문: "TCC를 언제 쓰나요?"  
  답변: "업무 트랜잭션 간 상호 의존이 높고 부분 실패 복구가 자주 필요한 결제/예약/포인트 시나리오에서 주로 씁니다. 2PC의 락 의존이 부담될 때 대안으로 사용합니다."
- 질문: "TCC의 가장 어려운 점은?"  
  답변: "보상(Cancel) 구현의 멱등성과 중복 호출 방지입니다. 네트워크 재시도에서 동일한 요청이 중복되면 잘못된 이중 취소가 될 수 있어서 idempotency key와 상태 검사가 필수입니다."

## 장점
- 분산 환경에서 흐름 제어를 서비스 코드로 명시할 수 있다
- 비즈니스 로직에 맞춘 보상 전략 설계가 가능하다
- 글로벌 잠금보다 유연하게 부분 실패를 처리한다

## 단점
- 서비스 간 인터페이스/상태 규격이 복잡해진다
- Confirm/Cancel 구현이 없으면 오히려 장애 포인트가 늘어난다
- 오케스트레이터/워크플로우 운영 비용이 증가한다

## 주의사항 / 실무 팁
- `Cancel`은 항상 멱등으로 구현한다. 동일 메시지가 여러 번 와도 상태가 깨지면 안 된다.
- 타임아웃 스케줄러(예: 보상 워커)는 장애 복구 계획과 함께 운영해야 한다.
- 로그/감사 기록을 남겨야 사후 분석과 고객 대응이 수월하다.
