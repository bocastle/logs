# 커버링 인덱스 정리

## 핵심 요약
- `커버링 인덱스`는 조회에 필요한 컬럼이 인덱스 안에 모두 들어 있어서 테이블 원본까지 가지 않아도 되는 경우를 말한다.
- 핵심은 인덱스를 탔느냐보다, 인덱스만으로 조회를 끝낼 수 있느냐이다.
- 면접에서는 랜덤 I/O 감소, 응답 속도 개선, 인덱스 크기 증가의 트레이드오프를 같이 설명하면 좋다.
- 모든 쿼리에 커버링 인덱스를 만들려 하면 쓰기 성능과 저장 공간이 악화될 수 있다.

## 개념 설명
일반적으로 DB는 인덱스를 이용해 조건에 맞는 행 위치를 찾은 뒤, 실제 데이터가 들어 있는 테이블 페이지를 다시 읽는다. 이 추가 접근이 많아지면 랜덤 I/O 비용이 커지고 조회가 느려질 수 있다.

그런데 `SELECT`에 필요한 컬럼과 `WHERE`, `ORDER BY`에 필요한 컬럼이 모두 인덱스 안에 있으면, DB는 테이블 본문까지 가지 않고 인덱스만 읽고 결과를 만들 수 있다. 이런 상황을 보통 커버링 인덱스라고 부른다.

즉, 인덱스는 단순히 "찾기 위한 구조"일 뿐 아니라, 경우에 따라서는 "결과를 바로 반환할 수 있는 작은 데이터 집합" 역할도 한다. 그래서 면접에서는 인덱스 탐색 비용뿐 아니라 테이블 접근을 줄이는 효과까지 같이 설명하는 것이 중요하다.

## 예시
주문 목록 화면에서 아래 쿼리를 자주 실행한다고 가정해보자.

- 조회 조건: `user_id`
- 정렬 기준: `created_at DESC`
- 반환 컬럼: `id`, `status`, `created_at`

이때 `(user_id, created_at, id, status)` 형태의 인덱스가 있다면, DB는 사용자별 최근 주문을 인덱스만 스캔해서 바로 반환할 가능성이 높다.

반대로 인덱스에 `status`가 없으면 조건 탐색은 인덱스로 하더라도 최종 결과를 만들기 위해 테이블을 다시 읽어야 한다. 데이터가 많을수록 이 차이가 커질 수 있다.

## 면접 답변 예시
- 질문: "커버링 인덱스는 무엇인가요?"  
  답변: "조회에 필요한 컬럼이 인덱스에 모두 포함돼 있어서 테이블 본문을 다시 읽지 않고 인덱스만으로 결과를 반환할 수 있는 상황을 말합니다. 그래서 랜덤 I/O를 줄이고 조회 성능을 개선하는 데 유리합니다."
- 질문: "무조건 좋은가요?"  
  답변: "아닙니다. 필요한 컬럼을 많이 넣을수록 인덱스가 커지고 쓰기 비용도 증가합니다. 그래서 자주 실행되는 핵심 조회 패턴에만 제한적으로 적용하는 것이 실무적입니다."

## 장점
- 테이블 추가 접근을 줄여 조회 성능을 높일 수 있다
- 읽기 중심 화면이나 목록 조회에서 큰 효과를 볼 수 있다
- 실행 계획을 설명할 때 인덱스 설계 의도를 명확히 보여주기 좋다

## 단점
- 인덱스 크기가 커져 메모리와 디스크 사용량이 늘 수 있다
- `INSERT`, `UPDATE`, `DELETE` 때 관리해야 할 인덱스 비용이 증가한다
- 쿼리 패턴이 바뀌면 설계한 인덱스 효율이 빠르게 떨어질 수 있다

## 주의사항 / 실무 팁
- "인덱스를 만들었다"와 "커버링 인덱스가 됐다"는 다른 말이라는 점을 구분해야 한다.
- 반환 컬럼이 너무 많으면 커버링 인덱스를 억지로 만들기보다 쿼리 자체를 줄이거나 조회 화면 요구사항을 조정하는 편이 낫다.
- 면접에서는 실행 계획에서 index only scan, 혹은 테이블 접근 감소 관점으로 설명하면 이해가 쉽다.
- 결국 중요한 것은 특정 쿼리 패턴 기준으로 읽기 이득과 쓰기 비용의 균형을 맞추는 것이다.
