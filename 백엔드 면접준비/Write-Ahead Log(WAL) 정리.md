# Write-Ahead Log(WAL) 정리

## 핵심 요약

- `Write-Ahead Log(WAL)`은 실제 데이터 파일을 바꾸기 전에 변경 내용을 로그에 먼저 기록하는 방식이다.
- 데이터베이스가 장애 후에도 커밋된 변경을 복구할 수 있게 해 주는 핵심 장치다.
- 트랜잭션의 원자성과 내구성을 설명할 때 자주 연결된다.
- 면접에서는 "데이터를 바로 덮어쓰기보다, 복구 가능한 로그를 먼저 남겨 안정성을 확보하는 방식"이라고 설명하면 좋다.

## WAL이란?

`Write-Ahead Log`를 직역하면 "먼저 쓰는 로그"다.  
데이터베이스는 데이터를 변경할 때 실제 테이블 파일이나 인덱스 파일을 바로 안전하게 바꾸기 어렵다.

그래서 먼저 "어떤 변경을 하려고 했는지"를 로그에 기록한다.  
그 다음 실제 데이터 페이지를 메모리나 디스크에 반영한다.

핵심 원칙은 간단하다.

> 데이터 변경보다 로그 기록이 먼저다.

이 원칙 덕분에 서버가 중간에 죽어도 로그를 보고 어디까지 반영해야 하는지 복구할 수 있다.

## 왜 필요한가?

데이터베이스는 성능 때문에 모든 변경을 즉시 데이터 파일에 완벽히 써 내려가지 않는다.  
많은 경우 변경된 페이지를 메모리에 두었다가 나중에 디스크에 반영한다.

그런데 이 상태에서 장애가 나면 문제가 생긴다.

- 사용자는 커밋 성공 응답을 받았다.
- 하지만 실제 데이터 파일에는 아직 반영되지 않았다.
- 서버가 재시작되면 그 변경이 사라질 수 있다.

WAL은 이런 상황을 막기 위해 커밋 전에 로그를 안정적으로 기록한다.  
재시작 후에는 로그를 재생해서 커밋된 변경을 다시 반영할 수 있다.

## 트랜잭션과의 관계

WAL은 ACID 중 특히 원자성과 내구성과 관련이 깊다.

- 원자성: 트랜잭션이 중간에 실패하면 일부만 반영되지 않도록 되돌릴 수 있다.
- 내구성: 커밋된 트랜잭션은 장애가 나도 사라지지 않도록 복구할 수 있다.

즉 WAL은 "커밋했다고 말한 데이터는 잃지 않는다"는 약속을 지키기 위한 기반이다.

## 동작 흐름

단순화하면 흐름은 이렇다.

1. 트랜잭션이 데이터를 변경한다.
2. 변경 내용이 WAL에 먼저 기록된다.
3. 커밋 시점에 필요한 로그가 디스크에 안전하게 내려간다.
4. 실제 데이터 파일은 즉시 또는 나중에 반영된다.
5. 장애가 나면 WAL을 보고 redo 또는 undo를 수행한다.

실제 DBMS마다 구현은 다르지만, "로그를 먼저 남기고 데이터 파일은 나중에 따라올 수 있다"는 생각은 공통적이다.

## Redo와 Undo

면접에서 조금 더 깊게 물어보면 `redo`와 `undo`를 같이 설명하면 좋다.

- `redo`: 커밋됐지만 데이터 파일에 완전히 반영되지 않은 작업을 다시 적용한다.
- `undo`: 커밋되지 않았는데 일부 반영된 작업을 되돌린다.

WAL은 장애 복구 시점에 어떤 작업을 다시 하고, 어떤 작업을 되돌릴지 판단하는 근거가 된다.

## 성능과 체크포인트

WAL을 쓰면 매번 큰 데이터 파일을 랜덤하게 쓰는 대신, 로그 파일에 순차적으로 쓰는 구조를 만들 수 있다.  
순차 쓰기는 디스크 입장에서 더 효율적인 경우가 많다.

하지만 로그가 계속 쌓이기만 하면 복구 시간이 길어진다.  
그래서 데이터베이스는 일정 시점에 체크포인트를 만들어 "여기까지는 데이터 파일에 반영됐다"는 기준을 남긴다.

체크포인트가 있으면 장애 복구 때 처음부터 모든 로그를 볼 필요가 줄어든다.

## 면접에서 어떻게 말하면 좋을까?

WAL은 데이터베이스가 변경 내용을 실제 데이터 파일에 쓰기 전에 로그에 먼저 기록해서 장애 복구를 가능하게 하는 방식이라고 설명하면 된다.  
그리고 커밋된 데이터의 내구성, redo/undo, 체크포인트까지 이어서 말하면 깊이가 있다.

## 면접 답변 예시

> WAL은 데이터베이스가 실제 데이터 파일을 변경하기 전에 변경 내용을 로그에 먼저 기록하는 방식입니다. 커밋 전에 로그가 안전하게 저장되어 있으면, 서버가 중간에 죽더라도 재시작 후 로그를 보고 커밋된 변경은 다시 적용하고, 커밋되지 않은 변경은 되돌릴 수 있습니다. 그래서 WAL은 트랜잭션의 원자성과 내구성을 지키는 데 중요한 역할을 합니다. 또 데이터 파일을 매번 즉시 갱신하지 않아도 되기 때문에 성능에도 도움이 되지만, 로그가 너무 길어지지 않도록 체크포인트 관리가 필요합니다.

## 장점

- 장애 후에도 커밋된 데이터를 복구할 수 있다.
- 데이터 파일을 즉시 모두 반영하지 않아도 되어 쓰기 성능에 도움이 된다.
- redo, undo 기반으로 트랜잭션 복구를 체계적으로 처리할 수 있다.

## 단점

- 로그 저장 공간과 관리 비용이 필요하다.
- 로그 flush 정책에 따라 성능과 안정성 사이의 trade-off가 생긴다.
- 체크포인트가 부적절하면 복구 시간이 길어질 수 있다.

## 주의사항 / 실무 팁

- WAL은 백업이 아니라 장애 복구를 위한 로그라는 점을 구분해야 한다.
- 커밋 응답 시점과 로그가 디스크에 안전하게 기록되는 시점을 함께 이해하면 좋다.
- 면접에서는 ACID의 내구성과 연결해서 말하면 자연스럽고, redo/undo를 덧붙이면 더 탄탄하다.
