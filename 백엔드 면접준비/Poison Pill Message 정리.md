# Poison Pill Message 정리

## 핵심 요약
- `Poison Pill Message`는 소비자가 아무리 재처리해도 계속 실패하게 만드는 문제성 메시지를 뜻한다.
- 메시지 큐 시스템에서는 이런 메시지 하나가 재시도를 무한히 유발하면서 큐 적체를 만들 수 있다.
- 면접에서는 "정상 메시지 흐름을 막는 독성 메시지"라고 설명하고, `DLQ`, 재시도 정책, 스키마 검증과 연결하면 좋다.
- 중요한 포인트는 실패 자체보다 "같은 실패가 반복되며 시스템 전체 처리량을 무너뜨린다"는 점이다.

## 개념 설명
비동기 메시지 처리 시스템은 보통 실패한 메시지를 재시도할 수 있게 설계한다. 이건 일시적 장애를 견디는 데 매우 유용하다. 예를 들어 외부 API가 잠깐 느리거나 DB 연결이 순간적으로 끊겼다면 몇 번의 재시도로 정상 처리될 수 있다.

하지만 메시지 본문 자체가 잘못됐거나, 필수 필드가 없거나, 소비자 코드가 특정 데이터 패턴을 처리하지 못한다면 재시도를 아무리 해도 결과는 달라지지 않는다. 이런 메시지가 바로 `Poison Pill Message`다.

문제는 시스템이 이 메시지를 "일시적 실패"로 착각하면 같은 메시지를 계속 다시 집어넣는다는 것이다. 그러면 소비자 스레드가 문제 메시지에 계속 묶이고, 뒤에 있는 정상 메시지 처리까지 지연될 수 있다.

그래서 핵심은 무조건 재시도하는 것이 아니라, 재시도해도 안 될 실패를 빨리 구분하는 것이다.

## 예시
주문 이벤트를 처리하는 소비자가 있다고 해 보자.

- 메시지에 `orderId`는 있지만 `userId`가 비어 있다
- 소비자 로직은 `userId`가 없으면 예외를 던진다
- 큐 시스템은 실패 메시지를 3초 후 재시도한다
- 데이터가 잘못된 것이므로 10번을 다시 받아도 계속 실패한다

이 상태가 길어지면 같은 메시지가 로그를 오염시키고, 재시도 큐를 채우고, 파티션 처리 지연까지 만든다. 결국 한 건의 잘못된 메시지가 시스템 전체의 지연 원인이 될 수 있다.

## 면접 답변 예시
- 질문: "`Poison Pill Message`는 무엇인가요?"  
  답변: "메시지 큐에서 소비자가 반복해서 처리에 실패하는 문제성 메시지입니다. 재시도로 해결되지 않기 때문에 계속 남아 있으면 큐 적체와 처리 지연을 유발합니다."
- 질문: "왜 위험한가요?"  
  답변: "일시적 장애와 달리 재처리해도 성공하지 않는데, 시스템이 그걸 구분하지 못하면 같은 메시지에 자원을 계속 소모하기 때문입니다."
- 질문: "어떻게 대응하나요?"  
  답변: "최대 재시도 횟수를 두고, 넘으면 `DLQ`로 보내며, 입력 스키마 검증과 에러 분류를 통해 재시도할 실패와 버려야 할 실패를 나눕니다."

## 장점
- 이 개념을 이해하면 재시도 정책을 더 현실적으로 설계할 수 있다
- `DLQ`, 스키마 검증, idempotency, 장애 관측성을 함께 설명하기 좋다
- 메시지 기반 시스템에서 운영 관점의 성숙도를 보여 줄 수 있다

## 단점
- 용어는 단순하지만 실제로는 "재시도 가능한 실패"와 "재시도 불가능한 실패"를 구분하는 기준이 어렵다
- 잘못 처리하면 정상적으로 복구될 수 있는 메시지까지 너무 빨리 포기할 수 있다
- 큐 솔루션마다 재시도와 격리 방식이 달라 구현 경험이 없으면 답변이 추상적으로 들릴 수 있다

## 주의사항 / 실무 팁
- 모든 실패 메시지를 `Poison Pill`로 보면 안 된다. 일시적 장애와 데이터 자체 오류를 구분해야 한다.
- 재시도 횟수, backoff, `DLQ` 이동 조건을 명확히 정하지 않으면 운영 중 장애가 길어진다.
- 면접에서는 "`독성 메시지 하나가 소비자 자원을 계속 점유해 정상 처리 흐름을 막는다`"고 설명하면 핵심이 잘 전달된다.
- 실무에서는 생산자 단계 스키마 검증, 컨슈머 단계 예외 분류, `DLQ` 모니터링 세 가지를 같이 가져가야 효과가 크다.
