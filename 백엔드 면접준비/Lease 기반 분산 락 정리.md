# Lease 기반 분산 락 정리

## 핵심 요약
- `Lease 기반 분산 락`은 락을 영구 점유하는 대신 일정 시간 동안만 소유권을 빌리는 방식의 분산 락이다.
- 장애가 나도 락이 무한정 풀리지 않는 문제를 줄이기 위해 `만료 시간`이 들어간다는 점이 핵심이다.
- 면접에서는 "분산 환경에서 작업 중복을 막되, 장애 시 자동 회복 가능성을 높이기 위한 시간 제한 락"이라고 설명하면 좋다.
- 락 자체보다도 만료 시간, 갱신, 소유자 검증을 어떻게 설계하느냐가 실무 포인트다.

## 개념 설명
여러 서버 인스턴스가 동시에 같은 작업을 실행하면 문제가 생기는 경우가 있다. 예를 들어 배치 작업 하나만 실행돼야 하거나, 같은 쿠폰 발급 로직이 중복 처리되면 안 되는 상황이다. 이런 때 분산 락을 사용한다.

그런데 분산 환경에서는 단순 락만으로는 부족하다. 락을 잡은 서버가 죽어 버리면 락이 영원히 남아 다른 서버가 아무 작업도 못 하는 상황이 생길 수 있기 때문이다. `Lease 기반 분산 락`은 이 문제를 줄이기 위해 "락은 일정 시간 동안만 유효하다"는 조건을 둔다.

즉, 소유자는 락을 획득할 때 만료 시간을 함께 갖고, 작업이 길어지면 갱신할 수 있다. 반대로 서버가 죽으면 갱신이 멈추고, 시간이 지나면 다른 서버가 락을 다시 획득할 수 있다.

핵심은 다음 세 가지다.

- 락은 영구 소유가 아니라 임대다
- 만료 전 갱신이 필요할 수 있다
- 해제할 때는 내가 잡은 락이 맞는지 소유자 검증이 필요하다

## 예시
예를 들어 여러 서버 중 한 대만 정산 배치를 실행해야 한다고 해 보자.

1. 서버 A가 Redis나 ZooKeeper 같은 저장소에 락을 획득한다.
2. 이 락은 30초 동안만 유효하다.
3. 서버 A가 배치를 수행하는 동안 주기적으로 lease를 갱신한다.
4. 만약 서버 A가 중간에 죽으면 갱신이 멈추고, 30초 후 서버 B가 락을 가져갈 수 있다.

이 구조 덕분에 장애가 나도 시스템이 영구 정지 상태에 빠질 가능성을 줄일 수 있다. 다만 작업 시간이 lease보다 길어질 수 있다면 안전한 갱신 로직이 필수다.

## 면접 답변 예시
- 질문: "`Lease 기반 분산 락`은 왜 쓰나요?"  
  답변: "분산 락은 중복 실행을 막기 위해 쓰지만, 락을 잡은 인스턴스가 장애 나면 영원히 안 풀릴 수 있습니다. Lease 방식은 락에 만료 시간을 둬서 장애 시 자동 회복 가능성을 높이는 방식입니다."
- 질문: "주의할 점은 무엇인가요?"  
  답변: "작업 시간이 길면 lease를 안전하게 갱신해야 하고, 락을 해제할 때는 다른 인스턴스가 이미 다시 잡은 락을 잘못 지우지 않도록 소유자 토큰 검증이 필요합니다."

## 장점
- 장애 상황에서 락이 영구 점유되는 문제를 줄일 수 있다
- 배치, 스케줄러, 중복 처리 방지 시나리오에 설명하기 좋다
- 단순 분산 락보다 운영 복구 관점이 분명하다

## 단점
- 만료 시간 설정이 잘못되면 중복 실행이나 조기 만료 문제가 생길 수 있다
- 갱신 로직, 소유자 토큰 검증 등 구현 복잡도가 올라간다
- 저장소 시간 기준과 네트워크 지연까지 고려해야 한다

## 주의사항 / 실무 팁
- lease 시간이 너무 짧으면 정상 작업 중에도 락이 만료돼 다른 인스턴스가 동시에 들어올 수 있다.
- 너무 길면 장애 복구가 느려진다. 그래서 작업 특성에 맞춘 TTL 산정이 중요하다.
- 락 해제는 단순 delete가 아니라 "내가 획득한 락인지"를 토큰으로 검증하는 방식이 안전하다.
- 면접에서는 "`분산 락`과의 차이는 만료 시간 기반 자동 회복성"이라고 비교해서 말하면 이해가 잘 된다.
