# Adaptive Concurrency Limit 정리

## 핵심 요약
- `Adaptive Concurrency Limit`은 서버가 현재 처리 가능한 동시 요청 수를 고정값이 아니라 상황에 따라 동적으로 조정하는 방식이다.
- 지연 시간이 늘어나거나 에러가 증가하면 허용량을 줄이고, 여유가 있으면 조금씩 늘리면서 최적점을 찾는다.
- 면접에서는 "과부하 직전의 처리량을 유지하려고 동시성 한도를 자동 튜닝하는 보호 전략"이라고 설명하면 이해가 빠르다.
- 단순 rate limiting과는 다르게, 현재 시스템 상태를 보고 내부 처리량을 조절한다는 점이 핵심이다.

## 개념 설명
서버는 무조건 많은 요청을 동시에 받는다고 좋은 것이 아니다. 어느 지점을 넘어서면 스레드, 커넥션, CPU, I/O 대기가 쌓이면서 오히려 응답 시간이 급격히 늘고 실패율도 올라간다. 이 구간에 들어가면 더 많은 요청을 받는 것이 처리량 증가가 아니라 장애 가속으로 이어질 수 있다.

`Adaptive Concurrency Limit`은 이런 문제를 줄이기 위해 현재 응답 시간, 큐 적체, 에러율 같은 신호를 보고 "지금 이 서버가 동시에 몇 개까지 받는 것이 적절한가"를 계속 조절한다. 상태가 좋으면 한도를 조금씩 늘려 보고, 응답이 느려지기 시작하면 다시 줄인다. 즉 정적인 임계값이 아니라 피드백 기반 제어에 가깝다.

이 전략은 특히 외부 API 호출, DB 의존성이 큰 서비스, 프록시나 게이트웨이 계층에서 유용하다. 목표는 단순히 요청을 많이 받는 것이 아니라, 시스템이 무너지기 전에 안정적인 처리 구간을 유지하는 것이다.

## 예시
추천 API가 DB와 외부 개인화 서비스를 함께 호출한다고 가정해 보자.

- 평소에는 동시 요청 100개까지 무난하게 처리된다
- 하지만 외부 서비스가 느려지는 순간 같은 100개도 대기열을 급격히 키울 수 있다
- 이때 adaptive limit이 동시 허용량을 60, 40처럼 낮추면 전체 실패 폭을 줄이고 회복 여지를 남길 수 있다

즉 일부 요청은 초기에 제한하더라도, 시스템 전체가 무너지는 것보다는 더 나은 선택이 될 수 있다.

## 면접 답변 예시
- 질문: "`Adaptive Concurrency Limit`은 무엇인가요?"  
  답변: "서버가 처리 가능한 동시 요청 수를 고정하지 않고, 현재 지연 시간이나 에러율을 보면서 동적으로 조정하는 방식입니다. 과부하 직전의 안정적인 처리량을 유지하려는 보호 장치라고 볼 수 있습니다."
- 질문: "Rate Limiting과는 뭐가 다른가요?"  
  답변: "Rate Limiting은 보통 초당 요청 수 같은 외부 유입량을 제한하는 데 초점이 있고, Adaptive Concurrency Limit은 서버 내부 상태를 보면서 현재 동시에 처리할 수 있는 양을 조절하는 데 더 가깝습니다."

## 장점
- 과부하가 시작될 때 시스템을 더 빠르게 보호할 수 있다
- 고정 동시성 값보다 트래픽 변화와 의존성 지연 변화에 유연하다
- 백프레셔, 서킷 브레이커, 큐잉 전략과 함께 설명하기 좋은 주제다

## 단점
- 제어 기준을 잘못 잡으면 지나치게 보수적이거나 불안정하게 흔들릴 수 있다
- 구현과 운영 관측 지표가 단순 고정 임계값보다 복잡하다
- 사용자 입장에서는 일부 요청이 더 일찍 거절될 수 있어 정책 설명이 필요하다

## 주의사항 / 실무 팁
- 동시성 제한은 항상 timeout, retry 정책과 함께 봐야 한다. 재시도 폭증이 있으면 제한 효과가 약해질 수 있다.
- CPU 바운드인지 I/O 바운드인지에 따라 최적 한도와 관측 지표가 달라질 수 있다.
- 면접에서는 `Backpressure`, `Bulkhead Pattern`, `Rate Limiting`과 구분해서 말하면 더 깔끔하다.
- 실무에서는 고정값으로 시작하더라도, 지표가 쌓이면 adaptive 전략으로 고도화하는 흐름을 말하면 현실감이 있다.
