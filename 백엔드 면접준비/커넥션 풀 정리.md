# 커넥션 풀 정리

## 핵심 요약

- 커넥션 풀은 DB 연결을 미리 여러 개 만들어 두고 재사용하는 방식이다.
- 요청마다 매번 DB 연결을 새로 맺으면 느리고 비용이 크기 때문에 풀을 쓴다.
- 풀 크기가 너무 작으면 대기 시간이 길어지고, 너무 크면 DB 자체에 부담을 줄 수 있다.
- 면접에서는 "왜 필요한가", "풀 크기를 어떻게 볼 것인가", "장애 시 어떤 증상이 생기는가"까지 말하면 좋다.

## 커넥션 풀이란?

애플리케이션이 데이터베이스와 통신하려면 커넥션이 필요하다. 그런데 TCP 연결을 열고, 인증하고, 세션을 준비하는 과정은 생각보다 비용이 있다.

만약 사용자 요청이 들어올 때마다 매번 새 커넥션을 만들고 닫는다면 응답 시간도 느려지고, DB 서버도 불필요한 연결 생성/해제 작업을 계속 처리해야 한다.

그래서 보통은 애플리케이션 시작 시점이나 필요 시점에 커넥션을 여러 개 만들어 풀(pool)에 넣어 두고, 요청이 오면 그중 하나를 꺼내 쓰고 다시 반납한다.

쉽게 말하면 "DB 연결을 재사용하기 위한 대기열"이라고 보면 된다.

## 왜 필요한가?

커넥션 풀의 목적은 크게 세 가지다.

1. 연결 생성 비용 절감
2. 응답 속도 안정화
3. DB 자원 사용량 통제

연결을 재사용하면 요청 처리 속도가 더 일정해진다. 동시에 애플리케이션이 DB에 무한정 연결을 여는 것을 막아서, DB 서버를 보호하는 효과도 있다.

## 동작 방식

대략적인 흐름은 이렇다.

1. 애플리케이션이 시작되면서 커넥션 풀을 초기화한다.
2. 요청이 들어오면 풀에서 사용 가능한 커넥션을 하나 빌린다.
3. 쿼리를 실행한다.
4. 작업이 끝나면 커넥션을 닫는 대신 풀에 반납한다.

여기서 중요한 점은 개발자가 `close()`를 호출하더라도, 많은 라이브러리에서는 실제 물리 연결을 끊는 게 아니라 풀에 반환하는 동작으로 처리한다는 것이다.

## 풀 크기는 왜 중요한가?

풀 크기를 너무 작게 잡으면 동시에 처리할 수 있는 요청 수가 제한된다. 사용 가능한 커넥션이 모두 바쁘면 뒤 요청은 기다려야 하고, 심하면 timeout이 난다.

반대로 너무 크게 잡는다고 무조건 좋은 것도 아니다. 애플리케이션이 DB에 과도하게 많은 연결을 유지하면, DB 메모리와 CPU를 낭비하고 문맥 전환 비용도 늘어난다. 특히 애플리케이션 인스턴스가 여러 대라면 각 인스턴스의 풀 크기가 합쳐져 DB 최대 연결 수를 넘길 수도 있다.

그래서 풀 크기는 보통 다음을 같이 보고 정한다.

- 애플리케이션 인스턴스 수
- DB의 최대 동시 연결 가능 수
- 요청당 평균 쿼리 시간
- 트래픽 패턴

## 실무에서 자주 보는 문제

### 1. 커넥션 반납 누락

커넥션을 빌려 놓고 제대로 반납하지 않으면 풀이 점점 고갈된다. 그러면 어느 순간부터 모든 요청이 대기하거나 timeout이 날 수 있다.

JDBC를 직접 다루는 코드나 트랜잭션 경계가 복잡한 코드에서 특히 조심해야 한다.

### 2. 느린 쿼리 때문에 풀 점유 시간 증가

쿼리 자체가 느리면 커넥션이 오래 점유된다. 이 경우 겉으로는 "풀이 부족하다"처럼 보이지만, 사실 근본 원인은 슬로우 쿼리일 수 있다.

그래서 풀 모니터링만 볼 게 아니라 쿼리 성능도 같이 봐야 한다.

### 3. DB 장애 전파

DB가 느려지거나 다운되면 커넥션 획득 시간이 늘어나고, 애플리케이션 스레드가 대기 상태로 쌓일 수 있다. 이때 웹 서버 전체가 응답 불가처럼 보이는 상황도 생긴다.

그래서 timeout 설정, 헬스체크, 장애 격리 전략이 중요하다.

## HikariCP 같은 풀 라이브러리에서 보는 설정

자바 진영에서는 HikariCP를 많이 쓴다. 면접에서 설정 이름을 외울 필요는 없지만 어떤 의미인지 정도는 알면 좋다.

- `maximumPoolSize`: 최대 커넥션 수
- `minimumIdle`: 유휴 커넥션 최소 개수
- `connectionTimeout`: 커넥션을 얻기 위해 기다리는 최대 시간
- `idleTimeout`: 오래 놀고 있는 커넥션 정리 시간
- `maxLifetime`: 커넥션 최대 수명

예를 들어 DB가 일정 시간 뒤 오래된 연결을 끊는 환경이라면 `maxLifetime`을 적절히 설정해서 애플리케이션이 먼저 정리하도록 맞추는 식의 운영이 필요하다.

## 면접 답변 예시

커넥션 풀은 데이터베이스 연결을 미리 여러 개 만들어 두고 요청이 올 때마다 재사용하는 방식입니다. 요청마다 새 연결을 생성하면 연결 수립 비용이 크고 DB에도 부담이 되기 때문에, 실무에서는 대부분 커넥션 풀을 사용합니다.

중요한 건 풀 크기를 너무 작게 잡으면 요청이 대기하다 timeout이 날 수 있고, 너무 크게 잡으면 여러 인스턴스에서 DB 연결을 과도하게 점유해서 오히려 DB가 힘들어질 수 있다는 점입니다. 그래서 애플리케이션 수, DB 최대 연결 수, 쿼리 시간, 트래픽을 함께 보고 조정해야 합니다.

## 장점

- 연결 생성 비용을 줄여 응답 시간을 안정화할 수 있다.
- DB 연결 수를 통제해서 자원 사용량을 관리하기 좋다.
- 재사용 구조라 고트래픽 환경에서 효율적이다.
- 모니터링 포인트가 분명해서 운영에 도움이 된다.

## 단점

- 설정이 부적절하면 오히려 병목이 된다.
- 커넥션 누수나 슬로우 쿼리가 있으면 장애가 빠르게 드러난다.
- 애플리케이션 인스턴스 수와 DB 최대 연결 수를 함께 봐야 해서 운영 판단이 필요하다.
- 단순히 풀 수치만 보고 문제를 해결하려 하면 원인을 놓칠 수 있다.

## 주의사항 / 실무 팁

- 커넥션 풀 문제처럼 보여도 실제 원인은 느린 쿼리일 수 있으니 같이 봐야 한다.
- 애플리케이션 한 대의 `maximumPoolSize`만 보지 말고 전체 인스턴스 합계를 봐야 한다.
- timeout, max lifetime, DB 서버의 유휴 연결 정책을 서로 맞춰야 한다.
- 면접에서는 "커넥션 풀은 성능 최적화 도구"라고만 말하지 말고, "DB 자원을 통제하는 운영 장치"라는 관점까지 얹으면 좋다.
