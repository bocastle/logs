# Bloom Filter 정리

## 핵심 요약

- `Bloom Filter`는 어떤 값이 집합에 "없다"는 것을 매우 빠르게 판별하기 위한 확률형 자료구조다.
- 메모리를 적게 쓰는 대신, "있다" 판정에는 오탐(false positive)이 생길 수 있고 "없다" 판정은 신뢰할 수 있다.
- 캐시, 중복 체크, 대용량 존재 여부 검사 같은 곳에서 자주 언급된다.
- 면접에서는 "왜 해시를 여러 개 쓰는지", "오탐은 가능하지만 누락은 왜 없다고 보는지"를 설명할 수 있으면 좋다.

## Bloom Filter란?

`Bloom Filter`는 대량의 데이터를 집합 형태로 관리할 때, 특정 값이 포함됐을 가능성이 있는지 아주 빠르게 확인하는 구조다.  
배열이나 해시셋보다 훨씬 적은 메모리로 동작할 수 있어서 대용량 시스템에서 자주 나온다.

다만 정확도와 메모리를 교환하는 구조라서, 일반적인 집합처럼 100% 정확한 포함 여부를 주진 않는다.

## 어떻게 동작할까?

기본 아이디어는 단순하다.

1. 비트 배열을 하나 준비한다.
2. 원소를 넣을 때 여러 개의 해시 함수에 통과시킨다.
3. 각 해시 결과 위치의 비트를 1로 표시한다.
4. 조회할 때도 같은 해시 함수들을 적용한다.
5. 해당 비트들 중 하나라도 0이면 그 값은 집합에 없는 것이다.

반대로 모든 비트가 1이면 "있을 가능성은 있다"라고만 말할 수 있다.  
왜냐하면 다른 원소들이 우연히 같은 비트를 이미 1로 만들어 놨을 수 있기 때문이다.

## 왜 오탐은 있는데 누락은 없을까?

면접에서 자주 묻는 핵심이다.

원소를 넣을 때는 관련된 비트들을 전부 1로 바꾼다.  
그래서 실제로 들어간 원소를 같은 해시 함수로 다시 조회하면, 원칙적으로 그 비트들은 모두 1이어야 한다.

즉, "없는 값을 있다고 착각"할 수는 있어도, "있는 값을 없다고 판단"하는 일은 기본 Bloom Filter에서는 없다고 본다.  
이 특성 때문에 존재하지 않는 값을 빠르게 걸러내는 용도로 유용하다.

## 어디에 쓰일까?

대표적인 예시는 아래와 같다.

- 캐시 조회 전에 아예 없는 키를 빠르게 걸러내기
- 웹 크롤러에서 이미 방문한 URL일 가능성 체크하기
- 중복 요청이나 중복 데이터 유입 가능성 줄이기
- 대용량 저장소 조회 전에 1차 필터로 사용하기

예를 들어 캐시 미스가 아주 많은 시스템이라면, 없는 키를 매번 데이터베이스까지 내려가서 확인하는 비용이 크다.  
이때 Bloom Filter로 "이 키는 거의 확실히 없다"를 빨리 판단하면 뒤쪽 부하를 줄일 수 있다.

## 한계는 무엇일까?

좋은 점만 있는 건 아니다.

- 오탐이 있기 때문에 정확한 포함 여부를 최종 보장하진 못한다.
- 원소가 계속 늘어나면 오탐 확률이 올라간다.
- 기본 Bloom Filter는 삭제에 약하다.

삭제가 필요한 경우에는 `Counting Bloom Filter` 같은 변형 구조를 쓰기도 한다.  
즉, 메모리 효율은 좋지만 문제 성격에 맞게 써야 한다.

## 면접 답변 예시

> Bloom Filter는 어떤 값이 집합에 없는지를 빠르게 판별하기 위한 확률형 자료구조입니다. 비트 배열과 여러 해시 함수를 사용해서 메모리를 적게 쓰면서도 대량 데이터 존재 여부를 빠르게 확인할 수 있습니다. 특징은 false positive는 가능하지만 false negative는 없다는 점입니다. 그래서 정확한 포함 여부를 확정하는 용도보다는, 캐시나 DB 조회 전에 없는 값을 빠르게 걸러내는 1차 필터로 많이 사용합니다.

## 장점

- 매우 적은 메모리로 대량 데이터 존재 여부를 빠르게 확인할 수 있다.
- 없는 값을 빠르게 걸러내서 뒤쪽 저장소 부하를 줄이는 데 유리하다.
- 분산 시스템이나 캐시 계층 앞단에서 효율적인 1차 필터 역할을 한다.

## 단점

- 오탐이 가능해서 단독으로 최종 판단을 맡기기 어렵다.
- 데이터가 많이 쌓일수록 오탐 확률 관리가 중요해진다.
- 기본 구조는 삭제에 적합하지 않다.

## 주의사항 / 실무 팁

- Bloom Filter는 "최종 진실"이 아니라 "빠른 1차 판별기"로 두는 게 안전하다.
- 비트 배열 크기와 해시 함수 개수는 예상 데이터 수와 허용 오탐률을 기준으로 정해야 한다.
- 캐시 보호 용도로 설명할 때는 "DB를 완전히 대체하는 게 아니라 불필요한 조회를 줄인다"는 점을 분명히 하는 게 좋다.
- 면접에서는 "메모리 효율과 정확도를 맞바꾼 구조"라고 먼저 정의하면 답변 흐름이 깔끔해진다.
