# gRPC와 REST 비교 정리

## 핵심 요약

- REST는 범용성과 디버깅이 쉬운 반면, gRPC는 고성능 바이너리 통신과 강한 계약에 유리하다.
- 면접에서 핵심은 "언제 REST를 택하고 언제 gRPC를 택하는지"에 대한 기준을 제시하는 것이다.
- 트랜잭션 일관성, 에러코드 설계, 스트리밍 지원, 클라이언트 생태계를 함께 비교해야 한다.

## 개념 설명

### 1) 메시지 포맷과 프로토콜

- REST: 주로 JSON + HTTP/1.1
- gRPC: Protocol Buffers + HTTP/2

`JSON`은 가독성이 높고 툴 연동이 쉬운 반면,
`protobuf`는 바이트 크기가 작고 직렬화 비용이 낮다.

### 2) API 계약

- REST: OpenAPI로 계약 관리
- gRPC: `.proto` 기반 스키마가 엄격

엄격 계약이 필요한 팀 환경에서는 gRPC로 인터페이스 변경 추적이 수월하다.

### 3) 통신 패턴

- REST: 요청-응답 위주
- gRPC: unary, server streaming, client streaming, bidirectional streaming 지원

실시간성/양방향 전달이 필요한 작업은 gRPC streaming이 유리하다.

## 사용 예시

- REST 적합: 공개 API, 외부 파트너 연동, 브라우저 직접 호출이 잦은 서비스
- gRPC 적합: 내부 마이크로서비스 간 대량 호출, 스트리밍 기반 처리, 엄격한 스키마 통제

## 면접 답변 예시

"두 방식은 배타적이 아니라 보완적입니다. 공개 API 중심의 서비스는 REST가 운영 진입 장벽이 낮고 도큐먼트 친화적입니다. 반면 내부 고성능 호출이 잦고 장애 영향 분석이 중요한 환경에서는 gRPC의 HTTP/2, 스트리밍, 스키마 엄격성이 유리합니다. 결국 트래픽 패턴과 운영 체계를 보고 혼합해 쓰는 판단이 중요합니다."

## 장점

### REST
- 쉬운 디버깅과 표준 도구 접근성
- 웹 브라우저/타사 클라이언트 호환성 높음

### gRPC
- 더 작은 페이로드와 빠른 직렬화
- 강한 계약과 스트리밍 지원

## 단점

### REST
- 오버헤드가 상대적으로 클 수 있음
- 대용량 실시간 스트림엔 불리

### gRPC
- 디버깅 진입 장벽이 높고 브라우저 직접 호환성이 제한적
- 클라이언트 언어별 코드 생성/운영 체계가 필요

## 주의사항 / 실무 팁

- 외부 공개 API 설계에서는 REST+OpenAPI, 내부 서비스 메시에서는 gRPC를 병행하는 패턴을 자주 쓴다.
- 에러는 gRPC status code와 REST status code 매핑 정책을 문서화해야 운영 모니터링이 쉬워진다.
- 스키마 변경은 버전 정책이 없으면 장애로 이어지므로, 하위 호환 계획을 먼저 만든다.
