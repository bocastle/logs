# 이벤트 스키마 호환성을 지키는 배포 순서 정리

## 핵심 요약

- 이벤트 필드 추가는 새 형식을 읽을 수 있는 consumer를 먼저 배포한 뒤 producer가 발행하는 순서가 안전하다.
- Schema Registry의 문법 호환성만으로 기본값과 의미 변화 같은 논리 호환성까지 보장되지는 않는다.
- Consumer lag와 과거 replay를 고려해 구 이벤트를 읽는 코드와 필드 제거 시점을 정한다.

## 개념 설명

이벤트 기반 시스템은 rolling deploy와 consumer lag 때문에 여러 schema version이 동시에 존재한다. 새 consumer가 구 이벤트를 읽는 backward compatibility와 구 consumer가 새 이벤트를 읽는 forward compatibility를 구분하고, 변경 유형에 맞는 배포 순서를 정해야 한다.

Optional 필드를 추가한다면 새 필드를 이해하면서도 구 이벤트를 읽을 수 있는 consumer를 먼저 배포한다. Decode 오류와 처리 지표를 확인한 뒤 producer가 새 필드를 발행한다. 필드 제거는 반대로 producer가 더 이상 쓰지 않은 뒤 모든 consumer와 replay 경로가 의존하지 않는지 확인하고 진행한다.

## 예시

```text
1 consumer: v1/v2 + unknown optional field 허용
2 schema registry compatibility 검사
3 producer: schema_version=v2 발행
4 구 consumer 소멸 뒤 필드 제거
```

Required 필드 추가, 타입 변경과 기존 필드 의미 재사용은 단순 추가보다 위험하다. Registry 검사를 통과해도 `status` 값의 의미가 바뀌면 오래된 consumer가 잘못 처리할 수 있다. 큰 의미 변화는 새 이벤트 타입이나 명시적 version과 단계적 전환을 검토한다.

## 면접 답변 예시

> 이벤트 스키마를 변경할 때는 여러 producer와 consumer version이 공존한다는 전제로 순서를 정합니다. 필드 추가는 구 이벤트도 읽을 수 있는 consumer를 먼저 배포하고, 확인 후 producer가 새 필드를 발행합니다. 필드 제거는 producer의 사용을 먼저 멈춘 뒤 lag와 replay 경로까지 구 consumer 의존이 없어졌을 때 진행합니다. Schema Registry 검사와 함께 실제 과거 fixture를 처리하는 계약 테스트로 의미 호환성도 확인하겠습니다.

## 장점

- Schema Registry에서 알려진 문법상 비호환 변경을 배포 전에 차단할 수 있다.
- Consumer-first 순서로 rolling deploy 중 decode 실패 범위를 줄일 수 있다.

## 단점

- Optional 필드라도 기본값과 의미가 consumer마다 다르면 논리 호환성이 깨진다.
- 여러 consumer 팀과 replay 보존 기간이 길수록 필드 제거까지 시간이 오래 걸린다.

## 주의사항 / 실무 팁

- 구·신 producer와 consumer 조합을 과거 이벤트 fixture로 각각 테스트한다.
- Schema version별 발행·소비량, decode 오류와 lag가 제거 기준을 만족하는지 확인한다.
