# 핫 파티션 정리

## 핵심 요약
- `핫 파티션`은 특정 파티션이나 샤드에 요청이 과도하게 몰려 병목이 생기는 현상이다.
- 전체 시스템은 분산돼 있어도, 키 분포가 고르지 않으면 일부 노드만 과부하가 걸릴 수 있다.
- 면접에서는 "분산은 했지만 트래픽이 균등하게 퍼지지 않아 특정 파티션이 뜨거워지는 문제"라고 설명하면 핵심이 잡힌다.
- 해결은 단순 증설보다 키 설계, 파티셔닝 전략, 캐시, 쓰기 분산 방식까지 같이 봐야 한다.

## 개념 설명
샤딩이나 파티셔닝을 도입하면 데이터를 여러 노드에 나눠 담을 수 있다. 하지만 나눈다고 해서 자동으로 부하가 고르게 분산되지는 않는다. 특정 사용자, 특정 시간대, 특정 상품처럼 요청이 한 키에 몰리면 그 키가 속한 파티션만 집중적으로 바빠질 수 있다.

이 상태가 `핫 파티션`이다. 예를 들어 전체 평균 CPU는 낮은데 일부 노드만 응답 시간이 급격히 늘어나거나, 한 샤드만 쓰기 지연과 락 대기가 심해지는 식으로 나타난다. 시스템 전체 용량은 남아 있어도 병목은 일부 지점에서 먼저 터진다.

핵심은 분산 설계의 문제를 인프라 문제로만 보면 안 된다는 점이다. 서버를 더 추가해도 키 분포가 그대로면 같은 파티션만 계속 뜨거워질 수 있다.

## 예시
실시간 좋아요 집계를 사용자 ID 기준으로 샤딩했다고 가정해보자.

- 일반 사용자 요청은 고르게 분산된다
- 그런데 대형 이벤트 중 특정 인플루언서 게시물에 좋아요가 집중된다
- 해당 게시물 관련 쓰기 요청이 특정 파티션에 몰린다

결과적으로 전체 클러스터는 여유가 있어도, 그 파티션이 있는 노드만 CPU와 I/O가 치솟고 지연이 발생한다. 이때 사용자는 "왜 일부 요청만 느리지?"라고 느낄 수 있는데, 내부적으로는 `핫 파티션`이 원인일 수 있다.

## 면접 답변 예시
- 질문: "`핫 파티션`은 무엇인가요?"  
  답변: "`핫 파티션`은 파티셔닝된 시스템에서 특정 키나 범위에 요청이 집중돼 일부 파티션만 병목이 생기는 현상입니다. 분산 저장 구조여도 키 분포가 고르지 않으면 충분히 발생할 수 있습니다."
- 질문: "어떻게 대응하나요?"  
  답변: "먼저 어떤 키가 부하를 몰아받는지 관찰해야 하고, 그다음에는 파티션 키 재설계, 랜덤 접미사나 버킷 분산, 캐시 도입, 비동기 큐 처리 같은 방법을 검토합니다. 단순 증설만으로는 해결되지 않는 경우가 많습니다."

## 장점
- 샤딩을 단순 개념이 아니라 운영 관점에서 이해하고 있다는 인상을 줄 수 있다
- 트래픽 패턴과 데이터 모델링을 함께 설명하기 좋은 주제다
- 성능 장애 원인을 평균 지표가 아닌 분포 관점에서 보는 사고를 보여 주기 좋다

## 단점
- 파티셔닝 전략, 캐시, 데이터 접근 패턴이 같이 얽혀 있어 설명이 복합적일 수 있다
- 해결책이 상황 의존적이라 단정적으로 말하면 오히려 약해 보일 수 있다
- 너무 인프라 이야기만 하면 애플리케이션 설계와 연결이 약해질 수 있다

## 주의사항 / 실무 팁
- 평균 CPU나 평균 응답 시간만 보면 놓치기 쉬워서 파티션별 지표를 봐야 한다.
- 인기 키가 생길 수 있는 도메인인지 먼저 파악하는 것이 중요하다.
- 면접에서는 `샤딩`, `캐시`, `큐잉`, `Consistent Hashing`과 연결하면 설명이 자연스럽다.
- 실무에서는 키 설계 단계에서부터 쏠림 가능성을 가정하고, 이벤트성 트래픽에 대한 완충 장치를 준비하는 편이 안전하다.
