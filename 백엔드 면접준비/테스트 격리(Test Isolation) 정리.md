# 테스트 격리(Test Isolation) 정리

## 핵심 요약

- 테스트 격리(Test Isolation)는 **각 테스트가 서로 독립적으로 실행**되도록 보장하는 것을 의미합니다.
- 격리가 깨지면 실행 순서/환경에 따라 결과가 달라지는 **비결정적(Non-deterministic) 테스트**가 발생할 수 있습니다.
- Spring에서 같은 DB를 사용하는 테스트는 보통 `@DirtiesContext`, `@Sql(TRUNCATE)`, `@Transactional(롤백)` 같은 방법으로 격리를 확보합니다.

## 테스트 격리란?

테스트 격리(Test Isolation)는 **어떤 테스트가 실행되더라도 다른 테스트의 결과나 상태에 영향을 주거나 받지 않도록** 만드는 것입니다.

## 테스트 격리가 중요한 이유

격리가 제대로 이루어지지 않으면 **비결정적 테스트(Non-deterministic test)** 가 발생할 수 있습니다.  
비결정적 테스트는 같은 테스트를 여러 번 실행했을 때 **항상 같은 결과를 내지 않는 테스트**를 말합니다.

예를 들어 테스트가 데이터베이스 같은 **공유 자원**에 의존하면:

- 테스트 실행 순서
- 다른 테스트 실행 여부/결과  
  에 따라 성공/실패가 달라질 수 있습니다.

이런 경우 테스트가 실패했을 때 **실제 코드 문제인지**, **환경/순서 같은 비결정적 요인 때문인지** 판단하기 어렵기 때문에, 예측 가능한 조건을 만들기 위한 격리가 중요합니다.

## Spring에서 같은 데이터베이스를 사용하는 테스트는 어떻게 격리할까요?

### 1) `@DirtiesContext`로 애플리케이션 컨텍스트 새로 로드

Spring 테스트 컨텍스트는 애플리케이션 컨텍스트를 **캐싱해서 재사용**합니다.  
`@DirtiesContext`를 사용하면 테스트마다 **새 애플리케이션 컨텍스트를 로드**하여 강한 격리를 확보할 수 있습니다.

```java
@DirtiesContext(classMode = DirtiesContext.ClassMode.AFTER_EACH_TEST_METHOD)
@SpringBootTest
class MyIntegrationTest {
    // ...
}
```

### 2) `@Sql`로 TRUNCATE 스크립트 실행

`@Sql`을 사용하면 테스트 실행 전/후로 SQL 스크립트를 실행할 수 있습니다.  
`TRUNCATE`로 테이블을 비워 테스트 간 상태를 분리할 수 있습니다.

```java
@Sql("/truncate.sql")
@SpringBootTest
class MyIntegrationTest {
    // ...
}
```

> `truncate.sql` 예시

```sql
TRUNCATE TABLE member;
TRUNCATE TABLE orders;
-- ...
```

### 3) `@Transactional`로 테스트 종료 시 롤백

테스트 클래스(또는 메서드)에 `@Transactional`을 붙이면 테스트 종료 후 트랜잭션을 롤백하여 DB 상태를 원복할 수 있습니다.

```java
@Transactional
@SpringBootTest
class MyIntegrationTest {
    // ...
}
```

#### `@Transactional` 사용 시 주의할 점

1. **프로덕션과 다른 조건으로 테스트될 수 있음**  
   예: OSIV를 꺼두고 `@Transactional`도 없는 프로덕션 코드에서는 지연 로딩 시 `LazyInitializationException`이 발생할 수 있는데, 테스트는 트랜잭션이 열려 있어 예외가 안 나 “실제로는 실패할 코드가 테스트에서는 통과”하는 경우가 생길 수 있습니다.

- 거짓 양성(False Positive): **프로덕션 코드는 정상 동작하지만 테스트는 실패**
- 거짓 음성(False Negative): **프로덕션 코드는 실패하지만 테스트는 통과**

2. `@SpringBootTest`의 `WebEnvironment`가 `DEFINE_PORT` 또는 `RANDOM_PORT`인 경우  
   서블릿 컨테이너가 **별도 스레드**에서 실행되므로 테스트 트랜잭션 롤백이 적용되지 않을 수 있습니다.

3. 프로덕션 코드 트랜잭션 전파 레벨이 `REQUIRES_NEW`인 경우  
   새 트랜잭션을 만들기 때문에 **테스트 트랜잭션과 무관**하여 롤백되지 않습니다.

4. **비동기 메서드**는 새 스레드에서 실행되므로 롤백되지 않습니다.

## 장점

### 테스트 격리의 장점

- 테스트가 항상 동일한 조건에서 실행되어 **예측 가능한 결과**를 얻기 쉽습니다.
- 실패 원인 분석 시 “환경/순서 문제”를 줄여 **디버깅 비용**이 감소합니다.

### Spring 격리 방식별 장점

- `@DirtiesContext`: 컨텍스트 단위로 **강한 격리**를 확보
- `@Sql(TRUNCATE)`: DB 상태를 **명시적으로 초기화**
- `@Transactional(롤백)`: 구현이 단순하고 **DB 상태 원복이 자동화**됨

## 단점

### Spring 격리 방식별 단점

- `@DirtiesContext`: 매번 컨텍스트를 새로 로드하므로 **비용이 크고 느릴 수 있음**
- `@Sql(TRUNCATE)`: 테이블 추가/변경 시 스크립트를 계속 관리해야 해 **유지보수 비용**이 발생
- `@Transactional(롤백)`: 프로덕션과 다른 실행 조건이 되어 **거짓 음성/거짓 양성** 가능성이 생길 수 있음  
  (또한 웹 서버 별도 스레드/`REQUIRES_NEW`/비동기 등으로 **롤백이 기대대로 안 될 수 있음**)

## 주의사항 / 실무 팁

- “강한 격리”가 필요한 테스트(인수/통합)와 “빠른 피드백”이 중요한 테스트(단위)를 구분해 격리 전략을 다르게 가져가면 운영이 수월합니다.
- DB 격리는 방법만 정하는 게 아니라, **실제로 격리가 유지되는지**(롤백 적용 범위, 별도 스레드, 전파 레벨, 비동기)까지 함께 점검하세요.
- 비결정적 테스트가 의심되면 “공유 자원(DB/캐시/파일/외부 API)”을 먼저 체크하고, 테스트 실행 순서를 바꿔가며 재현해 원인을 좁히는 것이 도움이 됩니다.
