# Write Through와 Write Behind 캐시 전략 정리

## 핵심 요약
- `Write Through`는 쓰기 요청이 들어오면 캐시와 DB를 함께 갱신하는 방식이고, `Write Behind`는 캐시에 먼저 쓰고 DB 반영은 나중에 비동기로 처리하는 방식이다.
- 둘 다 쓰기 경로를 최적화하는 전략이지만, 정합성과 지연 시간의 우선순위가 다르다.
- 면접에서는 Cache Aside와 비교해서 설명하면 훨씬 명확하다.
- 장애 시 데이터 유실 가능성과 복구 전략을 같이 말해야 답변의 완성도가 올라간다.

## 개념 설명
캐시 전략은 읽기 최적화만 있는 것이 아니다. 쓰기 시점에도 캐시를 어떻게 다룰지 결정해야 한다.

두 전략의 핵심 차이는 다음과 같다.
- `Write Through`: 애플리케이션이 데이터를 쓰면 캐시와 DB를 같이 갱신한다.
- `Write Behind`(또는 Write Back): 우선 캐시에 반영하고, 나중에 배치나 비동기 워커가 DB에 반영한다.

`Write Through`는 상대적으로 정합성을 설명하기 쉽다. 반면 `Write Behind`는 응답 속도에 유리하지만, 캐시 장애나 플러시 실패 시 데이터 유실 위험을 같이 관리해야 한다.

## 예시
예를 들어 사용자 프로필 수정 요청이 들어왔을 때:

- `Write Through`
  - API 서버가 캐시와 DB를 모두 성공시켜야 완료로 본다.
  - 읽기 직후 최신 값을 볼 확률이 높다.
- `Write Behind`
  - API 서버는 우선 캐시에만 기록하고 빠르게 응답한다.
  - 이후 큐/워커가 DB에 반영한다.
  - 장애가 나면 캐시에는 있는데 DB에는 없는 시점이 생길 수 있다.

## 면접 답변 예시
- 질문: "Write Through와 Write Behind의 차이는 무엇인가요?"  
  답변: "Write Through는 쓰기 시점에 캐시와 DB를 함께 갱신해서 정합성을 설명하기 쉽고, Write Behind는 캐시에 먼저 쓰고 나중에 DB에 반영해서 응답 속도와 쓰기 처리량에 유리합니다. 대신 Write Behind는 비동기 반영 실패나 유실 복구 전략이 꼭 필요합니다."
- 질문: "실무에서는 언제 Write Behind를 고려하나요?"  
  답변: "로그성 데이터나 약간의 지연 반영을 허용할 수 있는 대량 쓰기 시나리오에서 고려합니다. 반대로 결제나 재고처럼 강한 정합성이 필요한 영역은 신중해야 합니다."

## 장점
- Write Through는 읽기 직후 최신 데이터 일관성을 확보하기 쉽다
- Write Behind는 응답 지연을 줄이고 대량 쓰기 처리량을 높이기 좋다
- 시스템 요구사항에 따라 정합성 중심, 성능 중심 전략을 선택할 수 있다

## 단점
- Write Through는 매 쓰기마다 DB까지 같이 반영하므로 지연 시간이 늘 수 있다
- Write Behind는 장애 시 데이터 유실이나 반영 지연 이슈가 생길 수 있다
- 두 전략 모두 캐시 장애와 DB 장애를 따로 고려해야 해서 운영 복잡도가 올라간다

## 주의사항 / 실무 팁
- Write Behind를 쓸 때는 큐 적재 성공 여부, 재시도, DLQ 같은 운영 장치를 반드시 같이 설계한다.
- 캐시를 단순 성능 계층이 아니라 데이터 흐름 일부로 쓰는 순간 모니터링과 감사 로그 중요도가 올라간다.
- 면접에서는 "무조건 빠른 전략"이 아니라 데이터 정합성 요구 수준에 따라 선택한다는 점을 강조하면 좋다.
