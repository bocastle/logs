# 2PC(분산 트랜잭션) 정리

## 핵심 요약

- 2PC(2-Phase Commit)는 여러 데이터 저장소에 걸친 트랜잭션을 원자적으로 처리하기 위한 프로토콜이다.
- 첫 단계 `prepare`에서 커밋 가능 여부를 각 참여자가 보고하고, 둘째 단계 `commit/rollback`에서 최종 의사결정을 일괄 적용한다.
- 신뢰성은 높지만, 네트워크 지연/잠금 유지로 성능이 떨어질 수 있어 대규모 서비스에서는 대안(사가, 멱등 요청)과 함께 판단한다.

## 개념 정리

2PC는 **코디네이터(조정자)**와 여러 **파티시펀트(참여자)**가 협력한다.

1. 코디네이터가 `prepare`를 보내면 각 노드가 락을 잡고 커밋 준비를 한다.
2. 모두 `yes`면 `commit`, 하나라도 `no` 또는 타임아웃이면 `rollback`을 결정한다.

이때 `prepare` 성공 노드가 죽으면 코디네이터는 상태를 복구 로그로 다시 판단해야 하므로, 장애 상황에서의 복구 로직이 핵심이다.

## 예시

- 주문 생성 흐름: 주문 DB, 재고 DB, 포인트 DB를 함께 갱신해야 하는 경우
- 단계 1에서 3개 DB 모두 준비 완료 보고
- 단계 2에서 모두 커밋 후 최종 주문 확정, 실패 시 전부 롤백

쉽게 말해, "하나라도 실패하면 전부 실패"를 강제로 지키는 방식이다.

## 면접 답변 예시

"2PC는 분산 환경에서 트랜잭션 일관성을 보장하는 전통적 방법입니다. 참여자들을 먼저 준비 상태로 락을 걸어두고, 모든 참여자가 커밋 준비가 되면 코디네이터가 일괄 커밋합니다. 반대로 한쪽이라도 실패하면 전부 롤백해서 정합성을 지킵니다. 대신 응답 대기 시간이 길어지고 동시성 병목이 생길 수 있어, 최근엔 SAGA, Outbox, 멱등성 기반 재시도로 대체하거나 보완합니다."

## 장점

- 강한 정합성 보장: 부분 커밋을 방지
- 장애 상황에서도 복구 로그 기반 재조정 가능
- 트랜잭션 상태가 명확해 감사 추적(trace) 설계가 쉬움

## 단점

- 동기 블록킹으로 처리량 감소 가능성 큼
- 네트워크 분할/타임아웃 시 복구 복잡도 증가
- 파티시펀트 수가 늘수록 지연 증가
- 코디네이터 단일 장애점(SPOF) 위험

## 주의사항 / 실무 팁

- 코디네이터 로그(write-ahead log)는 영속 저장소에 남겨 재기동 후 상태를 복원한다.
- prepare 단계가 길어지지 않도록 비즈니스 로직은 최대한 짧게 만든다.
- 핵심 경로에서는 2PC 대신 사가/오케스트레이션을 검토해 락 경합을 줄인다.
- 운영 지표로 prepare 성공률, commit 대기시간, 롤백률을 꾸준히 모니터링한다.
