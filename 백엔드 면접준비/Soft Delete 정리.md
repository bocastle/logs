# Soft Delete 정리

## 핵심 요약
- `Soft Delete`는 데이터를 물리적으로 바로 삭제하지 않고, 삭제 여부를 나타내는 상태만 바꾸는 방식이다.
- 핵심 목적은 복구 가능성 확보, 감사 추적, 참조 무결성 문제 완화에 있다.
- 면접에서는 "행을 지우는 대신 삭제 플래그나 삭제 시각을 기록해 숨기는 전략"이라고 설명하면 된다.
- 장점뿐 아니라 조회 조건 누락, 인덱스 비대화, 보관 정책 부재 같은 운영 비용도 함께 말해야 답변이 탄탄해진다.

## 개념 설명
일반적인 삭제는 `DELETE`로 행을 실제로 제거하는 방식이다. 이 방식은 단순하지만, 나중에 복구가 필요하거나 "누가 언제 삭제했는가"를 추적해야 하는 서비스에서는 한계가 있다.

`Soft Delete`는 이런 문제를 줄이기 위해 실제 행은 남겨 두고 `deleted = true`나 `deleted_at = timestamp` 같은 컬럼만 갱신한다. 애플리케이션은 평소 조회 시 삭제된 데이터를 제외해서 사용자에게는 "삭제된 것처럼" 보이게 만든다.

이 전략은 게시글, 주문, 회원, 결제 이력처럼 감사와 복구가 중요한 도메인에서 자주 쓰인다. 특히 다른 테이블이 해당 데이터를 참조하고 있을 때, 물리 삭제보다 안전하게 운영할 수 있다.

하지만 데이터가 계속 쌓이기 때문에 시간이 지나면 테이블이 비대해질 수 있다. 또 모든 조회 쿼리에 삭제 조건을 넣어야 해서, 누락되면 버그나 보안 이슈로 이어질 수 있다.

## 예시
예를 들어 게시판 서비스에서 사용자가 글을 삭제했다고 해 보자.

- 물리 삭제를 하면 행이 사라져서 복구가 어렵다
- 댓글, 신고, 감사 로그가 해당 게시글 ID를 참조하고 있으면 후처리가 복잡해진다
- `Soft Delete`를 쓰면 `deleted_at`만 채우고 일반 목록 조회에서는 `WHERE deleted_at IS NULL` 조건으로 숨긴다
- 필요하면 관리자 화면에서 복구하거나, 일정 기간이 지난 뒤 배치로 실제 삭제할 수 있다

즉, 사용자 경험상 삭제는 즉시 반영하면서도 운영 관점에서는 안전망을 남겨 두는 방식이다.

## 면접 답변 예시
- 질문: "`Soft Delete`는 무엇인가요?"  
  답변: "데이터를 바로 물리 삭제하지 않고, 삭제 상태만 표시해서 애플리케이션에서 숨기는 방식입니다. 보통 `deleted_at` 같은 컬럼을 둡니다."
- 질문: "왜 쓰나요?"  
  답변: "복구 가능성을 남길 수 있고, 감사 추적이나 참조 무결성 관리가 쉬워지기 때문입니다. 특히 운영 중 실수 삭제 대응에 유리합니다."
- 질문: "주의할 점은 무엇인가요?"  
  답변: "모든 조회에 삭제 조건이 빠지지 않아야 하고, 오래 쌓인 삭제 데이터 정리 정책도 필요합니다. 그렇지 않으면 성능과 데이터 품질 문제가 생길 수 있습니다."

## 장점
- 실수 삭제에 대한 복구 가능성을 확보할 수 있다
- 감사 로그와 운영 추적에 유리하다
- 연관 데이터가 많은 도메인에서 물리 삭제보다 안전하게 다룰 수 있다

## 단점
- 조회 쿼리마다 삭제 조건을 신경 써야 한다
- 삭제 데이터가 계속 쌓여 저장 공간과 인덱스 비용이 증가한다
- "정말 삭제된 것인지"에 대한 정책이 불분명하면 운영이 혼란스러워질 수 있다

## 주의사항 / 실무 팁
- `deleted = true`보다 `deleted_at`을 두면 삭제 시점 추적이 쉬워서 실무에서 더 자주 쓰인다.
- 면접에서는 "복구와 감사 추적"을 장점으로 말하고, "조회 조건 누락 위험"을 단점으로 붙이면 균형이 좋다.
- 실무에서는 기본 쿼리 조건, ORM 글로벌 필터, 관리자 조회 예외 규칙을 함께 설계해야 한다.
- 결국 영구 보관인지, 일정 기간 뒤 실제 삭제할지까지 정해야 `Soft Delete`가 깔끔하게 운영된다.
