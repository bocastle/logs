# Outbox Pattern 정리

## 핵심 요약

- `Outbox Pattern`은 DB 변경과 메시지 발행을 하나의 흐름으로 묶어 데이터 불일치를 줄이는 패턴이다.
- 면접에서는 "왜 필요한가", "트랜잭션과 메시지 브로커 사이에서 어떤 문제가 생기나", "어떻게 발행을 보장하나"를 설명하면 좋다.
- 핵심 아이디어는 비즈니스 데이터 저장과 함께 outbox 테이블에도 이벤트를 기록하고, 별도 프로세스가 이를 읽어 메시지를 발행하는 것이다.
- 이 패턴을 써도 중복 발행 가능성은 남기 때문에 소비자 멱등성과 재처리 전략까지 같이 말해야 답변이 탄탄하다.

## 왜 Outbox Pattern이 필요하냐

백엔드에서는 주문 저장 후 이벤트를 발행하거나, 회원 가입 후 다른 서비스에 알림을 보내는 일이 많다.  
문제는 DB 트랜잭션과 메시지 브로커 전송이 서로 다른 시스템이라는 점이다.

예를 들어 DB 저장은 성공했는데 Kafka 발행이 실패하면 데이터와 이벤트가 어긋난다.  
반대로 이벤트는 나갔는데 DB 롤백이 되면 더 심각한 불일치가 생길 수 있다.

Outbox Pattern은 이런 틈을 줄이기 위한 방법이다.

## 기본 동작 방식

### 1. 비즈니스 데이터와 outbox 레코드를 같이 저장

주문 테이블에 주문을 저장하는 트랜잭션 안에서, outbox 테이블에도 "주문 생성 이벤트"를 함께 기록한다.

중요한 포인트는 두 작업이 같은 DB 트랜잭션 안에 있다는 점이다.  
그래서 주문 저장이 커밋됐으면 outbox 레코드도 반드시 남고, 주문 저장이 롤백되면 outbox도 같이 사라진다.

### 2. 별도 발행자가 outbox를 읽는다

배치나 백그라운드 워커, CDC 도구가 outbox 테이블을 읽어서 메시지 브로커로 이벤트를 발행한다.

발행이 성공하면 상태를 `SENT`로 바꾸거나, 재처리하지 않도록 마킹한다.

## 2PC 대신 왜 이걸 쓰나

이론적으로는 DB와 메시지 브로커를 2단계 커밋으로 묶는 방법을 떠올릴 수 있다.  
하지만 실제로는 구현 복잡도와 운영 부담이 크고, 지원 제약도 많다.

그래서 실무에서는 완전한 분산 트랜잭션보다, 로컬 트랜잭션 + 비동기 발행 조합인 Outbox Pattern을 더 현실적인 선택으로 본다.

## 어떤 방식으로 outbox를 읽나

### Polling

애플리케이션이 주기적으로 outbox 테이블을 조회해서 아직 발행되지 않은 이벤트를 가져가는 방식이다.  
구현은 단순하지만 조회 주기와 지연 시간이 생긴다.

### CDC(Change Data Capture)

DB 변경 로그를 읽어서 outbox 이벤트를 브로커로 전달하는 방식이다.  
Debezium 같은 도구가 여기서 자주 언급된다.

Polling보다 더 자연스럽게 실시간성에 가까운 구성이 가능하지만, 운영 복잡도는 올라갈 수 있다.

## 이 패턴을 써도 끝이 아닌 이유

### 중복 발행 가능성

메시지 발행 후 상태 업데이트 전에 장애가 나면, 같은 이벤트가 다시 읽혀 재발행될 수 있다.  
그래서 소비자 쪽 멱등성이 중요하다.

### outbox 테이블 관리

계속 쌓이기만 하면 테이블이 커져서 성능 문제가 생길 수 있다.  
보관 기간, 아카이빙, 삭제 정책도 함께 설계해야 한다.

### 순서 보장 고민

엔티티 단위 순서가 중요한 도메인이라면 event key, 파티션 전략, 조회 순서를 같이 봐야 한다.  
그냥 "발행만 하면 된다"로 끝내면 부족하다.

## 면접 답변 예시

> Outbox Pattern은 DB 변경과 메시지 발행 사이의 불일치를 줄이기 위한 패턴입니다. 비즈니스 데이터 저장과 함께 같은 트랜잭션 안에서 outbox 테이블에 이벤트를 기록하고, 이후 별도 프로세스가 그 이벤트를 읽어 Kafka 같은 브로커로 발행합니다. 이렇게 하면 DB 커밋은 됐는데 이벤트 발행이 누락되는 문제를 줄일 수 있습니다. 다만 재처리 과정에서 중복 발행이 생길 수 있어서 소비자 멱등성과 outbox 정리 전략까지 같이 설계해야 합니다.

## 장점

- DB 저장과 이벤트 기록을 같은 트랜잭션으로 묶어 일관성을 높일 수 있다.
- 분산 트랜잭션보다 현실적인 운영 모델을 만들기 쉽다.
- 이벤트 기반 아키텍처에서 장애 내성을 높이는 데 도움이 된다.

## 단점

- outbox 테이블과 발행 워커를 추가로 운영해야 한다.
- 결국 비동기 흐름이라 즉시 전파 보장은 어렵다.
- 중복 발행, 재처리, 정리 정책까지 고려해야 해서 설계가 길어진다.

## 주의사항 / 실무 팁

- 면접에서는 "DB 저장과 메시지 발행은 원자적으로 묶기 어렵다"는 문제 정의부터 꺼내면 설명이 좋다.
- outbox만 말하고 소비자 멱등성을 빼면 답변이 반쪽짜리가 되기 쉽다.
- polling과 CDC는 둘 다 알고 있되, 운영 복잡도 차이까지 같이 말하면 더 좋다.
- 테이블 청소 전략과 실패 재시도 정책을 빠뜨리지 않는 게 실무적으로 중요하다.
