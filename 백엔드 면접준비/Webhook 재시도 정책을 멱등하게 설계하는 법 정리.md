# Webhook 재시도 정책을 멱등하게 설계하는 법 정리

## 핵심 요약

- 면접에서 "Webhook 재시도 정책을 멱등하게 설계하는 법" 질문을 받으면 핵심은 "정상 처리 뒤에 실패와 복구 기준이 준비되어 있는가?"이다.
- 운영에서 어떤 알림을 보고 어떤 순서로 확인할지 말하면 답변이 실무적으로 들린다.
- 기술 선택의 장점과 함께 비용, 복잡도, 되돌림 기준을 같이 정리해야 한다.

## 개념 설명

백엔드 설계 주제는 정상 처리보다 실패와 복구 기준을 함께 말해야 깊이가 생긴다.

운영자가 원인을 찾을 수 있는 로그와 지표, 사용자가 받는 영향, 되돌릴 방법을 같이 잡는 편이 좋다.

"Webhook 재시도 정책을 멱등하게 설계하는 법"은 단독 지식으로 외우기보다 주변 시스템과 사용자 흐름에 어떤 영향을 주는지 연결해서 보는 편이 좋다.

## 예시

```text
검증 순서:
1. timestamp 허용 범위 확인
2. HMAC signature 검증
3. event_id 중복 처리 여부 확인
4. 처리 결과 저장 후 응답
```

실무 적용 전에는 이 구조가 현재 서비스의 트래픽, 보안 요구사항, 배포 방식과 맞는지 확인해야 한다.

## 면접 답변 예시

> "Webhook 재시도 정책을 멱등하게 설계하는 법" 질문이라면 저는 성공 경로뿐 아니라 실패, 재시도, 롤백, 관측 지표를 함께 설계하겠다고 답하겠습니다. 단순 구현 방법보다 어떤 장애를 줄이는지, 문제가 생겼을 때 어떤 지표로 알아차릴지, 변경을 되돌릴 기준은 무엇인지까지 말하는 것이 중요합니다.

## 장점

- 운영 중 문제를 더 빨리 좁힐 수 있다.
- 장애 영향 범위를 줄이는 설계를 만들 수 있다.
- 팀 안에서 같은 기준으로 리뷰하기 좋다.

## 단점

- 운영 기준이 늘수록 설정과 문서 관리 비용이 생긴다.
- 잘못된 알림은 피로도를 높여 실제 장애 대응을 늦춘다.
- 복구 전략이 없으면 복잡한 구조가 오히려 위험하다.

## 주의사항 / 실무 팁

- 재시도 가능 여부와 중복 처리 기준을 항상 같이 말한다.
- 로그, 메트릭, trace 중 어떤 신호로 문제를 발견할지 정한다.
- 운영 변경은 작은 범위로 적용하고 되돌릴 조건을 먼저 적어 둔다.
