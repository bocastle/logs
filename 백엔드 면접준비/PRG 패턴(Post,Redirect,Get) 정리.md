# PRG 패턴(Post/Redirect/Get) 정리

## 핵심 요약

- PRG 패턴은 **Post/Redirect/Get**의 약자로, 폼 제출 후 **새로고침/뒤로가기** 등으로 인해 **중복 POST 요청이 발생하는 문제를 방지**하기 위한 패턴입니다.
- 일반적으로 **멱등성이 보장되지 않는 POST 요청**에서 유용합니다.
- 흐름은 **POST → Redirect(302) → GET** 입니다.

## PRG 패턴이란?

PRG 패턴은 웹 애플리케이션에서 폼 제출 후 페이지 새로 고침이나 브라우저 뒤로 가기 등의 문제를 방지하기 위해 사용하는 디자인 패턴입니다.  
일반적으로 멱등성이 보장되지 않는 POST 요청에 사용합니다.

예를 들어, 사용자가 웹 페이지에서 주문 버튼을 클릭하고 새로고침을 수행하면 2번의 POST 요청이 서버로 전달될 수 있는데요. 이러한 상황에서 PRG 패턴이 주로 사용됩니다.

## 동작 과정(단계)

PRG 패턴은 다음과 같은 단계를 따릅니다.

### 1) Post

- 사용자가 폼을 제출하면 클라이언트는 서버에 **POST 요청**을 보냅니다.
- 서버는 이 요청을 처리하여 데이터베이스를 업데이트하거나 다른 작업을 수행합니다.

### 2) Redirect

- 서버는 POST 요청을 처리한 후, 클라이언트에게 **새로운 URL로 리디렉션**하라는 응답을 보냅니다.
- 이 리디렉션은 클라이언트에게 **302 Found** 상태 코드와 함께 새로운 URL을 포함한 **Location 헤더**를 반환하여 수행됩니다.

### 3) Get

- 클라이언트는 서버의 응답을 받아 새로운 URL로 **GET 요청**을 보냅니다.
- 서버는 이 GET 요청을 처리하여 최종 결과 페이지를 클라이언트에게 반환합니다.

## 장점

- 폼 제출 후 새로고침/뒤로가기로 인한 **중복 POST 요청**을 방지할 수 있습니다.
- 최종 결과 페이지는 GET으로 조회되므로, 사용자가 새로고침해도 **의도치 않은 재요청(POST 재전송)** 가능성이 줄어듭니다.

## 단점

- 원문에서는 별도의 단점이 직접적으로 언급되지는 않았습니다.

## 주의사항 / 실무 팁

- PRG는 “POST 처리 결과 화면”을 직접 반환하기보다, **리다이렉트 대상 URL을 분리**하는 것이 핵심입니다.
- 리다이렉트 후 GET에서 필요한 데이터가 있다면, **URL 설계(리소스 식별자 포함)** 또는 **서버 측 조회**로 결과 화면을 구성하는 방식이 일반적입니다.
- 302 외에도 상황에 따라 다른 리다이렉트 상태 코드를 사용할 수 있으니, 서비스 요구사항과 HTTP 규칙에 맞춰 선택하세요.
