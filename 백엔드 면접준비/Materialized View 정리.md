# Materialized View 정리

## 핵심 요약
- `Materialized View`는 조회 결과를 미리 계산해 저장해 두는 뷰다.
- 일반 뷰가 쿼리를 실행할 때마다 원본 테이블을 다시 읽는다면, Materialized View는 결과를 별도로 보관한다는 점이 핵심 차이다.
- 면접에서는 "복잡한 집계나 조인을 조회 시점이 아니라 사전에 계산해 두어 읽기 성능을 높이는 방법"이라고 설명하면 좋다.
- 대신 최신 데이터와의 시차, 갱신 비용, 운영 복잡도를 함께 말해야 답변이 탄탄해진다.

## 개념 설명
서비스에서 대시보드, 통계 화면, 추천 집계처럼 매번 복잡한 조인과 집계를 수행해야 하는 경우가 있다. 이런 쿼리를 요청마다 실시간으로 계산하면 데이터는 가장 최신이지만, 응답 시간이 길어지고 원본 DB 부하도 커질 수 있다.

`Materialized View`는 이런 상황에서 자주 쓰이는 전략이다. 결과를 미리 계산해 저장해 두고, 조회는 그 저장된 결과를 읽게 한다. 그래서 읽기 성능은 좋아지지만, 대신 원본 데이터가 바뀔 때 이 결과를 언제 어떻게 갱신할지 결정해야 한다.

핵심 포인트는 세 가지다.

- 읽기 성능을 위해 계산 결과를 저장한다
- 저장된 결과는 원본 데이터와 시차가 생길 수 있다
- 갱신 주기와 방식이 설계의 핵심이다

즉, Materialized View는 성능을 위해 "계산 비용을 미리 내는" 전략이라고 볼 수 있다.

## 예시
예를 들어 쇼핑몰 관리자 화면에서 "카테고리별 일 매출", "최근 30일 주문 수", "상위 판매 상품"을 보여준다고 해 보자.

- 매 요청마다 주문 테이블을 조인하고 집계하면 느릴 수 있다
- 배치나 이벤트 기반으로 집계 결과를 Materialized View에 저장해 두면 조회는 매우 빨라진다
- 대신 주문이 막 들어온 직후에는 뷰가 아직 갱신되지 않아 수치가 약간 늦을 수 있다

이런 식으로 실시간 정합성이 조금 느슨해도 되는 분석성 조회에서는 매우 실용적이다.

## 면접 답변 예시
- 질문: "`Materialized View`는 무엇인가요?"  
  답변: "복잡한 조회 결과를 미리 계산해 물리적으로 저장해 두는 뷰입니다. 일반 뷰와 달리 조회할 때마다 원본 쿼리를 다시 계산하지 않아서 읽기 성능을 높일 수 있습니다."
- 질문: "언제 적합한가요?"  
  답변: "대시보드나 통계처럼 조회가 무겁고 반복적이지만, 약간의 지연 갱신을 허용할 수 있는 경우에 적합합니다. 대신 갱신 비용과 최신성 요구를 같이 판단해야 합니다."

## 장점
- 복잡한 조인과 집계를 빠르게 조회할 수 있다
- 원본 테이블 부하를 줄이고 읽기 전용 화면 응답성을 높이기 좋다
- 대시보드, 리포트, 분석성 조회 사례로 설명하기 좋다

## 단점
- 원본 데이터와 완전히 실시간으로 맞지 않을 수 있다
- 갱신 전략이 필요해 운영 복잡도가 올라간다
- 저장 공간이 추가로 필요하고, 갱신 시 쓰기 비용도 든다

## 주의사항 / 실무 팁
- "빠르다"만 말하면 부족하다. 어떤 최신성 요구를 포기하고 무엇을 얻는지 같이 설명해야 한다.
- 갱신 방식은 배치, 주기 refresh, 이벤트 기반 반영 등 여러 선택지가 있고 시스템 특성에 따라 달라진다.
- 주문/정산처럼 강한 실시간 정합성이 필요한 값은 Materialized View만 믿지 않고 원본 검증 경로를 두는 편이 안전하다.
- 면접에서는 "`일반 뷰는 논리적, Materialized View는 계산 결과를 저장한 물리적 뷰`"라고 비교해서 말하면 이해가 쉽다.
