# 백엔드 면접준비: 멀티테넌시 정리

## 핵심 요약
- 멀티테넌시는 하나의 애플리케이션이 여러 고객(테넌트)을 분리해 동시에 지원하는 구조입니다.
- 구현 방식은 `독립형(DB 분리)`, `스키마 분리`, `공유 DB+스키마`로 크게 나뉩니다.
- 비용 절감과 운영 효율은 높지만, 데이터 분리와 장애 전파 제어가 핵심 난이도입니다.

## 개념 설명
멀티테넌시의 목표는 동일한 소프트웨어 인스턴스로 여러 조직/고객을 운영하는 것입니다.  
예를 들어 SaaS에서 A사와 B사가 같은 서비스 코드를 쓰지만 서로의 데이터가 섞이면 안 됩니다.

대표 구현 방식

- 독립형(Separate Database): 가장 강한 분리, 운영비·복잡도는 높음
- 스키마 분리(Shared DB, Separate Schema): 관리 효율과 분리의 균형
- 공유형(Shared DB/Schema): 가장 저렴하지만 코드/권한 계층이 가장 까다로움

추가로 `tenant_id`를 요청마다 전달해 권한 필터를 적용하는 형태가 가장 흔합니다.

## 실용 예시
테넌트 ID 기반 인증 후 미들웨어에서 쿼리에 자동으로 `tenant_id = :tenantId` 조건을 붙입니다.
이 경우 ORM/Hibernate 필터, 쿼리 스코프, 또는 DB Row Level Security(RLS)를 함께 쓰면 실수 가능성을 줄일 수 있습니다.

## 면접 답변 예시
"멀티테넌시에서는 단순히 데이터 저장 구조만 분리하면 끝이 아니라, 인증/권한 경계, 트래픽 격리, 백업·모니터링 정책까지 함께 설계해야 합니다. 비용 최적화가 필요하면 공유형을, 보안 규제가 강하면 독립형을 우선 검토합니다."

## 장점
- 인프라 공유로 고객 온보딩 비용 감소
- 운영 자동화가 쉽고 배포 단위를 통일하기 유리
- 업데이트/장애 대응 속도가 높음

## 단점
- 한 버그나 쿼리 실수가 여러 테넌트에 영향을 줄 수 있음
- tenant 경계가 깨지면 보안 사고로 연결될 수 있음
- 용량/트래픽이 특정 테넌트에 치우치면 전체 성능에 영향

## 주의사항 / 실무 팁
- tenant 식별자는 애플리케이션 입장/인증 미들웨어/DB 계층에서 일관되게 검증하세요.
- 청구/요금제별로 쿼리량 제약을 두는 정책을 먼저 잡아야 급증 위험을 줄입니다.
- 장애 전파 방지를 위해 tenant별 할당량, 회로차단, 큐 격리 전략을 함께 도입하세요.
- `RLS`, 감사 로그, 접근권한 감사 추적을 운영 표준으로 만드세요.
