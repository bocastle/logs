# Infrastructure as Code(IaC) 정리

## 핵심 요약

- **IaC(Infrastructure as Code)**는 수동 설정 대신 **코드로 인프라를 프로비저닝/관리**하는 방식입니다.
- 반복 작업과 휴먼 에러를 줄이고, **일관성**과 **운영 효율**을 높이기 위해 사용합니다.
- 접근 방식은 크게 **선언적(Declarative)**과 **명령형(Imperative)**으로 나뉩니다.

## Infrastructure as Code(IaC)란?

코드형 인프라(Infrastructure as Code, IaC)는 수동 프로세스 대신 **코드를 통해 인프라를 프로비저닝하고 관리**하는 방법입니다.

기존의 수동 설정 방식은 다음과 같은 문제가 있습니다.

- 반복 작업이 많음
- 휴먼 에러가 발생하기 쉬움
- 인프라 설정을 별도로 문서화해 관리해야 하는 번거로움

IaC는 인프라를 코드로 관리함으로써 **일관성을 보장**하고 **운영 효율**을 높일 수 있도록 위 문제를 해결하기 위해 등장했습니다.

## IaC의 방식

### 선언적(Declarative) 방식

- **최종 상태**를 정의하면 IaC 도구가 이를 **자동으로 구성**하는 방식입니다.
- 사용자는 원하는 결과를 기술하고, **수행 과정은 도구가 처리**합니다.
- 대표 도구: **Terraform**, **AWS CloudFormation**

### 명령형(Imperative) 방식

- 인프라의 **구성 방법(단계)**을 직접 정의하는 방식입니다.
- 사용자가 인프라 설정 단계를 코드로 정의하며 **명령어 기반으로 실행**됩니다.
- 대표 도구: **Ansible**, **AWS CDK**

## 장점

- Git과 같은 **형상 관리 도구**를 활용해 변경 사항을 추적할 수 있습니다.
- **코드 자체가 문서**가 되며, 협업 시 **코드 리뷰**로 인프라 변경 사항을 검토할 수 있습니다.
- 수동 작업 없이 **코드 실행만으로 인프라 구축을 자동화**할 수 있습니다.
- 코드를 **재사용**할 수 있어 비슷한 인프라를 구축할 때 시간을 절약할 수 있습니다.

## 단점

- 다양한 도구의 사용법을 익혀야 해 **러닝 커브**가 발생할 수 있습니다.
- 인프라의 **상태(state) 관리**가 복잡할 수 있습니다.
- 인프라 변경 시 문제가 발생했을 때 **디버깅이 어려울 수** 있습니다.

## 주의사항/실무 팁

- IaC 도구는 보통 **상태(state)**를 기반으로 리소스를 추적합니다. 팀 단위 운영에서는 상태 파일의 **공유/잠금/백업** 전략을 함께 고려하세요.
- 코드 변경이 곧 인프라 변경으로 이어지므로, PR 기반으로 **리뷰 → 적용(Apply/Deploy)** 흐름을 만들고 변경 이력을 남기는 것이 좋습니다.
- 선언적/명령형은 도구에 따라 철학과 사용 방식이 달라지므로, 팀의 운영 방식(표준화/유연성/학습 비용)에 맞춰 선택하세요.
