# 트랜잭션 전파 정리

## 핵심 요약

- 트랜잭션 전파는 "이미 트랜잭션이 있을 때 새 메서드가 그 트랜잭션에 어떻게 참여할지"를 정하는 규칙이다.
- 스프링 면접에서는 `REQUIRED`, `REQUIRES_NEW`, `MANDATORY`, `SUPPORTS` 정도를 우선 이해하면 된다.
- 실무에서는 전파 옵션 자체를 외우는 것보다, 어떤 작업을 한 트랜잭션으로 묶고 어떤 작업을 분리할지 판단하는 게 더 중요하다.
- 특히 로그 저장, 이력 적재, 외부 연동 같은 작업은 `REQUIRES_NEW`가 왜 등장하는지 예시로 설명하면 좋다.

## 트랜잭션 전파가 왜 필요할까?

서비스 메서드는 서로를 자주 호출한다. 예를 들어 주문 생성 메서드가 재고 차감 메서드와 결제 기록 저장 메서드를 부를 수 있다. 이때 이미 바깥쪽에서 트랜잭션이 시작된 상태라면, 안쪽 메서드가 그 트랜잭션에 같이 들어갈지 아니면 새로 시작할지를 정해야 한다.

이 규칙이 바로 트랜잭션 전파다.

즉, 트랜잭션 전파는 "메서드 호출 관계 속에서 트랜잭션 경계를 어떻게 가져갈지"에 대한 정책이라고 보면 된다.

## REQUIRED

가장 기본값이다.

- 현재 트랜잭션이 있으면 거기에 참여한다.
- 현재 트랜잭션이 없으면 새로 시작한다.

실무에서는 대부분 이 옵션으로 충분하다. 하나의 비즈니스 작업을 한 덩어리로 처리하고 싶을 때 자연스럽기 때문이다.

예를 들어 주문 생성과 재고 차감, 주문 상세 저장이 모두 같은 성공/실패 운명을 가져야 한다면 같은 트랜잭션으로 묶는 게 맞다.

## REQUIRES_NEW

이 옵션은 현재 트랜잭션이 있어도 잠시 중단하고, 새로운 트랜잭션을 따로 시작한다.

이게 필요한 대표적인 상황은 "본 작업과 별개로 꼭 남겨야 하는 기록"이다.

예를 들면 주문 처리 중 예외가 나서 전체 주문은 롤백돼야 하더라도, 실패 이력 로그는 남기고 싶을 수 있다. 이럴 때 로그 저장을 `REQUIRES_NEW`로 분리하면 바깥 트랜잭션이 롤백돼도 해당 로그 트랜잭션은 커밋될 수 있다.

다만 남발하면 트랜잭션 흐름이 복잡해지고 성능 부담도 생긴다.

## SUPPORTS

- 현재 트랜잭션이 있으면 참여한다.
- 없으면 그냥 트랜잭션 없이 실행한다.

읽기 전용 조회처럼 트랜잭션이 있으면 같이 타고 들어가도 되지만, 없어도 크게 문제 없는 경우에 볼 수 있다. 다만 무심코 쓰기보다는 의도를 분명히 하는 게 중요하다.

## MANDATORY

- 현재 트랜잭션이 반드시 있어야 한다.
- 없으면 예외를 던진다.

즉, 이 메서드는 혼자 호출되면 안 되고, 꼭 상위 비즈니스 트랜잭션 안에서만 실행돼야 한다는 뜻이다.

핵심 도메인 로직에서 호출 계약을 강하게 걸고 싶을 때 의미가 있다.

## 그 밖에 자주 언급되는 것들

- `NOT_SUPPORTED`: 현재 트랜잭션이 있으면 중단하고 트랜잭션 없이 실행
- `NEVER`: 트랜잭션이 있으면 오히려 예외
- `NESTED`: 중첩 트랜잭션처럼 동작하지만 실제 지원 여부와 제약을 확인해야 함

면접에서는 모든 옵션을 길게 외우기보다, 자주 쓰는 옵션의 차이를 선명하게 설명하는 편이 낫다.

## 실무 예시

### 주문 저장과 재고 차감

주문 저장, 주문 상품 저장, 재고 차감은 하나라도 실패하면 전체가 롤백되는 게 자연스럽다. 이 경우 보통 같은 트랜잭션으로 묶으므로 `REQUIRED` 흐름이 맞다.

### 실패 로그 저장

주문 처리 도중 예외가 발생했을 때 장애 분석용 로그는 꼭 남기고 싶을 수 있다. 이럴 때 실패 로그 저장을 `REQUIRES_NEW`로 두면 본 작업과 분리할 수 있다.

### 외부 API 호출

DB 트랜잭션을 오래 잡은 상태로 외부 API를 기다리면 락 점유 시간과 커넥션 점유 시간이 길어질 수 있다. 그래서 외부 호출까지 무조건 같은 트랜잭션 안에 넣는 게 좋은지는 따져봐야 한다.

## 많이 헷갈리는 포인트

### 1. 전파 옵션이 곧 설계는 아니다

전파 옵션을 바꾸는 것만으로 아키텍처 문제가 해결되지는 않는다. 메서드 경계가 어색하거나 트랜잭션 범위가 너무 크면 전파 옵션을 손보기 전에 구조를 먼저 봐야 한다.

### 2. `REQUIRES_NEW`는 강력하지만 비싸다

새 트랜잭션을 따로 만들고 커밋 흐름도 분리되므로, 남발하면 성능과 복잡도가 같이 올라간다.

### 3. 프록시 기반 동작 주의

스프링 `@Transactional`은 보통 프록시 기반이라 같은 클래스 안에서 자기 메서드를 직접 호출하면 기대한 전파가 적용되지 않는 경우가 있다. 실무에서 꽤 자주 부딪히는 함정이다.

## 면접 답변 예시

트랜잭션 전파는 이미 트랜잭션이 진행 중일 때 하위 메서드가 그 트랜잭션에 어떻게 참여할지를 정하는 규칙입니다. 기본값인 `REQUIRED`는 기존 트랜잭션이 있으면 참여하고 없으면 새로 시작합니다. 반면 `REQUIRES_NEW`는 기존 트랜잭션과 별개로 새 트랜잭션을 시작해서, 예를 들어 본 작업이 롤백돼도 실패 로그는 남기고 싶은 상황에 쓸 수 있습니다. 실무에서는 옵션 이름을 외우는 것보다 어떤 작업을 함께 커밋할지, 어떤 작업을 분리할지를 판단하는 게 더 중요하다고 생각합니다.

## 장점

- 비즈니스 흐름에 맞춰 트랜잭션 경계를 세밀하게 설계할 수 있다.
- 일부 작업을 독립 커밋시켜 운영 관점의 요구를 반영할 수 있다.
- 상위 서비스와 하위 서비스의 호출 계약을 명확하게 만들 수 있다.

## 단점

- 옵션을 잘못 쓰면 롤백 흐름이 예상과 다르게 흘러갈 수 있다.
- `REQUIRES_NEW` 같은 옵션은 성능 부담과 복잡도를 높일 수 있다.
- 프록시 동작 방식까지 이해하지 않으면 설정해도 기대대로 안 움직일 수 있다.

## 주의사항 / 실무 팁

- 전파 옵션을 고르기 전에 "같이 성공해야 하는 작업인지"부터 먼저 정리하는 게 좋다.
- 외부 API 호출이나 오래 걸리는 작업을 큰 트랜잭션 안에 오래 묶지 않는 편이 안전하다.
- 실패 로그, 감사 이력 같은 보조성 작업은 분리 가치가 있는지 따져보고 `REQUIRES_NEW`를 검토할 수 있다.
- 면접에서는 `REQUIRED`와 `REQUIRES_NEW` 차이를 예시와 함께 설명하면 가장 전달력이 좋다.
