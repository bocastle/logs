# Write Skew 정리

## 핵심 요약
- `Write Skew`는 각 트랜잭션이 서로 다른 행을 수정해서 직접 충돌은 없지만, 함께 반영되면 비즈니스 규칙이 깨지는 동시성 이상 현상이다.
- `Snapshot Isolation`에서는 발생할 수 있고, 단순한 행 단위 충돌 검사만으로는 막기 어렵다.
- 면접에서는 "같은 데이터를 동시에 쓰지 않아도, 같은 조건을 보고 다른 데이터를 수정하면서 규칙이 깨질 수 있는 문제"라고 설명하면 전달이 잘 된다.
- 해결하려면 더 강한 격리 수준이나 명시적 락, 제약 조건 설계가 필요할 수 있다.

## 개념 설명
많은 사람이 동시성 문제를 이야기할 때 같은 행을 동시에 업데이트하는 충돌만 떠올린다. 하지만 실제로는 서로 다른 행을 수정하더라도, 수정 전에 참조한 조건이 같다면 시스템 규칙이 깨질 수 있다.

`Write Skew`는 대표적으로 "현재 조건을 읽고, 그 조건을 믿고, 서로 다른 대상을 수정하는" 흐름에서 나온다. 각 트랜잭션은 자신이 본 스냅샷 기준으로는 문제가 없다고 판단하지만, 둘 다 커밋되고 나면 전체 상태는 잘못된 결과가 된다.

핵심은 데이터 충돌 여부보다 "비즈니스 불변식이 유지되느냐"다. 그래서 단순히 같은 row를 건드리지 않았다고 안전하다고 보면 안 되고, 읽은 조건과 쓰는 결과가 어떤 규칙으로 연결되는지 봐야 한다.

## 예시
병원 당직 스케줄 예시가 자주 나온다.

- 규칙: 특정 시간대에는 반드시 의사 1명 이상이 당직 상태여야 한다
- 현재 상태: 의사 A, 의사 B 둘 다 당직 중이다

이때 두 트랜잭션이 동시에 실행된다.

1. 트랜잭션 1은 "B가 당직 중이니 A를 쉬게 해도 된다"라고 판단하고 A를 off로 바꾼다
2. 트랜잭션 2는 "A가 당직 중이니 B를 쉬게 해도 된다"라고 판단하고 B를 off로 바꾼다

두 트랜잭션은 서로 다른 행을 수정했기 때문에 행 충돌은 없다. 하지만 둘 다 커밋되면 최종적으로 당직 의사가 0명이 되어 규칙이 깨진다. 이게 `Write Skew`다.

## 면접 답변 예시
- 질문: "`Write Skew`는 어떤 문제인가요?"  
  답변: "`Write Skew`는 트랜잭션들이 같은 조건을 읽고 서로 다른 행을 수정하면서, 직접적인 쓰기 충돌은 없는데 최종적으로 비즈니스 규칙이 깨지는 동시성 이상 현상입니다. 그래서 row 충돌 검사만으로는 막히지 않을 수 있습니다."
- 질문: "어떻게 대응할 수 있나요?"  
  답변: "규칙을 보장해야 하는 데이터를 더 강한 격리 수준에서 처리하거나, 관련 행을 함께 잠그는 방식이 필요할 수 있습니다. 경우에 따라서는 DB 제약 조건이나 애플리케이션 레벨 검증을 추가해 불변식을 강제해야 합니다."

## 장점
- 동시성 이상 현상을 깊이 있게 이해하고 있다는 인상을 줄 수 있는 주제다
- 단순 락 충돌과 비즈니스 규칙 위반을 구분해 설명하기 좋다
- `Snapshot Isolation`의 한계를 말할 때 실전적인 예시가 된다

## 단점
- 개념 자체가 추상적이라 예시 없이 설명하면 상대가 바로 이해하기 어렵다
- DB마다 격리 수준 구현 차이가 있어 설명이 모호해질 수 있다
- 해결책이 항상 하나로 고정되지 않아 상황 맥락을 같이 말해야 한다

## 주의사항 / 실무 팁
- 면접에서는 반드시 당직표, 좌석 예약, 승인 인원 제한 같은 예시를 함께 드는 편이 좋다.
- "같은 row를 안 건드리니 안전하다"는 가정이 왜 틀릴 수 있는지 강조하면 답변이 선명해진다.
- `Snapshot Isolation`, `Serializable`, `낙관적 락`, `제약 조건`과 연결해 설명하면 깊이가 생긴다.
- 실무에서는 규칙을 만족해야 하는 데이터 집합이 무엇인지 먼저 정의하고, 그 범위에 대해 락이나 검증을 적용해야 한다.
