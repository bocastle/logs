# Document Picture-in-Picture API 정리

## 핵심 요약
- `Document Picture-in-Picture API`는 웹페이지 일부를 작은 떠 있는 별도 창으로 띄워 계속 보여줄 수 있게 하는 브라우저 API다.
- 기존 동영상 전용 Picture-in-Picture와 달리, 영상뿐 아니라 HTML 문서 단위 UI를 별도 창에 렌더링할 수 있다는 점이 핵심이다.
- 면접에서는 "미니 플레이어처럼 웹 UI 일부를 독립 창으로 분리해 보여주는 기능"이라고 설명하면 빠르게 전달된다.
- 멀티태스킹 경험을 개선할 수 있지만 브라우저 지원 범위, 상태 동기화, 접근성 고려가 함께 따라온다.

## 개념 설명
기존 Picture-in-Picture는 주로 비디오 요소를 작은 떠 있는 창으로 띄우는 기능으로 알려져 있다. 그런데 실제 서비스에서는 영상만이 아니라 채팅 패널, 회의 컨트롤, 실시간 모니터링 위젯처럼 "계속 봐야 하는 작은 UI" 자체를 분리하고 싶은 요구가 자주 생긴다.

`Document Picture-in-Picture API`는 이런 요구를 해결하기 위해 나온 기능이다. 현재 문서의 일부를 별도 브라우저 창처럼 띄우되, 사용자는 다른 탭이나 앱으로 이동해도 그 작은 창을 계속 볼 수 있다.

중요한 점은 이 창이 단순 이미지가 아니라 별도의 문서 컨텍스트라는 점이다. 그래서 버튼, 텍스트, 스타일 등 일반 DOM UI를 그대로 구성할 수 있다. 대신 원본 페이지와 분리된 창 사이에서 어떤 상태를 공유할지, 닫힐 때 어떻게 정리할지까지 설계해야 한다.

즉, 이 API는 "사용자 멀티태스킹에 맞춰 UI 일부를 별도 창으로 분리하는 도구"로 이해하면 된다.

## 예시
예를 들어 화상 회의 웹앱에서 발표 중인 사용자가 다른 문서를 보면서도 회의 제어 버튼은 항상 보고 싶다고 해 보자.

- 메인 페이지에는 전체 회의 화면이 있다
- 사용자가 "미니 컨트롤 창 열기"를 누른다
- 마이크 음소거, 카메라 토글, 참가자 수 같은 핵심 UI만 작은 창으로 띄운다
- 사용자가 다른 탭으로 이동해도 그 창은 계속 떠 있다

이때 핵심은 "전체 페이지를 복제하는 것"이 아니라 "작게 유지할 가치가 있는 인터랙션만 추려서 별도 창으로 제공하는 것"이다.

## 면접 답변 예시
- 질문: "`Document Picture-in-Picture API`는 무엇인가요?"  
  답변: "웹페이지 일부 UI를 작은 떠 있는 별도 창으로 띄울 수 있게 해 주는 브라우저 API입니다. 기존 영상 중심 Picture-in-Picture를 문서 UI까지 확장한 개념으로 이해하면 됩니다."
- 질문: "어디에 쓰기 좋나요?"  
  답변: "영상 플레이어 컨트롤, 화상 회의 조작 패널, 실시간 대시보드 위젯처럼 사용자가 다른 작업을 하면서도 계속 봐야 하는 UI에 적합합니다."
- 질문: "주의할 점은 무엇인가요?"  
  답변: "브라우저 지원이 아직 제한적일 수 있고, 원본 문서와 별도 창 사이 상태 동기화나 닫힘 처리, 접근성 설계를 함께 고려해야 합니다."

## 장점
- 사용자가 다른 작업을 하면서도 핵심 UI를 계속 볼 수 있다
- 단순 비디오가 아니라 일반 DOM 기반 인터페이스를 띄울 수 있다
- 멀티태스킹 경험이 중요한 서비스에서 차별화된 UX를 만들기 좋다

## 단점
- 브라우저 지원 범위에 제약이 있을 수 있다
- 별도 창과 메인 문서 상태를 맞추는 로직이 필요하다
- 잘못 쓰면 UI가 분산돼 사용자에게 더 혼란스러울 수 있다

## 주의사항 / 실무 팁
- 면접에서는 기존 비디오 PiP와의 차이를 먼저 말하면 이해를 빠르게 얻을 수 있다.
- 어떤 UI를 분리할지 기준이 중요하다. 전체 화면을 옮기기보다 "항상 보여야 하는 핵심 조작부"만 떼는 편이 자연스럽다.
- 별도 창이 닫혔을 때 메인 화면 상태를 어떻게 복구할지까지 설명하면 실무 감각이 드러난다.
- 접근성 측면에서는 포커스 이동, 키보드 조작, 창 전환 경험을 같이 고려해야 한다.
