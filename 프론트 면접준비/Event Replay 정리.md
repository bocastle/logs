# Event Replay 정리

## 핵심 요약
- `Event Replay`는 UI가 아직 완전히 활성화되기 전 사용자가 발생시킨 입력 이벤트를 임시로 보관했다가, 준비가 끝난 뒤 다시 처리하는 개념이다.
- 서버 렌더링과 하이드레이션이 분리된 환경에서 사용자 입력 손실을 줄이는 데 도움이 된다.
- 핵심은 "화면은 먼저 보이지만 아직 완전히 동작하지 않을 수 있는 시간"을 부드럽게 넘기는 것이다.
- 면접에서는 "초기 인터랙션 유실을 줄여 SSR 기반 UX를 안정화하는 전략"이라고 정리하면 좋다.

## 개념 설명
SSR이나 스트리밍 렌더링을 적용하면 사용자는 화면을 빨리 볼 수 있다. 하지만 화면이 보인다고 해서 즉시 모든 인터랙션이 동작하는 것은 아니다. 자바스크립트 번들이 아직 내려오는 중이거나, 특정 컴포넌트의 하이드레이션이 끝나지 않았을 수 있기 때문이다.

이때 사용자가 버튼을 누르거나 입력을 시작하면, 아무 처리도 안 되거나 예상과 다르게 동작할 수 있다. `Event Replay`는 이런 간극을 줄이기 위해 등장한 개념이다. 시스템이 이벤트를 즉시 처리하지 못하는 동안 입력을 기억해 두고, 해당 컴포넌트가 준비되면 원래 순서대로 다시 전달하는 방식이다.

핵심은 단순한 성능 최적화가 아니라 사용자 경험의 연속성을 지키는 데 있다. "보이는 순간부터 만질 수 있을 것 같다"는 사용자 기대를 최대한 깨지 않게 만드는 장치라고 이해하면 쉽다.

## 예시
상품 상세 페이지에서 상단 수량 선택 버튼이 SSR로 먼저 보였다고 가정해보자.

1. 사용자는 페이지가 보이자마자 `+` 버튼을 누른다
2. 하지만 해당 컴포넌트의 하이드레이션은 아직 끝나지 않았다
3. `Event Replay`가 없으면 클릭이 무시되거나 사용자가 다시 눌러야 할 수 있다
4. `Event Replay`가 있으면 클릭 이벤트를 잠시 저장해 두었다가, 준비가 끝난 직후 동일한 이벤트를 다시 처리한다

이렇게 하면 사용자는 "첫 클릭이 먹히지 않았다"는 불편을 덜 느낀다.

## 면접 답변 예시
- 질문: "`Event Replay`는 어떤 상황에서 필요한가요?"  
  답변: "SSR이나 스트리밍 렌더링에서는 화면이 먼저 보이고 인터랙션은 나중에 준비될 수 있습니다. 이때 사용자가 초기에 발생시킨 클릭이나 입력을 임시 보관했다가 하이드레이션 후 다시 처리하는 개념이 `Event Replay`입니다."
- 질문: "장점이 무엇인가요?"  
  답변: "사용자 입장에서는 첫 입력이 사라지지 않는다는 점이 가장 큽니다. 특히 네트워크가 느리거나 번들이 큰 환경에서 초기 인터랙션 손실을 줄여 UX를 더 안정적으로 만들 수 있습니다."

## 장점
- 초기 화면 노출과 인터랙션 준비 사이의 간극을 줄이는 데 유리하다
- SSR, 스트리밍, 부분 하이드레이션 환경에서 체감 UX를 개선하기 좋다
- "보이는데 안 눌리는 화면"이 주는 거부감을 완화할 수 있다

## 단점
- 이벤트 순서와 재실행 타이밍을 잘못 다루면 예측하기 어려운 버그가 생길 수 있다
- 모든 이벤트를 재생하는 것이 안전한 것은 아니어서 제약을 이해해야 한다
- 프레임워크 지원 방식에 따라 개발자가 직접 제어할 수 있는 범위가 제한될 수 있다

## 주의사항 / 실무 팁
- 이벤트 재생은 만능이 아니라, 하이드레이션 지연 자체를 줄이는 최적화와 같이 가야 한다.
- 재생 가능한 이벤트와 그렇지 않은 이벤트를 구분해서 이해하는 편이 좋다.
- 면접에서는 `SSR`, `Hydration`, `Streaming SSR`과 연결해 설명하면 맥락이 살아난다.
- 실무에서는 중요한 CTA 버튼, 폼 입력, 장바구니 같은 초기 상호작용 구간에서 특히 체감 차이가 크다.
