# useEffectEvent 정리

## 핵심 요약
- `useEffectEvent`는 effect 안에서 최신 state와 props를 안전하게 읽고 싶을 때 쓰는 React 패턴이다.
- 핵심은 effect의 재실행 조건과, effect 내부에서 사용하는 최신 값의 참조를 분리하는 것이다.
- 면접에서는 "불필요한 effect 재실행을 줄이면서 stale closure를 피하는 방법"으로 설명하면 좋다.
- 이벤트 핸들러 대체재라기보다, effect 내부 로직을 더 명확하게 나누는 도구에 가깝다.

## 개념 설명
React에서 `useEffect`를 쓰다 보면 두 가지 요구가 자주 충돌한다. 하나는 구독이나 연결은 최소한으로 다시 만들고 싶다는 점이고, 다른 하나는 그 안에서 최신 state나 props를 읽고 싶다는 점이다.

예를 들어 채팅 서버 연결은 `roomId`가 바뀔 때만 다시 만들고 싶을 수 있다. 그런데 연결 성공 후 띄우는 토스트 메시지에는 최신 `theme` 값을 반영하고 싶다면 문제가 생긴다. `theme`를 effect 의존성에 넣으면 연결도 같이 다시 만들어지고, 빼면 오래된 값이 캡처될 수 있다.

`useEffectEvent`는 이런 상황에서 쓴다. effect 안에서 호출할 함수는 따로 두되, 그 함수가 최신 값은 읽을 수 있게 만들어 effect 재실행 조건과 최신 값 참조를 분리한다. 그래서 stale closure를 피하면서도 불필요한 재구독을 줄일 수 있다.

## 예시
채팅방 연결 예시로 보면 이해가 쉽다.

- 나쁜 경우
  - `useEffect` 안에서 연결 생성과 알림 표시를 같이 처리한다.
  - 알림에 `theme`가 필요해서 의존성 배열에 넣는다.
  - 결과적으로 테마만 바뀌어도 소켓 연결이 다시 만들어진다.
- 개선 경우
  - 연결 생성은 `roomId` 기준으로만 effect에서 관리한다.
  - 연결 성공 시 실행할 로직은 `useEffectEvent`로 분리한다.
  - 이렇게 하면 연결은 유지하면서도 알림은 최신 테마 기준으로 보여줄 수 있다.

## 면접 답변 예시
- 질문: "useEffectEvent는 왜 필요한가요?"  
  답변: "effect 안에서 최신 state와 props를 읽고 싶지만, 그 값 때문에 effect 자체가 매번 다시 실행되면 비효율적인 경우가 있습니다. `useEffectEvent`는 이런 상황에서 effect의 재실행 조건과 최신 값 참조를 분리해서 stale closure와 불필요한 재구독 문제를 함께 줄이는 데 도움이 됩니다."
- 질문: "useCallback이나 ref와는 어떻게 다른가요?"  
  답변: "`useCallback`은 함수 identity를 유지하는 데 초점이 있고, `ref`는 직접 최신 값을 넣어 관리해야 해서 코드 의도가 흐려질 수 있습니다. `useEffectEvent`는 effect 내부에서 쓸 최신 로직이라는 의미를 더 명확히 드러내는 장점이 있습니다."

## 장점
- stale closure 문제를 설명할 때 실무적인 대안을 제시하기 좋다
- 구독, 연결, 타이머 같은 effect성 로직을 더 명확하게 나눌 수 있다
- 의존성 배열을 억지로 늘리거나 줄이는 코드를 줄이는 데 도움이 된다

## 단점
- 일반 이벤트 핸들러와 effect event의 역할을 혼동하면 오히려 코드가 헷갈릴 수 있다
- React 버전과 팀의 사용 패턴에 따라 아직 낯선 개념일 수 있다
- 모든 effect 문제를 해결하는 만능 도구처럼 쓰면 설계가 흐트러질 수 있다

## 주의사항 / 실무 팁
- 정말 effect 내부에서만 필요한 로직인지 먼저 구분하는 것이 중요하다.
- "최신 값을 읽고 싶다"와 "effect를 다시 실행해야 한다"는 요구를 분리해서 생각하면 면접 답변이 좋아진다.
- 실무에서는 `ref`로 우회하던 코드를 더 읽기 쉽게 바꾸는 용도로 설명하면 설득력이 있다.
- 면접에서는 "의존성 배열을 속이는 도구"가 아니라 "effect의 책임을 분리하는 도구"라고 표현하는 편이 안전하다.
