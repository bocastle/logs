# Error Boundary 정리

## 핵심 요약

- `Error Boundary`는 React 컴포넌트 트리에서 렌더링 중 발생한 에러를 잡아서 화면 전체가 깨지는 것을 막는 장치다.
- 주로 `대체 UI를 보여 주고, 에러를 기록하고, 장애 범위를 좁히는 역할`로 설명하면 된다.
- 면접에서는 "비동기 에러나 이벤트 핸들러 에러까지 다 잡는 만능 장치가 아니다"라는 한계를 같이 말해야 답변이 탄탄해진다.
- `Suspense`가 로딩 상태를 다룬다면, `Error Boundary`는 렌더링 단계의 실패를 다룬다고 구분하면 기억하기 쉽다.

## Error Boundary가 뭐냐

React 앱에서 어떤 컴포넌트가 렌더링 도중 예외를 던지면, 예전에는 트리 전체가 깨지거나 빈 화면이 보이기도 했다.  
`Error Boundary`는 이런 에러를 상위 경계에서 잡아서, "문제가 난 영역만 대체 UI로 바꾸는 방식"으로 복구할 수 있게 해 준다.

즉 핵심은 에러를 없애는 기능이 아니라, `에러가 났을 때 사용자 경험과 장애 범위를 관리하는 장치`라는 점이다.

## 왜 필요한가

- 일부 위젯이나 페이지 섹션만 실패해도 전체 화면이 무너지는 일을 줄일 수 있다.
- 사용자에게 "문제가 생겼다"는 메시지와 재시도 수단을 줄 수 있다.
- 에러 로깅 시스템과 연결해 어디서 문제가 났는지 추적하기 좋다.

면접에서는 "프론트엔드도 장애 격리가 필요하다"는 관점으로 풀어내면 좋다.

## 언제 잡을 수 있나

`Error Boundary`는 주로 아래 같은 상황을 잡는다.

- 자식 컴포넌트의 렌더링 중 에러
- 생명주기 메서드에서 발생한 에러
- constructor 단계에서 발생한 에러

반대로 아래는 기본적으로 잡지 못한다.

- 이벤트 핸들러 내부 에러
- `setTimeout`, `Promise`, `async/await` 같은 비동기 에러
- 서버 사이드 렌더링 단계의 에러
- 경계 컴포넌트 자기 자신에서 난 에러

그래서 "모든 에러를 한 번에 처리한다"라고 답하면 과장된 설명이 된다.

## 어디에 두는 게 좋나

보통은 앱 전체를 감싸는 전역 경계 하나와, 장애 영향이 큰 구간을 나누는 지역 경계를 같이 쓴다.

- 전역 경계: 앱이 완전히 죽는 걸 막는다.
- 지역 경계: 채팅창, 대시보드 카드, 결제 위젯처럼 독립적인 영역만 따로 보호한다.

경계를 너무 잘게 쪼개면 구조가 복잡해지고, 너무 크게 두면 작은 에러도 페이지 전체를 대체 UI로 바꿔 버릴 수 있다.

## Suspense와는 뭐가 다르냐

둘 다 fallback UI를 보여 줄 수 있어서 헷갈리기 쉽다.

- `Suspense`: 아직 준비되지 않은 상태, 즉 로딩을 다룬다.
- `Error Boundary`: 실패한 상태, 즉 에러를 다룬다.

실무에서는 로딩과 실패를 함께 다뤄야 하므로 둘을 인접하게 배치하는 경우가 많다.

## 간단한 예시

```tsx
class SectionErrorBoundary extends React.Component<
  { children: React.ReactNode },
  { hasError: boolean }
> {
  constructor(props: { children: React.ReactNode }) {
    super(props);
    this.state = { hasError: false };
  }

  static getDerivedStateFromError() {
    return { hasError: true };
  }

  componentDidCatch(error: Error) {
    console.error("section render failed", error);
  }

  render() {
    if (this.state.hasError) {
      return <div>이 영역을 불러오는 중 문제가 생겼습니다.</div>;
    }

    return this.props.children;
  }
}
```

면접에서는 코드를 외우기보다 "`getDerivedStateFromError`로 대체 UI 상태를 만들고, `componentDidCatch`에서 로깅한다" 정도로 설명하면 충분하다.

## 면접 답변 예시

> Error Boundary는 React에서 렌더링 과정 중 발생한 에러를 상위 컴포넌트 경계에서 잡아서, 화면 전체가 깨지는 대신 대체 UI를 보여 주도록 만드는 장치입니다. 주로 장애 범위를 줄이고 에러 로깅을 붙이기 위해 사용합니다. 다만 이벤트 핸들러나 비동기 코드 에러까지 전부 잡는 건 아니고, 렌더링 계층의 에러를 다루는 데 강점이 있습니다. 그래서 저는 전역 경계 하나와, 실패해도 다른 영역에 영향이 적어야 하는 주요 섹션 경계를 나눠서 두는 방식으로 설명하는 편입니다.

## 장점

- 프론트엔드 장애 대응을 사용자 경험 관점으로 설명하기 좋다.
- `Suspense`, 로깅, 장애 격리까지 연결해서 답변을 확장하기 쉽다.
- "에러가 나도 전체 앱은 살린다"는 운영 관점을 보여 주기 좋다.

## 단점

- 잡을 수 있는 범위를 정확히 모르면 만능 에러 처리처럼 말하기 쉽다.
- 클래스 컴포넌트 기반 개념이라 익숙하지 않으면 설명이 끊길 수 있다.
- 경계 위치를 잘못 잡으면 대체 UI 범위가 너무 넓거나 너무 잘게 쪼개질 수 있다.

## 주의사항 / 실무 팁

- 면접에서는 `렌더링 에러를 잡는 장치`라고 먼저 정의하면 설명이 흔들리지 않는다.
- 이벤트 핸들러와 비동기 에러는 별도 처리 전략이 필요하다고 덧붙이면 좋다.
- `Suspense는 로딩`, `Error Boundary는 실패`로 구분해서 말하면 전달이 빠르다.
- 실제 서비스에서는 대체 UI만 보여 주지 말고 로깅과 모니터링까지 연결한다고 말하면 실무 감각이 난다.
