# Trusted Types 정리

## 핵심 요약
- `Trusted Types`는 XSS를 줄이기 위해 브라우저가 위험한 문자열 주입 지점을 더 엄격하게 다루도록 만드는 보안 메커니즘이다.
- 핵심은 `innerHTML`, `script src`, `eval` 계열처럼 문자열 기반으로 DOM이나 스크립트를 주입하는 지점을 통제하는 데 있다.
- 면접에서는 "문자열을 바로 넣지 못하게 하고, 검증된 타입만 넣게 강제하는 정책"이라고 설명하면 전달이 빠르다.
- CSP와 함께 쓰는 경우가 많고, 레거시 코드가 많을수록 도입 난도가 올라간다는 점까지 같이 말하면 답변이 균형 잡힌다.

## 개념 설명
프론트엔드에서 XSS가 자주 생기는 이유는 결국 "신뢰할 수 없는 문자열이 브라우저가 실행 가능한 형태로 들어가는 순간"이 있기 때문이다. 예를 들어 사용자 입력을 그대로 `innerHTML`에 넣거나, 외부 데이터를 조합해서 스크립트 URL을 만드는 방식이 대표적이다.

`Trusted Types`는 이런 문제를 줄이기 위해 브라우저 차원에서 "아무 문자열이나 위험한 API에 넣지 못하게" 제한한다. 대신 개발자가 직접 만든 정책(policy)을 통과해 생성된 `TrustedHTML`, `TrustedScriptURL` 같은 신뢰 가능한 타입만 허용한다.

즉, 보안 검사를 코드 리뷰나 개발자 습관에만 맡기지 않고, 런타임 정책으로 한 번 더 강제하는 셈이다. 이 점 때문에 대규모 프론트엔드 애플리케이션이나 여러 팀이 동시에 수정하는 서비스에서 특히 의미가 있다.

다만 `Trusted Types`가 XSS를 자동으로 전부 해결해 주는 것은 아니다. 어디서 어떤 정책으로 문자열을 정제할지 정해야 하고, 서드파티 라이브러리가 문자열 주입 방식에 의존하면 마이그레이션 작업이 필요하다.

## 예시
예를 들어 댓글 내용을 강조 표시하려고 아래처럼 구현했다고 해 보자.

- 사용자가 입력한 문자열을 서버에서 받아 온다
- 특정 키워드를 강조하려고 HTML 태그를 조합한다
- 그 결과를 `element.innerHTML = result`로 바로 넣는다

이 방식은 편하지만 XSS 위험이 있다. `Trusted Types`를 적용하면 단순 문자열은 바로 넣을 수 없고, 미리 정의한 정책을 거쳐 `TrustedHTML` 객체로 변환한 값만 넣을 수 있다. 그래서 "어떤 정제 로직을 거쳤는지"가 코드 상에서 더 분명해진다.

## 면접 답변 예시
- 질문: "`Trusted Types`는 무엇인가요?"  
  답변: "`Trusted Types`는 브라우저가 위험한 DOM 주입 지점에 임의 문자열이 들어가는 것을 막고, 검증된 타입만 허용하게 하는 보안 메커니즘입니다. XSS를 줄이는 데 목적이 있습니다."
- 질문: "왜 필요한가요?"  
  답변: "프론트엔드에서는 `innerHTML` 같은 API를 잘못 쓰면 XSS가 발생하기 쉽습니다. `Trusted Types`는 이런 부분을 개발자 주의에만 맡기지 않고 런타임 정책으로 강제할 수 있다는 장점이 있습니다."
- 질문: "도입할 때 어려운 점은 무엇인가요?"  
  답변: "레거시 코드나 서드파티 라이브러리가 문자열 주입을 많이 쓰고 있으면 수정 범위가 커질 수 있습니다. 그래서 보통 CSP와 함께 단계적으로 적용합니다."

## 장점
- XSS가 자주 생기는 문자열 주입 지점을 구조적으로 통제할 수 있다
- 보안 검증 흐름이 정책 객체 중심으로 명확해진다
- 대규모 코드베이스에서 위험한 API 사용을 일관되게 관리하기 좋다

## 단점
- 레거시 코드가 많으면 도입 비용이 크다
- 서드파티 라이브러리와 충돌할 수 있다
- 정책 설계를 잘못하면 우회 지점이 남거나 개발 생산성이 떨어질 수 있다

## 주의사항 / 실무 팁
- `Trusted Types`는 보통 CSP와 같이 설명하는 편이 자연스럽다.
- 면접에서는 "`innerHTML` 같은 위험한 API에 아무 문자열이나 못 넣게 한다"는 문장으로 시작하면 이해가 빠르다.
- 실무에서는 먼저 어떤 API가 위험 지점인지 인벤토리화하고, 점진적으로 정책을 적용하는 방식이 현실적이다.
- "보안을 강화한다"는 말만 하지 말고, 레거시 마이그레이션 비용과 서드파티 호환성 문제를 같이 언급하면 답변이 더 실무적으로 들린다.
