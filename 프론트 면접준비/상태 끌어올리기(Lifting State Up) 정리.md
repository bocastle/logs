# 상태 끌어올리기(Lifting State Up) 정리

## 핵심 요약

- 상태 끌어올리기는 여러 하위 컴포넌트가 같은 데이터를 써야 할 때 공통 부모로 state를 올려서 관리하는 방식이다.
- 핵심 목적은 데이터의 출처를 하나로 만들고, 화면 간 불일치를 줄이는 데 있다.
- React 면접에서는 "왜 필요한지", "Prop Drilling과의 관계", "언제 Context나 전역 상태로 넘어가는지"까지 이어서 설명하면 좋다.
- 무조건 위로 올리는 게 답은 아니고, 정말 공유가 필요한 최소 범위까지만 올리는 감각이 중요하다.

## 상태 끌어올리기가 무엇인가

React에서는 보통 state를 그 값을 가장 잘 아는 컴포넌트 가까이에 둔다.  
그런데 형제 컴포넌트 둘 이상이 같은 값을 기준으로 움직여야 하는 순간이 온다.

예를 들어 이런 경우다.

- 검색어 입력창과 검색 결과 목록이 같은 키워드를 봐야 한다.
- 섭씨 입력창과 화씨 입력창이 같은 온도를 다른 표현으로 보여줘야 한다.
- 체크박스 선택 상태를 상단 요약 영역과 하단 리스트가 같이 써야 한다.

이럴 때 각각이 state를 따로 가지면 값이 쉽게 어긋난다.  
그래서 공통 부모가 state를 가지고, 자식들에게 props로 내려주는 방식이 상태 끌어올리기다.

## 왜 필요한가

핵심은 single source of truth, 즉 진실의 원천을 하나로 만드는 것이다.

같은 데이터를 여러 곳에서 각자 관리하면 동기화 코드가 늘어나고 버그가 생기기 쉽다.  
반대로 부모 한 곳에서만 값을 관리하면 누가 값을 바꾸고 누가 그 값을 보여주는지가 분명해진다.

면접에서는 아래처럼 말하면 자연스럽다.

- 공유 데이터는 한 곳에서 관리해야 예측 가능성이 높아진다.
- 형제 컴포넌트 간 직접 의존보다 부모를 통한 전달이 흐름을 이해하기 쉽다.
- 디버깅할 때도 상태 출처가 하나면 훨씬 편하다.

## 간단한 흐름

상태 끌어올리기는 보통 이런 식으로 설명할 수 있다.

1. 여러 자식이 공통으로 써야 하는 값을 찾는다.
2. 그 값을 가장 가까운 공통 부모의 state로 올린다.
3. 자식은 props로 값을 받는다.
4. 값 변경이 필요하면 부모가 내려준 함수로 요청한다.

즉, 데이터는 위에서 아래로 흐르고, 이벤트는 아래에서 위로 올라오는 구조다.

## 예시로 이해하기

예를 들어 검색 입력창과 검색 결과 컴포넌트가 있다고 하자.

- `SearchInput`은 사용자가 입력한 값을 바꾸는 역할
- `SearchResults`는 현재 검색어로 결과를 보여주는 역할

이 둘이 각각 검색어를 따로 들고 있으면 한쪽만 갱신될 수 있다.  
그래서 부모인 `SearchPage`가 `query` state를 가지고, 입력창에는 값과 변경 함수를 내려주고, 결과 목록에는 현재 `query`만 내려준다.

이 구조가 되면 어떤 컴포넌트가 값을 읽고 바꾸는지 분리가 명확해진다.

## Prop Drilling과의 관계

상태 끌어올리기를 말하다 보면 바로 나오는 질문이 Prop Drilling이다.

공통 부모가 가까우면 문제가 없다.  
하지만 상태를 너무 높은 곳으로 올리면 중간 컴포넌트들이 쓰지도 않는 props를 계속 전달하게 된다.

이때는 선택지가 생긴다.

- 상태 범위를 다시 더 작게 나눈다.
- Context API를 쓴다.
- 전역 상태 관리 도구를 쓴다.

즉, 상태 끌어올리기는 기본 해법이고, 트리가 깊어져 전달 비용이 커질 때 다른 도구를 검토하는 흐름으로 이해하면 된다.

## 언제 과한가

모든 상태를 위로 올리면 오히려 부모가 비대해지고, 불필요한 렌더링 범위도 넓어진다.

예를 들어 특정 입력 컴포넌트 안에서만 쓰는 임시 포커스 상태나 열림 여부까지 전부 상위로 올리면 관리가 더 복잡해질 수 있다.

그래서 실무에서는 "공유가 필요한 최소 공통 조상까지만 올린다"는 기준이 중요하다.

## 면접 답변 예시

> 상태 끌어올리기는 여러 컴포넌트가 같은 데이터를 공유해야 할 때 그 state를 가장 가까운 공통 부모로 올려서 관리하는 방식입니다. 이렇게 하면 데이터 출처가 하나가 돼서 형제 컴포넌트 간 값 불일치를 줄일 수 있습니다. 부모가 state를 가지고 자식에게 값과 변경 함수를 props로 내려주는 구조가 기본이고, React의 단방향 데이터 흐름과도 잘 맞습니다. 다만 상태를 너무 높은 곳으로 올리면 Prop Drilling이 심해질 수 있어서, 공유 범위가 넓거나 트리가 깊어지면 Context나 전역 상태 관리 도구를 같이 검토합니다.

## 장점

- 데이터 출처가 하나라서 상태 일관성을 유지하기 쉽다.
- 컴포넌트 간 책임이 비교적 명확해진다.
- 디버깅과 테스트가 쉬워진다.

## 단점

- 부모 컴포넌트가 비대해질 수 있다.
- props 전달이 깊어지면 Prop Drilling 문제가 생긴다.
- 상태 범위를 잘못 잡으면 불필요한 리렌더링이 늘 수 있다.

## 주의사항 / 실무 팁

- 상태를 올리기 전에 정말 "공유 상태"인지 먼저 구분하는 게 좋다.
- 가장 가까운 공통 부모까지만 올리고, 필요 이상으로 루트까지 끌어올리지 않는 편이 낫다.
- props가 여러 단계로 길게 내려가면 Context, 전역 상태, 컴포넌트 분리 중 무엇이 더 단순한지 다시 판단해야 한다.
- 면접에서는 "single source of truth"와 "최소 공통 조상"이라는 표현을 같이 쓰면 답변이 단단해 보인다.
