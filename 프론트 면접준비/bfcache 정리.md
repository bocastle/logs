# bfcache 정리

## 핵심 요약
- `bfcache`는 브라우저가 페이지 전체 상태를 메모리에 보관했다가 뒤로 가기/앞으로 가기 때 즉시 복원하는 기능이다.
- 일반 HTTP 캐시와 달리 HTML, JS 상태, 스크롤 위치까지 통째로 되살리는 것이 핵심이다.
- 면접에서는 "사용자 체감 속도를 크게 개선하지만, 특정 이벤트 리스너나 페이지 생명주기 코드가 있으면 bfcache 적중률이 떨어질 수 있다"라고 설명하면 좋다.
- 성능 최적화 관점에서는 네트워크 최적화만큼이나 브라우저 복원 흐름을 이해하는 것이 중요하다.

## 개념 설명
사용자가 목록 페이지에서 상세 페이지로 갔다가 다시 뒤로 가기를 눌렀는데, 목록이 거의 즉시 복원되는 경험을 해본 적이 있다. 이때 자주 작동하는 것이 `bfcache`다.

`bfcache`는 브라우저가 이전 페이지를 단순히 다시 요청하는 것이 아니라, 페이지의 실행 상태를 메모리에 보관했다가 그대로 복원하는 방식이다. 그래서 스크롤 위치, 입력 중이던 값, 일부 자바스크립트 상태까지 빠르게 살아날 수 있다.

HTTP 캐시는 정적 리소스나 응답 데이터를 다시 쓰는 개념에 가깝고, `bfcache`는 페이지 세션 자체를 되돌리는 개념에 가깝다. 그래서 둘은 비슷해 보여도 역할이 다르다. 면접에서는 "캐시된 파일을 다시 읽는 것"과 "페이지 상태를 통째로 복원하는 것"을 구분해서 말하면 이해도가 높아 보인다.

## 예시
전자상거래 서비스에서 상품 목록 페이지를 본 뒤 상품 상세로 이동했다고 가정해보자.

- `bfcache`가 잘 동작하면
  - 뒤로 가기 시 목록이 거의 즉시 나타난다.
  - 사용자가 보고 있던 스크롤 위치도 유지된다.
  - 다시 API 요청과 렌더링을 처음부터 하지 않아도 된다.
- `bfcache`가 막히면
  - 목록 페이지를 새로 불러오면서 흰 화면이나 로딩이 보일 수 있다.
  - 스크롤 위치가 초기화될 수 있다.
  - 사용자는 "뒤로 갔는데 왜 다시 느리게 뜨지?"라고 느낄 수 있다.

## 면접 답변 예시
- 질문: "bfcache는 무엇인가요?"  
  답변: "`bfcache`는 브라우저가 방문했던 페이지의 상태를 메모리에 보관했다가 뒤로 가기나 앞으로 가기 시 즉시 복원하는 기능입니다. 일반 캐시가 리소스를 재사용하는 개념이라면, bfcache는 페이지 실행 상태 자체를 통째로 복원한다는 점이 다릅니다."
- 질문: "왜 프론트엔드 면접에서 중요한가요?"  
  답변: "사용자 체감 성능에 직접 영향을 주기 때문입니다. 뒤로 가기 복원이 빠르면 UX가 좋아지고, 반대로 불필요한 이벤트 처리나 생명주기 코드 때문에 bfcache가 깨지면 페이지가 매번 다시 로드돼 성능이 떨어질 수 있습니다."

## 장점
- 뒤로 가기와 앞으로 가기 경험이 매우 빨라진다
- 스크롤 위치와 페이지 상태가 유지돼 사용자 경험이 좋아진다
- 불필요한 재렌더링과 재요청을 줄여 체감 성능을 높일 수 있다

## 단점
- 브라우저별 동작 차이와 제약을 이해해야 한다
- 페이지 종료 시점에 무거운 정리 로직을 넣어두면 오히려 복원 최적화를 해칠 수 있다
- 개발자가 HTTP 캐시와 혼동하면 원인 분석이 꼬이기 쉽다

## 주의사항 / 실무 팁
- `unload` 이벤트 사용은 bfcache 적중률을 떨어뜨릴 수 있어서 피하는 편이 좋다.
- 페이지가 다시 열리는 것이 아니라 "복원"될 수 있다는 전제를 두고 생명주기 코드를 설계해야 한다.
- 면접에서는 `pageshow`, `pagehide` 같은 이벤트를 함께 언급하면 이해도가 높아 보인다.
- 실무에서는 뒤로 가기 UX가 중요한 목록/상세 구조에서 특히 효과가 크다고 설명하면 좋다.
