# useTransition 정리

## 핵심 요약

- `useTransition`은 급한 업데이트와 덜 급한 업데이트를 나눠서 사용자 입력 반응성을 지키도록 돕는 React 훅이다.
- 검색 필터링, 탭 전환, 큰 리스트 렌더링처럼 화면 갱신 비용이 큰 경우에 특히 설명하기 좋다.
- 모든 상태 업데이트를 빠르게 만드는 기능이 아니라, 우선순위를 나누는 도구에 가깝다.
- 면접에서는 "입력은 바로 받고, 무거운 화면 갱신은 조금 뒤로 미룬다"라고 말하면 이해가 빠르다.

## useTransition이 왜 필요할까?

React에서 상태를 바꾸면 화면이 다시 렌더링된다.  
그런데 어떤 업데이트는 즉시 보여야 하고, 어떤 업데이트는 아주 약간 늦어도 사용자가 불편하지 않다.

예를 들어 검색창에 글자를 입력할 때 입력창 반응은 바로 와야 한다.  
반면 입력할 때마다 수천 개 리스트를 다시 그리는 작업은 약간 뒤로 밀어도 된다.

이럴 때 모든 업데이트를 같은 급으로 처리하면 타이핑이 버벅이는 느낌이 날 수 있다.  
`useTransition`은 이런 상황에서 덜 급한 렌더링을 transition으로 표시해 우선순위를 조정한다.

## 기본 개념

`useTransition`을 쓰면 보통 두 가지를 받는다.

- `isPending`: transition 작업이 아직 끝나지 않았는지 알려 준다.
- `startTransition`: 덜 급한 상태 업데이트를 감싸는 함수다.

```jsx
const [isPending, startTransition] = useTransition();

function handleChange(e) {
  const next = e.target.value;
  setKeyword(next);

  startTransition(() => {
    setFilteredKeyword(next);
  });
}
```

이 예시에서는 입력값 자체는 바로 반영하고, 리스트 필터링에 쓰는 값은 transition으로 넘긴다.

## 어떤 상황에 어울릴까?

아래처럼 "사용자 반응성"과 "무거운 렌더링"이 같이 있는 경우가 대표적이다.

- 검색어 입력과 결과 목록 필터링
- 탭 전환 시 큰 영역 다시 그리기
- 정렬 기준 변경 후 복잡한 리스트 재구성
- 차트나 표가 많은 화면 전환

반대로 단순한 폼 입력이나 가벼운 컴포넌트에는 굳이 쓸 필요가 없다.

## useDeferredValue와 차이는?

둘 다 급한 값과 덜 급한 값을 분리하는 데 쓰이지만 결이 조금 다르다.

- `useTransition`: 상태 업데이트를 "이건 덜 급하다"라고 직접 표시한다.
- `useDeferredValue`: 이미 있는 값을 조금 늦게 따라가게 만든다.

면접에서는 "`useTransition`은 업데이트를 감싸서 우선순위를 낮추고, `useDeferredValue`는 값 소비를 늦춘다" 정도로 구분하면 충분하다.

## 면접에서 자주 나오는 포인트

중요한 건 `useTransition`이 비동기 요청 자체를 늦추는 기능은 아니라는 점이다.  
핵심은 React 렌더링 우선순위를 조정해 입력 경험을 지키는 것이다.

즉, 네트워크 속도를 빠르게 만드는 도구가 아니라, 화면이 무거울 때 "무엇을 먼저 부드럽게 보여 줄지"를 정하는 도구라고 보는 편이 맞다.

## 면접 답변 예시

> useTransition은 React에서 우선순위가 낮은 상태 업데이트를 구분할 수 있게 해 주는 훅입니다. 예를 들어 검색창 입력은 즉시 반영해야 하지만, 그 입력을 바탕으로 큰 리스트를 다시 렌더링하는 작업은 조금 늦어도 괜찮습니다. 이때 입력 상태는 바로 업데이트하고, 무거운 리스트 갱신은 startTransition으로 감싸면 사용자는 덜 버벅이는 입력 경험을 얻게 됩니다. 그래서 저는 useTransition을 성능 최적화라기보다 사용자 반응성을 지키기 위한 우선순위 제어 도구로 설명하는 편입니다.

## 장점

- 입력 반응성과 무거운 화면 갱신을 분리해 UX를 개선하기 좋다.
- React concurrent rendering 개념을 실무 예시와 함께 설명하기 좋다.
- `isPending`으로 로딩 감각을 부드럽게 표현할 수 있다.

## 단점

- 렌더링 비용 자체를 없애는 건 아니라서 근본 구조가 나쁘면 한계가 있다.
- 어디까지를 transition으로 둘지 기준이 없으면 오히려 흐름이 복잡해진다.
- 작은 화면이나 단순한 상태에서는 체감 이점이 거의 없다.

## 주의사항 / 실무 팁

- 먼저 실제 병목이 큰 렌더링인지 확인하고 적용하는 편이 낫다.
- 입력값과 결과 렌더링 값을 분리하는 패턴을 같이 이해해야 설명이 자연스럽다.
- 면접에서는 "사용자 입력은 urgent, 큰 렌더링은 non-urgent"라는 표현을 쓰면 핵심이 잘 전달된다.
