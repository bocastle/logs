# CSS @scope 정리

## 핵심 요약
- `@scope`는 특정 DOM 범위 안에서만 CSS 규칙이 적용되도록 제한하는 기능이다.
- 면접에서는 전역 CSS 충돌, 명시도 관리, 컴포넌트 경계라는 세 가지 관점으로 설명하면 깔끔하다.
- Shadow DOM처럼 완전한 캡슐화는 아니지만, 스타일 적용 범위를 선언적으로 줄일 수 있다는 점이 핵심이다.
- 브라우저 지원과 폴백 전략까지 같이 언급하면 실무 감각이 좋아 보인다.

## 개념 설명
기존 CSS는 선택자를 잘 작성해도 전역 문서 전체를 기준으로 해석된다. 그래서 프로젝트가 커질수록 "이 스타일이 어디까지 영향을 주는지"를 추적하기 어려워진다.

`@scope`는 이런 문제를 줄이기 위해 등장한 기능이다. 시작 루트와 필요하면 종료 경계를 지정해서, 그 안에서만 스타일 규칙이 유효하도록 만든다.

즉, `@scope`는 "선택자를 더 복잡하게 쓰는 것"이 아니라 "스타일이 살아 있는 범위를 먼저 선언하는 것"에 가깝다.

비슷한 개념과 비교하면 다음처럼 정리할 수 있다.
- BEM: 네이밍 규칙으로 충돌을 줄인다
- CSS Modules: 빌드 단계에서 클래스명을 지역화한다
- Shadow DOM: 브라우저 차원에서 강한 스타일 캡슐화를 제공한다
- `@scope`: 표준 CSS 문법 안에서 적용 범위를 제한한다

## 예시
```css
@scope (.profile-card) {
  .title {
    font-size: 20px;
    font-weight: 700;
  }

  .description {
    color: #555;
  }
}
```

이 경우 `.title`, `.description`이라는 클래스명이 문서 어디에 있든 전부 바뀌는 것이 아니라, `.profile-card` 내부에 있을 때만 적용된다.

조금 더 엄격하게 경계를 줄 수도 있다.

```css
@scope (.dialog) to (.dialog-footer) {
  button {
    background: black;
    color: white;
  }
}
```

이런 식이면 특정 다이얼로그 영역 안에서만 버튼 규칙을 관리하기 쉬워진다.

## 면접 답변 예시
- 질문: "`@scope`는 왜 필요한가요?"  
  답변: "`@scope`는 CSS 규칙이 적용되는 DOM 범위를 제한해서 전역 충돌을 줄이기 위한 기능입니다. 프로젝트가 커지면 같은 클래스명이나 선택자 패턴이 여러 화면에서 재사용되는데, 이때 범위를 선언적으로 묶어 두면 명시도 경쟁을 덜 만들고 유지보수가 쉬워집니다."
- 질문: "CSS Modules나 Shadow DOM이 있는데도 `@scope`가 의미가 있나요?"  
  답변: "네, 다릅니다. CSS Modules는 빌드 도구에 의존하고 Shadow DOM은 캡슐화가 강한 대신 전역 테마 연동이 불편할 수 있습니다. `@scope`는 표준 CSS 안에서 범위를 제한하는 방식이라 점진적으로 도입하기 좋다는 장점이 있습니다."

## 장점
- 스타일 영향 범위를 줄여 전역 충돌을 완화할 수 있다
- 선택자를 과하게 복잡하게 만들지 않아도 된다
- 컴포넌트 단위로 CSS 의도를 읽기 쉬워진다

## 단점
- 브라우저 지원 범위를 반드시 확인해야 한다
- 완전한 캡슐화가 아니므로 외부 스타일과의 상호작용을 완전히 차단하지는 못한다
- 팀이 개념에 익숙하지 않으면 기존 CSS 구조와 혼용할 때 혼란이 생길 수 있다

## 주의사항 / 실무 팁
- `@scope`를 도입한다고 해서 명시도 문제가 자동으로 사라지는 것은 아니므로, 기존 레이어 구조나 네이밍 규칙과 함께 봐야 한다.
- 디자인 시스템 컴포넌트처럼 재사용 범위가 분명한 구간부터 점진적으로 적용하는 편이 안전하다.
- 면접에서는 "전역 CSS 충돌을 줄이는 표준 기능"이라고 먼저 말하고, CSS Modules나 Shadow DOM과의 차이를 짧게 덧붙이면 전달력이 좋다.
