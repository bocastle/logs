# useSyncExternalStore와 tearing 정리

## 핵심 요약
- `tearing`은 같은 순간을 보고 있어야 하는 UI 조각들이 서로 다른 상태를 보여 주는 문제다.
- React에서 외부 스토어를 직접 구독할 때 이 문제가 생길 수 있고, 이를 안정적으로 맞추기 위한 도구가 `useSyncExternalStore`다.
- 면접에서는 "`useSyncExternalStore`는 외부 상태를 React 렌더링 타이밍과 안전하게 맞춰 읽기 위한 훅"이라고 설명하면 정리가 잘 된다.
- 특히 동시성 렌더링 환경에서 "구독은 했지만 언제 어떤 값을 읽는가"가 중요하다는 점까지 말하면 깊이가 생긴다.

## 개념 설명
React의 `useState`, `useReducer`, Context처럼 React 내부 상태만 쓰면 렌더링 시점에 어떤 값이 보이는지가 비교적 일관적이다. 그런데 Zustand 같은 외부 스토어, 브라우저 API, 커스텀 pub/sub 저장소처럼 React 바깥에 있는 상태를 직접 읽기 시작하면 이야기가 달라진다.

이때 문제가 되는 것이 `tearing`이다. 예를 들어 화면의 A 컴포넌트와 B 컴포넌트가 같은 전역 상태를 보고 있어야 하는데, 렌더링 중간에 외부 값이 바뀌면서 A는 이전 값, B는 다음 값을 보게 되면 한 화면 안에서 상태가 찢어진 것처럼 보일 수 있다. 사용자는 같은 시점의 화면인데도 숫자나 텍스트가 서로 안 맞는다고 느낀다.

`useSyncExternalStore`는 이런 문제를 줄이기 위해 React가 외부 스토어를 읽는 방식을 표준화한 훅이다. 핵심은 구독 함수와 현재 스냅샷을 읽는 함수를 분리해서, React가 렌더링 중 외부 상태를 일관된 방식으로 읽도록 돕는 것이다. 즉 단순히 "스토어가 바뀌면 다시 렌더링"하는 수준이 아니라, "렌더링 시점에 어떤 값을 기준으로 UI를 그릴 것인가"를 안정적으로 맞추는 역할이 있다.

## 예시
예를 들어 실시간 주식 가격을 보여 주는 화면이 있다고 하자.

- 상단 요약 카드와 하단 차트가 같은 전역 스토어의 가격 데이터를 본다
- 렌더링 도중 실시간 소켓 메시지로 스토어 값이 갱신된다
- 잘못 구독하면 상단은 10,100원을, 하단은 10,120원을 동시에 보여 줄 수 있다

이런 상태가 바로 `tearing`에 가깝다. 반면 `useSyncExternalStore`를 사용하면 React가 렌더링 타이밍에 같은 스냅샷을 기준으로 읽도록 도와서 화면 일관성을 높일 수 있다.

## 면접 답변 예시
- 질문: "`useSyncExternalStore`는 왜 필요한가요?"  
  답변: "React 외부에 있는 스토어를 구독할 때 렌더링 시점과 상태 읽기 시점이 어긋나면 같은 화면 안에서도 서로 다른 값이 보일 수 있습니다. `useSyncExternalStore`는 이런 외부 상태를 React와 안전하게 연결해서 일관된 스냅샷 기준으로 렌더링하도록 돕는 훅입니다."
- 질문: "`tearing`이 무엇인가요?"  
  답변: "동일한 상태를 봐야 하는 UI 일부가 렌더링 중간의 상태 변화 때문에 서로 다른 값을 보여 주는 현상입니다. 특히 외부 스토어와 동시성 렌더링이 섞일 때 주의해야 합니다."

## 장점
- React 내부 상태와 외부 상태 관리의 차이를 명확하게 설명하기 좋다
- 동시성 렌더링, 상태 일관성, 스토어 구독 설계를 함께 말할 수 있다
- 단순 훅 암기가 아니라 React 렌더링 모델 이해도를 보여 주기 좋다

## 단점
- 실무에서 라이브러리가 감춰 주는 경우가 많아 직접 체감하지 못하면 설명이 추상적으로 들릴 수 있다
- `useSyncExternalStore` 사용법만 외우고 왜 필요한지 모르면 답변이 얕아진다
- `tearing`, stale closure, 불필요한 리렌더링을 한꺼번에 섞어 말하면 주제가 흐려질 수 있다

## 주의사항 / 실무 팁
- 모든 전역 상태가 곧바로 `tearing` 문제를 만든다고 말하면 과장될 수 있다. 외부 스토어와 렌더링 타이밍 이슈를 함께 묶어 설명하는 게 정확하다.
- Zustand, Redux 같은 라이브러리도 내부적으로 이런 문제를 어떻게 다루는지 한 번 이해해 두면 면접 답변이 단단해진다.
- 면접에서는 `useDeferredValue`, `useTransition`, `bfcache` 같은 성능 주제와는 구분해서, 이 글은 "반응 속도"보다 "상태 일관성"에 가깝다고 정리하면 좋다.
- 실무에서는 외부 스토어를 직접 만들 경우 `subscribe`, `getSnapshot`의 안정성을 먼저 점검해야 한다.
