# React 렌더링 최적화 정리

## 핵심 요약

- React에서 렌더링은 컴포넌트 함수가 다시 실행되고, 그 결과를 바탕으로 UI 변경 사항을 계산하는 과정입니다.
- 렌더링이 항상 나쁜 것은 아닙니다. 문제는 **불필요한 렌더링이 너무 자주 발생하거나, 렌더링 비용이 큰 경우**입니다.
- 대표적인 최적화 방법으로는 `React.memo`, `useMemo`, `useCallback`, 상태 위치 조정, 리스트 가상화 등이 있습니다.
- 면접에서는 “무조건 memo를 붙인다”보다 **왜 렌더링이 발생했고, 어디의 비용이 큰지 먼저 확인한다**는 관점이 중요합니다.

## React 렌더링이란?

React에서 렌더링은 컴포넌트가 화면에 표시될 UI를 계산하는 과정입니다.

함수 컴포넌트 기준으로 보면 렌더링은 컴포넌트 함수가 다시 호출되는 것에 가깝습니다.  
컴포넌트 함수가 실행되면 JSX를 반환하고, React는 이전 결과와 새 결과를 비교해서 실제 DOM에 반영할 변경 사항을 찾습니다.

여기서 중요한 점은 렌더링이 곧바로 DOM 변경을 의미하지는 않는다는 것입니다.  
컴포넌트가 다시 실행되더라도 실제 DOM에 바뀔 내용이 없으면 DOM 변경은 최소화될 수 있습니다.

## 렌더링은 언제 발생할까?

React 컴포넌트는 보통 이런 상황에서 다시 렌더링됩니다.

- 자신의 state가 변경될 때
- 부모 컴포넌트가 렌더링될 때
- 전달받은 props가 변경될 때
- 사용 중인 context 값이 변경될 때

특히 자주 놓치는 부분은 부모 컴포넌트 렌더링입니다.  
부모가 다시 렌더링되면 자식 컴포넌트도 기본적으로 다시 렌더링될 수 있습니다. props 값이 실제로 같아 보여도, 새 객체나 새 함수가 매번 만들어지면 자식 입장에서는 값이 바뀐 것처럼 보일 수 있습니다.

## 불필요한 렌더링이 문제 되는 경우

렌더링 자체는 React가 원래 하는 일입니다.  
그래서 작은 컴포넌트가 몇 번 더 렌더링된다고 해서 바로 문제가 되지는 않습니다.

문제가 되는 경우는 보통 이런 상황입니다.

- 렌더링되는 컴포넌트가 매우 많다.
- 렌더링 중 무거운 계산을 한다.
- 큰 리스트를 한 번에 그린다.
- props로 전달되는 객체나 함수가 매번 새로 만들어진다.
- context 변경으로 넓은 범위의 컴포넌트가 함께 렌더링된다.

즉, 최적화는 “렌더링 횟수”만 보는 것이 아니라 “렌더링 비용”까지 같이 봐야 합니다.

## React.memo

`React.memo`는 props가 바뀌지 않았다면 컴포넌트 렌더링을 건너뛸 수 있게 도와줍니다.

```jsx
const UserItem = React.memo(function UserItem({ user }) {
  return <li>{user.name}</li>;
});
```

부모가 다시 렌더링되더라도 `UserItem`이 받는 props가 이전과 같다면 React는 렌더링을 건너뛸 수 있습니다.

다만 `React.memo`가 항상 좋은 것은 아닙니다.  
props 비교 비용도 있고, props로 매번 새 객체나 새 함수가 넘어오면 memo 효과가 거의 없을 수 있습니다.

```jsx
<UserItem user={{ name: "Kim" }} />
```

위처럼 객체를 JSX 안에서 바로 만들면 렌더링할 때마다 새 객체가 만들어집니다.  
겉보기에는 같은 값이어도 참조가 다르기 때문에 memo가 기대만큼 동작하지 않을 수 있습니다.

## useMemo

`useMemo`는 계산 결과를 기억해두고, 의존성이 바뀔 때만 다시 계산하게 해줍니다.

```jsx
const filteredUsers = useMemo(() => {
  return users.filter((user) => user.active);
}, [users]);
```

이런 방식은 배열 필터링, 정렬, 복잡한 계산처럼 비용이 있는 작업에 사용할 수 있습니다.

하지만 단순한 계산에까지 모두 `useMemo`를 붙이면 코드만 복잡해질 수 있습니다.  
`useMemo`는 “계산 비용이 있거나, 참조 안정성이 필요한 경우”에 쓰는 편이 좋습니다.

## useCallback

`useCallback`은 함수를 기억해두는 Hook입니다.

```jsx
const handleClick = useCallback(() => {
  onSelect(id);
}, [id, onSelect]);
```

주로 memo된 자식 컴포넌트에 함수를 props로 넘길 때 도움이 됩니다.  
부모가 렌더링될 때마다 새 함수가 만들어지면 자식의 props도 바뀐 것으로 판단될 수 있기 때문입니다.

다만 `useCallback`도 무조건 붙이는 것이 답은 아닙니다.  
자식 컴포넌트가 memo되어 있지 않거나, 함수 생성 비용이 문제가 되지 않는다면 효과가 거의 없을 수 있습니다.

## 상태 위치 조정

렌더링 최적화에서 꽤 중요한 것이 상태를 어디에 둘지 결정하는 일입니다.

state를 너무 상위 컴포넌트에 두면 작은 상태 변화에도 넓은 범위가 다시 렌더링될 수 있습니다.  
반대로 state를 실제로 필요한 컴포넌트 근처에 두면 렌더링 범위를 줄일 수 있습니다.

예를 들어 검색 input의 입력값이 페이지 전체와 관련 없는데 최상위 컴포넌트에 있다면, 글자를 입력할 때마다 페이지 전체가 다시 렌더링될 수 있습니다.  
이런 경우 검색 영역 내부로 상태를 내리면 훨씬 단순해질 수 있습니다.

## key와 리스트 렌더링

리스트를 렌더링할 때 `key`도 중요합니다.

```jsx
{users.map((user) => (
  <UserItem key={user.id} user={user} />
))}
```

`key`는 React가 리스트의 각 항목을 구분하는 기준입니다.  
안정적인 `id`를 key로 사용하면 항목 추가, 삭제, 정렬이 일어났을 때 React가 어떤 항목이 유지되고 어떤 항목이 바뀌었는지 더 잘 판단할 수 있습니다.

반대로 배열 index를 key로 쓰면 순서가 바뀌는 상황에서 불필요한 렌더링이나 상태 꼬임이 생길 수 있습니다.

## 리스트 가상화

목록이 아주 많다면 렌더링 자체를 줄이는 것이 가장 효과적일 때가 많습니다.

예를 들어 1만 개의 항목을 한 번에 DOM에 그리면 아무리 memo를 잘 써도 부담이 큽니다.  
이럴 때는 화면에 보이는 항목만 렌더링하는 리스트 가상화를 고려할 수 있습니다.

대표적으로 `react-window`, `react-virtualized` 같은 방식의 라이브러리를 떠올릴 수 있습니다.

핵심은 “안 보이는 것을 굳이 지금 렌더링하지 않는다”입니다.

## Context 사용 시 주의할 점

Context 값이 바뀌면 해당 Context를 사용하는 컴포넌트들이 다시 렌더링될 수 있습니다.

그래서 하나의 거대한 Context에 여러 값을 모두 넣으면, 일부 값만 바뀌어도 넓은 영역이 같이 영향을 받을 수 있습니다.  
이럴 때는 Context를 관심사별로 나누거나, 자주 바뀌는 값과 거의 바뀌지 않는 값을 분리하는 것이 좋습니다.

## 면접 답변 예시

React 렌더링 최적화는 불필요한 렌더링을 줄이거나 렌더링 비용을 낮추는 작업입니다.  
렌더링은 state, props, context 변경이나 부모 컴포넌트 렌더링으로 발생할 수 있습니다. 다만 렌더링 자체가 항상 문제는 아니고, 실제로 비용이 큰 부분을 확인한 뒤 최적화하는 것이 중요합니다.

대표적으로 props가 변하지 않는 컴포넌트에는 `React.memo`를 사용할 수 있고, 비용이 큰 계산에는 `useMemo`를 사용할 수 있습니다. memo된 자식에게 함수를 넘겨야 한다면 `useCallback`으로 함수 참조를 안정화할 수도 있습니다. 또 state를 필요한 위치에 가깝게 두거나, 큰 리스트는 가상화를 적용해서 렌더링 범위를 줄일 수 있습니다.

중요한 점은 무조건 memo를 많이 쓰는 것이 아니라, React DevTools Profiler 같은 도구로 어떤 컴포넌트가 자주 렌더링되고 비용이 큰지 확인한 뒤 적용하는 것입니다.

## 장점

- 화면 반응성을 개선할 수 있습니다.
- 큰 리스트나 복잡한 컴포넌트에서 렌더링 비용을 줄일 수 있습니다.
- 컴포넌트 구조와 상태 위치를 더 명확하게 설계하는 데 도움이 됩니다.

## 단점

- `memo`, `useMemo`, `useCallback`을 과하게 쓰면 코드가 오히려 복잡해질 수 있습니다.
- 잘못된 의존성 배열은 오래된 값을 참조하는 문제를 만들 수 있습니다.
- 최적화 자체에도 비교 비용과 관리 비용이 있으므로 모든 컴포넌트에 적용하는 것은 좋지 않습니다.

## 주의사항 / 실무 팁

- 먼저 Profiler로 병목을 확인하고, 실제로 비용이 큰 곳부터 최적화하세요.
- 단순한 컴포넌트에는 memo를 붙이지 않아도 충분한 경우가 많습니다.
- props로 객체나 함수를 넘길 때는 참조가 매번 바뀌는지 확인해보세요.
- state는 가능한 한 실제로 필요한 범위에 가깝게 두는 것이 좋습니다.
- 리스트가 매우 크다면 memo보다 가상화가 더 큰 효과를 줄 수 있습니다.
