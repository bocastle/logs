# 리스트 가상화 정리

## 핵심 요약

- `리스트 가상화`는 긴 목록 전체를 한 번에 그리지 않고, 화면에 보이는 범위만 렌더링하는 최적화 기법이다.
- 데이터 개수가 많을수록 DOM 노드 수가 성능에 직접 영향을 주기 때문에, 가상화는 렌더링 비용을 크게 줄일 수 있다.
- 대표 라이브러리로는 `react-window`, `react-virtualized`가 자주 언급된다.
- 면접에서는 "왜 필요한지", "무엇을 아끼는지", "언제 오히려 불편해지는지"를 같이 말하면 좋다.

## 리스트 가상화란 무엇인가

예를 들어 1만 개의 아이템이 있는 목록을 생각해 보면, 실제로 사용자는 한 번에 몇십 개 정도만 화면에서 본다.  
그런데도 모든 항목을 DOM에 올리면 브라우저가 초기 렌더링과 업데이트에 큰 비용을 쓰게 된다.

리스트 가상화는 이 문제를 줄이기 위해 "보이는 영역 근처의 아이템만 렌더링하고, 나머지는 공간만 계산해 두는 방식"을 쓴다.

즉 데이터는 1만 개여도, 실제 DOM에는 지금 필요한 일부만 존재하게 만드는 전략이다.

## 왜 성능에 도움이 되는가

프론트엔드 성능에서 중요한 것은 단순히 자바스크립트 실행 시간만이 아니다.  
DOM 노드 수가 많아지면 브라우저가 레이아웃 계산, 페인팅, 업데이트 과정에서 부담을 크게 느낀다.

리스트 가상화는 이런 부담을 줄여 준다.

- 초기 렌더링할 노드 수가 줄어든다.
- 스크롤 중 업데이트 대상이 줄어든다.
- 메모리 사용량이 감소한다.
- 리렌더링 비용을 통제하기 쉬워진다.

그래서 대시보드, 관리자 페이지, 채팅 목록, 로그 뷰어 같은 화면에서 자주 쓰인다.

## 동작 원리

핵심 아이디어는 단순하다.

1. 전체 리스트 높이 또는 길이는 유지한다.
2. 현재 스크롤 위치를 기준으로 보여야 하는 아이템 범위를 계산한다.
3. 그 범위에 해당하는 아이템만 실제로 렌더링한다.
4. 스크롤이 움직이면 렌더링 범위를 다시 계산한다.

사용자는 긴 리스트를 스크롤한다고 느끼지만, 실제 DOM은 아주 일부만 계속 갈아 끼우는 구조다.

## 예시로 이해하기

10000개의 행이 있는 테이블이 있다고 해 보자.  
한 행 높이가 40px이고 화면에는 15개 정도만 보인다면, 굳이 10000개 행을 모두 렌더링할 필요는 없다.

이때 가상화 라이브러리는 현재 스크롤 위치를 보고 예를 들어 12번부터 30번까지 정도만 렌더링한다.  
스크롤을 내리면 기존 DOM 일부를 재사용하거나 교체하면서 다음 구간을 보여 준다.

면접에서는 이걸 "전체 데이터를 다 갖고 있어도, 전체 DOM을 다 만들 필요는 없다"라고 설명하면 전달이 잘 된다.

## 언제 쓰면 좋은가

다음 같은 경우에 리스트 가상화의 효과가 크다.

- 수백 개 이상 아이템이 한 화면에 몰리는 경우
- 테이블, 로그, 피드처럼 길게 이어지는 목록
- 스크롤 성능이 중요한 관리자 화면
- 모바일 환경처럼 렌더링 자원이 더 민감한 경우

반대로 목록이 짧은데도 무조건 가상화를 넣는 것은 과할 수 있다.

## 단순 최적화와 다른 점

React에서 `memo`, `key`, pagination 같은 기법도 목록 성능에 영향을 준다.  
하지만 리스트 가상화는 "렌더링할 대상 자체를 줄인다"는 점이 핵심이다.

즉 리렌더링을 줄이는 최적화와, DOM 개수를 줄이는 최적화는 성격이 다르다.  
면접에서 이 차이를 구분해서 말하면 이해도가 높아 보인다.

## 단점과 트레이드오프

리스트 가상화는 강력하지만 구현 난도가 조금 올라간다.

- 아이템 높이가 일정하지 않으면 계산이 복잡해진다.
- 스크롤 위치 복원 처리가 까다로울 수 있다.
- 검색, 포커스 이동, 스크린 리더 대응에서 추가 고려가 필요하다.
- SEO가 중요한 정적 콘텐츠 목록에는 잘 맞지 않을 수 있다.

또 사용자가 "페이지 전체에서 Ctrl+F"처럼 브라우저 기본 동작을 기대하는 경우, 렌더링되지 않은 항목은 검색 대상이 아니라는 점도 실무에서 문제될 수 있다.

## 면접 답변 예시

> 리스트 가상화는 긴 목록을 전부 DOM에 올리지 않고, 화면에 보이는 범위만 렌더링하는 최적화 기법입니다. 아이템 수가 많아지면 자바스크립트 계산보다도 DOM 수 자체가 렌더링 성능에 큰 부담이 되는데, 가상화는 이 부담을 줄여 줍니다. 예를 들어 수천 개의 행이 있는 테이블에서도 실제 DOM은 현재 보이는 수십 개만 유지하도록 만드는 방식입니다. 다만 아이템 높이가 가변적이면 구현이 복잡해지고, 접근성이나 검색 같은 기본 동작에 추가 고려가 필요해서 데이터 양과 화면 목적을 보고 적용하는 편이 좋습니다.

## 장점

- 대량 데이터 목록에서 렌더링 성능을 크게 개선할 수 있다.
- DOM 수와 메모리 사용량을 줄이기 좋다.
- 스크롤 성능이 중요한 화면에서 체감 효과가 크다.

## 단점

- 구현 복잡도가 올라간다.
- 가변 높이, 포커스, 접근성 같은 예외 처리가 까다롭다.
- 짧은 리스트에는 오버엔지니어링이 될 수 있다.

## 주의사항 / 실무 팁

- 먼저 pagination만으로 충분한지, 정말 한 화면에 긴 목록이 필요한지부터 판단하는 게 좋다.
- 가상화는 `보이는 것만 그린다`는 원리로 설명하면 면접에서 명확하다.
- 접근성, 키보드 탐색, 검색 UX를 같이 언급하면 실무 감각이 드러난다.
- React에서는 `리렌더링 최적화`와 `가상화`를 별개 축으로 구분해 말하는 편이 좋다.
