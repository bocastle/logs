# Event Timing API 정리

## 핵심 요약
- `Event Timing API`는 사용자의 입력 이벤트가 실제로 처리되고 화면 반응으로 이어지기까지의 지연을 측정하는 브라우저 성능 API다.
- Core Web Vitals의 `INP`를 이해할 때 같이 나오는 개념이라, "입력 지연을 계측하는 도구"라고 설명하면 빠르다.
- 단순히 요청 시간이 아니라 클릭, 탭, 키 입력 같은 상호작용의 반응성을 본다는 점이 핵심이다.
- 면접에서는 `Long Task`, 메인 스레드 점유, 렌더링 지연과 연결해서 말하면 실무 감각이 드러난다.

## 개념 설명
사용자는 버튼을 눌렀는데 화면이 늦게 반응하면 서비스가 느리다고 느낀다. 그런데 기존의 네트워크 지표나 페이지 로드 지표만으로는 이런 체감 지연을 설명하기 어렵다. 이때 등장하는 것이 `Event Timing API`다.

이 API는 브라우저가 입력 이벤트를 언제 받았고, 실제로 해당 이벤트 처리가 끝나 화면 업데이트에 연결될 준비가 되었는지를 측정하게 도와준다. 쉽게 말해 "사용자가 행동한 시점부터 브라우저가 반응하기까지 얼마나 막혀 있었는가"를 보는 도구다.

여기서 중요한 포인트는 세 가지다.

- 입력 자체가 대기한 시간
- 이벤트 핸들러가 실행되는 시간
- 렌더링이나 후속 작업 때문에 체감 반응이 늦어진 시간

그래서 `Event Timing API`는 프론트엔드 성능을 단순 로딩 최적화에서 한 단계 더 나아가, 실제 인터랙션 품질 관점으로 보게 해 준다.

## 예시
예를 들어 검색 화면에서 사용자가 필터 버튼을 클릭했는데 결과 목록이 1초 뒤에야 바뀐다고 해 보자.

- 네트워크는 빠를 수 있다
- 하지만 메인 스레드에서 무거운 계산이 먼저 돌고 있을 수 있다
- 혹은 클릭 핸들러 안에서 동기 작업이 길게 실행될 수 있다

이런 경우 `Event Timing API`를 보면 특정 입력 이벤트의 지연이 크게 잡히고, 이를 통해 "문제는 API 응답이 아니라 메인 스레드 점유"라는 식으로 원인을 좁힐 수 있다.

또 면접에서는 `Long Task`와 묶어서 설명하면 좋다. 긴 자바스크립트 실행이 있으면 입력 이벤트가 바로 처리되지 못하고 밀리기 때문에, 사용자 체감 성능이 나빠진다.

## 면접 답변 예시
- 질문: "`Event Timing API`는 무엇인가요?"  
  답변: "사용자 입력 이벤트가 발생한 뒤 실제로 브라우저가 반응하기까지 걸린 시간을 측정하는 성능 API입니다. 클릭이나 키 입력 같은 상호작용 지연을 볼 수 있어서, 체감 성능이나 INP 같은 지표를 분석할 때 중요합니다."
- 질문: "왜 실무에서 의미가 있나요?"  
  답변: "로딩 속도만 빨라도 인터랙션이 느리면 사용자는 답답하게 느낍니다. Event Timing API를 보면 메인 스레드가 막혀서 입력 처리가 늦어지는지 확인할 수 있어서, 핸들러 최적화나 작업 분할 포인트를 찾는 데 도움이 됩니다."

## 장점
- 사용자 체감 성능을 입력 지연 관점에서 볼 수 있다
- `INP`, `Long Task`, 메인 스레드 병목과 연결해 설명하기 좋다
- 단순 로드 성능이 아닌 인터랙션 품질 개선 포인트를 찾는 데 유용하다

## 단점
- 지표를 수집해도 원인을 바로 해결해 주는 것은 아니다
- 네트워크, 렌더링, 자바스크립트 실행이 섞여 있어 해석이 필요하다
- 브라우저 지원 범위와 수집 방식에 따라 운영 설계가 필요하다

## 주의사항 / 실무 팁
- `Event Timing API`는 "느리다"를 보여주는 계측 도구이지, 자동 최적화 기능은 아니다.
- 수치만 보지 말고 어떤 입력에서 지연이 큰지, 공통적으로 긴 작업이 있는지를 함께 봐야 한다.
- React나 대형 SPA에서는 무거운 상태 계산, 동기 렌더링, 과도한 리렌더링이 입력 지연으로 이어지기 쉽다.
- 면접에서는 "`사용자 입력 반응성`을 측정한다"는 한 문장으로 먼저 정의하고, 이어서 `INP`나 메인 스레드 병목 이야기를 붙이면 답변 흐름이 좋다.
