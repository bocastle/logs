# OffscreenCanvas 정리

## 핵심 요약
- `OffscreenCanvas`는 캔버스 렌더링을 메인 스레드 밖으로 옮길 수 있게 해 주는 브라우저 API다.
- 핵심 목적은 무거운 그래픽 작업 때문에 UI가 끊기는 문제를 줄이는 것이다.
- 면접에서는 "캔버스 작업과 화면 상호작용을 분리해 반응성을 높이는 도구"라고 설명하면 전달이 빠르다.
- 특히 `Web Worker`와 함께 언급하면 실무 감각이 더 잘 드러난다.

## 개념 설명
일반적인 `canvas` 렌더링은 메인 스레드에서 돌아간다. 그래서 차트가 자주 다시 그려지거나, 이미지 후처리나 간단한 게임 루프처럼 계산량이 많은 작업이 들어오면 스크롤이나 클릭 반응까지 같이 느려질 수 있다.

`OffscreenCanvas`는 이 문제를 줄이기 위해 등장했다. 이름 그대로 브라우저 화면에 직접 붙어 있는 캔버스가 아니라, 화면 밖에서 그리기 작업을 수행할 수 있는 캔버스 객체다. 이 객체를 `Web Worker`로 넘기면 렌더링 계산을 메인 스레드와 분리할 수 있다.

핵심 포인트는 세 가지다.

- 캔버스 연산을 메인 스레드와 분리할 수 있다
- UI 이벤트 처리와 그래픽 계산이 덜 충돌한다
- 성능 개선 여지가 있지만, 모든 캔버스 작업이 자동으로 빨라지는 것은 아니다

즉, `OffscreenCanvas`는 "캔버스를 더 빨리 그리는 API"라기보다 "캔버스가 UI를 덜 막게 만드는 구조적 수단"에 가깝다.

## 예시
예를 들어 대시보드에서 실시간 차트를 초당 여러 번 다시 그린다고 해 보자.

- 메인 스레드에서 렌더링하면 스크롤과 버튼 클릭이 같이 버벅일 수 있다
- `OffscreenCanvas`와 `Worker`를 쓰면 차트 계산과 드로잉을 별도 스레드에서 처리할 수 있다
- 메인 스레드는 사용자 입력과 DOM 업데이트에 더 집중할 수 있다

또 다른 예시는 이미지 편집 기능이다. 필터 적용, 픽셀 단위 변환, 썸네일 생성 같은 작업은 계산량이 커서 UI를 쉽게 막는다. 이런 경우 `OffscreenCanvas`는 사용자 체감 성능 개선 포인트가 될 수 있다.

## 면접 답변 예시
- 질문: "`OffscreenCanvas`는 무엇인가요?"  
  답변: "캔버스 렌더링 작업을 메인 스레드 밖에서도 처리할 수 있게 해 주는 API입니다. 특히 Web Worker와 함께 사용하면 무거운 그래픽 연산이 UI 반응성을 해치는 문제를 줄일 수 있습니다."
- 질문: "언제 쓰면 좋나요?"  
  답변: "실시간 차트, 이미지 처리, 게임 렌더링처럼 canvas 작업이 자주 일어나고 계산량이 큰 경우에 적합합니다. 이런 작업을 메인 스레드에서 돌리면 클릭이나 스크롤까지 끊길 수 있어서, 분리할 가치가 있습니다."

## 장점
- 무거운 캔버스 작업이 메인 스레드를 덜 점유하게 할 수 있다
- 사용자 입력 반응성과 그래픽 처리량을 분리해서 설계하기 좋다
- `Web Worker`, 렌더링 최적화, 체감 성능 개선 이야기를 함께 풀기 좋다

## 단점
- 브라우저 지원과 구현 방식 차이를 고려해야 한다
- DOM 접근은 여전히 워커에서 직접 할 수 없어서 구조가 복잡해질 수 있다
- 작업 분리 비용이 있어 단순한 캔버스에는 오히려 과할 수 있다

## 주의사항 / 실무 팁
- `OffscreenCanvas`는 만능 성능 버튼이 아니다. 병목이 DOM인지, 네트워크인지, 캔버스 연산인지 먼저 구분해야 한다.
- 메인 스레드와 워커 사이 메시지 전달 비용도 있으니, 아주 작은 작업까지 무조건 옮기는 것은 비효율적일 수 있다.
- 실무에서는 차트 라이브러리나 이미지 처리 파이프라인처럼 반복 계산이 큰 곳에서 후보로 보는 편이 현실적이다.
- 면접에서는 "`canvas 렌더링을 worker로 분리해 UI 끊김을 줄이는 전략`"이라고 한 문장으로 먼저 정의하면 안정적으로 답할 수 있다.
