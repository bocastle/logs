# useRef 정리

## 핵심 요약

- `useRef`는 렌더링과 무관하게 값을 유지하거나 DOM에 직접 접근할 때 쓰는 React 훅이다.
- 상태처럼 값을 기억하지만, `ref.current`를 바꿔도 컴포넌트가 다시 렌더링되지는 않는다.
- 그래서 포커스 이동, 이전 값 저장, 타이머 ID 보관, 외부 라이브러리 인스턴스 관리 같은 용도에 잘 맞는다.
- 면접에서는 "state와 뭐가 다른지", "언제 ref를 쓰고 언제 state를 써야 하는지"를 같이 설명하면 좋다.

## useRef가 무엇인가

`useRef`는 `{ current: ... }` 형태의 객체를 하나 만들어 두고, 그 객체를 컴포넌트가 다시 렌더링돼도 계속 유지하게 해 준다.

가장 많이 보는 예시는 DOM 접근이다.

- input에 자동 포커스를 주고 싶다.
- 스크롤 위치를 읽고 싶다.
- 특정 요소의 크기나 위치를 직접 확인하고 싶다.

이럴 때 ref를 요소에 연결해 두고 `ref.current`로 실제 DOM 노드를 참조한다.

하지만 `useRef`는 DOM 전용 훅은 아니다. 렌더링에 반영할 필요는 없지만 컴포넌트 안에서 계속 들고 있어야 하는 값이라면 ref로 관리할 수 있다.

## state와 차이

면접에서 가장 자주 나오는 포인트다.

`useState`는 값이 바뀌면 UI도 바뀌어야 할 때 쓴다.  
반대로 `useRef`는 값이 바뀌어도 화면을 다시 그릴 필요가 없을 때 쓴다.

예를 들어 검색창에 입력한 텍스트를 화면에 보여줘야 하면 state가 맞다.  
반면 `setTimeout`의 반환값이나 이전 렌더의 값을 잠깐 저장하는 정도라면 ref가 더 자연스럽다.

즉, 기준은 단순하다.

- 변경이 UI에 반영돼야 하면 state
- 변경이 UI와 무관하면 ref

## 언제 많이 쓰나

실무에서 자주 만나는 사례는 아래 정도다.

### 1. DOM 직접 접근

```jsx
const inputRef = useRef(null);

useEffect(() => {
  inputRef.current?.focus();
}, []);
```

로그인 폼 첫 입력칸 자동 포커스 같은 경우가 대표적이다.

### 2. 이전 값 저장

렌더링 사이에 이전 props나 state를 기억해 두고 비교할 때 쓸 수 있다.

```jsx
const prevValueRef = useRef(value);

useEffect(() => {
  prevValueRef.current = value;
}, [value]);
```

### 3. 타이머나 외부 인스턴스 보관

`setInterval` ID, WebSocket 인스턴스, 차트 라이브러리 객체처럼 다시 렌더링할 필요는 없지만 계속 참조해야 하는 값을 보관할 때 유용하다.

## 왜 state 대신 ref를 쓰기도 하나

불필요한 렌더링을 줄이기 좋기 때문이다.

예를 들어 debounce 타이머 ID를 state에 넣으면 값이 바뀔 때마다 렌더링이 다시 일어난다. 그런데 화면에서 그 ID를 보여줄 일은 거의 없다. 이런 값은 ref에 넣는 편이 더 맞다.

또 ref는 클로저 문제를 완화할 때도 종종 쓴다. 최신 값을 ref에 넣어 두고 비동기 콜백에서 읽는 방식이다. 다만 이건 편리한 만큼 코드 흐름이 숨겨질 수 있어서 남용하면 읽기 어려워진다.

## 주의할 점

`useRef`는 편하지만, 아무 값이나 다 ref에 넣는 습관은 좋지 않다.

- `ref.current`가 바뀌어도 렌더링이 안 되기 때문에 화면과 데이터가 어긋날 수 있다.
- ref 값 변경은 React의 선언적 흐름 밖에서 일어나는 경우가 많아서 추적이 어렵다.
- DOM 조작을 과하게 하면 React가 화면을 관리하는 방식과 충돌할 수 있다.

특히 "이 값이 바뀌면 화면도 바뀌어야 하는데 ref에 넣어 둔 경우"가 흔한 실수다. 그때는 거의 항상 state가 맞다.

## 면접 답변 예시

> `useRef`는 렌더링과 상관없이 값을 유지하고 싶을 때 쓰는 React 훅입니다. 가장 흔한 용도는 DOM 접근인데, input 포커스 제어나 스크롤 위치 확인 같은 작업에 많이 씁니다. 또 이전 값 저장, 타이머 ID나 외부 라이브러리 인스턴스 보관처럼 UI에 바로 반영되지 않는 값도 ref로 관리할 수 있습니다. `useState`와 가장 큰 차이는 ref 값이 바뀌어도 리렌더링이 일어나지 않는다는 점입니다. 그래서 화면에 보여줄 데이터는 state로, 렌더링과 무관한 참조값은 ref로 나누어 쓰는 게 핵심입니다.

## 장점

- 렌더링 없이 값을 유지할 수 있다.
- DOM 접근이 간단하다.
- 타이머, 외부 인스턴스, 이전 값 저장 같은 실무 패턴에 잘 맞는다.

## 단점

- 값이 바뀌어도 UI가 자동으로 갱신되지 않는다.
- 선언적 흐름에서 벗어나기 쉬워서 추적이 어려워질 수 있다.
- 남용하면 state로 관리해야 할 데이터까지 ref로 숨기게 된다.

## 주의사항 / 실무 팁

- "이 값이 바뀌면 화면도 바뀌어야 하나?"를 먼저 묻고 state와 ref를 구분하는 게 좋다.
- DOM 접근은 꼭 필요한 경우에만 하고, 가능하면 React 흐름 안에서 해결하는 편이 유지보수에 유리하다.
- 비동기 콜백에서 최신 값을 읽기 위해 ref를 쓸 때는 왜 필요한지 주석이나 변수명으로 의도를 드러내는 게 좋다.
- 면접에서는 "ref는 리렌더링을 일으키지 않는다"는 점을 중심으로 state와 비교해서 답하면 전달력이 좋다.
