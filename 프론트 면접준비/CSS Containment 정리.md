# CSS Containment 정리

## 핵심 요약

- `CSS Containment`는 브라우저가 한 영역의 변경 영향을 문서 전체로 퍼뜨리지 않도록 경계를 주는 방법이다.
- 잘 쓰면 리플로우, 리페인트 범위를 줄여서 복잡한 UI의 성능을 안정시키는 데 도움이 된다.
- 대신 레이아웃과 스타일 전파를 일부 끊기 때문에, 이유 없이 남발하면 오히려 예상과 다른 동작이 생길 수 있다.
- 면접에서는 "브라우저 최적화 힌트를 주는 기능" 정도로 먼저 설명하고, 어떤 비용을 줄이는지 이어서 말하면 전달이 좋다.

## CSS Containment란?

페이지 안에 카드, 사이드바, 모달처럼 비교적 독립적인 영역이 있을 때가 많다.  
이런 영역 안에서 변경이 일어나도 브라우저는 "혹시 바깥까지 영향이 번질까?"를 계산해야 한다.

`contain` 속성은 이 요소의 변화가 어디까지 영향을 미치는지 범위를 좁혀 주는 선언이다.  
쉽게 말하면 브라우저에게 "이 박스는 어느 정도 독립적으로 다뤄도 된다"라고 알려 주는 셈이다.

## 왜 성능에 도움이 될까?

브라우저는 DOM과 CSSOM을 바탕으로 레이아웃과 페인팅을 계산한다.  
문제는 작은 변화 하나가 상위나 형제 요소에 영향을 줄 수 있으면 계산 범위가 커진다는 점이다.

예를 들어 무한 스크롤 목록의 각 카드가 독립적이라면, 카드 하나 내부의 변화 때문에 페이지 전체를 다시 따질 필요는 없다.  
이때 containment를 적절히 주면 브라우저가 계산 범위를 더 공격적으로 줄일 수 있다.

특히 아래 같은 경우에 효과를 기대할 수 있다.

- 길고 복잡한 리스트의 아이템 카드
- 대시보드 위젯
- 외부 임베드 영역
- 자주 갱신되는 독립 섹션

## 어떤 값들이 있나?

자주 언급되는 값은 `layout`, `paint`, `size`, `style`이다.

- `layout`: 내부 레이아웃 변경이 바깥 레이아웃 계산에 미치는 영향을 줄인다.
- `paint`: 요소 바깥으로 그리기 영향이 퍼지지 않도록 범위를 제한한다.
- `size`: 내부 콘텐츠가 부모 크기 계산에 직접 영향을 주지 않게 한다.
- `style`: 스타일 계산 전파 범위를 줄이는 개념인데 실무에서는 다른 값보다 덜 자주 이야기된다.

보통 면접에서는 세부 명세를 다 외우기보다 "레이아웃, 페인트 같은 계산 범위를 분리한다" 정도로 설명해도 충분하다.

## 예시

```css
.widget {
  contain: layout paint;
}
```

이런 식으로 선언하면 `widget` 내부에서 생긴 레이아웃 변경이나 그리기 변경을 더 독립적으로 처리할 여지가 생긴다.

예를 들어 실시간 차트 옆에 여러 위젯이 있는 화면이라면, 한 위젯의 변화가 다른 위젯의 계산에 덜 영향을 주게 만들 수 있다.

## 면접에서 어떻게 말하면 좋을까?

핵심은 "브라우저 최적화 힌트"와 "독립 영역"이라는 두 단어다.

CSS Containment는 성능을 마법처럼 올리는 옵션이 아니라, 독립적인 UI 구역에 대해 브라우저가 계산 범위를 줄일 수 있게 돕는 힌트다.  
그래서 컴포넌트 구조가 명확하고 변경 범위가 좁은 화면일수록 효과를 기대하기 쉽다.

## 면접 답변 예시

> CSS Containment는 특정 요소를 비교적 독립적인 렌더링 영역처럼 다루게 해서 브라우저의 레이아웃이나 페인트 계산 범위를 줄이는 방법입니다. 예를 들어 대시보드 위젯이나 긴 리스트 카드처럼 다른 영역과 영향이 적은 컴포넌트에 적용하면 성능 최적화에 도움이 될 수 있습니다. 다만 size나 layout 제약을 잘못 주면 원래 기대한 배치와 달라질 수 있어서, 독립성이 높은 구간에만 의도적으로 적용하는 게 중요합니다.

## 장점

- 변경 영향 범위를 줄여서 복잡한 화면의 렌더링 비용을 낮추는 데 도움이 된다.
- 독립적인 UI 블록을 성능 관점에서도 분리해 생각하게 만들어 준다.
- 대시보드, 카드 리스트, 임베드 영역처럼 경계가 분명한 컴포넌트에 잘 맞는다.

## 단점

- 레이아웃이나 크기 계산이 예상과 다르게 동작할 수 있다.
- 브라우저 성능 문제를 구조 개선 없이 속성 하나로 해결하려는 식의 오용이 생기기 쉽다.
- 독립성이 낮은 영역에 적용하면 효과보다 디버깅 비용이 더 커질 수 있다.

## 주의사항 / 실무 팁

- 먼저 정말 독립적인 컴포넌트인지 판단하고 적용하는 편이 낫다.
- 성능 문제는 `contain` 하나보다 DOM 구조, 페인트 비용, 과도한 리렌더링을 같이 봐야 한다.
- 면접에서는 "리플로우 범위 축소"와 "독립적인 UI 섹션"을 연결해서 설명하면 실무 감각이 있어 보인다.
