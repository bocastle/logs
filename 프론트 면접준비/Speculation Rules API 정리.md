# Speculation Rules API 정리

## 핵심 요약

- Speculation Rules API는 브라우저에게 어떤 페이지를 미리 가져오거나 미리 렌더링할지 알려주는 기능이다.
- `prefetch`는 리소스를 미리 받아두는 것이고, `prerender`는 다음 페이지를 거의 열어둔 상태로 준비하는 것에 가깝다.
- 사용자가 실제로 이동할 가능성이 높은 경로에만 적용해야 네트워크와 서버 비용을 낭비하지 않는다.
- 로그인 상태, 개인화 데이터, 분석 이벤트처럼 부작용이 있는 페이지에는 신중하게 적용해야 한다.

## 개념 설명

기존에는 `<link rel="prefetch">` 같은 방식으로 브라우저에게 힌트를 줄 수 있었다. Speculation Rules API는 이를 더 구조적으로 표현한다. 보통 JSON 규칙을 `<script type="speculationrules">` 안에 넣고, 브라우저가 조건에 맞는 링크를 미리 준비하게 한다.

면접에서는 "빠르게 보이게 하는 기술"이라고만 말하면 부족하다. 중요한 포인트는 브라우저가 사용자의 다음 행동을 추측해서 준비한다는 점과, 그 준비 수준이 `prefetch`와 `prerender`로 나뉜다는 점이다.

## 예시

```html
<script type="speculationrules">
{
  "prefetch": [
    {
      "source": "list",
      "urls": ["/products", "/pricing"]
    }
  ],
  "prerender": [
    {
      "source": "document",
      "where": {
        "href_matches": "/articles/*"
      },
      "eagerness": "moderate"
    }
  ]
}
</script>
```

이 예시는 `/products`, `/pricing`을 미리 가져오고, 문서 안의 `/articles/*` 링크는 조건에 따라 미리 렌더링할 수 있게 한다. 실제 서비스에서는 모든 링크를 대상으로 삼기보다, 전환 가능성이 높은 주요 경로부터 제한적으로 적용하는 편이 안전하다.

## 면접 답변 예시

"Speculation Rules API는 사용자가 다음에 방문할 가능성이 높은 페이지를 브라우저가 미리 준비하도록 힌트를 주는 API입니다. `prefetch`는 필요한 리소스를 미리 다운로드하는 수준이고, `prerender`는 페이지를 백그라운드에서 렌더링해 이동 시 거의 즉시 보여줄 수 있습니다. 다만 서버 요청이 미리 발생할 수 있으므로 장바구니, 결제, 개인화 페이지처럼 부작용이 있거나 비용이 큰 경로에는 신중하게 적용해야 합니다."

## 장점

- 페이지 전환 체감 속도를 크게 줄일 수 있다.
- 사용자가 자주 이동하는 경로에 적용하면 UX 개선 효과가 크다.
- 선언형 규칙이라 특정 라우터 코드와 강하게 결합하지 않고 적용할 수 있다.
- `eagerness` 같은 설정으로 적극도를 조절할 수 있다.

## 단점

- 잘못 적용하면 불필요한 네트워크 요청과 서버 부하가 늘어난다.
- 개인화 페이지를 미리 렌더링하면 상태나 보안 이슈가 생길 수 있다.
- 모든 브라우저에서 동일한 수준으로 지원되는 것은 아니다.
- 분석 이벤트나 페이지 진입 로직이 미리 실행되는지 확인해야 한다.

## 주의사항 / 실무 팁

- 전환 가능성이 높은 링크부터 제한적으로 적용한다.
- 결제, 로그아웃, 상태 변경이 포함된 페이지에는 적용하지 않는다.
- 서버 로그에서 speculative request를 구분할 수 있는지 확인한다.
- Core Web Vitals 개선 효과만 보지 말고 서버 비용도 같이 본다.
- SPA 라우터의 자체 prefetch 기능과 중복되지 않는지 점검한다.
