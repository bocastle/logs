# CSS Cascade Layers 정리

## 핵심 요약

- `CSS Cascade Layers`는 CSS 우선순위를 레이어 단위로 관리하게 해 주는 기능이다.
- 선택자 명시도를 계속 높이는 대신, 스타일의 출처와 의도를 먼저 나눌 수 있다.
- 디자인 시스템, 외부 라이브러리 스타일, 앱 전용 스타일이 섞일 때 특히 유용하다.
- 면접에서는 "명시도 싸움을 줄이고, 스타일 우선순위를 구조적으로 관리하는 방법"이라고 설명하면 좋다.

## CSS Cascade Layers란?

CSS는 원래 여러 규칙이 같은 요소에 적용될 때 캐스케이딩 규칙으로 최종 스타일을 결정한다.  
이때 명시도, 선언 순서, `!important` 같은 요소가 얽히면 작은 수정도 점점 어려워질 수 있다.

`Cascade Layers`는 여기에 "레이어"라는 기준을 하나 더 둔다.  
예를 들어 reset, component, utility, override처럼 스타일을 층으로 나누고, 어떤 층이 더 나중에 이길지를 미리 정하는 방식이다.

## 왜 필요한가?

프로젝트가 커지면 CSS 문제는 단순히 "선택자를 잘 쓰자"로 끝나지 않는다.

- 브라우저 기본 스타일을 초기화하는 reset CSS가 있다.
- 디자인 시스템 컴포넌트 스타일이 있다.
- 화면별 예외 스타일이 있다.
- Tailwind 같은 유틸리티 클래스가 섞일 수도 있다.

이런 상황에서 우선순위를 명시도만으로 해결하면 `.page .section .button.primary`처럼 선택자가 점점 길어진다.  
`Cascade Layers`는 "어떤 종류의 스타일이 이겨야 하는가"를 레이어 순서로 표현하게 해 준다.

## 기본 문법

```css
@layer reset, base, components, utilities;

@layer reset {
  button {
    all: unset;
  }
}

@layer components {
  .button {
    background: royalblue;
    color: white;
  }
}

@layer utilities {
  .bg-red {
    background: red;
  }
}
```

위 예시에서는 `utilities` 레이어가 `components`보다 뒤에 선언되어 있으므로, 같은 속성을 다룰 때 더 높은 우선순위를 가진다.  
중요한 점은 단순히 파일에 나중에 적었다는 뜻이 아니라, 레이어 순서로 의도를 고정한다는 것이다.

## 명시도와의 관계

레이어가 다르면 명시도가 높은 선택자보다 나중 레이어의 낮은 명시도 선택자가 이길 수 있다.  
이게 `Cascade Layers`의 핵심이다.

즉 아래처럼 생각하면 편하다.

- 먼저 레이어 순서를 본다.
- 같은 레이어 안에서는 기존 CSS 규칙처럼 명시도와 선언 순서를 본다.

그래서 레이어를 잘 나누면 `!important`를 덜 쓰고도 예측 가능한 스타일 구조를 만들 수 있다.

## 어디에 쓰면 좋을까?

실무에서는 보통 아래처럼 나눌 수 있다.

- `reset`: 브라우저 기본 스타일 정리
- `base`: 전역 폰트, 색상, 기본 태그 스타일
- `components`: 버튼, 모달, 카드 같은 컴포넌트
- `utilities`: 여백, 색상, 표시 여부 같은 유틸리티 클래스
- `overrides`: 어쩔 수 없는 예외 처리

이름 자체가 중요한 것은 아니고, 팀에서 "어떤 레이어가 어떤 역할을 하는지" 합의하는 것이 더 중요하다.

## 면접에서 어떻게 말하면 좋을까?

`Cascade Layers`는 CSS의 우선순위를 레이어 단위로 선언해서 명시도 경쟁을 줄이는 기능이라고 설명하면 된다.  
특히 외부 라이브러리, 디자인 시스템, 화면별 스타일이 같이 있을 때 예측 가능한 순서를 만들 수 있다는 점을 덧붙이면 좋다.

## 면접 답변 예시

> CSS Cascade Layers는 스타일을 reset, base, components, utilities 같은 레이어로 나누고, 레이어 순서에 따라 우선순위를 관리하는 기능입니다. 기존에는 스타일 충돌이 생기면 선택자를 더 구체적으로 만들거나 `!important`를 쓰는 경우가 많았는데, 레이어를 쓰면 어떤 종류의 스타일이 이겨야 하는지 구조적으로 표현할 수 있습니다. 같은 레이어 안에서는 기존처럼 명시도와 선언 순서를 따르지만, 레이어가 다르면 레이어 순서가 먼저 영향을 줍니다. 그래서 규모가 큰 프로젝트에서 CSS 충돌을 줄이고 유지보수성을 높이는 데 도움이 됩니다.

## 장점

- 선택자 명시도를 불필요하게 높이지 않아도 된다.
- reset, 컴포넌트, 유틸리티 스타일의 역할을 분리하기 좋다.
- 외부 CSS와 내부 CSS가 섞일 때 우선순위를 더 예측하기 쉽다.

## 단점

- 팀에서 레이어 구조를 합의하지 않으면 오히려 헷갈릴 수 있다.
- 기존 프로젝트에 도입할 때는 현재 스타일 우선순위를 다시 점검해야 한다.
- 브라우저 지원 범위를 확인해야 하는 환경에서는 별도 검토가 필요하다.

## 주의사항 / 실무 팁

- 레이어를 너무 많이 만들면 장점보다 복잡도가 커진다.
- `!important`를 완전히 없애는 기능이라기보다는, 남용을 줄이는 도구로 보는 편이 좋다.
- 면접에서는 명시도, 선언 순서, `!important`와 어떻게 관계되는지까지 짧게 말하면 답변이 탄탄해진다.
