# INP(Interaction to Next Paint) 정리

## 핵심 요약
- `INP`는 사용자의 클릭, 탭, 키 입력 같은 상호작용이 화면 반응으로 이어지기까지의 지연을 보는 사용자 경험 지표다.
- 예전에는 `FID`가 많이 언급됐지만, 지금은 실제 인터랙션 전체 반응성을 더 잘 보여 주는 지표로 `INP`를 설명하는 편이 좋다.
- 핵심은 "이벤트가 얼마나 빨리 시작됐는가"보다 "사용자가 반응했다고 느끼는 시점까지 얼마나 걸렸는가"에 있다.
- 면접에서는 "입력 이벤트 처리, 메인 스레드 점유, 렌더링 지연까지 함께 보는 반응성 지표"라고 말하면 정리가 잘 된다.

## 개념 설명
프론트엔드 성능을 이야기할 때 로딩 속도만 강조하면 실제 사용 경험을 놓치기 쉽다. 페이지가 빨리 떠도 버튼을 눌렀을 때 한참 뒤에 반응하면 사용자는 느리다고 느낀다. `INP`는 바로 이런 인터랙션 품질을 측정하려는 지표다.

사용자가 버튼을 클릭하면 브라우저는 이벤트를 큐에 넣고, 자바스크립트 핸들러를 실행하고, 필요한 레이아웃 계산과 페인트를 거쳐 화면을 갱신한다. 이 과정 중 메인 스레드가 다른 작업으로 길게 점유돼 있거나, 이벤트 처리 코드가 무겁거나, 렌더링 비용이 크면 반응이 늦어진다. `INP`는 이런 전체 흐름을 반영해서 사용자가 체감하는 응답성을 본다.

즉, `INP`는 단순히 이벤트 리스너가 호출되는 시점만 보는 것이 아니라, 실제로 다음 화면 변화가 일어날 때까지를 본다는 점이 중요하다. 그래서 인터랙션이 많은 서비스일수록 더 실무적인 지표로 받아들여진다.

## 예시
상품 목록 페이지에서 필터 버튼을 누르면 다음과 같은 일이 일어날 수 있다.

- 클릭 이벤트가 발생한다
- 자바스크립트가 대량 데이터 필터링을 메인 스레드에서 수행한다
- 상태 변경으로 많은 컴포넌트가 다시 렌더링된다
- 레이아웃 계산과 페인트가 이어진다

이 과정이 길어지면 사용자는 버튼이 눌렸는지조차 헷갈릴 수 있다. 이런 경우 `INP`가 나빠질 가능성이 높다. 반대로 무거운 계산을 분리하거나, 렌더링 범위를 줄이거나, 즉각적인 로딩 상태를 보여 주면 체감 반응성과 지표가 함께 좋아질 수 있다.

## 면접 답변 예시
- 질문: "`INP`는 무엇인가요?"  
  답변: "`INP`는 사용자의 클릭이나 입력이 실제 화면 반응으로 이어지기까지의 지연을 측정하는 반응성 지표입니다. 이벤트 시작만 보는 게 아니라, 처리와 렌더링이 끝나서 사용자가 반응을 확인하는 시점까지 본다는 점이 중요합니다."
- 질문: "프론트엔드에서 `INP`를 개선하려면 어떻게 하나요?"  
  답변: "먼저 메인 스레드를 오래 점유하는 작업을 줄여야 합니다. 무거운 연산을 쪼개거나 비동기화하고, 불필요한 리렌더링을 줄이고, 인터랙션 직후 즉시 피드백을 주는 방식으로 체감 반응성을 개선할 수 있습니다."

## 장점
- 단순 로딩 속도보다 실제 사용성을 설명하기 좋은 성능 주제다
- 이벤트 처리, 렌더링, 메인 스레드 병목을 함께 연결해서 말할 수 있다
- Core Web Vitals 맥락과 사용자 체감 성능을 같이 설명하기 좋다

## 단점
- 수치만 외우고 설명하면 왜 느린지 원인 분석이 약해 보일 수 있다
- `FID`와 차이를 정확히 구분하지 못하면 답변이 흐려질 수 있다
- 지표 개선과 체감 개선이 항상 같은 방식으로 보이지는 않아 상황 설명이 필요하다

## 주의사항 / 실무 팁
- `INP`는 특정 한 번의 클릭보다 전체 인터랙션 경험을 보는 관점으로 이해하는 것이 좋다.
- 메인 스레드에서 긴 동기 작업이 있으면 반응성이 쉽게 나빠진다.
- 면접에서는 `Long Task`, `scheduler.postTask API`, `useTransition` 같은 주제와 연결하면 답변이 풍부해진다.
- 실무에서는 측정값만 보지 말고, 어떤 화면과 어떤 사용자 흐름에서 지연이 발생하는지 같이 확인해야 한다.
