# 가상 DOM과 Reconciliation 정리

## 핵심 요약

- 가상 DOM은 UI의 현재 모습과 다음 모습을 메모리 안에서 비교하기 위한 자바스크립트 객체 구조다.
- Reconciliation은 이전 가상 DOM과 새로운 가상 DOM을 비교해서 무엇이 바뀌었는지 찾는 과정이다.
- 핵심은 "화면 전체를 다시 그리는 것"이 아니라 "바뀐 부분만 효율적으로 반영하는 것"이다.
- 면접에서는 가상 DOM 자체보다도 왜 필요한지, diff 과정에서 `key`가 왜 중요한지까지 같이 설명하면 좋다.

## 가상 DOM이란?

가상 DOM은 실제 브라우저 DOM을 그대로 복제한 것은 아니고, 화면 구조를 자바스크립트 객체 형태로 표현한 추상화 계층에 가깝다.

프론트엔드에서 상태가 바뀌면 UI도 바뀌어야 한다. 이때 매번 개발자가 직접 `document.querySelector()`로 DOM을 찾고 수정하면 코드가 복잡해지고, 변경 지점을 놓치기도 쉽다. React 같은 라이브러리는 상태가 바뀌면 먼저 새로운 가상 DOM 트리를 만들고, 이전 트리와 비교한 뒤 실제 DOM에는 필요한 수정만 적용한다.

즉, 가상 DOM은 DOM 조작을 "자동화하고 예측 가능하게 만들기 위한 중간 표현"이라고 보면 된다.

## Reconciliation은 무엇을 하나?

Reconciliation은 이전 렌더 결과와 다음 렌더 결과를 비교해서 어떤 노드가 추가, 삭제, 수정됐는지 판단하는 과정이다.

예를 들어 이런 상황을 생각해볼 수 있다.

```jsx
<ul>
  <li>사과</li>
  <li>바나나</li>
</ul>
```

여기서 상태가 바뀌어서 아래처럼 됐다고 하자.

```jsx
<ul>
  <li>사과</li>
  <li>바나나</li>
  <li>포도</li>
</ul>
```

이 경우 React는 전체 `ul`을 갈아끼우는 것이 아니라, 마지막 `li` 하나가 추가됐다고 판단하고 그 부분만 실제 DOM에 반영하려고 한다.

이게 가능한 이유가 가상 DOM 비교와 Reconciliation 과정 때문이다.

## 왜 실제 DOM을 바로 건드리지 않을까?

실제 DOM 조작은 비용이 큰 편이다. DOM이 바뀌면 브라우저가 레이아웃 계산, 페인트, 컴포지팅 같은 작업을 다시 할 수 있기 때문이다.

물론 "가상 DOM이 무조건 빠르다"라고 말하면 과장이다. 비교 작업도 비용이 든다. 다만 상태 변화가 자주 일어나는 UI에서 사람이 수동으로 DOM 변경 지점을 관리하는 것보다, 선언적으로 렌더 결과를 만들고 라이브러리가 변경분만 적용하는 방식이 유지보수성과 안정성 면에서 훨씬 유리하다.

면접에서는 이렇게 정리하면 좋다.

> 가상 DOM의 목적은 실제 DOM보다 무조건 빠르기 때문이라기보다, 상태 변화에 따른 UI 업데이트를 선언적으로 관리하고 변경분만 반영하기 쉽게 만드는 데 있습니다.

## Diffing에서 `key`가 중요한 이유

리스트 렌더링에서는 `key`가 특히 중요하다. React는 형제 노드들을 비교할 때 `key`를 기준으로 어떤 항목이 유지되고, 어떤 항목이 새로 생겼는지 판단한다.

예를 들어 배열 앞쪽에 새 항목이 하나 추가됐는데 `key`가 없거나 index를 `key`로 쓰면, React가 기존 항목들을 엉뚱하게 재사용할 수 있다. 그러면 input 포커스가 튀거나, 체크박스 상태가 다른 행으로 옮겨가는 식의 버그가 생길 수 있다.

그래서 실무에서는 가능하면 데이터의 고유 id를 `key`로 쓰는 게 안전하다.

## 가상 DOM이 만능은 아니다

가상 DOM이 있다고 해서 성능 문제가 자동으로 해결되는 건 아니다.

예를 들어 아래 상황에서는 여전히 비효율이 생길 수 있다.

- 부모 컴포넌트가 자주 렌더링되면서 불필요하게 자식까지 계속 다시 렌더링될 때
- 리스트가 아주 크고 변경도 자주 일어날 때
- 애니메이션처럼 프레임 단위 최적화가 중요한 화면일 때

이럴 때는 컴포넌트 구조를 잘 나누고, 상태 범위를 줄이고, 경우에 따라선 virtualization 같은 별도 기법도 같이 써야 한다.

## 실무 예시

검색창에 입력할 때마다 상품 목록이 필터링되는 화면을 생각해보자.

- 사용자가 한 글자를 입력한다.
- 상태가 바뀌면서 컴포넌트가 다시 렌더링된다.
- 새로운 가상 DOM 트리가 만들어진다.
- 이전 트리와 비교해서 바뀐 목록 항목만 찾는다.
- 실제 DOM에는 달라진 텍스트나 노드만 반영한다.

개발자는 "입력 상태가 바뀌면 이런 UI가 보여야 한다"만 선언하면 되고, 어떤 노드를 직접 수정할지는 React가 맡는다.

## 면접 답변 예시

가상 DOM은 실제 DOM을 직접 바로 조작하지 않고, UI 상태를 메모리 안의 자바스크립트 객체 트리로 표현한 개념입니다. 상태가 바뀌면 새로운 가상 DOM을 만들고 이전 가상 DOM과 비교하는데, 이 비교 과정이 Reconciliation입니다. 여기서 변경된 부분만 찾아 실제 DOM에 반영합니다.

핵심은 실제 DOM보다 무조건 빠르다는 주장보다는, 상태 기반 UI를 선언적으로 관리하고 DOM 업데이트를 예측 가능하게 만든다는 점입니다. 특히 리스트 렌더링에서는 `key`를 통해 어떤 항목이 유지되는지 판단하므로, 안정적인 고유값을 `key`로 주는 것이 중요합니다.

## 장점

- 상태 변화와 UI 업데이트를 선언적으로 연결하기 좋다.
- 직접 DOM을 조작하는 코드가 줄어서 유지보수가 쉬워진다.
- 변경분 중심으로 업데이트하므로 복잡한 화면에서도 관리가 편하다.
- 컴포넌트 기반 구조와 잘 맞는다.

## 단점

- 비교 작업 자체도 비용이어서 무조건 빠르다고 볼 수는 없다.
- 렌더링 구조가 비효율적이면 가상 DOM이 있어도 성능 문제가 난다.
- `key`를 잘못 쓰면 디버깅하기 까다로운 UI 버그가 생긴다.
- 내부 동작을 잘 모르면 "왜 다시 렌더링되는지" 파악이 어려울 수 있다.

## 주의사항 / 실무 팁

- 가상 DOM과 실제 DOM을 같은 개념처럼 설명하지 말고, "중간 표현"이라고 구분해서 말하는 게 좋다.
- "가상 DOM이라서 빠르다"보다는 "변경분 관리와 선언적 UI에 유리하다"라고 설명하는 편이 더 정확하다.
- 리스트에서는 index보다 고유 id를 `key`로 쓰는 게 안전하다.
- 성능 이슈가 있으면 가상 DOM만 믿지 말고 렌더링 범위, 상태 위치, 리스트 크기를 함께 봐야 한다.
