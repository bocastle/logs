# useEffect와 cleanup 정리

## 핵심 요약

- `useEffect`는 렌더링이 끝난 뒤 부수 효과를 처리할 때 쓴다.
- 대표적인 부수 효과는 데이터 요청, 이벤트 리스너 등록, 타이머 실행 같은 것들이다.
- cleanup 함수는 이전 effect가 남긴 흔적을 정리해서 메모리 누수나 중복 실행을 막는다.
- 면접에서는 "언제 실행되는지", "왜 cleanup이 필요한지", "의존성 배열에 따라 뭐가 달라지는지"를 같이 설명하면 좋다.

## useEffect는 왜 쓰나?

React 컴포넌트는 기본적으로 UI를 그리는 역할에 집중하는 게 좋다. 그런데 실제 화면에서는 UI만 그리면 끝나지 않는다. API를 호출해야 할 때도 있고, 스크롤 이벤트를 붙여야 할 때도 있고, `setInterval`로 주기적인 작업을 돌려야 할 때도 있다.

이런 작업은 렌더링과는 별개로 다뤄야 한다. 그래서 React는 `useEffect`로 "렌더링 이후에 실행할 작업"을 분리하게 해 둔다.

## 언제 실행되나?

가장 기본은 이 흐름이다.

```tsx
useEffect(() => {
  console.log("effect 실행");
});
```

의존성 배열이 없으면 렌더링이 끝날 때마다 실행된다.

```tsx
useEffect(() => {
  fetchUser();
}, []);
```

빈 배열이면 마운트 직후 한 번 실행되는 패턴으로 많이 쓴다.

```tsx
useEffect(() => {
  syncTheme(theme);
}, [theme]);
```

의존성 배열에 값이 있으면 그 값이 바뀔 때마다 다시 실행된다.

면접에서는 단순히 "마운트 때 실행됩니다" 정도로 끝내기보다, 의존성 배열에 따라 실행 시점이 달라진다고 말해 주는 편이 더 정확하다.

## cleanup은 왜 필요한가?

effect 안에서 등록한 작업은 컴포넌트가 사라지거나 다음 effect가 실행되기 전에 정리해 줘야 할 때가 많다. 이때 반환하는 함수가 cleanup이다.

```tsx
useEffect(() => {
  const id = setInterval(() => {
    console.log("polling...");
  }, 1000);

  return () => {
    clearInterval(id);
  };
}, []);
```

이 cleanup이 없으면 컴포넌트가 사라진 뒤에도 타이머가 계속 돌 수 있다. 이벤트 리스너도 마찬가지다.

```tsx
useEffect(() => {
  const handleResize = () => {
    console.log(window.innerWidth);
  };

  window.addEventListener("resize", handleResize);

  return () => {
    window.removeEventListener("resize", handleResize);
  };
}, []);
```

핵심은 "내가 effect에서 만든 외부 연결을 내가 직접 닫아 줘야 한다"는 점이다.

## cleanup이 실행되는 시점

cleanup은 컴포넌트가 언마운트될 때만 실행된다고 오해하기 쉽다. 실제로는 그것보다 조금 더 넓다.

- 컴포넌트가 언마운트될 때 실행된다.
- 의존성이 바뀌어서 effect가 다시 실행되기 직전에도 이전 cleanup이 먼저 실행된다.

즉, 예전 effect를 정리한 뒤 새 effect를 붙이는 구조라고 보면 된다.

## 실무에서 자주 나오는 문제

### 1. 의존성 배열 누락

effect 안에서 쓰는 값이 의존성 배열에 빠지면 오래된 값을 바라보는 버그가 생길 수 있다. 반대로 무조건 다 넣으면 effect가 너무 자주 돌아서 비효율적일 수도 있다.

그래서 실무에서는 ESLint의 hooks 규칙을 기본으로 따르되, effect 자체를 더 작게 나누거나 이벤트 핸들러 구조를 바꾸는 식으로 해결하는 경우가 많다.

### 2. 데이터 요청 중복

검색어나 필터가 바뀔 때마다 effect에서 요청을 날리면 응답 순서가 꼬일 수 있다. 이럴 때는 `AbortController`로 이전 요청을 취소하거나, 최신 요청만 반영하도록 방어 로직을 둔다.

### 3. effect에 너무 많은 책임을 몰아넣음

하나의 effect에서 API 요청, 이벤트 등록, 상태 동기화까지 다 해 버리면 읽기도 어렵고 버그도 잘 생긴다. 목적이 다른 부수 효과는 나눠 두는 편이 낫다.

## 면접 답변 예시

`useEffect`는 React에서 렌더링이 끝난 뒤 실행해야 하는 부수 효과를 처리할 때 사용하는 훅입니다. 예를 들면 데이터 요청, 이벤트 리스너 등록, 타이머 실행 같은 작업이 여기에 들어갑니다. 그리고 effect에서 등록한 외부 자원은 cleanup 함수로 정리할 수 있는데, 이 cleanup은 컴포넌트가 언마운트될 때뿐 아니라 의존성이 바뀌어 effect가 다시 실행되기 직전에도 호출됩니다. 그래서 메모리 누수나 중복 이벤트 등록을 막는 데 중요합니다.

## 장점

- 렌더링 로직과 부수 효과 로직을 분리할 수 있다.
- 외부 자원 연결과 해제를 한곳에서 관리하기 좋다.
- 의존성 배열을 통해 실행 조건을 비교적 명확하게 표현할 수 있다.

## 단점

- 의존성 배열을 잘못 다루면 무한 실행이나 stale closure 문제가 생길 수 있다.
- effect에 로직이 몰리면 컴포넌트가 빠르게 복잡해진다.
- 비동기 요청이 많아지면 취소 처리나 경쟁 상태 관리까지 신경 써야 한다.

## 주의사항 / 실무 팁

- `useEffect`를 "아무거나 나중에 처리하는 곳"처럼 쓰지 말고 목적별로 나누는 게 좋다.
- 이벤트 리스너, 타이머, 소켓 연결처럼 정리가 필요한 자원은 cleanup을 거의 습관처럼 붙이는 편이 안전하다.
- 의존성 배열 경고를 무작정 끄기보다 effect 구조 자체를 다시 보는 게 더 좋은 해결책인 경우가 많다.
- 서버 컴포넌트나 데이터 패칭 도구를 쓰는 환경에서는 effect가 아닌 더 적절한 방식이 있는지도 같이 판단해야 한다.
