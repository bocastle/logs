# 브라우저 뒤로가기와 폼 상태 복구 설계 정리

## 핵심 요약

- 폼 복구는 검색 조건과 긴 초안처럼 재입력 비용이 큰 값만 대상으로 하고 민감정보는 제외한다.
- 브라우저의 bfcache 복원과 애플리케이션 초기화가 같은 값을 두 번 덮어쓰지 않게 복원 우선순위를 정한다.
- 뒤로가기, 새로고침, 직접 URL 진입과 제출 실패는 수명이 달라 각각 테스트한다.

## 개념 설명

폼 상태 복구는 사용자가 잠깐 다른 화면으로 갔다 돌아오거나 제출에 실패했을 때 입력을 다시 하지 않게 하는 UX다. 모든 값을 무조건 보존하기보다 재입력 비용과 노출 위험을 비교해 필드별 수명을 정한다.

검색 필터처럼 공유할 상태는 URL, 같은 방문 entry에서만 필요한 작은 값은 History state, 새로고침 뒤에도 남길 초안은 제한된 임시 저장소를 검토한다. 비밀번호, 결제정보와 주민번호는 자동 복구 대상에서 제외한다. Bfcache가 DOM을 그대로 복원할 수 있으므로 React 초기화가 복원된 값을 기본값으로 다시 덮지 않게 한다.

## 예시

```ts
history.replaceState({ draft: { q, filters } }, "", location.href);
window.addEventListener("popstate", (event) => restore(event.state?.draft));
```

이 예시는 검색 조건처럼 민감하지 않은 짧은 draft에만 맞다. 개인 작성 초안이라면 계정과 tab 수명, 만료와 명시적 삭제 정책이 추가로 필요하다.

## 면접 답변 예시

> 폼 상태 복구는 재입력 비용이 큰 값과 민감한 값을 먼저 구분하겠습니다. 공유 가능한 검색 조건은 URL, 같은 entry의 UI 상태는 History state, 새로고침 뒤에도 필요한 초안만 제한된 저장소에 둘 수 있습니다. 비밀번호와 결제정보 같은 값은 복구하지 않습니다. 뒤로가기에서 bfcache가 이미 DOM을 복원했는데 애플리케이션 초기화가 값을 덮지 않도록 우선순위를 정하고 각 탐색 경로를 따로 테스트합니다.

## 장점

- 검색과 긴 입력 흐름에서 사용자가 이전 맥락을 유지하고 재입력 비용을 줄일 수 있다.
- 저장 위치와 수명을 필드별로 정하면 복구 정책을 설명하고 테스트하기 쉽다.

## 단점

- 서버 검증 뒤 반환된 값과 오래된 local draft가 충돌할 수 있다.
- History state나 storage에 민감정보를 넣으면 session 복원과 로그·도구를 통해 남을 수 있다.

## 주의사항 / 실무 팁

- 뒤로가기, bfcache 복귀, 새로고침, 직접 URL 진입과 제출 실패를 각각 테스트한다.
- 복원 데이터의 만료, 제출 성공 후 삭제와 계정 전환 시 격리 정책을 정한다.
