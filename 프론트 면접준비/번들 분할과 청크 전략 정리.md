# 번들 분할과 청크 전략 정리

## 핵심 요약

- 번들 분할은 "한 번에 모든 코드를 받는 방식"을 깨고, 화면 동작에 필요한 코드만 우선 로드해 초기 체감 속도를 높이는 기법이다.
- 큰 단위로 분할하면 전송량이 줄고, 너무 쪼개면 요청 수가 늘어나 오히려 느려질 수 있다.
- 면접에서는 **정적 분할**(entry/동기 import)과 **동적 분할**(lazy import, code-splitting) 차이를 구분해 설명하면 좋다.

## 개념 정리

번들러가 하나의 `app.js`를 만드는 기본 동작은 단순하지만, 초기 화면에 필요 없는 코드까지 함께 내려오면 렌더링 시작이 늦어진다. 번들 분할은 아래처럼 동작한다.

1. 공통 코어는 초기 번들에 남김
2. 라우트, 모달, 관리자 페이지처럼 무거운 기능은 별도 청크로 분리
3. 사용 시점에 `import()`로 추가 청크를 받아 실행

예시:

```js
const AdminPage = React.lazy(() => import('./pages/AdminPage'))

function AppRouter() {
  return (
    <Routes>
      <Route path="/admin" element={
        <Suspense fallback={<div>로딩중...</div>}>
          <AdminPage />
        </Suspense>
      } />
    </Routes>
  )
}
```

실무에서는 번들 크기 분석 도구를 이용해 "공통 청크", "중복 청크", "초기 청크"를 계속 모니터링한다.

## 면접 답변 예시

"번들 분할은 초기 렌더링 경로의 JS량을 줄이기 위한 전략입니다. 큰 기능은 별도 청크로 떼어 lazy loading하고, 핵심 경로는 작게 유지해 TTI와 FCP를 개선합니다. 단, 모든 컴포넌트를 다 분할하면 요청 개수가 늘고 오버헤드가 커지므로, 라우트/기능 단위로 균형 있게 자른 뒤 캐시 전략과 병행해야 합니다."

## 장점

- 초기 로딩 지연 완화로 사용자 경험 개선
- 캐시 효율 개선(변경되지 않은 청크 재사용)
- 팀별 개발 병목 완화: 독립 청크 단위 QA/릴리스 가능
- A/B 테스트 시 특정 기능만 재빌드 및 배포 가능

## 단점

- 청크 수가 너무 많아지면 네트워크 오버헤드 상승
- 라우팅 타이밍에 따라 번들 로딩 지연이 가시적으로 보일 수 있음
- 런타임 `Suspense` 처리 실패 시 사용자 체감 에러 증가

## 주의사항 / 실무 팁

- `splitChunks` 임계값을 고정하지 말고 성능 예산(예: Initial JS ≤ 200KB gzip)을 두고 조정한다.
- 중요한 라우트는 사전 프리패치(preload/preload)로 지연을 줄인다.
- 공통 라이브러리는 별도 공통 청크로 뽑되, 너무 이르게 분리하면 다시 합쳐지는 오버헤드가 생길 수 있다.
- 핵심 사용자 흐름에서 청크 로딩 실패 시 대체 UI(재시도 버튼)를 반드시 둔다.
