# 브라우저 캐시 전략 정리

## 핵심 요약

- 브라우저 캐시는 같은 리소스를 다시 내려받지 않도록 해서 로딩 속도와 서버 부담을 줄이는 방식이다.
- 면접에서는 `Cache-Control`, `ETag`, `Last-Modified`, 그리고 강한 캐시와 검증 캐시 차이를 같이 설명하면 흐름이 깔끔하다.
- 정적 파일은 오래 캐시하고, 자주 바뀌는 데이터는 재검증 가능하게 두는 식으로 나누는 게 실무에서 흔하다.
- 캐시를 잘못 잡으면 "배포했는데 사용자 화면이 안 바뀐다" 같은 문제가 생기므로 무효화 전략까지 같이 말하는 게 좋다.

## 브라우저 캐시가 왜 중요하냐

같은 JS, CSS, 이미지 파일을 매번 새로 받으면 네트워크도 낭비되고 초기 렌더링도 느려진다.  
브라우저 캐시는 이미 받은 리소스를 일정 조건 아래 재사용해서 이 비용을 줄인다.

면접에서는 "성능 최적화"를 이야기할 때 코드 스플리팅만 말하기보다, 네트워크 레벨 최적화로 캐시까지 묶어서 설명하면 더 실무적으로 들린다.

## 강한 캐시와 검증 캐시

### 1. 강한 캐시

유효 기간 안에는 서버에 다시 묻지 않고 브라우저가 바로 캐시를 사용한다.  
대표적으로 `Cache-Control: max-age=31536000, immutable` 같은 설정이 여기에 가깝다.

정적 해시 파일(`app.abc123.js`)처럼 파일명이 바뀌는 자산에 잘 맞는다.

### 2. 검증 캐시

브라우저가 캐시를 들고는 있지만, 다시 써도 되는지 서버에 확인한다.  
이때 `ETag`, `If-None-Match`, `Last-Modified`, `If-Modified-Since` 조합이 자주 나온다.

서버가 "안 바뀌었어"라고 판단하면 `304 Not Modified`를 주고 본문은 다시 보내지 않는다.

## 자주 나오는 헤더

### Cache-Control

캐시 정책의 중심이다.

- `max-age`: 몇 초 동안 캐시를 유효하게 볼지
- `no-cache`: 바로 재사용하지 말고 서버에 재검증하라는 의미
- `no-store`: 저장 자체를 하지 말라는 의미
- `public`, `private`: 공유 캐시에 저장 가능한지 범위를 정할 때 사용

면접에서 헷갈리기 쉬운 포인트는 `no-cache`가 "캐시 금지"가 아니라 "재검증 후 사용"에 가깝다는 점이다.

### ETag

리소스 버전표 같은 값이다.  
브라우저가 다음 요청 때 `If-None-Match`로 이전 ETag를 보내고, 서버가 같다고 판단하면 `304`로 응답한다.

### Last-Modified

마지막 수정 시각 기반 검증 방식이다.  
간단하지만 초 단위 정밀도나 복잡한 버전 판별에서는 ETag보다 덜 정확할 수 있다.

## 실무에서 흔한 전략

### 정적 파일

번들 파일 이름에 해시를 붙이고 아주 길게 캐시한다.  
파일 내용이 바뀌면 이름 자체가 바뀌므로 안전하게 강한 캐시를 걸 수 있다.

### HTML 문서

HTML은 배포 때 바뀔 가능성이 높아서 너무 길게 캐시하면 위험하다.  
보통 짧게 캐시하거나 재검증 가능하게 둬서 최신 자산 경로를 빠르게 받게 한다.

### API 응답

모든 API를 무조건 캐시하는 건 아니다.  
조회성 데이터는 조건부 캐시를 검토할 수 있지만, 사용자별 데이터나 실시간성이 중요한 데이터는 더 조심해서 본다.

## 면접에서 같이 묻기 좋은 포인트

### 캐시 무효화는 어떻게 하냐

가장 흔한 답은 "정적 파일은 파일명 해시로 무효화한다"다.  
이 한 문장이 나오면 단순히 개념만 외운 답변보다 훨씬 안정적으로 들린다.

### 304 응답이 왜 의미가 있냐

본문 전송은 줄어들지만, 네트워크 왕복 자체는 남아 있다.  
그래서 재검증 캐시도 유용하지만, 바뀌지 않는 정적 자산은 강한 캐시가 더 효율적일 수 있다.

## 면접 답변 예시

> 브라우저 캐시는 같은 리소스를 반복 다운로드하지 않도록 해서 성능을 높이는 전략입니다. 실무에서는 강한 캐시와 검증 캐시를 나눠서 생각하는데, 해시가 붙은 JS나 CSS 파일은 `Cache-Control`로 오래 캐시하고, HTML이나 자주 바뀌는 리소스는 `ETag`나 `Last-Modified`로 재검증하게 두는 경우가 많습니다. 특히 배포 후 최신 파일을 확실히 받게 하려면 캐시 무효화 전략까지 같이 설계해야 합니다.

## 장점

- 네트워크 요청과 다운로드 용량을 줄일 수 있다.
- 초기 로딩과 재방문 성능 개선에 직접적이다.
- 서버 부하를 줄이는 데도 도움이 된다.

## 단점

- 정책을 잘못 잡으면 오래된 화면이나 리소스를 보여줄 수 있다.
- HTML, 정적 파일, API 응답을 같은 기준으로 다루면 문제가 생기기 쉽다.
- CDN, 프록시 캐시까지 섞이면 디버깅이 까다로워진다.

## 주의사항 / 실무 팁

- 정적 자산은 파일명 해시와 장기 캐시를 함께 가져가는 조합이 가장 안전하다.
- `no-cache`와 `no-store`는 의미가 다르니 면접에서 구분해서 말하는 게 좋다.
- "캐시를 쓴다"에서 끝내지 말고 "언제 무효화할 건가"까지 말하면 답변 밀도가 올라간다.
- 서비스워커 캐시는 브라우저 기본 HTTP 캐시와 별개 층이라, 질문 범위를 구분해서 답하는 게 좋다.
