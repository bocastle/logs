# Streaming SSR 정리

## 핵심 요약
- `Streaming SSR`은 서버가 완성된 HTML을 한 번에 다 보내지 않고, 준비된 부분부터 잘라서 점진적으로 내려보내는 렌더링 방식이다.
- 사용자는 페이지 전체가 끝날 때까지 기다리지 않고, 먼저 준비된 UI를 더 빨리 볼 수 있다.
- React `Suspense`와 함께 자주 설명되며, 느린 데이터가 있는 구간만 뒤에 채워 넣는 흐름으로 이해하면 쉽다.
- 면접에서는 "첫 바이트 이후 화면을 단계적으로 보여 줘 체감 성능을 올리는 SSR 전략"이라고 말하면 핵심이 잡힌다.

## 개념 설명
기본적인 SSR은 서버에서 HTML을 만들어 내려준다는 점에서는 빠르지만, 실제 응답은 템플릿과 데이터가 모두 준비된 뒤에 한 번에 전송되는 경우가 많다. 이때 일부 API가 느리면 페이지 전체 응답도 같이 늦어진다.

`Streaming SSR`은 이 문제를 줄이기 위해 서버가 준비된 마크업부터 먼저 보내고, 아직 준비되지 않은 영역은 뒤에 이어서 전달한다. 그래서 헤더, 본문 골격, 상단 콘텐츠처럼 먼저 보여 줄 수 있는 부분은 빨리 그려지고, 추천 목록이나 댓글처럼 늦게 와도 되는 영역은 나중에 붙는다.

핵심은 "SSR을 더 잘게 나눠서 사용자에게 보여 주는 순서를 제어한다"는 점이다. 단순히 서버에서 렌더링한다는 수준을 넘어서, 어떤 정보를 먼저 보여 주고 어떤 구간은 기다리게 할지 설계하는 방식이라고 보면 된다.

## 예시
뉴스 상세 페이지를 생각해보자.

- 바로 보여 줄 수 있는 영역
  - 헤더
  - 기사 본문
  - 작성자 정보
- 늦게 와도 되는 영역
  - 관련 기사 추천
  - 실시간 댓글
  - 개인화 추천 배너

기존 SSR에서는 관련 기사 API가 느리면 전체 HTML 전송이 늦어질 수 있다. 반면 `Streaming SSR`을 쓰면 본문과 헤더는 먼저 보내고, 추천 영역은 로딩 상태나 fallback을 먼저 보여 준 뒤 준비가 끝났을 때 이어서 채워 넣을 수 있다.

## 면접 답변 예시
- 질문: "`Streaming SSR`은 무엇인가요?"  
  답변: "`Streaming SSR`은 서버가 완성된 HTML 전체를 한 번에 보내는 대신, 준비된 부분부터 순차적으로 내려 보내는 서버 렌더링 방식입니다. 그래서 느린 데이터가 일부 있어도 사용자에게 기본 화면을 더 빨리 보여 줄 수 있습니다."
- 질문: "일반 SSR과 차이가 뭔가요?"  
  답변: "일반 SSR은 응답이 다 준비된 뒤 한 번에 보내는 경우가 많고, `Streaming SSR`은 준비된 조각부터 먼저 보냅니다. 그래서 전체 완성 시점은 비슷할 수 있어도 사용자가 체감하는 초기 화면 노출은 더 빨라질 수 있습니다."

## 장점
- 느린 데이터 때문에 페이지 전체 응답이 묶이는 문제를 줄이기 좋다
- 사용자에게 먼저 보여 줄 수 있는 콘텐츠를 빠르게 노출할 수 있다
- `Suspense` 같은 메커니즘과 결합해 로딩 경험을 더 자연스럽게 만들 수 있다

## 단점
- 렌더링 흐름이 단순 SSR보다 복잡해져 디버깅 난이도가 올라갈 수 있다
- 어떤 영역을 먼저 보여 줄지 설계가 부족하면 기대만큼 체감 성능이 좋아지지 않을 수 있다
- 서버와 프레임워크가 스트리밍 렌더링을 안정적으로 지원해야 한다

## 주의사항 / 실무 팁
- 먼저 보여 줄 가치가 큰 영역과 늦어도 되는 영역을 분리하는 것이 중요하다.
- fallback UI가 너무 많거나 어색하면 오히려 화면이 덜 안정적으로 느껴질 수 있다.
- 면접에서는 `SSR`, `Suspense`, `Hydration`과 연결해서 설명하면 답변이 자연스럽다.
- 실무에서는 SEO가 중요한 핵심 콘텐츠는 가능한 한 먼저 스트리밍되도록 설계하고, 개인화 영역은 뒤로 미루는 식의 우선순위 조정이 현실적이다.
