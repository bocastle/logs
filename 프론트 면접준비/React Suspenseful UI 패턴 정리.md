# React Suspenseful UI 패턴 정리

## 핵심 요약

- Suspenseful UI는 데이터 요청과 렌더링 상태를 분리해 사용자 체감 응답성을 높인다.
- 핵심은 로딩/에러/완료 상태를 선언적으로 다루는 것이다.
- 면접에서 중요한 것은 `Suspense`, `Error Boundary`, 라우팅/데이터 계층 결합을 어떻게 안정화하는지다.

## 개념 설명

### 1) Suspense의 목적

- 데이터 로딩 전, `fallback`으로 대체 UI를 보여주고 실제 컴포넌트는 준비되면 교체한다.
- 사용자에게 즉시 피드백을 주면서 화면 멈춤을 줄인다.

### 2) 경계 분리

- 데이터 계층: 언제 불러올지, 언제 재시도할지
- UI 계층: 로딩 스켈레톤/플레이스홀더 처리
- 오류 계층: 실패 시 우회 렌더링

### 3) SSR/CSR 연동

- 서버에서 미리 데이터 패칭이 가능하면 초기 공백을 줄일 수 있다.
- 클라이언트에서 하이드레이션 직후 추가 fetch면 순간 플리커를 줄이기 위한 전략 필요

## 사용 예시

- 피드 목록 로딩
- 사용자 프로필 카드
- 다단계 검색 결과 패널

### 예시 구조

- 상위: 페이지 레벨에서 Suspense 구간 분리
- 하위: 컨텐츠 컴포넌트는 데이터 의존성만 가짐

## 면접 답변 예시

"Suspenseful UI의 핵심은 UI를 블로킹 없이 상태 기반으로 구성하는 것입니다. 단순히 `fallback`을 넣는 게 아니라, 실패와 타임아웃 시 사용자 흐름이 끊기지 않도록 Error Boundary와 재시도 정책을 함께 설계해야 합니다. 특히 라우팅 전환 구간에서 과도한 재로딩이 생기면 오히려 UX가 악화되니, 캐시 정책과 결합해 조절해야 합니다."

## 장점

- 지연 시간 체감 개선
- 상태 분리가 명확해 유지보수성 향상
- 화면 체감 속도 개선

## 단점

- 중첩된 Suspense 관리 복잡
- 에러 경로가 늘어남
- 동적 라우트와 병행 시 디버깅 난이도 증가

## 주의사항 / 실무 팁

- Suspense 중첩 기준을 명확히 정의한다.
- 동일 UI 섹션에서 로딩/에러/성공 UI가 겹치지 않게 단일 책임 원칙을 지킨다.
- 로깅 시 fallback 표시 횟수와 에러 폴링을 추적한다.
