# 코드 스플리팅과 Lazy Loading 정리

## 핵심 요약

- 코드 스플리팅은 큰 번들을 여러 덩어리로 나눠서 필요한 시점에만 받게 만드는 방식이다.
- Lazy Loading은 지금 당장 필요 없는 코드를 나중에 불러오는 전략이다.
- 둘은 같이 자주 설명되지만, 코드 스플리팅은 "나누는 것", Lazy Loading은 "나중에 불러오는 것"에 더 가깝다.
- 초기 로딩 성능에 도움이 되지만, 분할이 과하면 요청 수나 사용자 체감 지연이 오히려 늘 수 있다.

## 왜 필요한가?

프론트엔드 애플리케이션이 커지면 첫 진입 시 내려받는 JavaScript 양도 커진다. 그러면 다운로드, 파싱, 실행 시간이 길어지고 초기 화면 표시나 상호작용 가능 시점이 늦어진다.

예를 들어 관리자 페이지, 차트 라이브러리, 파일 업로드 모듈 같은 무거운 기능이 메인 페이지 진입 시점부터 모두 번들에 들어 있으면, 실제로는 안 쓰는 코드까지 먼저 받게 된다.

이럴 때 필요한 것이 코드 스플리팅과 Lazy Loading이다.

## 코드 스플리팅이란?

코드 스플리팅은 하나의 큰 번들을 여러 청크로 나누는 것이다.

대표적으로 다음 기준으로 나눌 수 있다.

- 라우트 기준 분할: 페이지별로 나누기
- 기능 기준 분할: 차트, 에디터, 결제 모듈처럼 무거운 기능 단위로 나누기
- 라이브러리 기준 분할: 공통 벤더 코드 분리

핵심은 "처음부터 모든 코드를 한 번에 받지 않게 하는 것"이다.

## Lazy Loading이란?

Lazy Loading은 분리된 코드를 실제로 필요할 때 가져오는 방식이다.

예를 들어 사용자가 `/admin` 페이지에 들어갈 때만 관리자 관련 번들을 불러오거나, 모달을 열 때만 에디터 코드를 불러오는 식이다.

React에서는 `React.lazy`와 `Suspense`, 또는 프레임워크 차원의 dynamic import 기능을 자주 쓴다.

```jsx
const AdminPage = React.lazy(() => import("./AdminPage"));
```

이 코드는 `AdminPage`가 필요한 시점까지 해당 코드를 뒤로 미루는 예시다.

## 둘의 차이

면접에서는 이 둘을 섞어 말하기 쉬운데, 구분해서 말하면 더 깔끔하다.

- 코드 스플리팅: 번들을 여러 조각으로 나누는 것
- Lazy Loading: 그 조각을 나중에 가져오는 것

즉 나눠 놓기만 하고 전부 초기에 받아 버리면 Lazy Loading의 이점은 줄어든다. 반대로 늦게 불러오려면 먼저 적절히 나눠져 있어야 한다.

## 어디에 적용하면 효과적인가?

### 1. 라우트 단위 페이지

가장 흔하고 안전한 시작점이다. 홈, 마이페이지, 관리자 페이지가 한 번들에 다 묶여 있을 필요는 없는 경우가 많다.

### 2. 무거운 기능성 UI

차트, 에디터, 지도, PDF 뷰어처럼 라이브러리 용량이 큰 경우 효과가 크다.

### 3. 자주 안 쓰는 모달이나 설정 화면

사용 빈도가 낮고 진입이 명확한 화면은 늦게 불러와도 사용자 경험에 큰 문제가 없는 경우가 많다.

## 장점만 있는 것은 아니다

초기 번들 크기를 줄일 수 있다는 장점이 있지만, 분할이 과하면 청크 요청이 너무 많아질 수 있다. 그리고 사용자가 특정 화면으로 이동하는 순간 로딩이 한 번 더 생기기 때문에, 경우에 따라서는 "첫 화면은 빨라졌는데 페이지 이동이 끊긴다"는 체감이 생길 수 있다.

그래서 중요한 건 무조건 잘게 자르는 게 아니라, 사용 흐름을 기준으로 적당히 나누는 것이다.

## 실무 예시

예를 들어 일반 사용자 대부분은 상품 목록과 상세만 쓰고, 관리자만 통계 대시보드에 들어간다고 해보자.

이때 차트 라이브러리와 대시보드 코드를 메인 번들에 포함하면 일반 사용자도 불필요한 코드를 같이 받는다. 반대로 관리자 화면 진입 시점에만 해당 코드를 lazy load하면 일반 사용자 초기 로딩은 더 가벼워진다.

다만 관리자 입장에서는 첫 진입 시 로딩이 생길 수 있으니, 필요하면 prefetch나 skeleton UI를 같이 고려할 수 있다.

## 면접 답변 예시

코드 스플리팅은 큰 프론트엔드 번들을 여러 청크로 나누는 것이고, Lazy Loading은 그 청크를 실제로 필요할 때 불러오는 전략입니다. 목적은 초기 진입 시 불필요한 JavaScript를 덜 받게 해서 다운로드와 파싱 비용을 줄이는 데 있습니다. 보통은 라우트 단위나 무거운 기능 단위로 나누고, 관리자 페이지나 차트 같은 저빈도 기능에 적용하면 효과가 큽니다. 다만 너무 잘게 나누면 요청 수가 늘고 화면 이동 시 지연이 생길 수 있어서, 사용자 흐름을 기준으로 적절히 분할하는 게 중요합니다.

## 장점

- 초기 번들 크기를 줄여 첫 화면 로딩 개선에 도움이 된다.
- 자주 안 쓰는 기능의 비용을 뒤로 미룰 수 있다.
- 큰 라이브러리의 영향을 필요한 사용자에게만 제한할 수 있다.
- 페이지나 기능 단위 성능 최적화를 더 세밀하게 할 수 있다.

## 단점

- 분할이 과하면 요청 수가 늘고 관리가 복잡해질 수 있다.
- 특정 화면 첫 진입 시 로딩 지연이 생길 수 있다.
- fallback UI나 에러 처리까지 챙겨야 해서 구현 고려사항이 늘어난다.
- 번들 구조를 잘못 잡으면 체감 성능이 오히려 나빠질 수 있다.

## 주의사항 / 실무 팁

- 가장 먼저 라우트 단위 분할부터 적용하면 리스크가 낮다.
- 무거운 라이브러리가 실제로 어디서 비용을 크게 만드는지 번들 분석으로 확인하고 나눈다.
- Lazy Loading을 넣었다면 로딩 fallback UI와 실패 시 재시도 경험도 같이 생각해야 한다.
- 너무 작은 단위까지 쪼개기보다 사용 흐름상 진입 시점이 분명한 영역 위주로 나눈다.
- 초기 로딩만 보지 말고, 화면 이동 시 체감 성능과 캐시 효과까지 함께 봐야 한다.
