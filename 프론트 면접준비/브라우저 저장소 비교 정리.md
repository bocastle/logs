# 브라우저 저장소 비교 정리

## 핵심 요약

- 브라우저 저장소를 말할 때 보통 `cookie`, `localStorage`, `sessionStorage`를 같이 비교한다.
- 쿠키는 서버와 함께 주고받을 수 있고, `localStorage`와 `sessionStorage`는 브라우저에만 저장된다.
- `localStorage`는 브라우저를 닫아도 남고, `sessionStorage`는 탭 세션이 끝나면 사라진다.
- 면접에서는 보안, 만료, 서버 전송 여부까지 같이 설명해야 답이 탄탄해진다.

## 왜 구분해서 알아야 하나?

실무에서는 "브라우저에 저장한다"는 말만으로는 부족하다. 로그인 상태를 어디에 둘지, 임시 폼 데이터를 어디에 둘지, 사용자 설정값을 어디에 둘지에 따라 선택이 달라진다.

같은 저장소처럼 보여도 동작 방식과 보안 성격이 다르기 때문이다.

## Cookie

쿠키는 브라우저에 저장되지만, 요청을 보낼 때 서버로 함께 전송될 수 있다는 점이 가장 큰 특징이다.

예를 들어 세션 기반 로그인에서는 서버가 세션 ID를 쿠키로 내려 주고, 이후 요청마다 브라우저가 그 쿠키를 같이 보낸다. 서버는 그 값을 보고 사용자를 식별한다.

쿠키의 특징을 정리하면 이렇다.

- 만료 시간을 설정할 수 있다.
- 서버 요청에 자동 포함될 수 있다.
- 크기가 작고, 매 요청마다 붙으면 네트워크 부담이 생길 수 있다.
- `HttpOnly`, `Secure`, `SameSite` 같은 보안 옵션을 줄 수 있다.

## localStorage

`localStorage`는 브라우저에 key-value 형태로 데이터를 저장하는 공간이다. 브라우저를 닫았다가 다시 열어도 데이터가 남아 있다.

주로 이런 데 쓴다.

- 다크 모드 같은 사용자 설정 저장
- 최근 본 탭 정보 저장
- 새로고침 후에도 유지하고 싶은 간단한 클라이언트 상태 저장

다만 JavaScript에서 바로 접근할 수 있어서 XSS에 취약할 수 있다. 그래서 민감한 인증 토큰을 무심코 넣는 건 조심해야 한다.

## sessionStorage

`sessionStorage`도 key-value 저장소라는 점은 비슷하지만, 탭 세션 기준으로 동작한다. 같은 브라우저라도 탭이 달라지면 별개로 취급되고, 탭을 닫으면 데이터가 사라진다.

예를 들면 이런 경우에 어울린다.

- 여러 탭끼리 공유되면 안 되는 임시 입력값
- 한 번 열린 화면 안에서만 유지하면 되는 상태
- 결제나 신청 흐름에서 일시적으로 쓰는 값

## 셋의 차이를 한 번에 정리하면

### 1. 서버 전송 여부

- 쿠키: 요청 시 서버로 함께 갈 수 있다.
- `localStorage`, `sessionStorage`: 브라우저 안에서만 쓰고 자동 전송되지 않는다.

### 2. 수명

- 쿠키: 만료 시간 설정 가능
- `localStorage`: 직접 지우기 전까지 유지
- `sessionStorage`: 탭 세션 종료 시 제거

### 3. 보안 성격

- 쿠키는 `HttpOnly`를 걸면 JavaScript에서 접근하지 못하게 할 수 있다.
- `localStorage`, `sessionStorage`는 JavaScript에서 접근 가능하므로 XSS에 더 민감하다.

### 4. 대표 용도

- 쿠키: 인증 세션, 서버와 함께 관리하는 사용자 식별 정보
- `localStorage`: 사용자 설정, 장기 유지 클라이언트 데이터
- `sessionStorage`: 탭 단위 임시 상태

## 실무에서 자주 나오는 판단 기준

인증과 연결된 정보라면 보통 보안 속성을 가진 쿠키가 더 자주 언급된다. 반면 화면 설정값처럼 클라이언트에서만 쓰는 값은 `localStorage`가 편하다.

하지만 여기서 끝내면 조금 얕다. 예를 들어 JWT를 어디에 저장할지는 보안 정책, CSRF 방어 방식, XSS 위험도, 백엔드 인증 구조까지 같이 봐야 한다. 그래서 면접에서는 "무조건 localStorage"나 "무조건 쿠키"처럼 단정하지 않는 게 낫다.

## 예시

### 다크 모드 저장

사용자가 다크 모드를 켜면 다음 방문에도 유지되길 원한다. 이런 값은 서버에 자동 전송할 필요가 없으니 `localStorage`가 잘 맞는다.

### 세션 로그인

서버가 세션 ID를 관리하고 요청마다 인증이 필요하다면 쿠키가 자연스럽다. 특히 `HttpOnly` 쿠키를 쓰면 JavaScript에서 세션 값을 직접 읽지 못하게 할 수 있다.

### 신청 폼 임시 저장

한 탭 안에서만 잠깐 유지되면 충분하고 탭을 닫으면 없어져도 되는 값은 `sessionStorage`가 잘 맞는다.

## 면접 답변 예시

브라우저 저장소를 비교할 때는 쿠키와 `localStorage`, `sessionStorage`를 같이 봅니다. 쿠키는 서버 요청에 함께 전송될 수 있어서 세션 인증처럼 서버와 연동되는 정보에 자주 쓰이고, `HttpOnly` 같은 보안 속성도 줄 수 있습니다. 반면 `localStorage`와 `sessionStorage`는 브라우저 안에만 저장되고 자동 전송되지 않습니다. `localStorage`는 브라우저를 닫아도 남아서 사용자 설정 저장에 적합하고, `sessionStorage`는 탭 세션 동안만 유지돼 임시 상태 저장에 잘 맞습니다.

## 장점

- 용도에 따라 저장 전략을 세분화할 수 있다.
- 클라이언트 상태를 간단하게 유지하기 좋다.
- 쿠키는 서버 인증 구조와 자연스럽게 연결할 수 있다.

## 단점

- 저장 위치를 잘못 고르면 보안 문제가 생길 수 있다.
- 쿠키는 요청마다 전송돼 불필요한 네트워크 비용이 생길 수 있다.
- 브라우저 저장소는 어디까지나 클라이언트 저장소라 서버 기준 신뢰 데이터로 보기 어렵다.

## 주의사항 / 실무 팁

- 민감한 인증 정보는 XSS 위험까지 고려해서 저장 위치를 정해야 한다.
- 쿠키를 쓸 때는 `HttpOnly`, `Secure`, `SameSite`를 같이 보는 습관이 중요하다.
- 브라우저 저장소에는 너무 큰 데이터를 넣지 않는 편이 좋다.
- 면접에서는 "어떤 값을 어디에 왜 저장할지"를 상황 중심으로 설명하면 훨씬 설득력이 있다.
