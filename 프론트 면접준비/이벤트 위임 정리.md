# 이벤트 위임 정리

## 핵심 요약

- 이벤트 위임은 여러 자식 요소에 각각 이벤트를 붙이지 않고, 공통 부모에 한 번만 이벤트를 등록해서 처리하는 방식입니다.
- 핵심 전제는 브라우저 이벤트가 버블링된다는 점입니다. 아래에서 발생한 이벤트가 위로 올라오기 때문에 부모가 대신 받을 수 있습니다.
- 동적으로 추가되는 요소에도 별도 이벤트 등록 없이 대응하기 쉬워서 실무에서 자주 씁니다.
- 대신 `event.target`과 `currentTarget` 차이, `closest()` 사용, 버블링되지 않는 이벤트 예외는 같이 알고 있어야 면접 답변이 안정적입니다.

## 이벤트 위임이란?

이벤트 위임은 말 그대로 "처리를 부모에게 맡기는 것"입니다.  
예를 들어 목록 안에 버튼이 100개 있으면 버튼마다 클릭 이벤트를 100번 다는 대신, 목록 부모 하나에만 클릭 이벤트를 달아두고 어떤 버튼이 눌렸는지 내부에서 구분하는 식입니다.

면접에서는 보통 이렇게 말하면 자연스럽습니다.

> "이벤트가 버블링되는 특성을 이용해서, 자식마다 핸들러를 붙이지 않고 상위 요소 하나에서 공통으로 처리하는 방식입니다."

## 왜 쓰는가?

이벤트 위임을 쓰는 이유는 크게 세 가지입니다.

### 1. 코드가 단순해집니다

같은 역할을 하는 자식 요소가 많을수록 각각에 이벤트를 다는 건 번거롭습니다.  
부모 하나만 관리하면 되니 초기 코드도 단순하고 수정할 때도 편합니다.

### 2. 동적 요소에 강합니다

실무에서는 리스트 아이템이나 버튼이 나중에 API 응답으로 추가되는 경우가 많습니다.  
각 요소가 생길 때마다 이벤트를 다시 붙이면 누락되기 쉽습니다.

이벤트 위임을 쓰면 부모는 이미 살아 있으니, 새로 들어온 자식도 별도 처리 없이 동작합니다.

### 3. 이벤트 리스너 수를 줄일 수 있습니다

요소가 아주 많을 때는 리스너를 개별로 붙이는 것보다 부모 하나에 집중시키는 편이 관리상 유리합니다.  
다만 이것만 가지고 무조건 성능이 좋아진다고 과장해서 말할 필요는 없습니다. 요즘 브라우저는 리스너 처리도 꽤 최적화되어 있어서, "관리성과 동적 UI 대응에 더 큰 장점이 있다" 정도로 말하는 편이 좋습니다.

## 동작 원리: 버블링

이벤트 위임은 이벤트 버블링을 기반으로 합니다.

예를 들어 버튼을 클릭하면 실제 클릭은 버튼에서 시작되지만, 이벤트는 그 버튼의 부모, 조상 요소 쪽으로 계속 전파될 수 있습니다.  
그래서 부모에서 클릭 이벤트를 듣고 있다가 "방금 클릭된 게 내가 원하는 버튼인지" 확인해서 처리할 수 있습니다.

정리하면 흐름은 이렇습니다.

1. 자식 요소에서 이벤트가 발생합니다.
2. 이벤트가 부모 방향으로 올라갑니다.
3. 부모 핸들러에서 `event.target`을 보고 실제 발생 지점을 확인합니다.
4. 원하는 선택자와 맞으면 해당 로직을 실행합니다.

## `event.target`과 `currentTarget` 차이

이 부분은 면접에서 자주 꼬리 질문이 나옵니다.

- `event.target`: 실제로 이벤트가 발생한 요소
- `event.currentTarget`: 현재 핸들러가 등록된 요소

이벤트 위임에서는 보통 부모에 이벤트를 등록하므로 `currentTarget`은 부모입니다.  
반면 실제로 클릭된 버튼이나 아이콘은 `target`이 됩니다.

예를 들어 버튼 안에 `span` 아이콘을 눌렀다면 `target`은 `span`일 수 있습니다.  
그래서 보통은 `target.matches()`만 쓰기보다 `target.closest()`를 함께 써서 의도한 버튼을 찾습니다.

## 예시

```html
<ul id="menu">
  <li><button data-action="save">저장</button></li>
  <li><button data-action="delete">삭제</button></li>
  <li><button data-action="share">공유</button></li>
</ul>
```

```javascript
const menu = document.querySelector("#menu");

menu.addEventListener("click", (event) => {
  const button = event.target.closest("button");

  if (!button || !menu.contains(button)) {
    return;
  }

  const action = button.dataset.action;

  if (action === "save") {
    console.log("저장");
  }

  if (action === "delete") {
    console.log("삭제");
  }

  if (action === "share") {
    console.log("공유");
  }
});
```

이 코드의 포인트는 세 가지입니다.

- 클릭 핸들러는 `menu`에 한 번만 등록합니다.
- 실제 클릭된 위치가 버튼 내부 다른 태그일 수 있으니 `closest("button")`로 올바른 버튼을 찾습니다.
- 부모 바깥쪽 요소가 섞여 들어오는 걸 막기 위해 `contains()`로 한 번 더 방어합니다.

## 언제 특히 유용한가?

### 목록형 UI

댓글 목록, 상품 목록, 테이블 행 액션 버튼처럼 비슷한 요소가 반복될 때 잘 맞습니다.

### 동적 렌더링 UI

검색 결과, 무한 스크롤, 모달 내부 옵션처럼 나중에 요소가 추가되는 화면에서 편합니다.

### 공통 액션 처리

`data-*` 속성으로 액션 타입만 달리하고, 부모에서 분기해서 처리하는 패턴이 자주 쓰입니다.

## 모든 이벤트에 다 쓸 수 있는가?

아닙니다. 여기서 한 번 더 깊게 설명하면 답변이 좋아집니다.

이벤트 위임은 기본적으로 버블링되는 이벤트에 잘 맞습니다.  
대표적으로 `click`은 잘 됩니다.

반면 버블링되지 않는 이벤트는 그대로는 위임이 어렵습니다.  
예를 들어 `focus`, `blur`는 일반적인 버블링 이벤트가 아니라서 바로 같은 방식으로 쓰기 애매합니다.

이럴 때는 대체 이벤트를 씁니다.

- `focus` 대신 `focusin`
- `blur` 대신 `focusout`

즉, 면접에서는 "이벤트 위임은 만능이 아니라 버블링 여부를 봐야 한다"까지 말해주면 좋습니다.

## React에서는 어떻게 보나?

React 질문으로 이어지면 이렇게 연결하면 됩니다.

React는 내부적으로 이벤트를 관리할 때 루트 단에서 위임 비슷한 방식으로 처리해온 역사도 있어서, 이벤트 위임 개념 자체를 알고 있으면 React 이벤트 시스템 이해에도 도움이 됩니다.  
다만 실무에서는 React가 대부분의 이벤트 처리를 추상화해주기 때문에, 직접 DOM 이벤트 위임을 짜는 경우는 바닐라 JS나 특정 성능/통합 상황에서 더 자주 보게 됩니다.

여기서 중요한 건 React 내부 구현 세부사항을 장황하게 말하는 것보다, "브라우저 이벤트 전파 구조를 이해하고 있다"는 인상을 주는 것입니다.

## 장점

- 자식 요소가 많아도 부모 하나만 관리하면 돼서 코드가 단순합니다.
- 동적으로 생성되는 요소에도 추가 작업 없이 대응하기 쉽습니다.
- 공통 로직을 한곳에 모을 수 있어서 유지보수하기 좋습니다.
- 반복 UI에서 이벤트 핸들러 관리 비용을 줄이기 좋습니다.

## 단점

- 실제 이벤트 발생 지점을 구분하는 로직이 필요해서 처음엔 조금 헷갈릴 수 있습니다.
- 자식 구조가 복잡하면 `target`, `closest()`, 예외 처리 코드가 늘어날 수 있습니다.
- 버블링되지 않는 이벤트에는 그대로 적용하기 어렵습니다.
- 모든 상황에 위임을 강제하면 오히려 코드 가독성이 떨어질 수 있습니다.

## 주의사항 / 실무 팁

- `event.target`만 바로 믿지 말고 `closest()`로 의도한 요소를 찾는 습관이 중요합니다.
- 버튼 안에 아이콘, 텍스트, 중첩 태그가 있으면 실제 `target`은 버튼이 아닐 수 있습니다.
- 부모 하나에 너무 많은 분기 로직을 몰아넣으면 "이벤트 위임"이 아니라 "거대한 조건문"이 되기 쉽습니다. 액션별 함수 분리가 낫습니다.
- 버블링이 막혀 있으면 위임도 기대대로 동작하지 않습니다. 중간에 `stopPropagation()`을 쓰는 코드가 있는지 봐야 합니다.
- 성능 얘기를 할 때는 "리스너 수 감소"만 반복하지 말고, 동적 UI 대응과 유지보수성을 같이 묶어서 설명하는 편이 더 실무적입니다.

## 면접 답변 예시

이벤트 위임은 자식 요소마다 이벤트를 개별로 붙이지 않고, 상위 부모 요소에 한 번만 이벤트를 등록해서 처리하는 방식입니다. 핵심은 이벤트 버블링인데, 자식에서 발생한 클릭 이벤트가 부모까지 올라오기 때문에 부모에서 실제 발생 지점을 확인해서 분기할 수 있습니다.

이 방식은 리스트처럼 비슷한 요소가 많거나, 나중에 동적으로 요소가 추가되는 화면에서 특히 유용합니다. 예를 들어 버튼이 계속 추가되는 목록에서는 버튼마다 리스너를 다시 붙이지 않아도 부모 하나에서 처리할 수 있어서 관리가 편합니다.

다만 모든 이벤트에 그대로 적용되는 건 아니고, 버블링되지 않는 이벤트는 주의해야 합니다. 또 실제 클릭 위치가 버튼 내부 아이콘일 수 있어서 `event.target`만 보지 말고 `closest()`로 원하는 요소를 찾는 식으로 구현하는 경우가 많습니다.
