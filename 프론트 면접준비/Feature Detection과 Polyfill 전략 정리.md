# Feature Detection과 Polyfill 전략 정리

## 핵심 요약
- `feature detection`은 브라우저 이름이나 버전이 아니라 "지금 이 환경에 필요한 기능이 실제로 있는가"를 확인하는 방식이다.
- `polyfill`은 없는 기능을 최대한 비슷하게 메워서 구형 환경에서도 동작하게 만드는 코드다.
- 면접에서는 "지원 여부를 추측하지 말고 기능 존재를 확인한 뒤, 필요할 때만 대체 수단을 넣는 전략"이라고 설명하면 좋다.
- 핵심은 호환성 확보와 번들 크기 사이의 균형이다.

## 개념 설명
예전에는 브라우저 종류나 버전을 보고 분기하는 경우가 많았지만, 이 방식은 유지보수가 어렵고 예외가 많다. 같은 Chrome 계열이라도 지원 범위가 다를 수 있고, UA 문자열은 신뢰성이 떨어질 수 있다.

그래서 보통은 `if ('IntersectionObserver' in window)`처럼 필요한 기능이 실제로 존재하는지 확인하는 `feature detection`을 먼저 쓴다. 이렇게 하면 특정 브라우저를 찍어 누르기보다, 기능 유무를 기준으로 더 안전하게 동작을 나눌 수 있다.

만약 기능이 없는데도 서비스 핵심 흐름이 유지돼야 한다면 `polyfill`을 고려한다. 예를 들어 오래된 브라우저에서 `Promise`, `fetch`, `IntersectionObserver`가 필요하다면 해당 기능만 보완해 주는 식이다. 다만 모든 기능이 polyfill 가능한 것은 아니고, 성능이나 동작 차이 때문에 완전한 대체가 어려운 경우도 있다.

## 예시
무한 스크롤 페이지에서 `IntersectionObserver`를 쓴다고 가정해 보자.

- 지원 브라우저에서는 `IntersectionObserver`로 효율적으로 감지한다
- 미지원 브라우저에서는 polyfill을 로드하거나
- 더 단순한 fallback으로 "더 보기" 버튼을 보여 줄 수도 있다

이처럼 핵심은 "지원 안 되면 서비스가 아예 멈추는가, 아니면 조금 덜 편해져도 되는가"를 먼저 나누는 것이다.

## 면접 답변 예시
- 질문: "브라우저 호환성은 어떻게 대응하나요?"  
  답변: "보통 브라우저 이름으로 분기하기보다 feature detection으로 필요한 기능 존재 여부를 먼저 확인합니다. 그리고 서비스 핵심에 필요한 기능이 빠져 있으면 polyfill이나 fallback UI를 넣어서 점진적으로 대응합니다."
- 질문: "polyfill을 항상 넣는 게 좋은가요?"  
  답변: "항상 그렇지는 않습니다. polyfill은 번들 크기와 초기 로딩 비용을 늘릴 수 있어서, 실제 지원 대상 브라우저와 기능 중요도를 보고 선택적으로 넣는 게 좋습니다."

## 장점
- 호환성 대응을 단순 암기가 아니라 설계 관점으로 설명할 수 있다
- Progressive Enhancement, 번들 최적화, 사용자 경험을 함께 연결하기 좋다
- 면접에서 "지원 여부 확인 -> 대체 전략 선택" 흐름으로 말하기 쉬운 주제다

## 단점
- polyfill과 transpile의 차이를 헷갈리면 설명이 흐려질 수 있다
- 모든 기능을 polyfill로 해결할 수 있다고 과장하기 쉽다
- 실제 지원 대상 브라우저 기준 없이 말하면 추상적 답변이 되기 쉽다

## 주의사항 / 실무 팁
- `feature detection`과 브라우저 sniffing은 구분해서 말하는 게 좋다.
- 문법 변환은 주로 Babel 같은 transpile 단계의 역할이고, 런타임 API 보완은 polyfill에 가깝다는 점을 정리해 두면 좋다.
- 핵심 사용자 흐름이 깨지지 않도록 fallback UI를 함께 설계하는 습관이 실무에서 중요하다.
- 면접에서는 "`무조건 최신 기능을 쓰는 것`보다 `지원 환경을 보고 선택적으로 적용하는 것`이 더 현실적이다"라고 마무리하면 좋다.
