# AbortController와 요청 취소 정리

## 핵심 요약

- `AbortController`는 진행 중인 비동기 작업, 특히 `fetch` 요청을 중단할 수 있게 해 주는 브라우저 API다.
- 면접에서는 "왜 요청 취소가 필요한가", "컴포넌트 unmount와 어떤 관련이 있나", "경쟁 상태를 어떻게 줄이냐"를 말하면 좋다.
- 사용자가 빠르게 검색어를 바꾸거나 화면을 떠나는 상황에서 불필요한 응답 반영을 막는 데 유용하다.
- 요청 취소는 성능 문제이기도 하지만, 더 자주 나오는 포인트는 오래된 응답이 최신 UI를 덮어쓰는 버그를 줄이는 데 있다.

## 요청 취소가 왜 필요하냐

프론트에서는 비동기 요청이 자주 겹친다.  
예를 들어 검색창에서 `a`를 입력해 요청을 보냈는데, 사용자가 바로 `ab`를 입력해 두 번째 요청이 나갈 수 있다.

이때 첫 번째 요청이 더 늦게 도착하면, 오래된 응답이 최신 결과를 덮어쓰는 문제가 생긴다.  
그래서 "필요 없는 요청은 취소하고, 살아 있는 요청만 반영하는 구조"가 중요하다.

## AbortController가 하는 일

`AbortController`를 만들고, 거기서 꺼낸 `signal`을 `fetch`에 전달하면 된다.  
이후 `abort()`를 호출하면 해당 요청은 중단된다.

핵심은 "취소 의도를 전달하는 표준 통로"라고 이해하면 된다.

## 어떤 상황에서 많이 쓰나

### 1. 컴포넌트 unmount

페이지를 떠났는데 뒤늦게 응답이 와서 상태를 업데이트하려고 하면 경고나 버그 원인이 될 수 있다.  
React에서는 effect 정리 시점에 요청을 취소하는 패턴이 자주 나온다.

### 2. 검색 자동완성

이전 검색어 요청은 취소하고 마지막 입력 기준 요청만 남기는 방식이다.  
디바운스와 같이 쓰면 더 안정적이다.

### 3. 탭 전환, 필터 변경

사용자가 조건을 빠르게 바꾸는 UI에서는 이전 요청을 끝까지 기다리기보다 끊는 편이 낫다.

## 예시로 이해하기

흐름은 단순하다.

1. 요청 시작 전에 `AbortController`를 만든다.
2. `fetch`에 `signal`을 넘긴다.
3. 더 이상 필요 없으면 `abort()`를 호출한다.
4. 취소된 요청은 일반 에러와 구분해서 처리한다.

면접에서는 코드 한 줄보다도 이 흐름을 자연스럽게 설명하는 편이 더 중요하다.

## 자주 헷갈리는 포인트

### 요청 취소와 응답 무시는 다르다

요청 취소는 가능하면 네트워크 작업 자체를 중단하는 쪽이다.  
반면 응답 무시는 결과가 와도 반영하지 않는 방식이라, 불필요한 작업이 이미 진행됐을 수 있다.

실무에서는 둘을 같이 쓰기도 한다.  
예를 들어 라이브러리 특성상 완전 취소가 어려우면 요청 식별자를 두고 최신 요청만 반영하는 방식까지 보완한다.

### 취소된 에러는 일반 실패와 구분해야 한다

사용자가 페이지를 떠나서 요청을 취소한 건 서버 장애가 아니다.  
그래서 로깅이나 에러 UI에서 같은 수준으로 다루면 노이즈가 커진다.

### 모든 비동기 작업이 자동으로 취소되는 건 아니다

`fetch`는 대표 사례지만, 어떤 라이브러리는 별도 취소 방식을 쓰거나 지원 범위가 다를 수 있다.  
그래서 "AbortController는 표준 기반 취소 수단이고, 라이브러리 지원 여부는 확인해야 한다" 정도로 말하면 안전하다.

## React 관점에서 자주 나오는 질문

### useEffect cleanup과 왜 같이 묶이냐

effect 안에서 요청을 시작했다면, 다음 effect가 실행되거나 컴포넌트가 사라질 때 이전 요청을 정리해야 하기 때문이다.  
이 지점을 놓치면 오래된 응답 반영, 메모리 낭비, 상태 꼬임으로 이어질 수 있다.

### 경쟁 상태를 어떻게 줄이냐

취소 가능한 요청은 취소하고, 필요하면 마지막 요청만 유효하다는 규칙을 둔다.  
즉 "취소"와 "최신성 보장"을 같이 생각해야 한다.

## 면접 답변 예시

> AbortController는 진행 중인 비동기 요청을 취소할 수 있게 해 주는 브라우저 API입니다. 프론트에서는 검색 자동완성, 필터 변경, 컴포넌트 unmount 같은 상황에서 이전 요청이 더 이상 의미 없을 때 많이 씁니다. 이렇게 하면 불필요한 네트워크 작업을 줄일 수 있고, 더 중요한 건 오래된 응답이 최신 UI 상태를 덮어쓰는 문제를 줄일 수 있습니다. React에서는 보통 `useEffect` cleanup에서 이전 요청을 abort하는 패턴으로 설명하면 좋습니다.

## 장점

- 필요 없는 요청을 조기에 끊어 UI 일관성을 높일 수 있다.
- 검색, 필터, 페이지 전환 같은 상호작용이 많은 화면에서 특히 유용하다.
- effect 정리와 연결해서 설명하기 좋아 면접 답변 구조가 깔끔하다.

## 단점

- 취소 로직과 일반 에러 처리를 분리하지 않으면 코드가 지저분해질 수 있다.
- 라이브러리마다 지원 방식이 달라 일관성이 떨어질 수 있다.
- 취소만으로 모든 경쟁 상태가 해결되는 건 아니다.

## 주의사항 / 실무 팁

- `abort()` 이후 에러를 무조건 장애로 기록하지 말고 취소 여부를 구분하자.
- 빠른 입력 UI에서는 디바운스와 함께 쓰면 체감이 훨씬 좋아진다.
- "요청 취소"와 "마지막 응답만 반영"을 별개 개념으로 이해하면 실무 설명이 탄탄해진다.
- 면접에서는 코드 디테일보다 "왜 취소가 필요한지"를 먼저 말하는 편이 더 설득력 있다.
