# Web Worker 정리

## 핵심 요약

- `Web Worker`는 메인 스레드와 분리된 백그라운드 스레드에서 자바스크립트를 실행하게 해 주는 브라우저 기능이다.
- UI 렌더링과 무거운 계산을 분리할 수 있어서, 화면이 버벅이는 문제를 줄이는 데 유용하다.
- 대신 DOM에 직접 접근할 수 없고, 메인 스레드와는 `postMessage` 기반으로만 통신한다.
- 면접에서는 "언제 쓰는지", "무엇이 안 되는지", "비용은 무엇인지"까지 같이 말하면 답변이 단단해진다.

## Web Worker란 무엇인가

브라우저의 자바스크립트는 보통 메인 스레드에서 실행된다.  
이 메인 스레드는 화면 렌더링, 이벤트 처리, 자바스크립트 실행을 함께 담당한다.

그래서 아주 무거운 계산을 메인 스레드에서 돌리면 버튼 클릭이 늦게 먹거나 스크롤이 끊기는 현상이 생긴다.  
이럴 때 계산 작업을 별도 스레드로 보내서 메인 스레드를 덜 바쁘게 만드는 방법이 `Web Worker`다.

쉽게 말하면 "화면 담당"과 "계산 담당"을 나누는 개념이라고 보면 된다.

## 왜 필요한가

프론트엔드에서 무거운 작업이 꼭 서버에서만 일어나는 것은 아니다.

- 대용량 JSON 가공
- 이미지 필터 처리
- 파일 파싱
- 복잡한 수식 계산
- 텍스트 검색이나 정렬

이런 일을 메인 스레드에서 바로 처리하면 UX가 급격히 나빠질 수 있다.  
사용자는 페이지가 느린 것보다 "멈춘 것처럼 보이는 느낌"에 더 민감하다.

Web Worker는 이런 상황에서 메인 스레드의 응답성을 지키는 데 의미가 있다.

## 동작 방식

메인 스레드와 Worker는 서로 메모리를 직접 공유하는 방식이 아니라 메시지를 주고받는 방식으로 동작한다.

대략 흐름은 이렇다.

1. 메인 스레드에서 Worker를 생성한다.
2. Worker에게 데이터를 보낸다.
3. Worker가 백그라운드에서 계산한다.
4. 결과를 다시 메인 스레드로 보낸다.
5. 메인 스레드가 화면을 업데이트한다.

예를 들면 이런 식이다.

```js
const worker = new Worker("worker.js");

worker.postMessage({ numbers: [1, 2, 3, 4] });

worker.onmessage = (event) => {
  console.log(event.data);
};
```

Worker 내부는 보통 이렇게 생긴다.

```js
self.onmessage = (event) => {
  const sum = event.data.numbers.reduce((acc, cur) => acc + cur, 0);
  self.postMessage(sum);
};
```

핵심은 메인 스레드가 기다리며 멈춰 있지 않는다는 점이다.

## 장점이 분명한 상황

Web Worker는 모든 곳에 넣는 기능이 아니라, 병목이 계산 쪽에 있을 때 의미가 있다.

- 데이터 가공이 길어서 입력이나 스크롤이 끊길 때
- 큰 파일을 브라우저에서 직접 읽어야 할 때
- 차트 계산이나 통계 계산이 무거울 때
- 오프라인 환경에서 클라이언트 연산이 많은 앱일 때

특히 "사용자 인터랙션은 즉시 반응해야 하고, 계산은 뒤에서 처리해도 되는" 상황에 잘 맞는다.

## 한계와 주의할 점

Web Worker가 만능은 아니다.

가장 대표적인 제한은 DOM 접근이 안 된다는 점이다.  
Worker 안에서는 `document`, `window`를 직접 다룰 수 없으니 화면을 바로 바꾸는 일은 못 한다.

또한 메시지를 주고받는 과정 자체도 비용이 있다.  
작업이 아주 가볍다면 Worker를 쓰는 오버헤드가 더 클 수 있다.

추가로 주의할 점은 다음과 같다.

- 데이터 전달 비용이 있다.
- 디버깅이 메인 스레드보다 번거롭다.
- 코드 구조가 분리되어 복잡도가 올라간다.
- 상태 동기화를 잘못하면 흐름을 따라가기 어려워진다.

즉 "무거운 작업이면 무조건 Worker"가 아니라, 실제 병목이 계산인지 먼저 봐야 한다.

## Web Worker와 비동기의 차이

면접에서 자주 헷갈리는 포인트가 "어차피 자바스크립트는 비동기 처리하잖아요?"라는 질문이다.

`Promise`, `async/await`, `setTimeout`은 작업 순서를 비동기로 관리하는 도구이지, CPU가 무거운 계산을 다른 스레드로 옮겨 주는 것은 아니다.

반면 Web Worker는 실제로 실행 위치를 메인 스레드 밖으로 분리한다.  
그래서 네트워크 대기와 계산 병목은 구분해서 말하는 편이 좋다.

## 면접 답변 예시

> Web Worker는 메인 스레드와 분리된 백그라운드 스레드에서 자바스크립트를 실행하는 기능입니다. 브라우저는 보통 렌더링과 이벤트 처리, 스크립트 실행을 메인 스레드가 같이 담당하기 때문에 무거운 계산이 들어가면 UI가 멈춘 것처럼 보일 수 있습니다. 이럴 때 계산 작업을 Worker로 보내면 메인 스레드의 응답성을 유지할 수 있습니다. 다만 Worker는 DOM에 직접 접근할 수 없고, 메인 스레드와 메시지 기반으로 통신해야 해서 오버헤드도 있습니다. 그래서 대용량 데이터 가공이나 파일 파싱처럼 계산 비용이 큰 경우에 선택하는 것이 적절하다고 봅니다.

## 장점

- UI 반응성을 지키면서 무거운 연산을 처리할 수 있다.
- 메인 스레드 병목을 분리해서 사용자 경험을 개선하기 좋다.
- 브라우저에서 처리해야 하는 계산성 작업에 실전적으로 쓸 수 있다.

## 단점

- DOM에 직접 접근할 수 없다.
- 메시지 전달 비용과 구조 복잡도가 생긴다.
- 가벼운 작업에는 오히려 비효율적일 수 있다.

## 주의사항 / 실무 팁

- 먼저 성능 병목이 CPU 계산인지 확인하고 도입하는 것이 맞다.
- 큰 객체를 자주 주고받으면 Worker 도입 효과가 줄어들 수 있다.
- 렌더링 업데이트는 메인 스레드, 계산은 Worker처럼 역할을 분리해서 설명하면 좋다.
- 면접에서는 `비동기 처리`와 `멀티 스레드 처리`를 구분해서 말해야 답변이 깔끔하다.
