# 웹 폰트 최적화 정리

## 핵심 요약
- 웹 폰트 최적화는 텍스트가 보이기까지 걸리는 시간과 레이아웃 흔들림을 줄이는 작업이다.
- 면접에서는 `FOIT`, `FOUT`, `font-display`, preload, 서브셋팅을 한 세트로 설명하면 구조가 잘 잡힌다.
- 폰트 파일 크기만 줄이는 것이 아니라, 어떤 글자가 언제 로드되는지까지 설계해야 한다.
- 브랜딩과 성능 사이의 균형을 잡는 판단이 실무에서 중요하다.

## 개념 설명
웹 폰트는 디자인 일관성을 주지만, 잘못 쓰면 첫 렌더링이 늦어지거나 텍스트가 갑자기 바뀌면서 사용자 경험이 흔들릴 수 있다.

자주 같이 나오는 개념은 다음과 같다.
- `FOIT`(Flash of Invisible Text): 폰트가 올 때까지 글자가 안 보이는 상태
- `FOUT`(Flash of Unstyled Text): 우선 시스템 폰트로 보여주고, 나중에 웹 폰트로 바뀌는 상태
- `font-display`: 폰트 로딩 중 텍스트 표시 전략을 정하는 속성
- 서브셋팅: 실제 필요한 문자만 포함해 폰트 파일 크기를 줄이는 방법

즉, 웹 폰트 최적화는 "예쁜 폰트를 쓰되 텍스트 가독성과 초기 렌더링을 해치지 않도록 조정하는 일"이라고 보면 된다.

## 예시
```css
@font-face {
  font-family: "PretendardSubset";
  src: url("/fonts/pretendard-subset.woff2") format("woff2");
  font-display: swap;
}

body {
  font-family: "PretendardSubset", system-ui, sans-serif;
}
```

```html
<link
  rel="preload"
  href="/fonts/pretendard-subset.woff2"
  as="font"
  type="font/woff2"
  crossorigin
/>
```

이렇게 하면 중요한 폰트를 먼저 당겨오고, 늦어지더라도 시스템 폰트 fallback이 있어 본문 가독성을 유지하기 쉽다.

## 면접 답변 예시
- 질문: "웹 폰트 최적화는 왜 중요한가요?"  
  답변: "웹 폰트는 브랜딩에는 좋지만 초기 렌더링을 늦추고 CLS를 유발할 수 있습니다. 그래서 저는 `font-display`, preload, 서브셋팅, fallback 폰트 조합으로 텍스트가 빨리 보이면서도 디자인 차이가 과하지 않게 만드는 방향으로 설명합니다."
- 질문: "`font-display: swap`을 무조건 쓰면 되나요?"  
  답변: "대부분의 본문에는 유리하지만, 아주 짧은 브랜드 헤드라인처럼 폰트 교체가 눈에 띄는 영역은 전략을 다르게 볼 수 있습니다. 중요한 건 서비스 성격에 맞게 FOIT와 FOUT의 트레이드오프를 판단하는 것입니다."

## 장점
- 본문이 더 빨리 보여 체감 성능이 좋아진다
- 불필요한 폰트 용량을 줄여 네트워크 비용을 낮출 수 있다
- 브랜딩과 가독성을 함께 관리할 수 있다

## 단점
- fallback 폰트와 실제 폰트 차이가 크면 화면이 튀어 보일 수 있다
- 다국어 서비스는 서브셋 전략이 복잡해진다
- preload를 남발하면 다른 핵심 리소스 우선순위를 해칠 수 있다

## 주의사항 / 실무 팁
- 본문 폰트와 제목 폰트를 모두 preload하지 말고, 실제 LCP에 영향이 큰 폰트부터 우선순위를 정한다.
- 한글 서비스는 문자 수가 많아서 서브셋 범위를 잘못 잡으면 특정 글자가 깨질 수 있으니 로그나 QA 시나리오를 같이 준비한다.
- fallback 폰트의 자간, 높이 차이를 점검해야 CLS를 줄일 수 있다.
