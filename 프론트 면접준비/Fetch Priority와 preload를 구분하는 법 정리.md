# Fetch Priority와 preload를 구분하는 법 정리

## 핵심 요약

- 면접에서 "Fetch Priority와 preload를 구분하는 법" 질문을 받으면 핵심은 "어떤 사용자 지표를 개선하려고 이 기능을 쓰는가?"이다.
- 정의만 외우기보다 적용 전후의 사용자 흐름을 함께 설명해야 한다.
- 예시는 실제 화면 흐름이나 운영 지표와 연결해서 말하는 편이 좋다.

## 개념 설명

성능 주제는 특정 API를 쓰면 빨라진다는 식으로 답하면 약하다. 어떤 병목을 측정하고 어떤 사용자 지표를 개선할지 정해야 한다.

LCP, CLS, long task, resource timing처럼 관측 지표와 실제 코드 변경을 연결해서 설명하면 답변이 훨씬 설득력 있다.

이 글에서는 "Fetch Priority와 preload를 구분하는 법" 질문에 답할 때 어떤 실무 기준으로 판단하면 좋은지에 초점을 둔다.

## 예시

```text
지원 확인 -> 기본 동작 유지 -> 최신 기능 적용 -> 실패 상태 안내 -> 지표 확인
```

이 예시는 핵심 판단 지점을 보여주기 위한 축약된 형태다. 실제 코드에서는 예외 처리와 관측 지표를 같이 붙여야 한다.

## 면접 답변 예시

> "Fetch Priority와 preload를 구분하는 법"에 대해 묻는다면 저는 개선하려는 지표를 정한 뒤 측정 결과로 적용 범위를 넓힐지 판단한다고 답하겠습니다. 기능 이름만 말하기보다 어떤 사용자 문제를 줄이는지, 실패했을 때 화면이 어떻게 유지되는지, 적용 후 어떤 지표로 확인할지까지 연결해서 설명하겠습니다.

## 장점

- 성능 개선을 느낌이 아니라 지표로 판단할 수 있다.
- 병목을 컴포넌트나 리소스 단위로 좁히기 쉽다.
- 불필요한 최적화를 줄일 수 있다.

## 단점

- 지표 수집 자체가 비용이 될 수 있다.
- 실험 환경과 실제 사용자 환경의 결과가 다를 수 있다.
- 성능만 보다가 접근성이나 유지보수를 놓칠 수 있다.

## 주의사항 / 실무 팁

- 지원하지 않는 환경의 기본 흐름을 먼저 정하고 최신 기능을 얹는다.
- 면접에서는 API 이름보다 실패했을 때 사용자에게 무엇이 보이는지 설명한다.
- 변경 전후의 LCP, INP, CLS, long task 같은 지표를 비교한다.
