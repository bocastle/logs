# Server-Timing 헤더 정리

## 핵심 요약
- `Server-Timing`은 서버가 응답 헤더에 성능 측정값을 담아 브라우저 개발자도구에서 보이게 하는 표준 방식이다.
- 백엔드 내부 처리 시간을 프론트엔드 관점의 네트워크 타이밍과 연결해서 볼 수 있다는 점이 핵심이다.
- 면접에서는 "서버 내부 단계별 시간을 응답 헤더로 노출해 성능 분석을 돕는 메커니즘"이라고 설명하면 된다.
- 프론트와 백엔드가 같은 숫자를 보며 병목을 대화할 수 있게 해 준다는 점이 실무 포인트다.

## 개념 설명
웹 성능 문제를 볼 때 프론트엔드는 보통 브라우저 네트워크 패널을 보고, 백엔드는 APM이나 서버 로그를 본다. 문제는 두 관점이 분리되어 있으면 "느린 게 서버 때문인지, 네트워크 때문인지, 렌더링 때문인지"를 맞추기 어렵다는 점이다.

`Server-Timing` 헤더는 이 간극을 줄여 준다. 서버가 응답을 내려줄 때 "DB 조회 40ms", "캐시 확인 5ms", "템플릿 렌더링 12ms" 같은 값을 헤더에 같이 담아 주면, 브라우저 개발자도구에서 네트워크 요청과 함께 서버 내부 처리 시간을 볼 수 있다.

즉, 프론트엔드 개발자는 단순히 TTFB가 길다는 사실만 보는 게 아니라, 그 안에서 서버가 어디에 시간을 썼는지도 힌트를 얻을 수 있다.

## 예시
예를 들어 상품 상세 API가 느리다고 하자. 브라우저에서는 응답까지 700ms가 걸리는 것으로 보이는데, 원인이 애매할 수 있다.

이때 서버가 아래 같은 `Server-Timing` 정보를 내려준다고 생각해 보자.

- `cache;dur=3`
- `db;dur=180`
- `render;dur=25`

그러면 프론트엔드 입장에서도 "문제의 대부분은 DB 구간이네"라고 빠르게 감을 잡을 수 있다. 반대로 서버 내부 시간 합이 짧은데 전체 응답이 느리다면 네트워크 구간이나 프록시, CDN, 브라우저 측 이슈를 의심해 볼 수 있다.

## 면접 답변 예시
- 질문: "`Server-Timing` 헤더는 무엇인가요?"  
  답변: "서버가 응답 헤더에 내부 처리 시간을 담아 보내는 표준입니다. 브라우저 개발자도구에서 확인할 수 있어서, 프론트엔드도 DB 조회나 렌더링 같은 서버 측 병목 힌트를 함께 볼 수 있습니다."
- 질문: "왜 유용한가요?"  
  답변: "성능 이슈가 생기면 프론트와 백엔드가 서로 다른 도구만 보는 경우가 많은데, Server-Timing을 쓰면 같은 요청 기준으로 공통 숫자를 볼 수 있어 원인 분석이 빨라집니다."

## 장점
- 프론트와 백엔드가 같은 요청 기준으로 성능 대화를 할 수 있다
- 브라우저 개발자도구에서 확인 가능해 접근성이 좋다
- TTFB 같은 바깥 지표와 서버 내부 단계를 연결해 해석하기 좋다

## 단점
- 어떤 단위를 어떻게 측정할지 서버에서 별도 설계가 필요하다
- 내부 구조를 과도하게 노출하면 운영상 민감할 수 있다
- 숫자가 많아질수록 오히려 해석이 복잡해질 수 있다

## 주의사항 / 실무 팁
- 모든 내부 단계를 다 노출하기보다, 병목 후보가 되는 핵심 구간만 간결하게 담는 편이 낫다.
- 운영 환경에서는 보안이나 헤더 크기를 고려해 민감한 이름 대신 일반화된 지표명을 쓰는 것이 안전하다.
- APM, 로그, 분산 추적과 함께 써야 의미가 커지고, 헤더 하나만으로 전체 원인 분석이 끝나지는 않는다.
- 면접에서는 "`브라우저에서 서버 내부 처리 시간을 볼 수 있게 해 주는 헤더`"라고 정의한 뒤, 프론트와 백엔드 협업 관점까지 덧붙이면 좋다.
