# 자바스크립트의 메모리 관리 정리

## 핵심 요약(짧게)

- 자바스크립트는 개발자가 `malloc/free`로 직접 관리하지 않고, **가비지 컬렉션(GC)** 으로 메모리를 자동 해제합니다. citeturn0view0
- 메모리 관리는 크게 **메모리 할당**과 **메모리 해제**로 나눠 설명할 수 있습니다. citeturn0view0
- 원시 값은 **Stack**, 객체(참조 타입)는 **Heap**에 저장됩니다. citeturn0view0
- 대표 GC 알고리즘: **Reference-counting**, **Mark-and-sweep** citeturn0view0

## 개요

메모리 관리는 크게 **메모리 할당**과 **메모리 해제** 두 가지 과정으로 나눌 수 있습니다.  
C 같은 저수준 언어는 `malloc()`, `free()`로 개발자가 직접 관리하지만, 자바스크립트는 언어 차원에서 자동으로 이 작업이 이루어집니다. citeturn0view0

## 메모리 할당

변수나 객체를 생성할 때 메모리 할당이 발생합니다. 예를 들어 아래 코드에서 `name` 변수는 메모리에 할당됩니다. citeturn0view0

```js
const name = "maeil-mail";
```

### Stack vs Heap

- 문자열/숫자 같은 **원시 값(primitive)** 은 **Stack** 영역에 저장됩니다. 원시 값은 고정 크기를 가지기 때문에 엔진이 컴파일 타임에 크기를 알 수 있어 **정적 데이터**로 볼 수 있습니다. citeturn0view0
- **객체(참조 타입)** 는 **Heap** 영역에 저장됩니다. 객체는 실행 시점에 필요한 만큼 메모리가 동적으로 할당되기 때문에 **동적 데이터**로 볼 수 있습니다. citeturn0view0

## 메모리 해제

할당된 메모리가 더 이상 필요 없을 때 메모리 해제가 일어납니다. 자바스크립트는 **가비지 컬렉션(GC)** 을 사용해 언어 차원에서 메모리 할당을 추적하고, 필요하지 않게 된 메모리를 자동으로 해제합니다. citeturn0view0

### 대표 GC 알고리즘

#### Reference-counting

- 객체가 **참조되는 횟수**를 추적하다가, 참조 횟수가 **0**이 되면 메모리에서 해제합니다. citeturn0view0
- 한계: **순환 참조(circular reference)** 가 발생하면 참조 카운트가 0이 되지 않아 메모리에서 해제되지 않을 수 있습니다. citeturn0view0

#### Mark-and-sweep

- **루트(root) 객체**에서 시작해 도달 가능한(참조되는) 객체를 표시(mark)하고,
- 마지막까지 표시되지 않은 객체를 청소(sweep)하여 해제합니다. citeturn0view0
- 특징: 순환 참조가 있더라도 루트에서 **도달 불가능**하면 표시되지 않아 해제될 수 있어, Reference-counting의 한계를 보완합니다. citeturn0view0

## 장점

- 개발자가 `malloc/free`를 직접 호출하지 않아도 되어, 언어 차원에서 **자동 메모리 해제(GC)** 를 통해 관리 부담을 줄일 수 있습니다. citeturn0view0

## 단점

- 원문에서는 단점이 별도로 직접 언급되지는 않았습니다. citeturn0view0

## 주의사항 / 실무 팁

- **GC가 있다고 해서 메모리 누수가 “절대” 생기지 않는 것은 아닙니다.** 불필요한 참조(이벤트 리스너, 클로저, 전역 변수 등)가 유지되면 해제가 지연될 수 있습니다. (관련 문항: “자바스크립트에서 메모리 누수가 발생하는 상황들”) citeturn0view0
- “Stack/Heap에 무엇이 저장되는지”를 구분해두면, 객체가 예상보다 오래 살아남는 경우(참조 유지)를 추적할 때 도움이 됩니다. citeturn0view0
