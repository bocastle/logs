# Compound Component 패턴 정리

## 핵심 요약
- `Compound Component` 패턴은 여러 하위 컴포넌트가 하나의 큰 컴포넌트처럼 함께 동작하도록 설계하는 패턴이다.
- 대표 예시는 `Tabs`, `Accordion`, `Select`, `Modal`처럼 부모와 자식이 역할을 나눠 가지는 UI다.
- 면접에서는 "관련 있는 하위 컴포넌트들을 묶어 선언형 API를 제공하는 패턴"이라고 설명하면 된다.
- 장점만 말하지 말고, 내부 결합도와 학습 비용도 같이 언급해야 실무 감각이 드러난다.

## 개념 설명
React 컴포넌트를 설계하다 보면 prop이 계속 늘어나는 경우가 있다. 예를 들어 탭 UI를 하나의 컴포넌트로만 만들면 `items`, `activeKey`, `onChange`, `renderLabel`, `renderPanel` 같은 prop이 점점 복잡해질 수 있다.

이럴 때 `Compound Component` 패턴을 쓰면 부모와 자식의 역할을 나눌 수 있다. 예를 들어 이런 식이다.

- `Tabs`
- `Tabs.List`
- `Tabs.Trigger`
- `Tabs.Content`

사용자는 이 조합을 보면서 구조를 직관적으로 이해할 수 있고, 라이브러리 작성자는 내부적으로 `Context` 같은 메커니즘으로 상태를 공유한다.

핵심은 "서로 관련된 하위 컴포넌트들이 하나의 문맥을 공유하며 함께 동작한다"는 점이다. 그래서 단순한 컴포넌트 분리와는 다르게, API 설계 패턴으로 보는 것이 맞다.

## 예시
탭 UI를 만든다고 해 보자.

- 단순 prop 기반 구현은 빠르지만 유연성이 떨어질 수 있다
- `Compound Component` 패턴을 쓰면 사용자는 필요한 위치에 `Tabs.Trigger`, `Tabs.Content`를 선언적으로 배치할 수 있다
- 내부에서는 현재 선택된 탭 상태를 부모가 관리하고 자식들이 그 상태를 공유한다

이 방식은 디자인 시스템이나 공통 UI 라이브러리에서 특히 자주 보인다. Headless UI 계열 컴포넌트를 이해할 때도 도움이 된다.

## 면접 답변 예시
- 질문: "`Compound Component` 패턴은 무엇인가요?"  
  답변: "하나의 부모 컴포넌트 아래 여러 하위 컴포넌트가 같은 상태나 문맥을 공유하면서 함께 동작하도록 만드는 패턴입니다. Tabs나 Accordion 같은 선언형 UI를 설계할 때 자주 씁니다."
- 질문: "왜 쓰나요?"  
  답변: "관련 있는 UI 조각을 자연스럽게 묶어서 API를 읽기 쉽게 만들 수 있기 때문입니다. prop을 과하게 늘리는 대신 구조 자체로 의도를 드러낼 수 있습니다."

## 장점
- 선언형 API를 만들기 좋아서 사용성이 좋다
- 관련 있는 역할을 컴포넌트 단위로 분리해 가독성을 높일 수 있다
- 디자인 시스템이나 headless component 설계에 잘 맞는다

## 단점
- 내부적으로 `Context`나 자식 검사 로직이 들어가면 구현 복잡도가 올라간다
- 패턴을 처음 접한 팀원에게는 사용법이 직관적이지 않을 수 있다
- 잘못 설계하면 하위 컴포넌트가 부모 문맥에 지나치게 강하게 결합된다

## 주의사항 / 실무 팁
- 모든 컴포넌트에 적용할 패턴은 아니다. 단순 버튼이나 입력창에는 오히려 과하다.
- `Tabs`, `Accordion`, `Menu`, `Select`처럼 "여러 하위 요소가 한 상태를 공유하는 UI"에서 효과가 크다.
- 외부 API가 예쁜 것만 보고 도입하면 안 되고, 접근성 처리와 상태 동기화 책임이 어디에 있는지도 봐야 한다.
- 면접에서는 "`prop 덩어리를 줄이고 구조 자체로 의미를 드러내는 선언형 컴포넌트 패턴`"이라고 요약하면 전달력이 좋다.
