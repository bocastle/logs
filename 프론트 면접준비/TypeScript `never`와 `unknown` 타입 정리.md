# TypeScript `never`와 `unknown` 타입 정리

## 핵심 요약

- `never`: **절대 발생할 수 없는 값**을 나타내는 타입. 예외를 던지거나 무한 루프 등으로 **정상적으로 종료(반환)되지 않는 함수**의 반환 타입으로 사용된다.
- `unknown`: **알 수 없는 값**을 나타내는 타입. 런타임에 타입이 결정되는 값(예: 외부 API 응답)에 사용하며, `any`보다 **타입 안정성이 높다**.
- `void` vs `never`: `void`는 **값을 반환하지 않고 정상 종료**할 수 있는 경우, `never`는 **아예 반환될 수 없는 상태**를 의미한다.

## `never`

`never`는 절대 발생할 수 없는 값을 나타내는 타입입니다. 함수가 값을 반환하지 않고 항상 예외를 던지거나 무한 루프를 도는 경우, 그 반환 타입은 `never`가 됩니다. `never`는 **bottom type**이라고도 불리며, 모든 타입의 하위 타입입니다.

### 사용 예시

```ts
function throwError(message: string): never {
  throw new Error(message);
}
```

이 함수는 어떤 값도 반환하지 않으며, 실행될 경우 예외를 던지기 때문에 `never` 반환 타입을 갖습니다.

### 추가 활용 포인트

`if-else`, `switch` 문을 통한 타입 검사 과정에서 **허용된 케이스에 포함되지 않은 경우**에 대응하기 위해 `never`를 사용하기도 합니다.

## `unknown`

`unknown`은 알 수 없는 값을 나타내는 타입입니다. 외부 API 호출 결과처럼 구체적인 타입을 미리 알 수 없고, 런타임에 타입이 결정되는 경우에 사용합니다.

`any`와 비슷하지만 더 안전한 방식입니다. `any`는 어떤 값이든 허용하기 때문에 타입 안정성이 떨어질 수 있지만, `unknown`은 **특정한 타입으로 사용하려면 타입을 좁혀야** 하기 때문입니다. `unknown`은 `any`와 함께 **top type**이라고도 불리며, 모든 타입의 상위 타입입니다.

### 사용 예시

```ts
function processUnknownValue(value: unknown) {
  if (typeof value === "string") {
    //
  } else if (typeof value === "number") {
    //
  } else {
    //
  }
}
```

`unknown`은 직접 타입을 강제할 수 없으며, **타입을 좁혀야 사용할 수 있다**는 점에서 `any`보다 안전한 선택입니다. 이를 활용하면 예상치 못한 타입 오류를 방지할 수 있습니다.

## `never` 타입과 `void` 타입의 차이

- `void`: 함수가 **명시적으로 값을 반환하지 않음**을 의미하지만, **정상적인 종료**는 가능하다. (예: `return;`)
- `never`: **아예 반환될 수 없는 상태**를 나타낸다. 예외를 던지거나 무한 루프에 빠지는 함수에만 적용된다.

즉, `void`는 정상적으로 종료하지만 `never`는 실행이 끝날 수 없는 함수에 적용되는 타입입니다.

## 장점

### `never`

- 예외/무한루프 등 **정상 반환 불가능한 함수**를 타입으로 명확히 표현할 수 있다.
- `switch` 등 분기 처리에서 **처리되지 않은 케이스**를 드러내는 데 도움이 된다.

### `unknown`

- `any`와 달리 사용 시 **타입 좁히기**가 필요해 타입 안정성이 높다.
- 외부 입력/런타임 결정 값에 대해 **안전한 기본 타입**으로 활용할 수 있다.

## 단점

### `never`

- 일반적인 데이터 모델링 용도로는 사용할 일이 많지 않고, 사용 맥락(예외/무한루프/미도달 코드)이 명확해야 한다.

### `unknown`

- 사용하려면 타입 가드/검증 코드가 필요해 초기 구현이 번거로울 수 있다.

## 주의사항 및 실무 팁

- `unknown` 값을 바로 다른 타입처럼 사용하려 하지 말고, **타입 가드(`typeof`, `instanceof`, 사용자 정의 타입 가드)**로 먼저 좁히는 습관을 들입니다.
- `any`는 빠르게 개발할 때 편리하지만, **타입 안정성을 크게 낮출 수 있으므로** 가능하면 `unknown` + 검증으로 대체합니다.
- `never`는 “여기까지 도달하면 안 된다”를 타입으로 표현할 때 유용합니다. 특히 `switch`에서 유니언 타입을 다룰 때, 누락된 케이스를 컴파일 단계에서 발견하도록 설계할 수 있습니다.
