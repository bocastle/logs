# Intersection Observer 정리

## 핵심 요약

- `Intersection Observer`는 특정 요소가 화면 안에 얼마나 들어왔는지를 비동기로 알려주는 브라우저 API다.
- 스크롤 이벤트를 계속 계산하는 방식보다 부담이 적어서 무한 스크롤, 지연 로딩, 노출 추적에 많이 쓴다.
- 면접에서는 "왜 scroll 이벤트 대신 이걸 쓰는가", "threshold와 rootMargin이 뭐냐", "언제 주의해야 하냐" 정도를 말하면 좋다.
- 만능은 아니다. 아주 정밀한 픽셀 단위 제어나 복잡한 애니메이션 타이밍은 다른 방식이 더 맞을 수 있다.

## Intersection Observer가 뭐냐

쉽게 말하면 브라우저에게 "이 요소가 화면에 보이기 시작하면 알려줘"라고 맡기는 방식이다.

예전에는 스크롤할 때마다 `getBoundingClientRect()`로 위치를 계산하는 코드가 많았는데, 요소가 많아질수록 계산이 잦아지고 코드도 지저분해진다.  
`Intersection Observer`는 이런 관찰 작업을 브라우저 레벨에서 처리해 주기 때문에 더 선언적으로 쓸 수 있다.

## 언제 많이 쓰나

### 1. 무한 스크롤

목록 맨 아래 감시용 요소를 하나 두고, 그 요소가 보이면 다음 페이지를 불러오는 식이다.

### 2. 이미지 지연 로딩

이미지가 실제로 보이기 직전에 로드해서 초기 로딩 부담을 줄일 수 있다.

### 3. 광고나 콘텐츠 노출 추적

배너나 카드가 실제로 보였는지 기록할 때 쓴다. 단순히 DOM에 렌더링됐다고 해서 사용자가 본 건 아니기 때문이다.

### 4. 스크롤 기반 UI 트리거

특정 섹션이 화면에 들어오면 애니메이션을 시작하거나, 현재 보고 있는 섹션에 맞춰 내비게이션 강조를 바꿀 때도 자주 쓴다.

## 핵심 옵션

### root

기본값은 브라우저 뷰포트다.  
특정 스크롤 컨테이너 안에서 보이는지 판단하고 싶으면 그 컨테이너를 `root`로 지정한다.

### rootMargin

관찰 영역의 여백이다.  
예를 들어 이미지가 화면에 "보이기 직전" 미리 로드되게 하려면 `rootMargin`을 조금 크게 줄 수 있다.

### threshold

요소가 얼마나 보여야 콜백을 실행할지 정하는 값이다.

- `0`이면 조금만 보여도 감지한다.
- `1`이면 요소 전체가 다 보여야 감지한다.
- 배열로 주면 0.25, 0.5, 0.75처럼 여러 구간도 볼 수 있다.

## scroll 이벤트보다 왜 낫냐

핵심은 "계속 직접 계산하지 않아도 된다"는 점이다.

스크롤 이벤트는 매우 자주 발생해서 잘못 쓰면 성능에 부담이 간다.  
특히 여러 요소를 한꺼번에 감시하면 코드가 금방 복잡해진다.

반면 `Intersection Observer`는 관찰 대상을 등록해 두고 브라우저가 시점이 왔을 때 알려주는 구조라서 코드도 단순하고, 일반적인 노출 감지 용도에 더 잘 맞는다.

## 예시로 이해하기

무한 스크롤을 예로 들면 이런 흐름이다.

1. 목록 아래에 빈 `sentinel` 요소를 둔다.
2. 이 요소를 `Intersection Observer`로 관찰한다.
3. 해당 요소가 화면에 들어오면 다음 데이터를 요청한다.
4. 데이터가 붙으면 다시 맨 아래 요소를 계속 관찰한다.

이 구조의 장점은 스크롤 위치를 매번 계산하지 않아도 된다는 점이다.

## 주의할 점

### 1. 관찰 해제를 안 하면 메모리 관리가 지저분해질 수 있다

컴포넌트가 사라질 때 `disconnect()`나 `unobserve()`를 정리해야 한다.

### 2. 너무 많은 일을 콜백 안에서 하면 의미가 줄어든다

감지 자체는 효율적이어도, 콜백 안에서 무거운 연산을 하면 결국 느려질 수 있다.

### 3. 스크롤 컨테이너가 따로 있으면 root 설정을 놓치기 쉽다

뷰포트 기준인지, 특정 영역 기준인지 헷갈리면 "왜 콜백이 안 오지?" 같은 문제가 생긴다.

### 4. 정밀한 스크롤 제어에는 한계가 있다

"몇 픽셀 이동했을 때마다 값 계산"처럼 아주 세밀한 요구라면 스크롤 이벤트나 다른 방식이 더 적합할 수 있다.

## 면접 답변 예시

> Intersection Observer는 어떤 요소가 화면이나 특정 스크롤 컨테이너 안에 들어왔는지를 비동기로 감지하는 브라우저 API입니다. 무한 스크롤, 이미지 지연 로딩, 광고 노출 추적 같은 곳에서 많이 쓰고, scroll 이벤트로 매번 위치를 계산하는 방식보다 코드가 단순하고 일반적으로 부담이 적습니다. 다만 root, rootMargin, threshold 설정을 상황에 맞게 잡아야 하고, 컴포넌트가 unmount될 때 관찰 해제도 같이 챙겨야 합니다.

## 장점

- 스크롤 감지 코드를 선언적으로 작성하기 좋다.
- 무한 스크롤, lazy loading 같은 대표 패턴에 잘 맞는다.
- 여러 요소의 노출 여부를 비교적 깔끔하게 다룰 수 있다.

## 단점

- 세밀한 픽셀 단위 제어에는 적합하지 않을 수 있다.
- `root`, `threshold` 설정을 잘못 잡으면 동작이 직관적이지 않을 수 있다.
- 관찰만 효율적인 것이지, 콜백 내부 로직까지 자동으로 최적화되는 건 아니다.

## 주의사항 / 실무 팁

- 리스트 하단 감시용 요소 하나만 두는 패턴이 무한 스크롤에서 가장 단순하다.
- `rootMargin`을 적절히 줘서 "보이기 직전" 미리 로드되게 하면 체감이 좋아진다.
- React에서는 `useEffect` 정리 함수에서 observer 해제를 꼭 넣는 습관이 좋다.
- 성능 이야기까지 묻는 면접이라면 "scroll 이벤트 자체가 나쁘다"가 아니라 "노출 감지 목적에는 Intersection Observer가 더 적합한 경우가 많다" 정도로 말하는 게 안전하다.
