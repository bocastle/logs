# 다크 모드 토큰 전환에서 깜빡임을 줄이는 법 정리

## 핵심 요약

- `first paint` 전에 사용자 선택 또는 시스템 선호를 결정해 root의 theme 속성과 `color-scheme`을 맞춘다.
- SSR에서는 cookie로 같은 값을 렌더링하고, 클라이언트 boot script는 CSS보다 먼저 최소 작업만 수행한다.
- `localStorage` 접근 실패, CSP와 잘못된 저장값 때문에 초기 렌더링이 멈추지 않게 한다.

## 개념 설명

다크 모드 깜빡임은 브라우저가 기본 테마로 한 번 그린 뒤 hydration 과정에서 저장된 테마를 읽어 색상을 바꿀 때 생긴다. 테마 결정이 `first paint`보다 늦기 때문에 발생하는 순서 문제다.

서버가 cookie를 읽을 수 있다면 처음부터 같은 `data-theme`을 렌더링한다. 정적 페이지는 CSS보다 앞선 작은 boot script에서 사용자 선택, 시스템 선호, 기본값 순으로 결정한다. `color-scheme`도 함께 설정해 input과 scrollbar 같은 브라우저 UI가 다른 테마로 먼저 그려지지 않게 한다.

## 예시

```html
<script>
  try {
    const saved = localStorage.getItem("theme");
    const theme = saved === "light" || saved === "dark" ? saved : null;
    if (theme) document.documentElement.dataset.theme = theme;
  } catch {}
</script>
<meta name="color-scheme" content="light dark" />
```

저장값을 allowlist로 검증하고 storage 접근 실패가 전체 script를 중단시키지 않게 한다. Inline script를 쓴다면 CSP nonce나 hash 정책과 배포 방식을 함께 설계한다.

## 면접 답변 예시

> 테마 깜빡임은 기본 테마가 그려진 뒤 클라이언트가 저장값을 읽어 바꾸기 때문에 생깁니다. SSR에서는 cookie를 사용해 서버부터 같은 theme 속성을 렌더링하고, 정적 페이지는 CSS보다 앞선 최소 boot script에서 사용자 선택과 시스템 선호를 결정하겠습니다. 저장값을 검증하고 `localStorage` 실패를 처리하며 `color-scheme`도 함께 맞춥니다. CSP와 hydration 계약을 유지하고 cold load를 light·dark 모두에서 확인합니다.

## 장점

- CSS token과 native form control, scrollbar의 첫 테마를 함께 맞출 수 있다.
- 서버와 클라이언트가 같은 우선순위를 사용하면 hydration 차이를 줄일 수 있다.

## 단점

- 서버와 클라이언트의 테마 결정 규칙이 다르면 hydration 차이와 다시 그리기가 생긴다.
- Inline boot script가 커지면 파싱 비용과 CSP 관리 부담이 늘어난다.

## 주의사항 / 실무 팁

- Boot script를 저장값 검증과 root theme 속성 설정에만 제한한다.
- 저장소 거절, 잘못된 값, cold·cache load와 시스템 테마 변경을 각각 테스트한다.
