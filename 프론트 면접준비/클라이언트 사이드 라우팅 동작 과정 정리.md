# 클라이언트 사이드 라우팅 동작 과정 정리

## 핵심 요약

- 클라이언트 사이드 라우팅(CSR 라우팅)은 **전체 페이지를 다시 로드하지 않고**, 현재 로딩된 애플리케이션에서 **필요한 컴포넌트만 교체**해 화면을 전환합니다.
- 링크 클릭 시 기본 동작(전체 리로드)을 막고, `window.history.pushState()`로 **URL만 변경**한 뒤 **라우터가 경로에 매핑된 컴포넌트를 렌더링**합니다.
- 뒤로/앞으로 가기는 `popstate` 이벤트로 감지하며, 이때도 **전체 리프레시 없이** URL에 맞는 컴포넌트로 다시 렌더링합니다.

## 클라이언트 사이드 라우팅이란?

클라이언트 사이드 라우팅은 브라우저에서 페이지를 전환할 때 전체 페이지를 다시 불러오지 않고, 현재 로딩된 애플리케이션 내에서 필요한 컴포넌트만 교체하는 방식입니다.

## 동작 과정

### 1) 링크 클릭: 기본 리로드 동작 차단

기본적으로 사용자가 아래처럼 링크를 클릭하면 브라우저는 바뀐 URL에 따라 전체 페이지를 다시 로드하려고 합니다.  
하지만 `React Router` 같은 라이브러리를 사용하면 이 기본 동작이 막히게 됩니다.

```jsx
<Link to="/about">About</Link>
```

### 2) URL만 변경: `history.pushState()` 호출

라우터는 `window.history.pushState()` 메서드를 호출해 브라우저의 URL을 바꾸되, **서버로 요청은 보내지 않습니다.**

- 브라우저 히스토리에 URL을 “추가”할 뿐
- 이 동작 자체가 자동으로 렌더링을 일으키지는 않음
- 대신 라우터가 URL 변경을 감지하고, 그에 맞는 컴포넌트를 찾아 렌더링

```js
window.history.pushState({}, "", "/about");
```

### 3) 경로 매칭 후 컴포넌트 렌더링

예를 들어 `/about`으로 이동하면, 라우터가 내부적으로 정의된 라우트 목록에서 해당 경로를 찾아 매핑된 컴포넌트를 렌더링합니다.

```jsx
<Route path="/about" element={<AboutPage />} />
```

이렇게 하면 전체 HTML을 다시 불러오지 않아도 브라우저가 마치 새로운 페이지로 이동한 것처럼 보이게 됩니다.

### 4) 필요한 데이터 로딩/상태 업데이트

화면 전환 과정에서 필요한 데이터가 있다면, 새롭게 렌더링된 컴포넌트 내부에서 `useEffect()` 등을 통해 API 요청을 보내거나 상태를 업데이트합니다.

```tsx
import { useEffect, useState } from "react";

export default function AboutPage() {
  const [data, setData] = useState(null);

  useEffect(() => {
    fetch("/api/about")
      .then((res) => res.json())
      .then(setData);
  }, []);

  return <div>{JSON.stringify(data)}</div>;
}
```

이 동작이 JavaScript로 이뤄지기 때문에, 네트워크 요청은 서버 HTML이 아니라 **데이터에 집중**되고, 사용자에게는 **빠른 페이지 전환**을 제공할 수 있습니다.

### 5) 뒤로/앞으로 가기: `popstate` 처리

브라우저의 “뒤로 가기/앞으로 가기” 버튼을 누르면 `popstate` 이벤트가 발생합니다.  
`React Router`는 이 이벤트를 구독하고, 바뀐 URL을 다시 분석해서 그에 맞는 컴포넌트를 렌더링합니다.

```js
window.addEventListener("popstate", () => {
  // 현재 URL을 분석한 뒤
  // 매핑된 컴포넌트를 렌더링하도록 라우터가 처리
});
```

따라서 사용자가 뒤로 가기를 눌러도 전체 리프레시가 일어나지 않고, CSR 방식으로 화면 전환만 일어나게 됩니다.

## 장점

- 전체 페이지를 다시 로드하지 않아 **화면 전환이 빠르게 느껴질 수 있습니다.**
- 서버 HTML 재요청 대신 필요한 **데이터 중심**으로 통신이 이뤄질 수 있습니다.

## 단점

- 원문에서는 별도의 단점이 직접적으로 언급되지는 않았습니다.

## 주의사항 / 실무 팁

- CSR 라우팅은 “URL만 바꾸고”, 실제 화면 변경은 **라우터가 URL 변화를 감지해 컴포넌트를 렌더링**하는 방식임을 분리해서 이해하면 좋습니다.
- 데이터가 필요하면 화면 전환 시점에 `useEffect()` 등으로 가져오게 되므로, 로딩 상태/에러 처리(스켈레톤, fallback UI 등)를 함께 설계하면 UX가 좋아집니다.
- 뒤로/앞으로 가기 처리는 `popstate` 기반이므로, 히스토리 기반 라우팅을 사용할 때 브라우저 네비게이션과의 동작을 함께 점검하세요.
