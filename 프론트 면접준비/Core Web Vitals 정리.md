# Core Web Vitals 정리

## 핵심 요약
- Core Web Vitals는 사용자 체감 성능을 핵심 지표로 묶은 세트입니다.
- 면접에서는 `LCP`, `INP`, `CLS`의 개념과 기준값, 최적화 방법을 연결해 설명하면 좋습니다.
- 백엔드 응답 속도, 리소스 번들 크기, 프론트 번들 전략이 서로 영향을 줍니다.

## 개념 정리
Core Web Vitals는 구글이 정의한 핵심 웹 사용자 경험 지표입니다.

- `LCP`(Largest Contentful Paint): 최대 콘텐츠가 화면에 렌더링되는 시간. `2.5초 이내`가 목표입니다.
- `INP`(Interaction to Next Paint): 사용자 입력(클릭/탭) 후 다음 화면 반응까지 걸리는 시간. `200ms 이내`가 이상적입니다.
- `CLS`(Cumulative Layout Shift): 페이지 로딩 중 요소 이동 누적량. `0.1 이하`가 좋습니다.

예전에는 `FID`가 주로 쓰였지만, 지금은 `INP`가 더 포괄적으로 반응성을 봅니다.

## 실무 예시
- 뉴스/커머스 페이지에서 큰 이미지가 늦게 뜨고, 스크롤 중 버튼 위치가 이동한다면
  - LCP 악화 + CLS 악화가 동시 발생한다는 신호입니다.
- 사용자 입력 후 딜레이가 크다면
  - INP가 높아졌을 가능성이 있고, JS 실행 스레드 병목이나 과도한 렌더 비용을 점검해야 합니다.

## 면접 답변 예시
"Core Web Vitals는 체감 성능을 객관화한 지표군입니다. 저는 개발 초기부터 `LCP`, `INP`, `CLS`를 번들 크기, 캐싱 전략, 이미지 전략, 레이아웃 고정으로 같이 관리합니다. 예를 들어 LCP는 중요한 비주얼을 `preload`와 이미지 최적화로 개선하고, INP는 이벤트 핸들러 경량화와 메인 스레드 분산으로 관리하며, CLS는 레이아웃 placeholder와 고정 높이/너비 설정으로 줄입니다. 면접에서는 단일 지표보다 사용자 흐름과 연계해 개선 우선순위를 설명할 수 있어야 합니다."

## 장점
- 사업 지표와 개발 지표를 연결할 수 있어 실무 정렬이 쉽습니다.
- 개선 전/후를 정량적으로 검증하기 좋습니다.
- 프론트 성능 최적화 우선순위를 객관적으로 정할 수 있습니다.

## 단점
- 측정 환경과 네트워크 상태에 따라 수치 분산이 큽니다.
- 지표만 맞추면 안 되고 실제 비즈니스 전환율까지 함께 봐야 합니다.
- 과도한 최적화가 오히려 개발 속도를 떨어뜨릴 수 있습니다.

## 주의사항 / 실무 팁
- `LCP`만 낮추고 INP를 무시하면 체감 성능이 다시 떨어집니다.
- CLS는 작은 레이아웃 이동은 수치상 무시될 수 있으니 실제 사용자 흐름(체크아웃, 폼 작성)에서 먼저 점검하세요.
- 측정은 실기기 + 네트워크 시뮬레이션으로 재현 가능하도록 CI에서 기준치를 관리합니다.
