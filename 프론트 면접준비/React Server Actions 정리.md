# 프론트 면접준비: React Server Actions 정리

## 핵심 요약
- Server Actions는 **React Server Components 환경에서 폼 제출 등 서버 작업을 직접 처리**하기 위한 서버 함수입니다.
- 클라이언트에서 `use client` 없이도 서버 상태 변경이 가능해지며, API 경계를 단순화할 수 있습니다.
- 보안, 직렬화 가능성, 호출 경계 관리가 핵심입니다.

## 개념 설명
Server Action은 `async` 함수를 서버에서 선언하고, 클라이언트에서 이를 직접 호출해 폸링/리렌더/리디렉션 등을 처리하는 방식입니다.  

기본 아이디어는 두 가지입니다.

- 클라이언트 이벤트(예: 폼 제출, 버튼 클릭)를 서버 함수에 연결한다.
- 서버는 그 요청을 받아 DB/외부 API에 반영 후 필요한 응답을 반환한다.

이때 함수는 `use server` 지시어 또는 서버 파일 내에서의 선언 규칙을 통해 서버에서 실행됨을 보장합니다.  
클라이언트 쪽에서는 별도 API URL을 직접 만들지 않고 서버 액션을 호출해 상태를 변경할 수 있어, 코드가 비교적 직관적으로 보입니다.

또한 Next.js에서 Server Actions는 Form 동작과 잘 맞아, 액션 실행 결과로 캐시 재검증(`revalidate`) 또는 리다이렉트가 연결되는 흐름을 쉽게 구성합니다.

## 실용 예시
예시: 주문 취소 버튼 클릭 시  
```ts
// app/order/actions.ts
'use server'

export async function cancelOrder(orderId: string) {
  // 권한 확인
  // DB에서 주문 상태 변경
  // 감사 로그 남기기
  return { ok: true };
}
```

```tsx
// app/order/page.tsx
import { cancelOrder } from './actions';

export default function CancelButton({ orderId }) {
  return (
    <form action={async () => cancelOrder(orderId)}>
      <button type="submit">주문 취소</button>
    </form>
  );
}
```

면접에서는 "왜 API 라우트 대신 액션을 쓰는가"를 묻는 경우가 많은데, 핵심은 **서버 동작을 선언적으로 바인딩**해 네트워크 레이어를 줄이는 설계적 이점 때문입니다.

## 면접 답변 예시
"Server Actions는 클라이언트에서 서버 상태 변경을 직접 호출하는 방식으로, 폼 동작과 자연스럽게 결합되고 API 엔드포인트 관리 비용을 줄일 수 있습니다. 다만 호출되는 함수는 반드시 서버에서 실행되어야 하므로 직렬화 가능한 인자/반환값, 인증, 캐시 무효화 전략을 함께 설계해야 합니다."

## 장점
- API 계층을 줄여 코드 경로가 단순해짐
- 폼 제출 로직과 데이터 변경 로직의 결합도가 낮아짐
- 서버 상태 반영 후 즉시 캐시/리다이렉션 동작 구성에 유리
- 클라이언트 번들에 서버 전용 라이브러리 포함 위험 감소

## 단점
- 서버/클라이언트 경계가 애매해지면 디버깅 난이도 상승
- 직렬화 가능한 데이터 제약으로 복잡한 객체 전달이 어렵거나 비효율
- 호출 지점이 분산되면 트랜잭션 추적이 어려울 수 있음

## 주의사항 / 실무 팁
- 액션마다 권한 체크를 **반드시** 수행하고, 사용자 입력은 서버에서 재검증하세요.
- 실패 응답을 일관되게 반환해 클라이언트 상태 분기를 명확히 하세요.
- 캐시가 중요한 페이지라면 `revalidate`나 무효화 전략을 미리 정하세요.
- `use server` 경계 파일에서만 사용되도록 팀 규칙을 정하면 운영 안정성이 높아집니다.
