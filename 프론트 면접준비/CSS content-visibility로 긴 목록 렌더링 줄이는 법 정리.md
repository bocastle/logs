# CSS content-visibility로 긴 목록 렌더링 줄이는 법 정리

## 핵심 요약

- 이 주제는 브라우저 기능을 그냥 켜는 문제가 아니라 사용자 흐름과 실패 처리를 함께 설계하는 문제다.
- 면접에서는 어떤 문제를 줄이려고 쓰는지, 지원하지 않는 환경에서는 어떻게 유지할지까지 말하면 좋다.
- 작은 API라도 접근성, 보안, 성능 비용을 같이 보면 답변이 훨씬 실무적으로 들린다.

## 개념 설명

`CSS content-visibility로 긴 목록 렌더링 줄이는 법`을 볼 때는 먼저 사용자가 실제로 겪는 상황을 떠올리면 쉽다. 탭을 여러 개 열었을 때, 네트워크가 끊겼을 때, 화면 전환 중 상태가 꼬였을 때처럼 브라우저 기능은 대부분 경계 상황에서 값이 드러난다.

프론트엔드 면접에서는 기능 이름을 아는 것보다 적용 기준을 설명하는 편이 중요하다. 이 기능이 없을 때도 핵심 흐름이 동작하는지, 적용 후 어떤 지표로 좋아졌다고 판단할지, 접근성이나 보안 요구사항과 충돌하지 않는지를 같이 말하면 된다.

## 예시

```css
@media (prefers-reduced-motion: reduce) {
  .panel {
    animation: none;
  }
}
```

예시는 전체 구현이 아니라 판단 기준을 설명하기 위한 최소 형태다. 실제 코드에서는 지원 여부 확인, 오류 처리, 정리 로직을 같이 둬야 한다.

## 면접 답변 예시

> 저는 `CSS content-visibility로 긴 목록 렌더링 줄이는 법`을 설명할 때 먼저 어떤 사용자 문제를 줄이는 기능인지 말하겠습니다. 그다음 브라우저 지원 범위와 fallback을 확인하고, 적용 후에는 성능 지표나 사용자 흐름이 실제로 좋아졌는지 보겠습니다. 특히 화면 상태나 입력 흐름에 영향을 주는 기능이라면 접근성, 취소 처리, 실패 메시지까지 같이 설계해야 안전하다고 답하겠습니다.

## 장점

- 사용자 흐름을 더 예측 가능하게 만들 수 있다.
- 브라우저가 제공하는 기능을 활용하면 직접 구현해야 할 복잡도를 줄일 수 있다.
- 성능이나 접근성 문제를 기능 단위로 설명하기 좋아진다.

## 단점

- 브라우저 지원 범위가 다르면 fallback 코드가 필요하다.
- 기능을 과하게 쓰면 오히려 상태 관리나 디버깅이 어려워질 수 있다.
- 보안이나 개인정보와 관련된 기능은 사용자 동의와 권한 실패를 고려해야 한다.

## 주의사항 / 실무 팁

- 적용 전에 지원 범위와 실패 시나리오를 먼저 확인한다.
- 기능이 없어도 핵심 흐름은 유지되도록 점진적 향상 방식으로 설계한다.
- 면접에서는 "언제 쓰는가"와 함께 "언제 피하는가"를 같이 말하면 답변이 단단해진다.
