# Service Worker 정리

## 핵심 요약

- `Service Worker`는 브라우저와 네트워크 사이에서 동작하는 백그라운드 스크립트다.
- 주로 오프라인 대응, 캐시 제어, 푸시 알림, 백그라운드 동기화 같은 기능에 쓰인다.
- `Web Worker`와 이름은 비슷하지만 목적이 다르다. `Web Worker`가 계산 작업 분리에 가깝다면, `Service Worker`는 네트워크 요청 가로채기와 앱 셸 관리에 더 가깝다.
- 면접에서는 "PWA의 핵심 기술 중 하나"라고 먼저 말하고, 캐시 전략과 주의점까지 이어서 설명하면 안정적이다.

## Service Worker란?

`Service Worker`는 브라우저 탭 안쪽이 아니라 그 바깥에서 별도로 등록되어 동작하는 스크립트다.  
그래서 페이지를 새로고침하거나 탭을 잠깐 닫아도, 등록된 워커가 네트워크 요청이나 푸시 이벤트를 처리할 수 있다.

쉽게 말하면 브라우저에 "이 사이트의 요청과 캐시를 내가 중간에서 관리해도 된다"라고 맡기는 느낌이다.

## 왜 쓰는가?

가장 대표적인 이유는 사용자 경험을 안정적으로 만들기 위해서다.

- 네트워크가 잠깐 불안정해도 캐시된 화면을 보여줄 수 있다.
- 정적 리소스를 캐시에 저장해 로딩 속도를 개선할 수 있다.
- 푸시 알림이나 백그라운드 동기화 같은 기능을 붙일 수 있다.

특히 PWA에서는 앱처럼 빠르고 끊김 없는 경험을 주기 위해 `Service Worker`를 거의 기본 요소처럼 다룬다.

## 동작 방식

보통 흐름은 이렇다.

1. 페이지가 `Service Worker`를 등록한다.
2. 브라우저가 워커를 설치하고 활성화한다.
3. 이후 특정 요청이 발생하면 워커가 이를 가로채서 네트워크로 보낼지, 캐시에서 응답할지 결정한다.

즉 핵심은 `fetch` 이벤트를 다룰 수 있다는 점이다.

## 예시

```js
self.addEventListener("fetch", (event) => {
  event.respondWith(
    caches.match(event.request).then((cachedResponse) => {
      return cachedResponse || fetch(event.request);
    })
  );
});
```

이 코드는 요청이 들어오면 먼저 캐시를 보고, 없으면 네트워크로 보내는 가장 단순한 형태다.

## 캐시 전략을 같이 말하면 좋다

면접에서는 `Service Worker` 자체보다 "어떤 전략으로 캐시할 건가"를 같이 묻는 경우가 많다.

- `Cache First`: 정적 파일에 유리하다.
- `Network First`: 최신 데이터가 중요한 API 응답에 더 어울린다.
- `Stale While Revalidate`: 일단 빠르게 캐시를 주고 뒤에서 새로 갱신한다.

이 세 가지 정도를 목적에 따라 구분해 말하면 실무 감각이 살아난다.

## Web Worker와 차이

이 부분은 헷갈리기 쉬워서 짧게라도 구분해 두는 편이 좋다.

- `Web Worker`: 무거운 계산을 메인 스레드 밖으로 빼는 데 초점이 있다.
- `Service Worker`: 네트워크 요청, 캐시, 오프라인 지원 같은 브라우저 기능 확장에 초점이 있다.

즉 둘 다 백그라운드에서 동작하지만, 해결하는 문제가 다르다.

## 면접에서 어떻게 말하면 좋을까?

Service Worker는 브라우저의 요청을 중간에서 제어할 수 있는 백그라운드 스크립트라고 설명하면 시작이 좋다.  
그 다음 오프라인 지원, 캐시 전략, 푸시 알림 같은 활용 예를 붙이면 충분히 구조가 잡힌 답변이 된다.

## 면접 답변 예시

> Service Worker는 브라우저와 네트워크 사이에서 동작하는 백그라운드 스크립트입니다. 페이지가 직접 열려 있지 않아도 요청을 가로채거나 캐시를 활용할 수 있어서, 오프라인 지원이나 PWA 구현에 많이 쓰입니다. 예를 들어 정적 파일은 Cache First로 빠르게 제공하고, API 데이터는 Network First로 최신성을 챙기는 식으로 전략을 나눌 수 있습니다. 다만 캐시를 잘못 관리하면 오래된 리소스를 계속 보여줄 수 있어서 업데이트 전략까지 같이 설계해야 합니다.

## 장점

- 오프라인 대응과 빠른 재방문 경험을 만들기 좋다.
- 정적 리소스 캐싱으로 로딩 속도를 개선할 수 있다.
- 푸시 알림, 백그라운드 동기화 같은 기능의 기반이 된다.

## 단점

- 캐시 무효화가 어렵고, 잘못 설계하면 오래된 화면이 남는다.
- 브라우저 생명주기와 업데이트 타이밍을 이해해야 해서 디버깅 난도가 올라간다.
- 모든 요청을 무조건 캐싱하면 데이터 최신성 문제가 생길 수 있다.

## 주의사항 / 실무 팁

- 정적 파일과 API 응답을 같은 전략으로 다루지 않는 편이 안전하다.
- 버전이 바뀌었을 때 어떤 캐시를 지울지 명확히 정해야 한다.
- 면접에서는 `Web Worker`와 차이, 캐시 전략, 오프라인 지원을 한 세트로 묶어 말하면 정리가 잘 되어 보인다.
