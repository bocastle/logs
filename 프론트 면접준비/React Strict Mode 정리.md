# React Strict Mode 정리

## 핵심 요약

- `React Strict Mode`는 개발 환경에서 잠재적인 사이드 이펙트와 오래된 패턴을 더 빨리 발견하게 도와주는 점검 장치다.
- 프로덕션 빌드에서 동작을 바꾸는 기능이라기보다, 개발 중 실수를 드러내는 안전장치에 가깝다.
- `useEffect`가 두 번 실행되는 것처럼 보이는 이유를 설명할 수 있어야 면접에서 많이 막히지 않는다.
- 답변할 때는 "왜 일부러 불편하게 보이게 만드나"를 함께 설명하면 이해도가 높아 보인다.

## React Strict Mode란?

`Strict Mode`는 React가 개발 환경에서만 활성화하는 검사용 래퍼다.  
앱을 감싸면 위험한 라이프사이클, 예상치 못한 부작용, 정리되지 않은 effect 같은 문제를 더 빨리 드러내 준다.

중요한 점은 이게 사용자 기능을 추가하는 옵션은 아니라는 점이다.  
개발자가 코드를 더 안전하게 유지하도록 돕는 도구라고 보면 된다.

## 왜 필요할까?

React는 선언형 UI를 지향하지만, 실제 버그는 대개 "렌더링 중에 뭔가를 바꿨다"거나 "effect 정리를 빼먹었다"는 식으로 나온다.  
이런 문제는 평소에는 잘 숨어 있다가 비동기 흐름이 복잡해질수록 크게 터진다.

`Strict Mode`는 일부 로직을 의도적으로 한 번 더 확인하면서 이런 문제를 빨리 드러낸다.

- effect cleanup이 빠져 있지 않은지
- 렌더링 과정에서 부수 효과를 만들지 않는지
- 오래된 API나 위험한 패턴을 쓰고 있지 않은지

즉, "지금은 돌아가지만 나중에 문제 될 코드"를 개발 단계에서 잡아내려는 목적이 크다.

## 왜 useEffect가 두 번 실행되는 것처럼 보일까?

면접에서 가장 자주 나오는 포인트다.

React 18 기준으로 개발 환경의 `Strict Mode`에서는 컴포넌트를 마운트했다가 바로 정리하고, 다시 마운트하는 흐름을 흉내 내기도 한다.  
이 과정에서 `useEffect`가 한 번 더 실행되는 것처럼 보인다.

이건 버그가 아니라, effect가 재실행되더라도 안전한지 확인하기 위한 동작이다.  
예를 들어 이벤트 리스너를 등록만 하고 해제하지 않았다면 이 과정에서 중복 등록 문제가 바로 드러난다.

## 간단한 예시

```jsx
useEffect(() => {
  const onResize = () => console.log(window.innerWidth);
  window.addEventListener("resize", onResize);

  return () => {
    window.removeEventListener("resize", onResize);
  };
}, []);
```

이 코드는 cleanup이 있으니 `Strict Mode`에서도 비교적 안전하다.  
반대로 `removeEventListener`가 빠져 있으면 개발 중에 리스너가 중복으로 붙는 문제가 더 잘 드러난다.

## 실무에서 어떻게 받아들이면 좋을까?

`Strict Mode` 때문에 화면이 이상해 보인다고 바로 꺼버리면 근본 원인을 놓치기 쉽다.  
보통은 아래 순서로 보는 게 좋다.

1. effect가 정말 필요한지 본다.
2. cleanup이 빠지지 않았는지 확인한다.
3. 렌더링 중에 네트워크 요청이나 외부 상태 변경을 하지 않았는지 점검한다.

즉, "Strict Mode가 문제를 만든다"기보다 "원래 있던 문제를 잘 보이게 만든다"에 가깝다.

## 면접 답변 예시

> React Strict Mode는 개발 환경에서 잠재적인 부작용을 빨리 찾기 위한 검사 도구입니다. 프로덕션 동작을 바꾸기보다는 effect cleanup 누락, 렌더링 중 부수 효과, 오래된 패턴 같은 문제를 드러내는 역할을 합니다. React 18에서는 개발 환경에서 일부 effect가 두 번 실행되는 것처럼 보일 수 있는데, 이건 버그라기보다 재마운트 상황에서도 안전한 코드를 만들도록 점검하는 과정입니다. 그래서 Strict Mode 때문에 문제가 생겼다고 보기보다, 원래 있던 취약한 로직이 드러났다고 이해하는 편이 맞습니다.

## 장점

- 숨겨진 사이드 이펙트를 개발 단계에서 빨리 찾을 수 있다.
- effect cleanup 누락 같은 실수를 비교적 빨리 드러낸다.
- 장기적으로 안전한 React 코드를 유지하는 데 도움이 된다.

## 단점

- 개발 중 effect가 두 번 도는 것처럼 보여 처음엔 혼란스럽다.
- 외부 라이브러리 코드가 엄격한 점검 흐름과 잘 안 맞으면 디버깅 비용이 생길 수 있다.
- 개념을 모르고 보면 "React가 이상하게 동작한다"고 오해하기 쉽다.

## 주의사항 / 실무 팁

- 개발 환경과 프로덕션 환경의 차이를 구분해서 설명해야 한다.
- `Strict Mode`를 꺼서 증상을 숨기기보다 cleanup 누락이나 부수 효과를 먼저 의심하는 게 좋다.
- 면접에서는 "`useEffect`가 왜 두 번 도는 것처럼 보이냐"는 질문과 연결해서 답하면 자연스럽다.
- "개발 중 검사용 장치"라는 성격을 먼저 말하고, 그다음 effect 재실행 예시를 붙이면 전달력이 좋다.
