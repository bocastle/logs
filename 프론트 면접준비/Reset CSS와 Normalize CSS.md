# Reset CSS와 Normalize CSS

웹 개발에서는 브라우저가 제공하는 기본 스타일(User Agent Stylesheet)이 브라우저마다 조금씩 다르기 때문에, 동일한 HTML/CSS라도 화면이 미묘하게 달라질 수 있습니다. 이를 통일(또는 정규화)하기 위해 **Reset CSS**와 **Normalize CSS**를 사용합니다. 두 방법은 목적은 비슷하지만 접근 방식이 다릅니다.

---

## Reset CSS란?

**Reset CSS**는 브라우저가 기본으로 제공하는 스타일을 **거의 전부 제거(초기화)** 해서, 모든 요소를 “스타일이 없는 출발점”으로 만드는 방식입니다.

### 특징

- 기본 `margin`, `padding`, `font-size`, `list-style` 등 **대부분을 0 또는 기본값으로 초기화**
- 브라우저 기본 스타일의 영향이 거의 없어서 **완전한 통제**가 가능
- 대신, 제목(h1~h6), 리스트(ul/ol), 버튼 등도 기본 스타일이 사라져서 **기본 UI를 직접 다시 스타일링해야 함**

### 간단 예시

```css
/* 대표적인 Reset 스타일 예시(간단 버전) */
* {
  margin: 0;
  padding: 0;
  box-sizing: border-box;
}
```

### 장점

- 브라우저별 기본 스타일 차이를 강하게 제거 → **디자인 일관성 확보**
- “디자인 시스템”처럼 모든 스타일을 직접 정의하는 프로젝트에 유리

### 단점

- 기본 스타일을 모두 잃기 때문에, 기본 레이아웃/타이포 등을 **다 처음부터 작성**해야 함
- 초기 세팅이 과하면 오히려 생산성이 떨어질 수 있음

---

## Normalize CSS란?

**Normalize CSS**는 브라우저 기본 스타일을 **완전히 없애지 않고**, 브라우저 간에 “불일치하는 부분만” **일관되게 보정**하는 방식입니다.

### 특징

- 기본 스타일을 유지하면서, 브라우저별 차이(예: `h1` 크기/여백, `button` 렌더링 차이 등)만 조정
- 결과적으로 **자연스러운 기본 스타일을 유지하면서 안정적으로 통일**

### 예시(개념)

```css
/* 브라우저마다 다를 수 있는 기본값을 '일관되게' 맞추는 방향 */
h1 {
  font-size: 2em;
  margin: 0.67em 0;
}
a {
  background-color: transparent;
}
```

### 장점

- 기본 스타일을 활용하므로 **초기 스타일링 부담이 적음**
- “일반적인 웹앱/서비스”에서 빠르게 안정적인 UI를 만들기 유리

### 단점

- Reset보다 “완전한 통일”은 어려울 수 있음(기본값을 어느 정도 남기기 때문)
- 프레임워크/디자인 시스템이 강하게 정해진 경우에는 오히려 혼합이 복잡할 수 있음

---

## 무엇을 선택하면 좋을까요?

프로젝트 성격에 따라 선택 기준이 달라집니다.

### Reset CSS가 어울리는 경우

- 디자인 시스템 기반으로 **모든 컴포넌트 스타일을 직접 정의**하는 경우
- “완전히 동일한 출발점”이 꼭 필요한 경우
- 레거시 브라우저/환경 차이를 강하게 통제해야 하는 경우

### Normalize CSS가 어울리는 경우

- 빠르게 개발해야 하거나, 기본 HTML 요소 스타일을 일정 부분 활용하고 싶은 경우
- 기본적인 타이포/여백을 “자연스럽게” 유지하면서 크로스 브라우저 차이만 줄이고 싶은 경우
- 유지보수 비용을 줄이고 싶은 일반적인 서비스 개발

---

## 참고로: 요즘은 어떻게 하나요?

최근에는 다음처럼 **부분 Reset + Normalize 성격**을 함께 가져가는 방식도 흔합니다.

- `box-sizing: border-box;` 같은 필수 규칙만 먼저 통일
- 기본 `margin`을 일부만 초기화
- 프레임워크(예: Tailwind의 preflight, modern CSS reset 등)에서 제공하는 기본 설정 사용

즉, “전부 다 Reset” 또는 “그대로 Normalize”의 이분법이 아니라, **프로젝트 요구에 맞게 적당히 조합**하는 경우가 많습니다.
