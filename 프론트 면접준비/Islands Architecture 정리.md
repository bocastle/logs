# Islands Architecture 정리

## 핵심 요약
- `Islands Architecture`는 페이지 대부분을 정적인 HTML로 두고, 상호작용이 필요한 부분만 작은 "섬"처럼 자바스크립트를 붙이는 방식이다.
- 전체 페이지를 한 번에 하이드레이션하는 대신 필요한 컴포넌트만 활성화하므로 초기 로딩 부담을 줄이기 좋다.
- 면접에서는 "SSR을 하느냐 CSR을 하느냐"의 이분법보다, 인터랙션 범위를 최소화해 성능을 관리하는 설계라고 설명하면 좋다.
- 콘텐츠 중심 서비스나 마케팅 페이지처럼 정적인 영역이 많은 화면에서 특히 잘 맞는다.

## 개념 설명
전통적인 CSR은 첫 화면을 그리기 위해 자바스크립트 의존도가 높고, 반대로 전체 SSR 후 전면 하이드레이션 방식은 화면은 빨리 보이더라도 브라우저가 많은 컴포넌트를 다시 활성화해야 하는 부담이 있다.

`Islands Architecture`는 이 중간 지점을 노린다. 페이지의 본문, 이미지, 설명 텍스트처럼 정적인 부분은 서버에서 바로 HTML로 내려 보내고, 검색창 자동완성, 장바구니 버튼, 댓글 입력창처럼 상호작용이 필요한 부분만 개별적으로 하이드레이션한다.

핵심은 "페이지 전체를 앱처럼 다루지 않는다"는 점이다. 그래서 면접에서는 정적 콘텐츠 비중이 높은 화면에서 불필요한 자바스크립트 실행을 줄이는 아키텍처라고 말하면 이해가 쉽다.

## 예시
상품 상세 페이지를 생각해보자.

- 정적으로 보여도 되는 영역
  - 상품 설명
  - 상세 이미지
  - 리뷰 요약 텍스트
- 인터랙션이 필요한 영역
  - 옵션 선택 드롭다운
  - 장바구니 버튼
  - 리뷰 필터

이런 화면에서 전체를 한 번에 하이드레이션하면 상품 설명 영역까지 모두 자바스크립트 활성화 대상이 된다. 반면 `Islands Architecture`를 쓰면 실제로 클릭이나 입력이 필요한 컴포넌트만 따로 살리면 된다.

## 면접 답변 예시
- 질문: "`Islands Architecture`는 무엇인가요?"  
  답변: "`Islands Architecture`는 페이지 대부분을 정적 HTML로 렌더링하고, 상호작용이 필요한 일부 컴포넌트만 선택적으로 하이드레이션하는 방식입니다. 그래서 초기 자바스크립트 비용을 줄이고, 콘텐츠 중심 페이지의 성능을 개선하는 데 유리합니다."
- 질문: "SSR과 어떤 차이가 있나요?"  
  답변: "SSR은 서버 렌더링 방식 자체를 말하고, `Islands Architecture`는 그 위에서 어떤 영역을 인터랙티브하게 만들지 더 세밀하게 나누는 설계에 가깝습니다. 즉 서버 렌더링 후 전부 하이드레이션하는 대신 필요한 부분만 활성화하는 것이 차이입니다."

## 장점
- 초기 자바스크립트 다운로드와 실행량을 줄이기 좋다
- 콘텐츠 중심 페이지에서 빠른 첫 화면과 가벼운 인터랙션을 함께 가져가기 쉽다
- 어떤 영역이 진짜 인터랙티브한지 설계 관점에서 분리하게 만들어 구조가 선명해진다

## 단점
- 페이지를 여러 섬으로 나누는 기준이 불명확하면 오히려 구조가 복잡해질 수 있다
- 상태 공유가 많은 대규모 앱 화면에서는 섬 간 연결 비용이 커질 수 있다
- 프레임워크와 빌드 체인 지원 방식에 따라 적용 난이도가 달라진다

## 주의사항 / 실무 팁
- 모든 페이지에 무조건 적용하기보다 정적 콘텐츠 비중이 높은 화면인지 먼저 판단하는 편이 좋다.
- 섬을 너무 잘게 쪼개면 초기 비용은 줄어도 관리 포인트가 늘 수 있다.
- 면접에서는 `Hydration`, `SSR`, `Code Splitting`과 연결해서 설명하면 답변이 자연스럽다.
- 실무에서는 사용자 행동이 몰리는 핵심 인터랙션만 섬으로 남기고, 나머지는 가능한 한 정적으로 유지하는 방향이 현실적이다.
