# Tree Shaking 정리

## 핵심 요약

- `Tree Shaking`은 실제로 사용하지 않는 코드를 번들에서 제거해 최종 파일 크기를 줄이는 최적화 방식이다.
- 핵심은 `모듈 구조가 정적으로 분석 가능해야 한다`는 점이다.
- 면접에서는 "번들러가 안 쓰는 코드를 자동으로 없애 주는 것" 정도로만 말하지 말고, `ES Module 기반의 정적 분석`과 `부수 효과(side effect)`까지 같이 설명하면 답변이 탄탄해진다.
- 장점은 번들 크기 감소지만, import 방식이나 라이브러리 구조에 따라 기대만큼 안 될 수 있다는 점도 같이 말해야 한다.

## Tree Shaking이 뭐냐

프론트엔드 프로젝트가 커질수록 번들 크기는 성능에 직접적인 영향을 준다.  
이때 모든 코드를 무조건 다 묶는 대신, 실제로 사용한 코드만 남기고 안 쓰는 코드는 제거하려는 최적화가 Tree Shaking이다.

쉽게 말하면 `import는 했지만 실제로 쓰지 않는 export`, 혹은 아예 참조되지 않는 모듈 코드를 번들 단계에서 털어내는 과정이라고 보면 된다.

## 왜 이름이 Tree Shaking이냐

모듈 의존성 그래프를 나무처럼 보고, 쓰이지 않는 가지를 흔들어서 떨어뜨린다는 비유에서 나온 표현이다.

면접에서는 이 비유를 길게 설명할 필요는 없고, `의존성 그래프에서 불필요한 코드를 제거하는 최적화`라고 정리하면 충분하다.

## 어떻게 동작하나

Tree Shaking이 잘 되려면 번들러가 어떤 코드가 사용되고 어떤 코드가 사용되지 않는지 미리 분석할 수 있어야 한다.

여기서 중요한 게 `ES Module`이다.

- `import` / `export`는 정적 구조라서 분석이 쉽다.
- 반대로 CommonJS의 `require`는 런타임 동적 해석 성격이 강해 Tree Shaking이 불리하다.

예를 들어 아래처럼 유틸이 여러 개 있어도:

```ts
export const sum = (a, b) => a + b;
export const multiply = (a, b) => a * b;
```

실제로 `sum`만 import해서 사용한다면, 번들러 설정과 코드 구조가 맞을 때 `multiply`는 최종 번들에서 빠질 수 있다.

## 왜 기대만큼 안 되는 경우가 있나

Tree Shaking은 생각보다 조건이 있다.

- 라이브러리가 CommonJS 기반이면 잘 안 될 수 있다.
- `import * as utils`처럼 통으로 가져오면 제거 여지가 줄 수 있다.
- 모듈 import 시 실행되는 부수 효과 코드가 있으면 안전하게 제거하지 못할 수 있다.

예를 들어 import만 해도 전역 스타일을 주입하거나, 전역 객체를 수정하거나, 로그를 남기는 모듈은 번들러가 함부로 없애기 어렵다.

## sideEffects는 왜 같이 나오나

실무에서는 `package.json`의 `sideEffects` 설정이 같이 언급된다.

이 값은 `이 패키지의 파일들이 import될 때 부수 효과가 있는가`를 번들러에게 알려 주는 힌트다.

- `sideEffects: false`라면 미사용 코드를 더 공격적으로 제거하기 좋다.
- 다만 실제로 부수 효과가 있는데 false로 잘못 선언하면 필요한 코드까지 빠질 수 있다.

그래서 면접에서는 `Tree Shaking은 코드 구조뿐 아니라 패키지 메타데이터와 번들러 설정에도 영향을 받는다`고 말하면 좋다.

## 성능 최적화와는 어떻게 연결되나

번들 크기가 줄어들면 보통 다음에 도움이 된다.

- 네트워크 다운로드 크기 감소
- 파싱 및 실행 비용 감소
- 초기 로딩 성능 개선

특히 대형 라이브러리를 일부만 써야 하는 프로젝트에서 효과가 크다.  
다만 무조건 체감 성능이 좋아진다고 단정하기보다, `번들 분석과 함께 봐야 한다`고 말하는 편이 현실적이다.

## 면접 답변 예시

> Tree Shaking은 번들링 과정에서 실제로 사용하지 않는 코드를 제거해서 최종 번들 크기를 줄이는 최적화입니다. 보통 ES Module의 정적 import/export 구조를 기반으로 분석하고, side effect가 없는 코드일수록 더 잘 제거됩니다. 그래서 단순히 번들러가 알아서 최적화해 준다기보다, 모듈 방식이나 import 패턴, package.json의 sideEffects 설정까지 영향을 준다고 이해하고 있습니다.

## 장점

- 번들 최적화 원리를 이해하고 있다는 점을 보여 주기 좋다.
- ES Module, 번들러, 성능 최적화를 자연스럽게 연결할 수 있다.
- 실무에서 왜 import 습관이 중요한지도 같이 설명하기 좋다.

## 단점

- "안 쓰는 코드는 무조건 다 지워진다"라고 말하면 너무 단순한 답변이 된다.
- CommonJS나 side effect 이슈를 모르면 깊이가 얕아 보일 수 있다.
- 실제 측정 없이 막연히 성능 향상만 강조하면 실무 감각이 약해 보일 수 있다.

## 주의사항 / 실무 팁

- 면접에서는 `ES Module 기반 정적 분석`을 꼭 한 번 넣어 주는 게 좋다.
- `import *`나 무거운 라이브러리 전체 import가 Tree Shaking에 불리할 수 있다고 말하면 좋다.
- side effect가 있으면 제거가 어려워진다는 점을 같이 설명하면 답변이 훨씬 안정적이다.
- 번들 크기 최적화는 Tree Shaking 하나로 끝나는 게 아니라 코드 스플리팅, 캐싱 전략과 같이 봐야 한다.
