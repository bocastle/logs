# Hydration Mismatch 정리

## 핵심 요약
- `Hydration Mismatch`는 서버가 먼저 렌더링한 HTML과 클라이언트가 초기 렌더링하면서 기대한 결과가 달라 생기는 불일치 문제다.
- SSR이나 하이드레이션 기반 프레임워크를 사용할 때 자주 나오는 면접 주제다.
- 핵심은 "서버와 클라이언트가 같은 입력값으로 같은 결과를 만들어야 한다"는 점이다.
- 면접에서는 "초기 HTML과 클라이언트 렌더 결과가 다르면 경고가 나고, 경우에 따라 UI가 다시 그려지거나 이벤트 연결이 꼬일 수 있다"고 설명하면 된다.

## 개념 설명
SSR 환경에서는 서버가 먼저 HTML을 만들어 내려주고, 브라우저에서 자바스크립트가 로드되면 그 HTML 위에 이벤트를 연결하면서 인터랙션 가능한 상태로 만든다. 이 과정을 `Hydration`이라고 한다.

문제는 서버와 클라이언트가 처음 렌더링할 때 서로 다른 결과를 만들면 생긴다. 예를 들어 서버에서는 현재 시간을 기준으로 텍스트를 만들었고, 클라이언트에서는 조금 뒤의 시간을 기준으로 다시 렌더링하면 문자열이 달라질 수 있다. 혹은 서버에서는 브라우저 API를 못 쓰는데 클라이언트에서는 `window` 값을 기준으로 UI를 바꾸면 구조 자체가 달라질 수도 있다.

이런 불일치가 바로 `Hydration Mismatch`다. 단순 경고로 끝날 수도 있지만, 심하면 예상치 못한 재렌더링이 일어나고, 초기 성능 이점이 줄어들고, 사용자 입장에서 화면이 순간적으로 바뀌는 문제가 생길 수 있다.

## 예시
다음과 같은 컴포넌트는 위험할 수 있다.

- 서버 렌더링 시점에 `new Date()`로 시간을 문자열로 만든다
- 클라이언트가 실행될 때 다시 `new Date()`로 다른 값을 만든다

그러면 서버 HTML의 시간과 클라이언트 렌더 시간이 달라져 불일치가 발생할 수 있다. 또 `localStorage` 값 존재 여부에 따라 초기 마크업이 달라지는 경우도 비슷한 문제가 생긴다.

실무에서는 이런 값을 서버와 클라이언트가 공유 가능한 형태로 전달하거나, 클라이언트 전용 값은 마운트 이후에 반영하는 식으로 처리한다.

## 면접 답변 예시
- 질문: "`Hydration Mismatch`는 왜 발생하나요?"  
  답변: "서버가 만든 초기 HTML과 클라이언트가 하이드레이션하면서 기대한 렌더 결과가 다를 때 발생합니다. 시간, 랜덤 값, 브라우저 전용 API, 조건부 렌더링 차이 때문에 자주 생깁니다."
- 질문: "예방 방법은 무엇인가요?"  
  답변: "서버와 클라이언트가 동일한 입력으로 동일한 마크업을 만들게 해야 합니다. 랜덤 값이나 현재 시간처럼 달라지기 쉬운 값은 초기 렌더에서 조심하고, 브라우저 전용 처리는 마운트 이후로 미루는 편이 안전합니다."

## 장점
- SSR, CSR, 하이드레이션 차이를 한 번에 설명하기 좋은 주제다
- 서버와 클라이언트 렌더링 경계에 대한 이해를 보여 줄 수 있다
- Next.js 같은 프레임워크 실무 경험과 연결하기 쉽다

## 단점
- 단순히 "경고가 난다" 수준으로만 설명하면 깊이가 부족해 보일 수 있다
- 원인과 해결 전략이 여러 층에 걸쳐 있어 설명이 조금 복합적일 수 있다
- 프레임워크별 예외 케이스까지 들어가면 답변이 길어질 수 있다

## 주의사항 / 실무 팁
- 초기 렌더 결과가 서버와 클라이언트에서 같아야 한다는 원칙을 먼저 기억하는 것이 좋다.
- `Date`, `Math.random`, 브라우저 전용 API, 사용자 환경 의존 조건문은 대표적인 점검 대상이다.
- 면접에서는 `SSR과 CSR`, `하이드레이션(Hydration)`, `React Server Components` 주제와 연결하면 자연스럽다.
- 실무에서는 에러 로그만 보지 말고 실제 사용자 화면이 순간적으로 바뀌는지까지 확인해야 한다.
