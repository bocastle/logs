# 이벤트 루프 정리

## 핵심 요약

- JavaScript는 기본적으로 **한 번에 하나의 작업만 처리하는 싱글 스레드 언어**입니다.
- 그런데도 비동기 작업을 처리할 수 있는 이유는 브라우저가 Web API, 작업 큐, 이벤트 루프를 함께 사용하기 때문입니다.
- 이벤트 루프는 Call Stack이 비었을 때 큐에 쌓인 콜백을 꺼내 실행해주는 역할을 합니다.
- 면접에서는 **Call Stack, Web API, Task Queue, Microtask Queue**의 관계를 설명할 수 있으면 좋습니다.

## 이벤트 루프란?

이벤트 루프는 JavaScript의 비동기 작업이 실행되는 순서를 관리하는 구조입니다.

JavaScript 자체는 한 줄씩 실행됩니다.  
함수가 실행되면 Call Stack에 올라가고, 실행이 끝나면 Stack에서 빠져나갑니다. 여기까지만 보면 모든 작업이 순서대로만 실행될 것 같습니다.

그런데 실제 웹에서는 `setTimeout`, `fetch`, 이벤트 리스너, Promise 같은 비동기 작업이 많습니다.  
이 작업들을 JavaScript 엔진 혼자 처리하는 것이 아니라, 브라우저의 Web API와 큐가 함께 도와줍니다.

이 사이를 조율하는 역할이 바로 이벤트 루프입니다.

## 먼저 알아야 할 구성 요소

### Call Stack

Call Stack은 현재 실행 중인 함수들이 쌓이는 공간입니다.

```js
function first() {
  second();
}

function second() {
  console.log("second");
}

first();
```

위 코드가 실행되면 `first()`가 Stack에 올라가고, 그 안에서 `second()`가 다시 올라갑니다.  
`second()` 실행이 끝나면 빠지고, 이어서 `first()`도 끝나면서 빠집니다.

쉽게 말하면 Call Stack은 “지금 JavaScript가 처리 중인 일 목록”입니다.

### Web API

Web API는 브라우저가 제공하는 기능입니다.

예를 들면 이런 것들이 있습니다.

- `setTimeout`
- DOM 이벤트
- `fetch`
- `requestAnimationFrame`

JavaScript 코드에서 `setTimeout`을 호출하면 타이머 자체를 Call Stack에서 계속 들고 있는 것이 아닙니다.  
브라우저의 Web API 쪽에 맡겨두고, JavaScript는 다음 코드를 계속 실행합니다.

### Task Queue

Task Queue는 Web API에서 완료된 비동기 콜백이 대기하는 곳입니다.

예를 들어 `setTimeout`의 시간이 끝나면, 콜백 함수가 바로 실행되는 것이 아니라 Task Queue에 들어갑니다.  
이후 Call Stack이 비면 이벤트 루프가 이 콜백을 꺼내 Stack에 올립니다.

### Microtask Queue

Microtask Queue는 Promise 관련 콜백처럼 조금 더 우선순위가 높은 비동기 작업이 대기하는 곳입니다.

대표적으로 아래 작업들이 Microtask Queue에 들어갑니다.

- `Promise.then`
- `Promise.catch`
- `Promise.finally`
- `queueMicrotask`

중요한 점은 Microtask Queue가 일반 Task Queue보다 먼저 처리된다는 것입니다.

## 실행 순서 예시

아래 코드를 보면 이벤트 루프의 흐름이 잘 드러납니다.

```js
console.log("A");

setTimeout(() => {
  console.log("B");
}, 0);

Promise.resolve().then(() => {
  console.log("C");
});

console.log("D");
```

출력 순서는 다음과 같습니다.

```txt
A
D
C
B
```

처음 보면 `setTimeout` 시간이 0이니까 `B`가 먼저 나올 것 같지만 그렇지 않습니다.

흐름을 풀어보면 이렇습니다.

1. `console.log("A")`가 바로 실행됩니다.
2. `setTimeout` 콜백은 Web API를 거쳐 Task Queue로 갈 준비를 합니다.
3. `Promise.then` 콜백은 Microtask Queue에 들어갑니다.
4. `console.log("D")`가 바로 실행됩니다.
5. 현재 Call Stack이 비면 Microtask Queue의 `C`가 먼저 실행됩니다.
6. 그다음 Task Queue의 `B`가 실행됩니다.

그래서 결과는 `A → D → C → B`가 됩니다.

## setTimeout 0초는 바로 실행된다는 뜻일까?

`setTimeout(fn, 0)`은 “0초 뒤에 바로 실행”이라기보다 “최소 0ms 이후에 실행 가능한 상태가 되면 큐에 넣는다”에 가깝습니다.

즉, 현재 실행 중인 동기 코드가 모두 끝나고, Microtask Queue까지 처리된 뒤에야 실행될 수 있습니다.  
그래서 타이머 시간이 0이어도 동기 코드보다 먼저 실행되지는 않습니다.

## async/await와 이벤트 루프

`async/await`도 결국 Promise 기반으로 동작합니다.

```js
async function run() {
  console.log("A");
  await Promise.resolve();
  console.log("B");
}

run();
console.log("C");
```

출력은 다음과 같습니다.

```txt
A
C
B
```

`await`를 만나면 함수의 나머지 부분은 잠시 멈추고, 이후 처리는 Microtask Queue 쪽으로 넘어갑니다.  
그래서 `run()` 바깥에 있는 `console.log("C")`가 먼저 실행되고, 그다음 `B`가 출력됩니다.

## 렌더링과 이벤트 루프

브라우저는 JavaScript만 실행하는 것이 아니라 화면도 그려야 합니다.

문제는 JavaScript 실행이 너무 오래 걸리면 브라우저가 렌더링할 시간을 얻지 못한다는 점입니다.  
Call Stack에 무거운 동기 작업이 오래 남아 있으면 클릭, 스크롤, 애니메이션도 밀릴 수 있습니다.

그래서 긴 작업은 적절히 나누거나, 화면 갱신과 관련된 작업은 `requestAnimationFrame`을 활용하는 것이 좋습니다.

## 면접 답변 예시

이벤트 루프는 JavaScript의 비동기 작업 실행 순서를 관리하는 구조입니다.  
JavaScript는 싱글 스레드라서 Call Stack에서 한 번에 하나의 작업만 처리합니다. 하지만 `setTimeout`, DOM 이벤트, `fetch` 같은 작업은 브라우저의 Web API에 맡길 수 있고, 완료된 콜백은 큐에 들어갑니다.

이벤트 루프는 Call Stack이 비었는지 확인하고, 비어 있으면 큐에서 대기 중인 콜백을 꺼내 실행합니다.  
이때 Promise의 `then` 같은 콜백은 Microtask Queue에 들어가고, `setTimeout` 같은 콜백은 Task Queue에 들어갑니다. Microtask Queue가 일반 Task Queue보다 먼저 처리되기 때문에 `Promise.then`이 `setTimeout(..., 0)`보다 먼저 실행될 수 있습니다.

## 장점

- JavaScript가 싱글 스레드인데도 비동기 작업을 다룰 수 있는 이유를 이해할 수 있습니다.
- `setTimeout`, Promise, `async/await`의 실행 순서를 예측하기 쉬워집니다.
- 화면이 멈추거나 이벤트 처리가 늦어지는 원인을 파악하는 데 도움이 됩니다.

## 단점

- 큐의 종류와 우선순위를 처음에는 헷갈리기 쉽습니다.
- 실제 브라우저 환경에서는 렌더링, 네트워크, 사용자 이벤트까지 함께 고려해야 해서 단순 예제보다 복잡할 수 있습니다.

## 주의사항 / 실무 팁

- `setTimeout(fn, 0)`은 즉시 실행이 아니라 큐에 들어갈 수 있는 가장 빠른 시점을 의미합니다.
- Promise 콜백은 일반 Task보다 먼저 처리되는 Microtask입니다.
- 오래 걸리는 동기 작업은 이벤트 처리와 렌더링을 막을 수 있으니 주의해야 합니다.
- 실행 순서가 헷갈릴 때는 Call Stack, Microtask Queue, Task Queue를 나눠서 생각하면 훨씬 편합니다.
