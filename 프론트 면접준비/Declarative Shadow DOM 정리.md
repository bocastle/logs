# Declarative Shadow DOM 정리

## 핵심 요약
- `Declarative Shadow DOM`은 서버가 보낸 HTML 안에서 `shadow root`를 선언적으로 정의하는 방식이다.
- 기존 `Shadow DOM`은 보통 자바스크립트로 `attachShadow()`를 호출해야 했는데, 이 방식은 초기 렌더링 시점부터 컴포넌트 구조를 포함할 수 있다.
- 면접에서는 "웹 컴포넌트를 SSR 친화적으로 만들기 위한 장치"라고 설명하면 핵심이 잘 전달된다.
- 장점뿐 아니라 브라우저 지원 범위, 디버깅 난이도, 스타일 격리 전략도 같이 말해야 답변이 균형 잡힌다.

## 개념 설명
기존 `Shadow DOM`은 컴포넌트 내부 마크업과 스타일을 외부와 격리하는 데 유용하지만, 초기 렌더링 단계에서는 한계가 있었다. 서버가 HTML을 내려줘도 실제 `shadow root`는 클라이언트 자바스크립트가 실행된 뒤에야 붙기 때문이다. 그래서 SSR 환경에서는 초기 콘텐츠 표시나 하이드레이션 흐름이 어색해질 수 있다.

`Declarative Shadow DOM`은 이런 문제를 줄이기 위해 HTML 안에서 미리 `shadow root`를 선언할 수 있게 한다. 브라우저는 문서를 파싱하면서 해당 구조를 바로 `shadow tree`로 해석한다. 즉, 자바스크립트 실행 전에도 컴포넌트 내부 구조를 비교적 온전히 전달할 수 있다.

이 개념의 핵심은 "`Shadow DOM`의 장점을 유지하면서도 서버 렌더링 흐름과 더 잘 맞춘다"는 점이다. 디자인 시스템, 웹 컴포넌트 기반 UI, 서버에서 빠르게 첫 화면을 보여줘야 하는 경우에 특히 의미가 있다.

다만 모든 컴포넌트에 무조건 필요한 기능은 아니다. 상태 변화가 복잡하고 클라이언트 로직 의존도가 높은 경우라면, 선언형으로 초기 구조를 내려주는 것만으로는 충분하지 않을 수 있다.

## 예시
예를 들어 회사 디자인 시스템에서 공통 `user-card` 웹 컴포넌트를 사용한다고 해 보자.

- 기존 방식은 서버가 `<user-card>` 태그만 내려주고, 클라이언트가 실행된 뒤 `attachShadow()`로 내부 템플릿을 붙인다
- 이 경우 JS 실행 전에는 내부 콘텐츠가 비어 보이거나, SEO/초기 렌더링 관점에서 아쉬움이 생길 수 있다
- `Declarative Shadow DOM`을 쓰면 서버 HTML 자체에 shadow tree를 포함해 첫 렌더링부터 카드 구조와 스타일을 전달할 수 있다

즉, 웹 컴포넌트를 쓰면서도 SSR의 장점을 더 살리려는 상황에서 유용하다.

## 면접 답변 예시
- 질문: "`Declarative Shadow DOM`은 무엇인가요?"  
  답변: "서버가 내려주는 HTML 안에서 `shadow root`를 선언적으로 정의하는 방식입니다. 기존처럼 자바스크립트로 나중에 `attachShadow()`를 호출하지 않아도 돼서 웹 컴포넌트를 SSR 환경과 더 잘 결합할 수 있습니다."
- 질문: "왜 필요한가요?"  
  답변: "기존 `Shadow DOM`은 클라이언트 JS가 실행된 뒤에야 내부 구조가 완성되는 경우가 많아서 초기 렌더링과 SEO 측면에서 불리할 수 있습니다. 선언형 방식은 이 초기 공백을 줄여 줍니다."
- 질문: "주의할 점은 무엇인가요?"  
  답변: "브라우저 지원 범위를 확인해야 하고, 스타일 격리와 하이드레이션 전략을 전체 렌더링 구조 안에서 같이 설계해야 합니다."

## 장점
- 웹 컴포넌트와 SSR을 더 자연스럽게 결합할 수 있다
- 초기 렌더링 시점에 컴포넌트 내부 구조를 더 빨리 전달할 수 있다
- `Shadow DOM`의 캡슐화 장점을 유지하면서 서버 렌더링 경험을 개선할 수 있다

## 단점
- 브라우저 지원과 폴백 전략을 따로 고려해야 한다
- 선언형 구조와 클라이언트 로직이 어긋나면 하이드레이션 디버깅이 까다로울 수 있다
- 팀이 `Shadow DOM` 자체에 익숙하지 않으면 유지보수 난도가 올라갈 수 있다

## 주의사항 / 실무 팁
- `Declarative Shadow DOM`은 "웹 컴포넌트를 쓰는 상황"에서 빛난다. 일반적인 React 컴포넌트에 바로 대입하면 설명이 어색해질 수 있다.
- 면접에서는 "`Shadow DOM`의 SSR 친화 버전"이라고 축약해서 설명한 뒤, 왜 초기 렌더링에 유리한지 이어서 말하면 좋다.
- 실무에서는 SEO, 서버 템플릿 생성 비용, 폴리필 필요 여부를 함께 봐야 한다.
- 디자인 시스템 수준에서 도입할 때는 어떤 컴포넌트가 `Shadow DOM`을 써야 하는지부터 명확히 정해야 혼란이 줄어든다.
