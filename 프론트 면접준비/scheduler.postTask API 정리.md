# scheduler.postTask API 정리

## 핵심 요약
- `scheduler.postTask API`는 브라우저에게 "이 작업은 어느 정도 우선순위로 처리해 달라"라고 힌트를 주는 작업 스케줄링 API다.
- `setTimeout`처럼 단순히 나중에 실행하는 것과 달리, 사용자 입력 대응 같은 중요한 작업을 더 빨리 처리하도록 유도할 수 있다.
- 면접에서는 메인 스레드 작업을 잘게 나누고, 사용자 체감 성능을 지키기 위한 도구라고 설명하면 좋다.
- 아직 모든 환경에서 완전히 동일하게 쓰이지는 않아서 폴백 전략까지 같이 말하면 답변이 탄탄해진다.

## 개념 설명
프론트엔드 성능 문제는 "자바스크립트가 느리다"보다 "메인 스레드가 오래 점유된다"로 설명하는 편이 더 정확할 때가 많다. 화면 그리기, 클릭 처리, 자바스크립트 실행이 모두 같은 메인 스레드를 놓고 경쟁하기 때문이다.

`scheduler.postTask API`는 이런 상황에서 작업 우선순위를 표현할 수 있게 도와준다. 예를 들어 지금 당장 사용자 입력에 반응해야 하는 작업은 더 높은 우선순위로 보내고, 분석 로그 적재나 미리 계산해 둘 값은 낮은 우선순위로 미룰 수 있다.

핵심은 "비동기" 자체보다 "브라우저에게 작업 중요도를 전달한다"는 점이다. 그래서 면접에서는 `Promise`나 `setTimeout`과 비교하면서, 단순 지연이 아니라 스케줄링 의도를 표현하는 API라고 설명하면 좋다.

## 예시
검색 자동완성 UI를 생각해보자.

- 사용자가 키를 입력하면
  - 입력창 값 반영과 추천 목록 렌더링은 빠르게 처리되어야 한다.
  - 반면 검색 행동 로그 저장이나 백그라운드 프리페치 준비는 조금 늦어도 된다.

이때

- 높은 우선순위 작업
  - 입력 반영
  - 추천 결과 그리기
- 낮은 우선순위 작업
  - 분석 이벤트 전송
  - 다음 화면 데이터 사전 준비

처럼 나누면 사용자 입장에서는 "입력은 즉시 반응하고, 부가 작업은 뒤에서 조용히 처리되는" 경험을 만들기 쉽다.

## 면접 답변 예시
- 질문: "scheduler.postTask API는 왜 쓰나요?"  
  답변: "`scheduler.postTask API`는 메인 스레드에서 실행할 작업의 우선순위를 표현하기 위해 씁니다. 단순히 나중에 실행하는 것이 아니라, 사용자 입력처럼 중요한 작업과 부가 작업을 구분해서 브라우저가 더 나은 응답성을 유지하도록 돕는다는 점이 핵심입니다."
- 질문: "`setTimeout`과 차이가 뭔가요?"  
  답변: "`setTimeout`은 시간 지연 중심이고, `scheduler.postTask`는 작업 중요도 중심입니다. 그래서 인터랙션 성능을 다룰 때는 어떤 작업을 먼저 처리해야 하는지 의도를 더 명확하게 전달할 수 있습니다."

## 장점
- 메인 스레드 작업을 중요도 기준으로 나눠 사용자 반응성을 높이기 좋다
- 긴 작업을 여러 조각으로 나눠 처리하는 패턴과 잘 어울린다
- 성능 최적화를 설명할 때 단순 지연이 아니라 우선순위 설계 관점으로 답변할 수 있다

## 단점
- 브라우저 지원 상황을 확인하고 폴백을 준비해야 한다
- 우선순위를 세밀하게 나누더라도 근본적으로 무거운 작업 자체가 사라지는 것은 아니다
- 팀이 우선순위 기준을 합의하지 않으면 오히려 코드 의도가 불명확해질 수 있다

## 주의사항 / 실무 팁
- 중요한 것은 API 사용 자체보다 긴 작업을 쪼개고, 정말 급한 작업이 무엇인지 구분하는 설계다.
- 지원하지 않는 환경에서는 `setTimeout`, `requestIdleCallback`, 일반 비동기 분리 등으로 폴백 전략을 준비하는 편이 현실적이다.
- 면접에서는 `Long Task`, `useTransition`, 입력 반응성 개선과 연결해서 설명하면 맥락이 살아난다.
- 실무에서는 "모든 작업을 high priority로 두지 않는다"는 원칙을 함께 말하면 균형 잡힌 답변이 된다.
