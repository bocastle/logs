# 트리 쉐이킹(Tree Shaking) 정리

## 핵심 요약

트리 쉐이킹(Tree Shaking)은 **최종 번들 크기를 줄이기 위해 사용되지 않는 코드를 제거**하는 최적화 기법입니다. 주로 **모듈 시스템 기반**으로 동작하며, **ES 모듈(ESM)의 정적 구조**를 활용해 사용 여부를 분석합니다.

## 개념

- 트리 쉐이킹은 번들링 과정에서 **사용되지 않는 데드(dead) 코드**를 감지해 결과 번들에서 제거합니다.
- 목적은 **번들 용량 감소 → 로딩/파싱 비용 절감 → 성능 개선**입니다.

## 동작 조건

### ES 모듈(ESM) 사용

- 트리 쉐이킹이 잘 작동하려면 **ES 모듈을 사용**해야 합니다.
- ESM은 **정적인 import/export 구조**를 가지므로, 빌드 타임에 “무엇이 사용/미사용인지” 판단하기 쉽습니다.
- 반면 **CommonJS처럼 동적인 모듈 시스템**은 정적 분석이 어려워 트리 쉐이킹이 제대로 동작하지 않을 수 있습니다.

### 사이드 이펙트(side effects) 최소화

- 트리 쉐이킹을 적용하려면 코드가 **사이드 이펙트가 없는 순수 함수 형태**로 작성되어야 합니다.
- 모듈 내부 코드가 전역 상태를 변경하거나 예측 불가능한 동작을 하면, 번들러는 해당 코드 제거가 안전한지 확신할 수 없어 트리 쉐이킹이 제한될 수 있습니다.
- 이를 보완하기 위해, 패키지에서 사이드 이펙트가 없음을 명시하는 설정을 사용할 수 있습니다.

```json
// package.json
{
  "sideEffects": false
}
```

## 트리 쉐이킹 적용 방법

webpack, Rollup, esbuild 등의 번들러가 트리 쉐이킹 기능을 지원합니다. 설정 방식은 번들러마다 다를 수 있습니다.

### Webpack 예시

- 최신 webpack은 보통 `mode: 'production'`만으로도 기본 트리 쉐이킹이 활성화됩니다.
- `TerserPlugin` 같은 플러그인을 추가하면 더 강력한 최적화를 적용할 수 있습니다.

```js
// webpack.config.js (예시)
module.exports = {
  mode: "production",
  // 필요 시 minimizer 등에 TerserPlugin을 추가해 최적화를 강화할 수 있습니다.
};
```

## 트리 쉐이킹 효과 측정 방법

트리 쉐이킹 적용 여부를 확인하는 가장 일반적인 방법은 **번들 크기 비교**입니다.

- **번들 분석 도구 사용**
  - webpack의 `bundle analyzer` 같은 도구를 사용하면 번들 이후 코드 양을 시각적으로 확인할 수 있습니다.
- **번들 산출물 직접 비교**
  - 번들 파일을 직접 확인하여 원본 코드 대비 어떤 코드가 제거되었는지 비교합니다.

## 장점

- **번들 크기 감소**: 미사용 코드 제거로 다운로드/캐시 부담 감소
- **성능 개선 가능**: 로딩, 파싱, 실행 비용 감소(특히 클라이언트 환경)

## 단점

- **전제 조건이 있음**: ESM 기반 정적 분석이 전제(환경/라이브러리에 따라 제한)
- **사이드 이펙트에 민감**: “안전하게 제거 가능한 코드” 판단이 어려우면 최적화가 제한될 수 있음
- **모듈/빌드 설정 영향**: 번들러/설정/라이브러리 배포 형태에 따라 결과가 달라질 수 있음

## 주의사항 및 실무 팁

- **ESM 우선 사용**: 가능하면 `import/export` 기반으로 구성합니다.
- **사이드 이펙트 관리**
  - 모듈 최상단에서 전역 상태 변경/즉시 실행 로직을 남발하지 않도록 주의합니다.
  - 필요하다면 `package.json`의 `sideEffects` 설정을 통해 번들러가 최적화 판단을 더 적극적으로 할 수 있게 합니다.
- **측정 기반으로 검증**
  - 트리 쉐이킹은 “켰다”로 끝나지 않고, 번들 분석/비교로 실제 효과를 확인하는 것이 중요합니다.
