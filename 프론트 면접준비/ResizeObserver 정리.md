# ResizeObserver 정리

## 핵심 요약

- `ResizeObserver`는 특정 요소의 크기 변화를 감지할 수 있게 해 주는 브라우저 API다.
- 윈도우 전체 크기만 보는 `window.resize`와 달리, 개별 컴포넌트 자체의 크기 변화를 관찰할 수 있다는 점이 핵심이다.
- 반응형 UI, 차트, 가변 레이아웃, 텍스트 영역 자동 조절 같은 상황에서 유용하다.
- 면접에서는 "왜 `window.resize`만으로 부족한지", "언제 필요한지", "관찰 해제를 왜 해야 하는지"를 같이 말하면 좋다.

## ResizeObserver란 무엇인가

프론트엔드에서는 화면 크기에 따라 UI를 바꾸는 일이 많다.  
그런데 실제로 바뀌는 건 브라우저 창 크기만이 아니다.

- 사이드바가 열리거나 닫힐 수 있다.
- 부모 레이아웃이 변경될 수 있다.
- 비동기 데이터 로딩 뒤 카드 크기가 달라질 수 있다.

이때 특정 DOM 요소의 크기 변화를 직접 감지하고 싶을 때 쓰는 API가 `ResizeObserver`다.

쉽게 말하면 "창이 아니라 요소를 관찰하는 resize 감지기"라고 설명할 수 있다.

## 왜 필요한가

예전에는 보통 `window.addEventListener("resize", ...)`로 대응했다.  
하지만 이 방식은 창 크기 변화만 감지할 수 있고, 컴포넌트 자체 크기가 다른 이유로 바뀌는 경우는 놓치기 쉽다.

예를 들면 이런 경우다.

- 부모 컨테이너 폭이 레이아웃 변경으로 줄어든 경우
- 아코디언이 열리면서 내부 콘텐츠 높이가 바뀐 경우
- 차트 영역이 탭 전환 후 다시 계산돼야 하는 경우

즉 "요소 중심"으로 반응해야 하는 UI에서는 `ResizeObserver`가 더 정확하다.

## 기본 사용 방식

흐름은 단순하다.

1. 관찰할 요소를 준비한다.
2. `ResizeObserver`를 만든다.
3. 크기가 바뀌면 콜백에서 필요한 작업을 한다.
4. 컴포넌트가 사라질 때 `disconnect`로 정리한다.

예시는 이런 식이다.

```js
const box = document.querySelector(".box");

const observer = new ResizeObserver((entries) => {
  for (const entry of entries) {
    console.log(entry.contentRect.width, entry.contentRect.height);
  }
});

observer.observe(box);
```

React에서는 보통 `ref`와 함께 쓰고, 언마운트 시 정리해 주는 패턴으로 많이 구현한다.

## 어디에 쓰면 좋은가

- 반응형 카드나 패널 레이아웃
- 차트 라이브러리 리사이즈 처리
- 텍스트 영역이나 에디터 높이 계산
- 컨테이너 폭에 따라 UI 모양이 바뀌는 컴포넌트

특히 재사용 컴포넌트를 만들 때 유용하다.  
화면 전체가 아니라 "이 컴포넌트가 놓인 자리" 기준으로 동작을 바꿀 수 있기 때문이다.

## 주의할 점

`ResizeObserver`도 무조건 남발하면 좋지 않다.

- 관찰 대상이 너무 많으면 성능 부담이 생길 수 있다.
- 콜백 안에서 다시 레이아웃을 크게 바꾸면 연쇄 업데이트가 생길 수 있다.
- 정리하지 않으면 메모리 누수나 예기치 않은 동작이 생길 수 있다.

또한 크기 변화에 따라 곧바로 상태를 많이 바꾸면 렌더링 횟수가 늘 수 있으니, 실제로 필요한 최소 정보만 반응시키는 편이 좋다.

## `window.resize`와의 차이

이 비교는 면접에서 자주 유용하다.

`window.resize`는 브라우저 창 크기 변화를 감지한다.  
반면 `ResizeObserver`는 특정 요소의 실제 크기 변화를 감지한다.

그래서 전체 화면 반응형 처리에는 `window.resize`가 맞을 수 있고, 컴포넌트 단위 레이아웃 대응에는 `ResizeObserver`가 더 적합하다.

## 면접 답변 예시

> ResizeObserver는 특정 DOM 요소의 크기 변화를 관찰하는 브라우저 API입니다. 기존의 window resize 이벤트는 브라우저 창 크기 변화만 감지할 수 있어서, 부모 레이아웃 변경이나 비동기 콘텐츠 로딩처럼 요소 크기만 달라지는 상황을 정확히 잡기 어렵습니다. ResizeObserver를 쓰면 차트, 패널, 에디터 같은 컴포넌트가 실제로 차지하는 크기에 맞춰 동작을 바꿀 수 있습니다. 다만 관찰 대상을 과하게 늘리면 성능 부담이 생길 수 있고, 컴포넌트 언마운트 시 disconnect로 정리해 주는 습관이 필요합니다.

## 장점

- 요소 단위 반응형 처리에 정확하다.
- 재사용 컴포넌트 설계에 잘 맞는다.
- 레이아웃 변화 대응을 더 세밀하게 할 수 있다.

## 단점

- 관찰 대상이 많아지면 부담이 생길 수 있다.
- 콜백 설계를 잘못하면 렌더링이 잦아질 수 있다.
- 정리 코드를 빼먹으면 관리가 지저분해진다.

## 주의사항 / 실무 팁

- `window.resize`로 충분한 문제인지 먼저 구분하고 도입하는 편이 좋다.
- 콜백에서 상태를 바꿀 때는 꼭 필요한 값만 반영해 렌더링을 줄이는 게 좋다.
- 컴포넌트 언마운트 시 `disconnect` 또는 대상 해제를 꼭 해 둬야 한다.
- 면접에서는 `창 크기 감지`와 `요소 크기 감지`를 구분해서 설명하면 이해도가 높아 보인다.
