# HTTP의 ETag 정리

## 핵심 요약

- `ETag(Entity Tag)`는 웹 리소스의 **특정 버전**을 식별하는 **고유 식별자 헤더**입니다.
- 클라이언트는 `If-None-Match`로 조건부 요청을 보내고, 변경이 없으면 서버는 `304 Not Modified`로 응답해 **재다운로드를 방지**합니다.
- `Last-Modified`와 유사하지만 **시간 기반이 아닌 콘텐츠 기반**으로 동작하며, `Strong/Weak ETag`로 나뉩니다.

## ETag(Entity Tag)란?

`ETag(Entity Tag)`는 HTTP 프로토콜에서 사용되는 헤더로, 웹 리소스의 특정 버전을 식별하는 고유한 식별자입니다.

서버가 클라이언트에게 리소스를 전송할 때 `ETag` 헤더를 포함시키고, 클라이언트는 이후 요청 시 `If-None-Match` 헤더에 이 값을 포함하여 조건부 요청을 보낼 수 있습니다.

## ETag의 목적과 동작 방식

### 캐싱 효율성 향상

`ETag`의 주요 목적은 캐싱 효율성을 높이는 것입니다. 클라이언트가 이미 가지고 있는 리소스와 서버의 현재 버전이 동일한지 확인할 수 있어, 변경되지 않았다면 전체 콘텐츠를 다시 다운로드하지 않고 `304 Not Modified` 응답을 받습니다. 이를 통해 네트워크 트래픽과 서버 부하를 줄일 수 있습니다.

### `Last-Modified`와의 차이

`Last-Modified` 헤더와 비슷한 역할을 하지만, 타임스탬프 기반이 아닌 콘텐츠 기반으로 동작합니다. 그렇기 때문에 `ETag`는 1초 내에 여러 변경이 있거나 시간 기반 비교가 적합하지 않은 상황에서 더 정확한 캐싱을 제공합니다.

## ETag 값과 종류

### 생성 방식

`ETag` 값은 보통 리소스의 내용에 기반한 해시값이나 버전 번호로 생성됩니다.

### Strong ETag vs Weak ETag

`ETag`는 `Strong ETag`와 `Weak ETag(W/ 접두사 사용)` 두 종류로 나뉩니다.

- `Strong ETag`: 바이트 단위까지 정확히 일치해야 합니다.
- `Weak ETag`: 의미적으로 동등하면 일치하는 것으로 간주합니다.

## ETag를 `Cache-Control` 헤더와 함께 사용하기도 하나요?

`ETag`와 `Cache-Control`은 서로 보완적인 관계에 있으며, 함께 사용하면 더 효과적인 캐싱 전략을 구현할 수 있습니다.

`Cache-Control`은 리소스의 캐싱 정책을 직접적으로 지정하는 헤더로, 캐시 가능 여부, 유효 기간, 재검증 요구사항 등을 설정합니다. 반면 `ETag`는 리소스의 변경 여부를 확인하는 메커니즘을 제공합니다.

### 조합 예시 1: 정적 리소스(이미지, CSS, JS 등)

정적 리소스의 경우 아래처럼 설정하면 1년간 클라이언트 캐시를 사용하되, 리소스 갱신 시(ex. 배포) `ETag`가 변경되어 새 버전을 받아갑니다. 파일명에 해시를 포함시키는 방식으로도 구현할 수 있습니다.

```http
Cache-Control: max-age=31536000
ETag: "abc123"
```

### 조합 예시 2: 자주 변경되는 API 응답

자주 변경되는 API 응답의 경우 아래처럼 `no-cache` 설정으로 매번 서버에 재검증을 요청하게 하되, `ETag`를 통해 콘텐츠가 변경되지 않았다면 `304` 응답으로 데이터 전송을 줄일 수 있습니다.

```http
Cache-Control: no-cache
ETag: "xyz789"
```

## 장점

- 변경되지 않은 리소스에 대해 `304 Not Modified`를 활용해 **불필요한 다운로드를 줄일 수 있습니다.**
- 네트워크 트래픽과 서버 부하를 줄여 **캐싱 효율을 높일 수 있습니다.**
- 시간 기반 비교가 애매한 상황(예: 1초 내 다중 변경)에서 **더 정확한 검증**이 가능합니다.

## 단점

- 원문에서는 별도의 단점이 직접적으로 언급되지는 않았습니다.

## 주의사항 / 실무 팁

- `ETag`는 “캐시 저장 정책”을 정하는 `Cache-Control`과 역할이 다르므로, 보통 **함께 설계**하는 편이 좋습니다.
- 자주 바뀌는 응답은 `Cache-Control: no-cache`로 **항상 재검증**하게 하고, 변경이 없을 때는 `ETag`로 `304`를 받아 **전송량을 줄이는 전략**을 고려할 수 있습니다.
- 정적 자원은 긴 `max-age`와 함께 배포 시 `ETag`가 바뀌거나(또는 파일명 해시 변경) 새 버전으로 교체되는 흐름을 만들면 운영이 수월합니다.
