# React Compiler 정리

## 핵심 요약

- `React Compiler`는 React 코드의 메모이제이션 가능성을 컴파일 단계에서 분석해 자동 최적화를 돕는 도구다.
- 개발자가 모든 곳에 `useMemo`, `useCallback`, `React.memo`를 직접 덕지덕지 붙이지 않아도 되게 하려는 방향과 연결된다.
- 목적은 성능 최적화 코드의 수작업 부담을 줄이고, 렌더링 비용을 더 일관되게 관리하는 데 있다.
- 면접에서는 "React의 자동 최적화 방향"이라고 설명하고, 기존 수동 메모이제이션과 어떻게 관계가 바뀌는지를 말하면 좋다.

## React Compiler란?

기존 React에서는 불필요한 재렌더링을 줄이기 위해 개발자가 직접 최적화 포인트를 찾아야 하는 경우가 많았다.  
대표적으로 `React.memo`, `useMemo`, `useCallback` 같은 도구를 써서 "이 값은 다시 계산하지 말자", "이 함수 참조는 유지하자"를 손으로 표현해 왔다.

`React Compiler`는 이런 최적화 중 일부를 컴파일러가 코드 분석을 통해 자동으로 처리하려는 시도다.  
쉽게 말하면 "이 컴포넌트는 다시 그릴 필요가 없는 상황"을 사람이 아니라 도구가 더 많이 판단하게 만드는 흐름이다.

## 왜 중요한가?

React 최적화는 늘 비용이 있었다.

- 최적화가 필요한 지점을 개발자가 직접 찾아야 한다.
- 과한 메모이제이션은 코드 가독성을 해칠 수 있다.
- 잘못된 의존성 배열이나 불필요한 `useCallback`이 오히려 유지보수를 어렵게 만든다.

`React Compiler`는 이런 문제를 줄이면서, 기본 코드 스타일은 더 단순하게 유지하려는 방향과 맞닿아 있다.

## 기존 방식과 차이

기존 방식은 "개발자가 최적화를 선언하는 구조"에 가깝다.  
반면 `React Compiler`는 "컴파일러가 안전한 범위에서 최적화를 추론하는 구조"에 가깝다.

즉 핵심 차이는 다음과 같다.

- 예전: 사람이 최적화 포인트를 직접 지정
- 이후: 컴파일러가 자동 최적화 가능한 부분을 더 많이 처리

물론 이 말이 "이제 수동 최적화가 완전히 필요 없다"는 뜻은 아니다.  
특수한 성능 병목이나 외부 라이브러리 경계에서는 여전히 명시적 제어가 필요할 수 있다.

## 면접에서 자주 엮이는 포인트

이 주제는 보통 아래 질문들과 같이 묶인다.

- `useMemo`, `useCallback`은 언제 필요한가?
- `React.memo`는 어떤 상황에서 쓰는가?
- 과도한 메모이제이션이 왜 문제인가?

이때 답변 방향은 단순하다.  
React Compiler가 등장하면서 "기본은 단순하게 작성하고, 정말 필요한 최적화만 남기자"는 흐름이 더 강해졌다고 설명하면 된다.

## 예시로 이해하기

예전에는 부모가 렌더링될 때 자식에게 넘기는 함수 참조가 바뀌는 걸 막기 위해 아래처럼 많이 작성했다.

```jsx
const handleSubmit = useCallback(() => {
  submitForm(formData);
}, [formData]);
```

이런 코드는 분명 필요할 때가 있지만, 모든 곳에 습관적으로 넣으면 코드가 지저분해진다.  
React Compiler의 방향은 이런 반복적인 패턴 일부를 자동화해 개발자가 비즈니스 로직에 더 집중하게 만드는 쪽이다.

## 면접에서 어떻게 말하면 좋을까?

React Compiler는 React가 성능 최적화를 더 자동화하려는 흐름이라고 짧게 정의하면 된다.  
그 다음 "예전에는 `useMemo`와 `useCallback`을 많이 손으로 넣었는데, 앞으로는 컴파일러가 안전한 부분을 더 맡게 된다"라고 이어가면 이해가 쉽다.

## 면접 답변 예시

> React Compiler는 React 코드의 렌더링 특성을 컴파일 단계에서 분석해서 자동 메모이제이션 같은 최적화를 더 많이 수행하려는 도구입니다. 기존에는 `useMemo`, `useCallback`, `React.memo`를 개발자가 직접 판단해서 붙이는 경우가 많았는데, 이 과정은 실수도 많고 코드도 복잡해질 수 있습니다. React Compiler는 이런 수작업 부담을 줄이는 방향으로 이해하면 되고, 그렇다고 모든 수동 최적화가 사라지는 건 아니라서 실제 병목 구간은 여전히 프로파일링으로 확인해야 합니다.

## 장점

- 불필요한 수동 메모이제이션을 줄여 코드가 단순해질 수 있다.
- 성능 최적화 품질을 더 일관되게 가져갈 가능성이 있다.
- 개발자가 의존성 배열과 참조 안정성에 과몰입하지 않게 도와준다.

## 단점

- 컴파일러가 무엇을 어떻게 최적화하는지 이해하지 못하면 디버깅이 어렵다.
- 기존 코드와 팀 습관이 많이 수동 최적화 중심이면 전환 비용이 생긴다.
- 모든 병목을 자동으로 해결해 주는 만능 도구처럼 오해하기 쉽다.

## 주의사항 / 실무 팁

- 자동 최적화가 들어와도 실제 성능 병목은 측정 기반으로 봐야 한다.
- `useMemo`, `useCallback`을 무조건 제거하거나 무조건 유지하는 식보다 팀의 컴파일러 사용 기준을 정하는 편이 낫다.
- 면접에서는 "React가 개발자 수작업 최적화 부담을 줄이는 방향으로 간다"는 흐름을 중심으로 설명하면 말이 자연스럽다.
