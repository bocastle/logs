# React key와 리스트 렌더링 정리

## 핵심 요약

- `key`는 React가 리스트의 각 요소를 구분하기 위해 쓰는 식별자다.
- `key`가 안정적이면 필요한 부분만 효율적으로 업데이트할 수 있다.
- `index`를 `key`로 무조건 쓰면 정렬, 삽입, 삭제 상황에서 예상치 못한 버그가 생길 수 있다.
- 면접에서는 "왜 필요한가", "잘못 쓰면 무슨 문제가 생기나", "실무에서는 뭘 쓰나"까지 이어서 말하면 좋다.

## key가 왜 필요한가

React는 상태가 바뀌면 이전 렌더 결과와 새로운 렌더 결과를 비교해서 어떤 부분만 바꿀지 판단한다. 리스트에서는 항목이 여러 개라서, 무엇이 유지되고 무엇이 새로 생겼는지 구분할 기준이 필요하다. 그 역할을 하는 게 `key`다.

즉, `key`는 단순 경고 제거용 옵션이 아니라 React가 변경 대상을 추적하는 기준이다.

## key가 없으면 어떤 일이 생기나

`key`가 없거나 적절하지 않으면 React는 항목을 정확히 식별하기 어렵다. 그러면 기존 DOM을 불필요하게 재사용하거나, 반대로 필요 없는 재생성을 하게 된다.

그 결과로 아래 같은 문제가 생길 수 있다.

- 입력창 포커스가 이상하게 이동함
- 체크박스 선택 상태가 다른 행으로 옮겨감
- 애니메이션이 어색하게 동작함
- 성능 최적화 효과가 줄어듦

## 좋은 key의 조건

좋은 `key`는 아래 조건을 만족하는 편이 좋다.

- 항목마다 고유해야 한다
- 렌더링이 바뀌어도 가능하면 안정적이어야 한다
- 데이터 자체를 대표하는 값이어야 한다

보통은 DB의 `id`, UUID, 서버에서 내려주는 고유 식별자를 쓴다.

```jsx
{todos.map((todo) => (
  <TodoItem key={todo.id} todo={todo} />
))}
```

## index를 key로 쓰면 왜 위험한가

배열 `index`는 화면 순서가 바뀌면 같이 바뀐다. 그래서 정렬, 필터링, 중간 삽입, 삭제가 있는 리스트에서는 안정적인 식별자가 아니다.

예를 들어 첫 번째 항목을 삭제했는데 `index`를 `key`로 썼다면, 뒤에 있던 항목들이 전부 한 칸씩 앞으로 당겨진다. React 입장에서는 "기존 항목이 이동했다"가 아니라 "각 항목이 다른 항목으로 바뀌었다"처럼 해석될 수 있다.

이때 각 행에 입력 상태가 있었다면 값이 다른 항목에 붙어서 보일 수도 있다.

## 그럼 index는 절대 쓰면 안 되나

절대 금지는 아니다. 아래 조건을 모두 만족하면 비교적 안전하다.

- 리스트 순서가 절대 바뀌지 않음
- 삽입, 삭제가 없음
- 각 항목에 로컬 상태가 사실상 없음

예를 들어 정적인 약관 목록처럼 변하지 않는 단순 텍스트 렌더링이라면 큰 문제가 없을 수 있다. 그래도 가능하면 데이터 기반 식별자를 우선 쓰는 쪽이 낫다.

## 실무에서 자주 나오는 상황

### 1. 서버 데이터 목록

가장 무난한 선택은 서버에서 받은 `id`를 그대로 쓰는 것이다.

### 2. 클라이언트에서 임시 생성한 항목

아직 서버 저장 전이라도 UUID 같은 임시 식별자를 만들어서 유지하는 편이 낫다.

### 3. 정렬과 필터가 많은 UI

테이블, 칸반 보드, 드래그 앤 드롭 리스트는 `key`가 특히 중요하다. 이 영역에서 잘못된 `key`는 UI 버그로 바로 이어진다.

## 면접에서 자주 꼬리질문 나오는 포인트

### key는 props로 자식에게 전달되나

아니다. `key`는 React 내부 비교용 값이라서 일반 props처럼 자식 컴포넌트에서 바로 읽을 수 없다. 자식에서 필요하면 별도 prop으로 넘겨야 한다.

### key가 같으면 무조건 재렌더링이 안 되나

그것도 아니다. `key`는 항목 식별용이고, 렌더 여부는 props나 state 변화와도 관련이 있다. `key`가 같다고 렌더링 자체가 완전히 막히는 건 아니다.

## 면접 답변 예시

React에서 `key`는 리스트 항목을 구분하기 위한 식별자입니다. React는 이전 렌더 결과와 다음 결과를 비교할 때 `key`를 기준으로 어떤 항목이 유지되고, 추가되고, 삭제됐는지 판단합니다. 그래서 안정적인 `key`를 쓰면 필요한 부분만 효율적으로 업데이트할 수 있습니다. 반대로 `index`를 `key`로 쓰면 정렬이나 삭제가 일어나는 리스트에서 항목 상태가 엉뚱하게 섞일 수 있어서 보통은 데이터의 고유 ID를 사용하는 편이 안전합니다.

## 장점

- React 렌더링 원리와 연결해서 설명하기 좋아 면접 답변이 탄탄해진다.
- 단순 경고 해결이 아니라 상태 보존, 성능, 버그 예방과 연결할 수 있다.
- 테이블, 폼 리스트, 드래그 앤 드롭 같은 실무 예시를 붙이기 쉽다.

## 단점

- `key`만 맞는다고 성능 문제가 모두 해결되는 것은 아니다.
- 데이터에 고유 식별자가 없으면 임시 전략을 따로 설계해야 한다.
- `index` 사용 여부는 상황 의존적인데, 이를 지나치게 절대 규칙처럼 말하면 답변이 거칠어질 수 있다.

## 주의사항 / 실무 팁

- 가능하면 서버 데이터의 고유 ID를 그대로 쓰는 습관이 가장 안전하다.
- 정렬, 필터, 삽입, 삭제가 있는 리스트에서는 `index key`를 피하는 편이 좋다.
- `Math.random()`처럼 렌더마다 바뀌는 값은 더 나쁘다. 항목이 매번 새 것으로 인식될 수 있다.
- `key`는 형제 요소 사이에서만 고유하면 되지만, 실무에서는 의미 있는 식별자를 쓰는 편이 유지보수에도 좋다.
