# 웹뷰(WebView) 정리

## 핵심 요약

- **웹뷰(WebView)**는 **모바일 앱 내부에 포함된 웹 브라우저 컴포넌트**로, 앱 안에서 웹 페이지(HTML/CSS/JS)를 로드해 **앱의 일부처럼 동작**하게 합니다.
- **하이브리드 앱**에서 많이 쓰이며, **빠른 업데이트(스토어 심사 없이 배포)**가 가능하다는 장점이 있습니다.
- 반면 **성능/보안/네이티브 기능 제약**을 고려해야 합니다.

## 웹뷰(WebView)란 무엇인가요?

- 웹뷰는 **네이티브 앱 내부에서 웹 페이지를 불러와 실행**할 수 있게 해주는 구성 요소입니다.
- 웹 브라우저 엔진을 활용해 **HTML, CSS, JavaScript로 작성된 웹 페이지**를 앱 안에서 실행할 수 있습니다.
- 하나의 웹 코드 기반으로 여러 플랫폼에서 **동일한 UI/기능을 재사용**하기 쉬워, 특정 화면/기능을 웹으로 만들고 앱에서 웹뷰로 띄우는 형태로 활용됩니다.

### 플랫폼별 구현 컴포넌트

```text
Android: WebView
iOS: WKWebView
```

## 장점

### 1) 코드 재사용과 빠른 개발

- 웹으로 만든 화면을 앱에서 웹뷰로 띄우면 **한 번 개발한 웹 코드를 여러 플랫폼에서 재사용**할 수 있습니다.

### 2) 업데이트가 자유로움

- 네이티브 앱은 보통 **스토어 심사 후 업데이트 배포**가 필요합니다.
- 웹뷰 기반 화면은 **심사 없이도 빠르게 변경 사항을 배포**할 수 있어, 업데이트 빈도가 잦거나 신속한 수정이 필요한 경우 유용합니다.

## 단점

### 1) 성능 이슈

- 네이티브에 비해 성능이 떨어질 수 있습니다.
- 특히 **복잡한 애니메이션**이나 **높은 연산량**이 필요한 경우, 웹뷰보다 네이티브 구현이 더 적합할 수 있습니다.

### 2) 보안 고려 필요

- 웹뷰로 외부 웹사이트를 로드하면 **악성 코드/피싱 공격**에 취약해질 수 있습니다.
- 사용자 입력 처리 등에서 보안을 위해 **추가적인 노력**이 필요합니다.

### 3) 네이티브 기능 제약

- 웹뷰는 기본적으로 웹 콘텐츠를 렌더링하는 컴포넌트라 **네이티브 기능 활용에 제한**이 있습니다.
- 브릿지를 통해 일부 기능을 사용할 수 있지만 **네이티브와 동일한 수준으로 사용하기는 어렵습니다.**
- 예시 제약:
  - 파일 업로드/다운로드 제약
  - GPS/카메라/마이크 등 하드웨어를 **네이티브 수준으로 제어하기 어려움**

## 웹뷰와 PWA(Progressive Web App)의 차이점

- **PWA**는 웹 기술로 만들지만 네이티브 앱과 유사한 기능을 제공하는 **웹 애플리케이션**입니다.
- **웹뷰**는 네이티브 앱 안에서 웹을 실행하는 **하이브리드 접근**입니다.
- 핵심 차이:
  - 웹뷰: **네이티브 앱에 종속**
  - PWA: **독립 실행**, **스토어 등록 없이 배포 가능**

## 주의사항 / 실무 팁

- 성능이 중요한 화면(복잡한 애니메이션/고연산)은 **네이티브 구현**을 우선 검토합니다.
- 외부 페이지를 웹뷰로 로드한다면:
  - 신뢰할 수 있는 도메인만 허용(화이트리스트 등)
  - 입력/인증/결제 흐름은 **피싱·스크립트 인젝션** 등의 위협을 고려해 설계
- 네이티브 기능이 필요한 경우:
  - 웹-네이티브 **브릿지 설계**가 필수이며, 기능/권한/에러 처리 정책을 미리 정합니다.
  - 파일/하드웨어 연동 요구사항이 크면 **웹뷰 단독으로 해결하려고 고집하지 않기**(네이티브 전환 비용까지 포함해 판단)
