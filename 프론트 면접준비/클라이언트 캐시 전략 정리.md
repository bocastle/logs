# 클라이언트 캐시 전략 정리

## 핵심 요약

- 클라이언트 캐시는 `속도`, `비용`, `신선도`의 균형이다.
- 캐시 정책은 `stale-while-revalidate`, `stale-if-error`, `max-age`, `no-cache` 같은 TTL/검증 규칙으로 구성된다.
- 면접에서는 "어떤 화면에서 어떤 전략을 쓰는지"를 사례로 설명해야 한다.
- 캐시 실패는 단순 성능 이슈가 아니라 데이터 정합성 이슈로 이어질 수 있다.

## 개념 설명

### 1) 캐시 계층과 책임 분리

클라이언트 캐시는 보통 아래처럼 나눈다.

- 인메모리: 렌더 직전 빠른 재사용
- 브라우저 캐시: 정적 리소스 캐시
- 서비스워커 캐시: 오프라인/준오프라인 패턴

### 2) 유효성(Validity) 전략

- `Cache-Control: max-age`: 캐시가 유효한 시간
- `stale-while-revalidate`: 만료 임박 후 즉시 데이터 표시 + 백그라운드 갱신
- `stale-if-error`: 갱신 실패 시 이전 데이터라도 먼저 보존

### 3) 무효화(Invalidation)

- 시간 기반 TTL
- ETag/Last-Modified 기반 조건부 요청
- 이벤트 기반 무효화(저장, 삭제, 로그아웃 등)

## 사용 예시

- 검색 자동완성: 입력 직후 빠른 결과 표시 + 짧은 TTL
- 사용자 프로필: 로그인 직후 캐시, 5~10분 TTL + 수동 갱신 버튼
- 뉴스 피드: SWR + stale-if-error로 네트워크 불안정 대응

```ts
// 개념 예시 (의사 코드)
if (cache.exists(key) && !cache.isExpired(key)) {
  return cache.get(key)
}

const data = await fetchWithEtag(key)
cache.set(key, data, ttl)
return data
```

## 면접 답변 예시

"캐시는 사용자 체감 성능을 크게 개선하는 반면, stale 데이터 노출 리스크도 함께 관리해야 합니다. 그래서 일반 조회는 `stale-while-revalidate`로 즉시성은 살리고 정합성은 백그라운드 갱신으로 보완합니다. 또한 중요한 거래 데이터는 캐시 TTL을 짧게 두고, 이벤트 기반 무효화를 병행해 데이터 불일치 창을 줄입니다."

## 장점

- 최초 화면 응답 시간이 빨라짐
- 네트워크 트래픽 감소
- 간헐적 네트워크 장애 대응력 향상

## 단점

- 오래된 데이터 노출 가능
- 동시 갱신 시 레이스 컨디션 발생 가능성
- 캐시 정책이 복잡해질수록 디버깅 난이도 상승

## 주의사항 / 실무 팁

- `캐시 히트율`만 보지 말고 `오류율`도 같이 본다.
- 사용자 권한/토큰 기반 데이터는 캐시 스코프를 엄격히 나눈다.
- 로그아웃 시 모든 민감 캐시를 즉시 비운다.
