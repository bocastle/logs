# Cache-Control 헤더 정리

## 핵심 요약

- **Cache-Control**은 **브라우저(클라이언트)**와 **중간 서버(프록시, CDN 등)**가 **응답을 얼마나 오래/어떤 방식으로 캐싱할지**를 지정하는 HTTP 헤더입니다.
- 캐싱을 잘 활용하면 **불필요한 네트워크 요청을 줄이고** 웹 성능(로딩 속도, 서버 부하, 대역폭)을 개선할 수 있습니다.
- 반대로 캐싱 정책을 잘못 잡으면 **오래된 데이터가 노출**되거나 **불필요한 재요청이 증가**할 수 있어, **리소스 특성별 전략**이 중요합니다.

## Cache-Control이란?

Cache-Control은 클라이언트(브라우저)와 중간 서버(proxy, CDN 등)가 어떤 방식으로 캐싱할지를 지정하는 헤더입니다.  
즉, **“특정 응답을 얼마나 오래, 어떻게 캐싱할 것인가”**를 결정합니다.

Cache-Control의 핵심은 **특정 리소스를 얼마나 오래 저장할 수 있는지를 서버가 결정**하는 것입니다.  
웹 페이지를 방문할 때 브라우저는 서버에서 HTML, CSS, JavaScript, 이미지 등의 리소스를 다운로드하는데, 이 리소스를 매번 다시 다운로드하지 않고 저장된 데이터를 활용하면 성능이 향상될 수 있습니다.

하지만 모든 리소스를 무조건 캐시에 저장하면 **최신 데이터가 반영되지 않는 문제**가 발생할 수 있으므로, **리소스 유형별로 적절한 캐싱 정책**을 설정해야 합니다.

## 예시: 리소스 유형별 캐싱 정책

### 1) 변경이 거의 없는 정적 리소스 (CSS/JS/이미지 등)

자주 변경되지 않는 정적 리소스는 장기간 캐시에 저장해 불필요한 다운로드를 방지하는 것이 좋습니다.

```http
Cache-Control: public, max-age=31536000, immutable
```

### 2) 변경 가능성이 높은 HTML 문서

변경 가능성이 높은 HTML 문서는 항상 최신 상태를 유지할 수 있도록 **매번 서버 검증을 요구**하는 방식으로 관리할 수 있습니다.

```http
Cache-Control: no-cache, must-revalidate
```

## 장점

## 캐싱의 장점에 대해서 설명해주세요.

캐싱을 잘 활용하면 다음 효과를 얻을 수 있습니다.

- 사용자의 **대역폭 소비 감소**
- 그에 따른 **서버 부하 감소**
- **페이지 로딩 속도 개선**

특히 네트워크 환경이 열악한 **모바일 환경**에서는 캐싱이 더욱 중요한 역할을 합니다.

## 단점

- 캐싱 전략을 잘못 설정하면 **오래된 데이터를 계속 제공**할 수 있습니다.
- 반대로 너무 보수적으로 설정하면 **불필요한 네트워크 요청이 증가**해 성능이 떨어질 수 있습니다.

## 주의사항 / 실무 팁

- **리소스 특성(변경 빈도, 중요도, 민감도)**에 맞춰 캐싱 정책을 분리하세요.
- 정적 리소스는 **장기 캐싱 + 불변(immutable) 전략**이 효과적이며, 변경이 필요할 때는 **파일명 해시(버전) 기반 배포**와 함께 사용하는 것이 일반적입니다.
- HTML은 사용자에게 최신 화면을 제공해야 하는 경우가 많으므로, **항상 검증(no-cache, must-revalidate)** 쪽이 유리한 상황이 많습니다.
- 캐싱 정책을 잘못 설정하면 **“오래된 화면/자산이 남는 이슈”**가 발생하기 쉽기 때문에, 배포/캐시 무효화 전략을 함께 설계하는 것이 안전합니다.
