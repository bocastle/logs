# 리플로우와 리페인트 정리

## 핵심 요약

- 리플로우는 요소의 크기나 위치가 바뀌어서 레이아웃을 다시 계산하는 과정이다.
- 리페인트는 레이아웃은 그대로 두고 색상, 그림자처럼 화면에 다시 그리기만 하는 과정이다.
- 보통 리플로우가 리페인트보다 비용이 더 크다. 레이아웃 계산이 먼저 일어나고, 그 다음 다시 그려야 하기 때문이다.
- 면접에서는 "어떤 변경이 레이아웃을 흔드는지"와 "실무에서 어떻게 줄이는지"를 같이 말하면 답변이 좋아진다.

## 리플로우와 리페인트가 왜 중요한가

브라우저는 HTML과 CSS를 받아서 화면을 만든다. 이때 한 번 그렸다고 끝이 아니라, 자바스크립트나 스타일 변경 때문에 화면을 다시 계산해야 하는 순간이 계속 생긴다.

문제는 이 재계산이 잦아지면 스크롤이 끊기거나, 입력이 버벅이거나, 애니메이션이 어색해진다는 점이다. 그래서 프론트엔드 면접에서는 렌더링 성능 이야기로 자연스럽게 자주 나온다.

## 리플로우란

리플로우는 요소의 크기, 위치, 배치가 바뀌어서 브라우저가 레이아웃을 다시 계산하는 과정이다.

대표적인 예시는 이런 경우다.

- 요소의 `width`, `height`, `margin`, `padding` 변경
- 글자 수가 바뀌어서 박스 크기가 달라지는 경우
- DOM 노드 추가, 삭제
- 브라우저 창 크기 변경

예를 들어 어떤 카드의 너비를 `200px`에서 `300px`로 바꾸면, 그 카드 하나만 바뀌는 게 아니라 주변 요소 배치까지 다시 계산해야 할 수 있다. 그래서 비용이 커진다.

## 리페인트란

리페인트는 레이아웃 변화 없이 다시 그리기만 하는 과정이다.

대표적으로 아래 같은 변경이 해당된다.

- `color`
- `background`
- `visibility`
- `box-shadow`

예를 들어 버튼의 배경색을 파란색에서 초록색으로 바꾸는 건 보통 리페인트에 가깝다. 위치나 크기는 안 바뀌기 때문이다.

## 둘의 관계

리플로우가 일어나면 대개 그 결과를 화면에 다시 그려야 하므로 리페인트도 따라온다. 반대로 리페인트는 반드시 리플로우를 동반하지는 않는다.

면접에서는 이 정도로 정리하면 깔끔하다.

- 리플로우: 레이아웃 다시 계산
- 리페인트: 화면 다시 그리기
- 리플로우가 보통 더 무겁다

## 실무에서 자주 나오는 예시

### 1. 스타일을 반복해서 바꾸는 경우

반복문 안에서 `element.style.width`를 여러 번 바꾸면 변경이 누적되면서 브라우저가 계속 계산하게 된다.

이럴 때는 스타일 변경을 한 번에 모으는 편이 낫다.

### 2. 읽기와 쓰기를 섞는 경우

예를 들어 `offsetHeight` 같은 레이아웃 값을 읽고, 바로 스타일을 수정하고, 다시 읽는 패턴은 성능에 안 좋다. 브라우저 입장에서는 "지금 레이아웃이 최신인지" 보장하려고 강제로 계산을 앞당길 수 있다.

### 3. 애니메이션 속성 선택

`top`, `left`를 움직이는 애니메이션은 레이아웃에 영향을 줄 수 있다. 반면 `transform`, `opacity`는 비교적 비용이 적어서 성능 최적화 이야기에서 자주 권장된다.

## 어떻게 줄일 수 있나

### 1. 레이아웃을 건드리는 속성 변경을 줄인다

가능하면 `width`, `top`, `left` 대신 `transform`을 쓰는 식으로 접근한다.

### 2. DOM 조작을 모아서 한다

한 번에 여러 요소를 바꾸거나, 클래스 토글로 묶어서 처리하면 불필요한 재계산을 줄일 수 있다.

### 3. 레이아웃 읽기와 쓰기를 분리한다

먼저 필요한 값을 읽고, 그다음 변경 작업을 몰아서 적용하는 습관이 좋다.

### 4. 과한 애니메이션을 피한다

특히 리스트가 많거나 모바일 환경에서는 작은 스타일 변경도 체감 성능에 크게 영향을 줄 수 있다.

## 면접 답변 예시

리플로우는 요소의 위치나 크기가 바뀌어서 브라우저가 레이아웃을 다시 계산하는 과정이고, 리페인트는 레이아웃 변화 없이 색상이나 그림자처럼 화면만 다시 그리는 과정입니다. 일반적으로 리플로우가 더 비용이 큰데, 레이아웃 계산 후 다시 페인트까지 이어질 수 있기 때문입니다. 실무에서는 `top`, `left`보다 `transform`을 쓰고, DOM 조작을 묶어서 처리해서 이런 비용을 줄이려고 합니다.

## 장점

- 브라우저 렌더링 성능 질문에 답할 때 기본 개념을 명확하게 설명할 수 있다.
- 성능 저하 원인을 CSS 변경, DOM 조작 방식과 연결해서 말하기 좋다.
- 단순 이론이 아니라 애니메이션 최적화나 모바일 성능 이슈로 확장하기 쉽다.

## 단점

- 리플로우와 리페인트를 너무 이분법적으로만 외우면 실제 동작을 과하게 단순화할 수 있다.
- 브라우저 엔진 최적화가 있어서 모든 변경이 항상 같은 비용으로 발생하는 것은 아니다.
- 개념만 알고 측정 도구를 모르면 실무에서 설득력이 약해질 수 있다.

## 주의사항 / 실무 팁

- 무조건 모든 스타일 변경을 피할 필요는 없고, 잦고 큰 변경부터 줄이는 게 현실적이다.
- 성능 이슈는 추측보다 브라우저 DevTools의 Performance 패널로 확인하는 게 정확하다.
- 애니메이션은 가능하면 `transform`, `opacity` 중심으로 설계하는 편이 안전하다.
- 리스트나 무한 스크롤 화면에서는 작은 리플로우도 누적되면 체감 성능이 크게 떨어질 수 있다.
