# Long Task 정리

## 핵심 요약
- `Long Task`는 브라우저 메인 스레드를 50ms 이상 오래 점유하는 작업을 뜻한다.
- 사용자가 클릭하거나 스크롤해도 반응이 늦는 이유를 설명할 때 자주 연결된다.
- 면접에서는 "왜 느린가"보다 "메인 스레드를 얼마나 오래 막았는가" 관점으로 설명하면 좋다.
- 해결책은 작업을 쪼개고, 무거운 계산을 뒤로 미루고, 필요하면 워커로 분리하는 것이다.

## 개념 설명
브라우저의 메인 스레드는 렌더링, 이벤트 처리, 자바스크립트 실행을 같이 담당한다. 이 스레드가 한 번 오래 막히면 버튼 클릭, 입력, 애니메이션, 화면 갱신이 전부 늦어진다.

이때 50ms를 넘는 작업을 보통 `Long Task`라고 부른다. 예를 들어 큰 배열을 한 번에 정렬하거나, 복잡한 DOM 연산을 반복하거나, 초기 렌더링 때 무거운 계산을 몰아서 실행하면 생기기 쉽다.

중요한 점은 자바스크립트 실행 시간이 길다는 사실 자체보다, 그 시간 동안 사용자 입력을 처리하지 못한다는 것이다. 그래서 성능 면접에서는 FPS나 번들 크기뿐 아니라 메인 스레드 점유 시간을 같이 설명해야 답변이 탄탄해진다.

## 예시
예를 들어 검색 결과 5천 건을 한 번에 가공해서 화면에 그린다고 하자.

- 나쁜 경우
  - 클릭 직후 필터링, 정렬, 그룹화를 한 함수에서 한 번에 처리한다.
  - 그 사이 메인 스레드가 오래 점유돼 입력 반응이 끊긴다.
- 개선 방법
  - 계산을 여러 단계로 나눈다.
  - 화면에 꼭 필요한 데이터만 먼저 그린다.
  - 무거운 연산은 `requestIdleCallback`, `setTimeout`, `Web Worker` 같은 방식으로 분산한다.

## 면접 답변 예시
- 질문: "Long Task는 무엇인가요?"  
  답변: "브라우저 메인 스레드를 50ms 이상 오래 점유하는 작업을 말합니다. 이런 작업이 생기면 클릭, 입력, 스크롤 반응이 늦어지고 UI가 버벅이는 느낌을 주기 쉽습니다."
- 질문: "실무에서 어떻게 줄이나요?"  
  답변: "무거운 계산을 한 번에 몰아 하지 않고 작업을 쪼개거나, 우선순위가 낮은 일은 뒤로 미루고, CPU 비용이 큰 연산은 Web Worker로 분리합니다. 또 렌더링 범위를 줄여 DOM 업데이트 자체를 가볍게 만드는 것도 중요합니다."

## 장점
- 사용자 체감 성능 문제를 설명할 때 기준점이 분명하다
- 메인 스레드 병목을 찾는 데 유용한 관점이 된다
- 렌더링 최적화, 코드 분할, 워커 도입 같은 개선책과 연결하기 좋다

## 단점
- 50ms 기준만 외우면 실제 병목 구조를 놓칠 수 있다
- 네트워크 지연, 레이아웃 비용, 과도한 리렌더링 같은 원인을 함께 보지 않으면 분석이 단편적일 수 있다
- 무조건 작업을 쪼개는 방식은 코드 복잡도를 높일 수 있다

## 주의사항 / 실무 팁
- Chrome DevTools Performance 탭으로 실제 Long Task 구간을 확인한 뒤 원인을 좁히는 게 좋다.
- "렌더링이 느리다"라고 뭉뚱그리기보다 자바스크립트 실행, 스타일 계산, 레이아웃 중 어디가 긴지 나눠서 설명하면 면접 답변이 좋아진다.
- 최적화는 추측보다 측정이 우선이다. 불필요한 미세 최적화보다 사용자 입력이 끊기는 구간을 먼저 줄이는 게 효과적이다.
