# Shadow DOM 정리

## 핵심 요약

- `Shadow DOM`은 컴포넌트 내부 마크업과 스타일을 바깥 문서와 분리해 주는 브라우저 기능이다.
- 목적은 재사용 가능한 UI 조각을 더 독립적으로 만들고, 스타일 충돌을 줄이는 데 있다.
- 웹 컴포넌트 문맥에서 자주 나오지만, 면접에서는 "스타일 캡슐화"와 "DOM 격리" 관점으로 설명하면 충분하다.
- 다만 완전한 격리는 아니고, 디버깅이나 전역 스타일 연동이 불편해질 수 있다는 점도 같이 말하면 좋다.

## Shadow DOM이란?

보통 HTML은 하나의 전역 문서 트리 안에서 동작한다.  
그래서 어떤 컴포넌트 안에 있는 버튼이든, 바깥쪽 CSS 선택자나 자바스크립트가 쉽게 영향을 줄 수 있다.

`Shadow DOM`은 이 구조 안에 별도의 숨겨진 DOM 트리를 붙여서 컴포넌트 내부를 비교적 독립적인 공간처럼 다루게 해 준다.  
쉽게 말하면 브라우저 차원에서 "이 컴포넌트 안쪽은 좀 따로 관리하자"라는 경계를 만드는 셈이다.

## 왜 쓰는가?

재사용 가능한 컴포넌트를 만들 때 가장 흔한 문제 중 하나가 스타일 충돌이다.

- 전역 CSS가 우연히 내부 버튼 모양을 깨뜨리는 경우
- 같은 클래스 이름이 다른 화면과 섞이는 경우
- 컴포넌트를 다른 프로젝트로 옮겼을 때 예상과 다르게 보이는 경우

`Shadow DOM`을 쓰면 컴포넌트 내부 스타일이 바깥에 덜 흔들리고, 반대로 내부 스타일이 바깥으로 새는 일도 줄어든다.  
그래서 디자인 시스템이나 웹 컴포넌트 같은 재사용 단위에서 특히 의미가 크다.

## 어떻게 생겼나?

자바스크립트로 요소에 shadow root를 붙이고 그 안에 마크업을 넣는 방식으로 쓴다.

```js
const host = document.querySelector("#profile-card");
const shadowRoot = host.attachShadow({ mode: "open" });

shadowRoot.innerHTML = `
  <style>
    button {
      background: black;
      color: white;
    }
  </style>
  <button>팔로우</button>
`;
```

이 경우 버튼과 스타일은 shadow root 내부에 들어간다.  
그래서 바깥 문서의 CSS가 바로 영향을 주지 않는 경우가 많다.

## 장점은 무엇인가?

첫째, 스타일 캡슐화가 된다.  
컴포넌트 내부 스타일이 외부 CSS와 덜 섞이므로 재사용성이 좋아진다.

둘째, 구조가 독립적이다.  
컴포넌트를 라이브러리처럼 배포할 때 외부 환경에 덜 흔들린다.

셋째, 브라우저 기본 UI도 비슷한 개념을 쓴다.  
예를 들어 일부 기본 요소가 내부 구현을 감추는 방식과 연결해서 설명할 수 있다.

## 한계는 무엇인가?

`Shadow DOM`이 만능은 아니다.

- 전역 스타일 시스템과 자연스럽게 섞기 어렵다.
- 테스트나 디버깅에서 DOM 구조가 한 번 더 감춰져 불편할 수 있다.
- SEO, 접근성, 이벤트 전파 같은 부분은 단순히 "완전 분리"라고만 이해하면 오해가 생길 수 있다.

즉, 독립성은 높아지지만 통합 비용도 같이 생긴다.

## 면접에서 어떻게 설명하면 좋을까?

면접에서는 너무 구현 디테일보다 "왜 필요한가"를 먼저 말하는 편이 낫다.

Shadow DOM은 브라우저가 제공하는 DOM 격리 기능으로, 컴포넌트 내부의 마크업과 스타일을 외부 문서와 어느 정도 분리해 줍니다. 그래서 웹 컴포넌트처럼 재사용 가능한 UI를 만들 때 스타일 충돌을 줄이고 독립성을 높이는 데 유용합니다.

이후에 "다만 전역 CSS와 잘 섞이지 않을 수 있고 디버깅도 조금 불편하다" 정도를 덧붙이면 균형이 좋아진다.

## 면접 답변 예시

> Shadow DOM은 브라우저 차원에서 컴포넌트 내부 DOM과 스타일을 외부 문서와 분리하는 기능입니다. 목적은 주로 스타일 캡슐화와 컴포넌트 독립성 확보에 있습니다. 예를 들어 웹 컴포넌트를 여러 서비스에서 재사용할 때, 외부 CSS가 내부 UI를 깨뜨리거나 내부 스타일이 바깥으로 새는 문제를 줄일 수 있습니다. 다만 완전히 공짜인 건 아니어서 전역 스타일 시스템과 연결하기 어렵고, 테스트나 디버깅이 조금 복잡해질 수 있습니다.

## 장점

- 스타일 충돌을 줄여 재사용 가능한 UI를 만들기 좋다.
- 컴포넌트 경계가 명확해서 외부 환경에 덜 흔들린다.
- 웹 컴포넌트 기반 설계와 잘 맞는다.

## 단점

- 전역 테마나 공통 CSS를 적용할 때 불편할 수 있다.
- 디버깅, 테스트, 커스터마이징 난도가 올라갈 수 있다.
- 팀이 Shadow DOM 개념에 익숙하지 않으면 유지보수 장벽이 생긴다.

## 주의사항 / 실무 팁

- 단순한 컴포넌트라면 CSS Modules, CSS-in-JS 같은 방식만으로도 충분한지 먼저 판단하는 게 좋다.
- 디자인 시스템을 여러 제품에 배포하는 상황처럼 격리 이점이 큰 경우에 더 잘 맞는다.
- 면접에서는 "스타일을 숨긴다"보다 "스타일 충돌을 줄이고 컴포넌트 독립성을 높인다"라고 표현하는 편이 정확하다.
