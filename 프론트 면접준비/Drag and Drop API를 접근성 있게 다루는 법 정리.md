# Drag and Drop API를 접근성 있게 다루는 법 정리

## 핵심 요약

- 드래그 정렬은 포인터 제스처 외에 버튼이나 키보드로 같은 작업을 완료할 수 있어야 한다.
- 이동 뒤에는 시각적 순서만 바꾸지 말고 DOM 순서와 포커스도 함께 갱신한다.
- 선택, 이동, 놓기와 취소 상태는 짧은 live message로 알리되 움직일 때마다 과도하게 읽지 않게 한다.

## 개념 설명

접근 가능한 drag and drop은 마우스나 터치로 끄는 동작만 제공하지 않고, 키보드와 보조 기술 사용자도 같은 결과를 만들게 하는 패턴이다. 가장 단순한 대안은 각 항목에 위로·아래로 이동 버튼을 제공하는 것이다.

키보드 드래그 모드를 제공한다면 `Space`로 항목을 선택하고 화살표로 위치를 바꾸며 다시 `Space`로 놓는 규칙을 화면에 안내한다. Pointer와 키보드가 같은 정렬 상태 모델을 사용해야 결과가 엇갈리지 않는다. 변경된 위치는 `aria-live`로 필요한 시점에만 알린다.

## 예시

```html
<button aria-describedby="sort-help">선택한 항목 위로 이동</button>
<p id="sort-help">Space로 잡고 화살표로 이동합니다.</p>
<div aria-live="polite"></div>
```

제스처를 사용할 수 없는 사용자도 명시적인 버튼으로 정렬을 완료할 수 있다. 실제 UI에서는 현재 위치와 전체 항목 수를 알리고 이동할 수 없는 첫·마지막 위치의 버튼 상태도 표현한다.

## 면접 답변 예시

> 드래그 앤 드롭은 포인터 동작만 구현하지 않고 키보드나 이동 버튼으로 같은 결과를 만들 수 있게 하겠습니다. Pointer와 키보드는 같은 순서 상태를 갱신하고, 이동 후에는 CSS로 보이는 위치뿐 아니라 DOM 순서와 포커스도 함께 바꿉니다. 현재 위치와 이동 결과는 live region으로 짧게 알리되 반복 이동마다 지나치게 방해하지 않게 합니다. 잡기, 이동, 놓기, 취소를 키보드와 실제 스크린 리더로 테스트하겠습니다.

## 장점

- 포인터를 정밀하게 사용하기 어려운 사용자도 정렬 작업을 완료할 수 있다.
- 이동 상태와 결과를 보조 기술 사용자에게 전달할 수 있다.

## 단점

- 위치 알림이 너무 잦으면 스크린 리더의 현재 읽기를 계속 방해한다.
- 시각적 순서만 바꾸고 DOM 순서를 그대로 두면 키보드 탐색 결과가 화면과 달라진다.

## 주의사항 / 실무 팁

- 이동 뒤 DOM 순서와 포커스를 함께 갱신하고 첫·마지막 항목의 불가능한 동작을 비활성화한다.
- Pointer, touch, keyboard 입력이 같은 결과와 취소 동작을 만드는지 회귀 테스트한다.
