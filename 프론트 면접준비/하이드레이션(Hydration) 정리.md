# 하이드레이션(Hydration) 정리

## 핵심 요약

- 하이드레이션은 서버가 미리 만들어 둔 HTML에 클라이언트 JavaScript가 붙으면서 화면을 "살아 있는 상태"로 만드는 과정입니다.
- SSR이나 SSG를 쓰는 React, Next.js 계열에서 자주 나오는 개념이고, 초기 화면 표시 속도와 SEO에 도움이 됩니다.
- 다만 서버가 만든 결과와 클라이언트가 처음 렌더링한 결과가 다르면 hydration mismatch가 생길 수 있습니다.
- 면접에서는 "왜 필요한가", "CSR과 뭐가 다른가", "어떤 문제가 자주 생기나" 정도까지 같이 말해주면 답변이 탄탄해집니다.

## 하이드레이션이란?

하이드레이션은 이름만 보면 조금 추상적으로 느껴지는데, 생각보다 개념은 단순합니다.  
서버가 먼저 HTML을 만들어 브라우저에 보내고, 사용자는 JavaScript가 다 내려오기 전에도 일단 화면을 볼 수 있습니다.  
그다음 클라이언트 쪽 JavaScript가 실행되면서 버튼 클릭, 입력 이벤트, 상태 변경 같은 상호작용 기능이 붙습니다. 이 과정을 하이드레이션이라고 보면 됩니다.

즉, "HTML을 그리는 것"에서 끝나는 게 아니라, 이미 그려진 HTML을 React 같은 프레임워크가 이어받아 애플리케이션으로 연결하는 단계입니다.

## 왜 필요한가?

CSR만 쓰면 브라우저가 JavaScript를 다 받은 뒤에야 의미 있는 화면을 만들 수 있습니다.  
그래서 초기 로딩이 느려 보일 수 있고, 검색 엔진이 내용을 읽기 어려운 경우도 있습니다.

반대로 SSR이나 SSG를 쓰면:

- 첫 화면은 더 빨리 보일 수 있고
- 서버가 만든 HTML이 있어서 SEO에 유리하고
- 사용자는 로딩 중에도 뭔가 보이는 상태를 먼저 확인할 수 있습니다

하지만 서버가 HTML만 보내고 끝나면 버튼은 보여도 동작하지 않는 "정적인 문서"에 가깝습니다.  
여기에 이벤트와 상태 관리를 연결해서 실제 앱처럼 동작하게 만드는 단계가 바로 하이드레이션입니다.

## CSR, SSR, 하이드레이션의 관계

헷갈리는 포인트는 보통 여기입니다.

### CSR

CSR(Client Side Rendering)은 브라우저가 JavaScript를 받아서 화면을 직접 그리는 방식입니다.  
초기에는 비어 있는 HTML이나 최소한의 뼈대만 받고, 실제 UI는 클라이언트가 만듭니다.

### SSR

SSR(Server Side Rendering)은 서버가 요청 시점에 HTML을 만들어서 내려줍니다.  
사용자는 더 빨리 화면을 볼 수 있지만, 처음 받은 HTML만으로는 상호작용이 완전히 연결되지 않았을 수 있습니다.

### 하이드레이션

하이드레이션은 SSR이나 SSG로 먼저 만들어진 HTML에 클라이언트 로직을 붙이는 단계입니다.  
그래서 "SSR을 썼다"와 "하이드레이션이 일어난다"는 보통 같이 연결해서 이해하면 됩니다.

## 동작 흐름

하이드레이션은 보통 아래 흐름으로 이해하면 설명하기 쉽습니다.

1. 서버가 HTML을 만들어 응답합니다.
2. 브라우저는 HTML을 먼저 그려서 사용자가 화면을 보게 합니다.
3. 클라이언트 JavaScript 번들이 다운로드되고 실행됩니다.
4. React 같은 프레임워크가 현재 DOM과 자신이 기대하는 트리를 연결합니다.
5. 이벤트 핸들러, 상태, 라우팅 등이 붙으면서 상호작용 가능한 앱 상태가 됩니다.

중요한 점은 "처음부터 다시 그리는 것"보다는 "이미 있는 결과물에 연결하는 것"에 가깝다는 점입니다.

## 예시로 이해하기

예를 들어 상품 상세 페이지가 있다고 해보겠습니다.

- 서버는 상품명, 가격, 설명이 들어간 HTML을 미리 만들어서 보냅니다.
- 사용자는 페이지에 들어오자마자 상품 정보를 볼 수 있습니다.
- 하지만 장바구니 버튼, 수량 변경, 리뷰 정렬 같은 기능은 JavaScript가 붙어야 제대로 동작합니다.

이때 JavaScript가 실행되면서 해당 버튼에 클릭 이벤트가 연결되고, 상태 변경이 가능해지는 순간이 하이드레이션입니다.

## 하이드레이션 mismatch란?

면접에서 한 단계 더 들어가면 자주 나오는 게 mismatch입니다.  
서버가 만든 HTML과 클라이언트가 처음 렌더링한 결과가 다를 때 발생합니다.

예를 들어 이런 경우가 흔합니다.

- 서버에서는 현재 시간을 기준으로 문자열을 만들었는데, 클라이언트에서는 실행 시점이 달라 다른 시간이 찍히는 경우
- `window`, `localStorage` 같은 브라우저 전용 값을 첫 렌더링에서 바로 읽는 경우
- 랜덤 값이나 `Math.random()` 결과를 바로 렌더링에 쓰는 경우
- 지역 시간대나 사용자 환경에 따라 출력 문자열이 달라지는 경우

이런 상황이 생기면 경고가 뜨거나, 기대한 대로 이벤트가 연결되지 않거나, 화면이 다시 그려지는 비용이 추가될 수 있습니다.

## mismatch를 줄이는 방법

실무에서는 "서버와 클라이언트의 첫 렌더링 결과를 최대한 같게 만든다"가 핵심입니다.

- 첫 렌더링에서 시간, 랜덤 값, 브라우저 전용 API에 바로 의존하지 않습니다.
- 브라우저에서만 알아야 하는 값은 `useEffect` 이후에 처리합니다.
- 사용자 환경마다 달라질 수 있는 값은 서버와 클라이언트에서 일관되게 맞춥니다.
- 리스트 key나 조건부 렌더링이 서버와 클라이언트에서 다르게 나오지 않게 주의합니다.

예를 들어 로그인 여부를 `localStorage`에서만 읽는 구조라면, 첫 렌더링에서 바로 UI를 갈라버리기보다 클라이언트 마운트 후 상태를 확정하는 식으로 푸는 경우가 많습니다.

## 부분 하이드레이션, 선택적 하이드레이션

요즘은 모든 UI를 한 번에 하이드레이션하지 않고, 필요한 부분만 나눠서 처리하려는 흐름도 자주 언급됩니다.  
페이지 전체를 한꺼번에 연결하면 JavaScript 비용이 커질 수 있기 때문입니다.

그래서 프레임워크들은:

- 필요한 컴포넌트만 늦게 활성화하거나
- 우선순위가 높은 영역부터 하이드레이션하거나
- 아예 서버 컴포넌트를 활용해 클라이언트 JavaScript 양을 줄이는 방향

으로 발전하고 있습니다.

면접에서는 너무 깊게 들어가기보다 "최근에는 전체를 다 붙이기보다 필요한 부분만 효율적으로 활성화하려는 흐름이 있다" 정도로 말해도 충분합니다.

## 장점

- 첫 화면이 빨리 보이는 느낌을 줄 수 있습니다.
- SEO 측면에서 CSR보다 유리한 경우가 많습니다.
- 서버가 만든 HTML을 활용하므로 초기 사용자 경험이 좋아질 수 있습니다.
- React, Next.js 기반 서비스에서 렌더링 전략을 설명할 때 핵심 연결 고리 역할을 합니다.

## 단점

- 서버와 클라이언트 결과가 다르면 mismatch 문제가 생길 수 있습니다.
- SSR 자체의 서버 비용과 복잡도가 추가됩니다.
- JavaScript 번들이 결국 필요하므로, HTML만 빠르다고 전체가 다 끝난 것은 아닙니다.
- 렌더링 전략이 섞이면 디버깅 포인트가 늘어날 수 있습니다.

## 주의사항 / 실무 팁

- 하이드레이션은 "초기 HTML을 빨리 보여주는 것"과 "상호작용 가능하게 만드는 것"을 분리해서 생각하면 이해가 쉽습니다.
- 서버에서 만든 값과 클라이언트에서 만든 값이 달라질 가능성이 있는지 먼저 의심해보면 mismatch 원인을 빨리 찾을 수 있습니다.
- 날짜, 랜덤 값, 브라우저 저장소, 사용자별 환경값은 특히 조심해야 합니다.
- Next.js 같은 프레임워크를 쓴다고 자동으로 끝나는 문제가 아니라, 컴포넌트 설계와 데이터 처리 방식이 맞아야 안정적입니다.
- 성능을 볼 때는 FCP만 보지 말고, 실제로 언제 상호작용 가능한 상태가 되는지도 같이 봐야 합니다.

## 면접 답변 예시

하이드레이션은 서버가 미리 만들어서 내려준 HTML에 클라이언트 JavaScript가 붙으면서 화면을 상호작용 가능한 상태로 만드는 과정입니다. 예를 들어 SSR을 사용하면 사용자는 JavaScript가 다 로드되기 전에도 화면을 먼저 볼 수 있는데, 그 시점에는 버튼이나 상태 변경이 완전히 연결되지 않았을 수 있습니다. 이후 React가 기존 HTML을 이어받아 이벤트와 상태를 연결하는데, 이게 하이드레이션입니다.

CSR과 비교하면 초기 화면 표시나 SEO 측면에서 유리할 수 있지만, 서버 렌더 결과와 클라이언트 첫 렌더 결과가 다르면 hydration mismatch가 생길 수 있습니다. 그래서 실무에서는 첫 렌더링에서 시간, 랜덤 값, 브라우저 전용 API에 바로 의존하지 않도록 설계하는 게 중요합니다.
