# React Synthetic Event 정리

## 핵심 요약

- `Synthetic Event`는 React가 브라우저 이벤트를 감싼 크로스 브라우저용 이벤트 객체다.
- 면접에서는 "왜 굳이 한 번 감싸는가", "이벤트 위임과 어떤 관계가 있나", "요즘 React에서는 예전과 무엇이 달라졌나"를 같이 말하면 좋다.
- 핵심은 브라우저마다 조금씩 다른 이벤트 동작을 React가 일관된 인터페이스로 맞춰 준다는 점이다.
- 예전에는 이벤트 풀링을 많이 이야기했지만, 최근 React에서는 그 제약이 사라졌다는 점도 같이 정리해 두면 좋다.

## Synthetic Event가 뭔가

React에서 `onClick`, `onChange` 같은 핸들러를 쓰면 브라우저의 원시 이벤트를 그대로 받는 게 아니라 React가 만든 이벤트 객체를 받는다.  
이걸 `Synthetic Event`라고 부른다.

이 객체는 브라우저 이벤트와 비슷한 인터페이스를 제공하지만, React가 한 번 감싸서 브라우저 차이를 줄여 준다는 점이 다르다.

## 왜 이런 걸 만들었나

브라우저 이벤트는 기본적으로 표준이 있지만, 세부 동작이나 지원 방식이 미묘하게 다르던 시절이 있었다.  
React는 개발자가 브라우저별 차이를 직접 흡수하지 않도록 일관된 이벤트 API를 제공하려고 했다.

또 React 내부 이벤트 시스템과 잘 연결해서 업데이트 흐름을 제어하기에도 유리하다.

## 이벤트 위임과의 관계

React는 전통적으로 이벤트를 각 DOM 노드에 전부 직접 붙이기보다, 상위 레벨에서 모아서 처리하는 이벤트 위임 방식을 활용해 왔다.  
그래서 많은 컴포넌트가 있어도 이벤트 리스너를 효율적으로 관리할 수 있다.

면접에서는 "Synthetic Event는 일관된 이벤트 객체, 이벤트 위임은 이벤트를 처리하는 방식"이라고 구분해서 말하면 좋다.  
둘은 연결돼 있지만 같은 개념은 아니다.

## 실제로 자주 쓰는 메서드

### `preventDefault()`

기본 동작을 막는다.  
예를 들어 폼 제출 시 페이지 새로고침을 막을 때 자주 쓴다.

### `stopPropagation()`

이벤트 버블링을 막는다.  
부모와 자식이 모두 클릭 핸들러를 가질 때 의도치 않은 연쇄 실행을 막는 데 쓴다.

### `nativeEvent`

정말 브라우저의 원시 이벤트 객체가 필요하면 `event.nativeEvent`로 접근할 수 있다.  
다만 대부분은 Synthetic Event 인터페이스만으로 충분하다.

## 예전 React와 지금 React 차이

예전에는 이벤트 객체를 재사용하는 `event pooling`이 있어서, 비동기 콜백 안에서 이벤트를 늦게 읽으면 값이 비어 있는 문제가 있었다.  
그래서 `event.persist()` 같은 메서드를 기억해야 했다.

하지만 최근 React에서는 이벤트 풀링이 제거돼서, 예전만큼 이 제약을 신경 쓰지 않아도 된다.  
면접에서 이걸 같이 언급하면 오래된 정보와 최신 정보를 구분해서 알고 있다는 인상을 줄 수 있다.

## 언제 헷갈리기 쉬운가

### 브라우저 이벤트와 완전히 같다고 생각할 때

비슷하지만 완전히 같은 객체는 아니다.  
React가 추상화한 계층이라는 점을 이해해야 한다.

### 이벤트 버블링 흐름을 놓칠 때

React에서도 이벤트 전파 개념은 중요하다.  
특히 리스트, 모달, 드롭다운처럼 클릭 계층이 깊은 UI에서 자주 문제를 만난다.

## 면접 답변 예시

> React Synthetic Event는 브라우저 이벤트를 React가 한 번 감싼 이벤트 객체로, 브라우저 차이를 줄이고 일관된 인터페이스를 제공하기 위한 개념입니다. 개발자는 `onClick`, `onChange` 같은 핸들러에서 이 객체를 받아 `preventDefault`, `stopPropagation` 같은 메서드를 공통적으로 사용할 수 있습니다. React는 이벤트 위임 방식과 함께 이 시스템을 활용해 이벤트를 효율적으로 처리해 왔고, 예전에 있던 event pooling은 최근 React에서는 제거돼 이전보다 사용이 단순해졌습니다.

## 장점

- 브라우저별 이벤트 차이를 덜 신경 써도 된다.
- React 이벤트 시스템과 자연스럽게 연결된다.
- 인터뷰에서 이벤트 위임, 버블링, 기본 동작 제어까지 한 번에 묶어 설명하기 좋다.

## 단점

- 원시 DOM 이벤트와 완전히 같다고 오해하기 쉽다.
- 오래된 자료를 보면 event pooling 정보가 섞여 있어 혼란스럽다.
- 이벤트 전파를 정확히 모르면 디버깅이 어려울 수 있다.

## 주의사항 / 실무 팁

- 면접에서는 `nativeEvent`를 언급하되, 대부분은 Synthetic Event만으로 충분하다고 덧붙이면 좋다.
- 이벤트 위임과 Synthetic Event를 같은 개념으로 말하지 않도록 구분해 두는 편이 안전하다.
- 오래된 React 자료의 `event.persist()`를 그대로 외우기보다, "예전엔 필요했지만 지금은 풀링이 제거됐다"까지 정리해 두면 답변 품질이 올라간다.
