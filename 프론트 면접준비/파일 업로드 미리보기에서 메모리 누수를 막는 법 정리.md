# 파일 업로드 미리보기에서 메모리 누수를 막는 법 정리

## 핵심 요약

- `URL.createObjectURL()`로 만든 URL은 브라우저가 자동으로 즉시 해제하지 않으므로 사용이 끝나면 직접 revoke해야 한다.
- 파일 교체, 미리보기 삭제와 컴포넌트 unmount가 모두 같은 cleanup 경로를 지나게 한다.
- 이미지 `load` 직후 무조건 해제하지 말고 사용자가 저장·확대할 필요가 끝나는 시점을 기준으로 수명을 정한다.

## 개념 설명

`URL.createObjectURL()`은 `File`이나 `Blob`을 가리키는 임시 URL을 만든다. 이 URL이 살아 있는 동안 브라우저는 원본 데이터를 참조할 수 있어, 새 파일을 고를 때마다 만들고 해제하지 않으면 탭 메모리가 계속 늘 수 있다.

Object URL은 미리보기가 제거되거나 다른 파일로 교체되고 컴포넌트가 사라질 때 해제한다. 너무 일찍 revoke하면 로딩이 실패하거나 사용자의 이미지 저장 같은 후속 동작이 막힐 수 있다. 로드 실패와 업로드 취소도 cleanup을 지나도록 URL과 해제 함수를 한 상태로 관리한다.

## 예시

```ts
const previewUrl = URL.createObjectURL(file);
image.src = previewUrl;
cleanup.push(() => URL.revokeObjectURL(previewUrl));
```

새 파일을 선택할 때 이전 cleanup을 먼저 실행하고 새 URL을 만든다. 여러 파일이라면 파일 ID별 URL을 map으로 관리해 삭제된 항목만 정확히 revoke한다.

## 면접 답변 예시

> 파일 미리보기에서는 `createObjectURL()`로 만든 URL의 수명을 파일 상태와 함께 관리하겠습니다. 파일 교체, 항목 삭제, 업로드 취소와 컴포넌트 unmount에서 이전 URL을 revoke합니다. 다만 이미지가 로드되자마자 무조건 해제하면 사용자의 확대나 저장 동작을 방해할 수 있어 미리보기 사용이 끝난 시점을 기준으로 정합니다. 같은 대용량 파일을 반복 선택하는 테스트에서 object URL 수와 탭 메모리가 계속 증가하지 않는지 확인합니다.

## 장점

- 대용량 파일을 반복 선택하는 화면에서 불필요한 탭 메모리 증가를 막을 수 있다.
- 파일별 URL과 cleanup을 함께 관리하면 교체와 삭제 수명 주기가 명확해진다.

## 단점

- 컴포넌트 unmount나 로드 실패 경로를 놓치면 참조가 계속 남는다.
- 너무 이른 revoke는 이미지 로딩과 사용자의 후속 동작을 깨뜨릴 수 있다.

## 주의사항 / 실무 팁

- 파일 교체 전에 이전 URL을 해제하고 unmount, 삭제, 취소가 같은 cleanup 함수를 호출하게 한다.
- 반복 업로드 시나리오를 DevTools 메모리와 살아 있는 object URL 수로 확인한다.
