# React Server Components 정리

## 핵심 요약

- `React Server Components(RSC)`는 일부 컴포넌트를 서버에서 렌더링하고, 그 결과를 클라이언트에 전달하는 React 컴포넌트 모델이다.
- 핵심은 `모든 컴포넌트를 브라우저에서 실행하지 않아도 된다`는 점이다.
- 면접에서는 SSR과 완전히 같은 개념으로 말하지 말고, `서버에서 실행되는 컴포넌트와 클라이언트에서 상호작용하는 컴포넌트를 분리하는 방식`이라고 설명하는 게 좋다.
- 번들 크기 절감, 데이터 접근 단순화가 장점이지만, 경계 관리와 사고방식 변화가 필요하다는 점도 함께 말해야 한다.

## React Server Components가 뭐냐

기존 React에서는 많은 컴포넌트가 기본적으로 브라우저에서 실행됐다.  
그래서 데이터 조회, 렌더링, 상호작용 코드가 한쪽에 몰리면서 번들이 커지거나 클라이언트 부담이 커지는 문제가 생겼다.

RSC는 이 중 일부 컴포넌트를 서버에서만 실행하고, 결과만 클라이언트로 보내는 방식이다.  
즉 `인터랙션이 필요 없는 UI 조각은 서버에서 처리하고, 사용자 상호작용이 필요한 부분만 클라이언트 컴포넌트로 남기는 구조`라고 보면 된다.

## SSR과는 뭐가 다르냐

이 질문이 면접에서 자주 붙는다.

- `SSR`: 페이지 HTML을 서버에서 먼저 만들어 보내는 방식
- `RSC`: 컴포넌트 단위로 서버 전용 실행 모델을 도입하는 방식

둘은 함께 쓰일 수도 있지만 완전히 같은 개념은 아니다.  
SSR은 "초기 HTML을 어디서 만들지"에 가깝고, RSC는 "어떤 컴포넌트를 어디서 실행할지"에 더 가깝다.

## 왜 쓰는가

- 클라이언트 번들 크기를 줄이기 좋다.
- 서버에서 바로 데이터 접근이 가능해진다.
- 민감한 로직이나 의존성을 브라우저로 보내지 않아도 된다.

특히 상품 상세, 블로그, 대시보드 읽기 화면처럼 인터랙션보다 데이터 표현 비중이 큰 화면에서 장점이 크다.

## 클라이언트 컴포넌트와 어떻게 나뉘나

RSC 환경에서는 보통 두 종류로 나뉜다.

- 서버 컴포넌트: 서버에서 실행되고 브라우저 JS 번들에 포함되지 않는다.
- 클라이언트 컴포넌트: 상태, 이벤트, 브라우저 API가 필요한 컴포넌트다.

예를 들어 목록 데이터 조회와 마크업 구성은 서버 컴포넌트에서 하고,  
좋아요 버튼이나 필터 토글처럼 상호작용이 필요한 부분만 클라이언트 컴포넌트로 분리할 수 있다.

## 장점은 뭐냐

가장 실무적으로 체감되는 장점은 번들 감소와 데이터 접근 방식 단순화다.

- 굳이 클라이언트에서 fetch 후 상태를 세팅하지 않아도 되는 구간이 늘어난다.
- 브라우저에 보내지 않을 코드가 많아져 초기 로딩 부담이 줄 수 있다.
- 서버 리소스를 적극 활용하면서도 UI는 React 컴포넌트 단위로 유지할 수 있다.

## 단점과 어려운 점은 뭐냐

RSC가 만능은 아니다.

- 서버 컴포넌트에서는 상태 훅, 이벤트 핸들러, 브라우저 API를 직접 쓸 수 없다.
- 서버/클라이언트 경계를 잘못 나누면 오히려 구조가 헷갈릴 수 있다.
- 데이터 캐싱, 스트리밍, 프레임워크 규칙을 함께 이해해야 해서 학습 비용이 있다.

즉 "성능이 좋아진다"보다 `어떤 코드를 서버에 두고 어떤 코드를 클라이언트에 둘지 설계 감각이 필요하다`가 더 정확하다.

## 언제 설명하면 좋나

면접에서는 Next.js App Router 경험과 같이 묶어서 말하면 자연스럽다.

- 서버에서 데이터 가져오는 화면
- SEO가 중요한 읽기 중심 페이지
- 대규모 클라이언트 번들이 부담인 서비스

반대로 거의 모든 요소가 복잡한 상호작용을 갖는 화면이라면 효과가 제한적일 수도 있다고 말하면 균형이 잡힌다.

## 면접 답변 예시

> React Server Components는 일부 컴포넌트를 서버에서 실행하고 결과만 클라이언트에 전달하는 방식입니다. 그래서 모든 UI 로직을 브라우저 번들에 실을 필요가 없고, 데이터 조회와 렌더링을 서버 쪽에 더 가깝게 둘 수 있습니다. 저는 이 개념을 SSR과 구분해서 설명하는 편인데, SSR이 초기 HTML 생성에 가깝다면 RSC는 컴포넌트 실행 위치를 서버와 클라이언트로 나누는 모델에 가깝습니다. 장점은 번들 크기 감소와 데이터 처리 단순화지만, 상호작용이 필요한 컴포넌트와의 경계를 잘 설계해야 합니다.

## 장점

- 최신 React 생태계 이해도를 보여 주기 좋다.
- SSR, 번들 최적화, 서버/클라이언트 경계 설계로 답변을 확장하기 쉽다.
- Next.js 경험과 연결하면 실무감이 살아난다.

## 단점

- SSR과 혼동하면 설명이 금방 무너진다.
- 프레임워크 맥락 없이 추상적으로만 말하면 공허해 보일 수 있다.
- 클라이언트 컴포넌트 제약을 모르면 실제 사용 이해도가 약해 보인다.

## 주의사항 / 실무 팁

- 면접에서는 `서버 전용 컴포넌트`와 `클라이언트 컴포넌트`를 구분해서 말하는 게 핵심이다.
- 상태 훅, 이벤트 핸들러, 브라우저 API는 클라이언트 컴포넌트 쪽이라고 정리해 두면 좋다.
- SSR과 같지 않다는 점을 분명히 하고, "함께 쓸 수는 있다"라고 이어 가면 자연스럽다.
- 번들 최적화만 강조하지 말고 경계 설계 비용도 같이 말하면 답변이 균형 잡힌다.
