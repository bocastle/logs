# Page Visibility API로 polling 비용 줄이는 법 정리

## 핵심 요약

- 면접에서 "Page Visibility API로 polling 비용 줄이는 법" 질문을 받으면 핵심은 "기능 지원 여부와 fallback을 어떻게 설계할 것인가?"이다.
- 실무에서는 적용 자체보다 관측 가능한 기준과 되돌릴 방법이 더 중요하다.
- 예시는 실제 화면 흐름이나 운영 지표와 연결해서 말하는 편이 좋다.

## 개념 설명

웹 API 주제는 브라우저가 어디까지 대신해 주고 애플리케이션이 어디서 책임져야 하는지를 나누는 문제다.

지원 범위와 fallback을 먼저 확인하고, 핵심 사용자 흐름을 망치지 않는 선에서 점진적으로 얹는 접근이 안전하다.

"Page Visibility API로 polling 비용 줄이는 법"은 단독 지식으로 외우기보다 주변 시스템과 사용자 흐름에 어떤 영향을 주는지 연결해서 보는 편이 좋다.

## 예시

```text
지원 확인 -> 기본 동작 유지 -> 최신 기능 적용 -> 실패 상태 안내 -> 지표 확인
```

실무 적용 전에는 이 구조가 현재 서비스의 트래픽, 보안 요구사항, 배포 방식과 맞는지 확인해야 한다.

## 면접 답변 예시

> "Page Visibility API로 polling 비용 줄이는 법"에 대해 묻는다면 저는 브라우저 지원 여부를 확인하고 핵심 흐름은 fallback으로 유지한 뒤 최신 기능을 점진적으로 적용한다고 답하겠습니다. 기능 이름만 말하기보다 어떤 사용자 문제를 줄이는지, 실패했을 때 화면이 어떻게 유지되는지, 적용 후 어떤 지표로 확인할지까지 연결해서 설명하겠습니다.

## 장점

- 브라우저 기능을 활용해 구현 복잡도를 줄일 수 있다.
- 점진적 향상 방식으로 지원 범위가 다른 환경을 다룰 수 있다.
- 사용자 흐름을 플랫폼 기능과 연결해 설명하기 좋다.

## 단점

- 지원 범위가 제한적이면 대체 흐름을 유지해야 한다.
- 권한 실패나 정책 차이 때문에 환경별 테스트가 필요하다.
- 기능 자체보다 예외 처리 코드가 더 길어질 수 있다.

## 주의사항 / 실무 팁

- 지원하지 않는 환경의 기본 흐름을 먼저 정하고 최신 기능을 얹는다.
- 면접에서는 API 이름보다 실패했을 때 사용자에게 무엇이 보이는지 설명한다.
- 작은 데모 코드와 함께 실제 확인 지표를 하나 붙이면 답변이 좋아진다.
