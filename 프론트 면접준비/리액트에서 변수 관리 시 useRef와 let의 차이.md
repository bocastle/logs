# React에서 `useRef`와 `let`으로 변수 관리하기

## 핵심 요약

- 컴포넌트 **내부**에서 선언한 `let` 변수는 **리렌더링 때마다 초기화**되어 이전 값을 잃습니다.
- `useRef()`로 만든 값은 **리렌더링되어도 유지**되며, 값이 바뀌어도 **리렌더링을 유발하지 않습니다**.
- 컴포넌트 **외부**에 `let`을 두면 리렌더링 영향은 피할 수 있지만, **인스턴스 간 공유/예측 불가/단방향 데이터 흐름 위배**로 권장되지 않습니다.

## 차이점 정리

### 리렌더링 시 값 유지 여부

- `let`(컴포넌트 내부): 리렌더링될 때마다 **초기화**
- `useRef()`(컴포넌트 내부): 리렌더링되어도 **값 유지**

### 값 변경 시 리렌더링 유발 여부

- `useState()`: 값 변경 시 보통 **리렌더링 유발**
- `useRef()`: 값 변경(`ref.current = ...`) 시 **리렌더링 유발하지 않음**

## 사용 사례

### DOM 요소 접근

`useRef()`는 DOM 요소에 접근할 때 자주 사용합니다.

- 예: `input`에 focus 주기
- 예: 스크롤 위치 조정

### 타이머 ID 보관

`setTimeout()` / `setInterval()`의 ID를 저장할 때도 유용합니다.

- 컴포넌트가 리렌더링되어도 타이머 ID가 유지되어야
- cleanup 함수에서 해당 ID로 타이머를 정리할 수 있음

## 컴포넌트 외부에 `let`을 선언하면 괜찮을까?

컴포넌트 외부에서 선언된 `let` 변수는 리렌더링의 영향을 받지 않습니다. 하지만 이런 방식은 권장되지 않는 패턴입니다.

### 권장되지 않는 이유

1. **모든 컴포넌트 인스턴스가 공유**
   - 같은 컴포넌트를 여러 번 사용할 때 예상치 못한 동작이 발생할 수 있음
2. **전역 변수처럼 동작**
   - 코드의 예측 가능성과 유지보수성이 떨어짐
3. **리액트 단방향 데이터 흐름 원칙 위배**
   - 상태 관리가 리액트 생명주기 밖에서 이루어짐

## 장점

### `useRef()`의 장점

- 리렌더링에도 값이 **유지**
- 값이 바뀌어도 **리렌더링을 유발하지 않음**
- DOM 접근, 타이머 ID 보관 등 **실무 패턴**에 적합

## 단점

### `let`(컴포넌트 내부)의 단점

- 리렌더링 시 **초기화**되어 값 유지가 어려움

### 컴포넌트 외부 `let`의 단점

- 컴포넌트 인스턴스 간 **상태 공유**로 버그 유발 가능
- 유지보수성/예측 가능성 저하
- 리액트의 데이터 흐름 원칙과 어긋날 수 있음

## 주의사항 및 실무 팁

- 컴포넌트의 상태를 관리해야 한다면, 리액트에서 제공하는 공식적인 방법인 `useState()` 또는 `useRef()`를 우선 고려하는 것이 좋습니다.
- `useRef()`는 **값을 바꿔도 화면이 자동으로 갱신되지** 않으므로, UI에 반영해야 하는 값은 `useState()`를 사용하세요.
- 컴포넌트 외부 변수를 이용한 상태 공유는 디버깅이 어려워질 수 있으니 지양하세요.

## 관련 코드/키워드 (코드블록 정리)

```ts
// useRef: 리렌더링에도 유지되며 변경해도 리렌더링을 유발하지 않음
const ref = useRef(initialValue);
ref.current = nextValue;
```

```ts
// let: 컴포넌트 내부에서 선언 시 리렌더링 때마다 초기화될 수 있음
let value = 0;
value += 1;
```

```ts
// 타이머 ID 보관 예시 (개념)
const timerIdRef = useRef<number | null>(null);

timerIdRef.current = window.setInterval(() => {
  // ...
}, 1000);

// cleanup에서 timerIdRef.current로 정리
```
