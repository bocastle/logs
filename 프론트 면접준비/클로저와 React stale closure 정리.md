# 클로저와 React stale closure 정리

## 핵심 요약

- 클로저는 함수가 선언될 때의 외부 변수 환경을 함께 기억하는 특성이다.
- React에서 `stale closure`는 예전 상태값을 캡처한 함수가 나중에도 그 값을 계속 바라보는 문제다.
- 면접에서는 클로저 자체 설명과 함께 `setInterval`, 이벤트 핸들러, 비동기 콜백에서 왜 문제가 생기는지 연결해서 말하면 좋다.
- 해결은 최신 값을 다시 읽게 만들거나, 함수형 업데이트와 올바른 의존성 관리로 흐름을 바꾸는 쪽이 핵심이다.

## 클로저를 어떻게 설명하면 좋나

클로저는 "함수가 만들어질 때 주변 스코프를 같이 기억하는 것"이라고 설명하면 된다.  
자바스크립트 함수는 단순히 코드 덩어리가 아니라, 그 함수가 선언된 시점의 변수 환경까지 묶어서 들고 간다.

예를 들어 카운터를 하나 만드는 함수를 생각해 보면, 내부 함수가 바깥 함수의 `count`를 계속 참조할 수 있다.  
이게 바로 클로저다.

면접에서는 "정보 은닉"이나 "상태 유지" 같은 장점도 같이 붙이면 설명이 더 자연스럽다.

## React에서는 왜 stale closure가 문제냐

React 컴포넌트는 렌더링이 다시 일어날 때마다 새로운 값과 새로운 함수들이 만들어진다.  
그런데 어떤 콜백이 예전 렌더 시점의 상태를 캡처한 채 오래 살아남으면, 화면은 바뀌었는데 콜백 안에서는 예전 값만 보는 상황이 생길 수 있다.

이걸 흔히 `stale closure`라고 부른다.

즉, 문제의 핵심은 "클로저가 틀렸다"가 아니라 "예전 렌더의 값을 잡은 함수가 예상보다 오래 사용됐다"는 데 있다.

## 자주 나오는 예시

### 1. setInterval 안에서 오래된 state를 읽는 경우

`useEffect`에서 `setInterval`을 등록했는데 의존성 배열을 비워 두고, 콜백 안에서 `count`를 직접 읽으면 처음 렌더의 `count`만 계속 볼 수 있다.

이럴 때는 아래처럼 함수형 업데이트가 안전하다.

```tsx
setCount((prev) => prev + 1);
```

이 방식은 최신 상태를 기준으로 계산하므로 stale closure 문제를 피하기 쉽다.

### 2. 비동기 요청이 끝난 뒤 예전 값을 사용하는 경우

검색어를 기준으로 요청을 보냈는데, 응답이 늦게 돌아오는 동안 사용자가 입력을 더 바꾸면 콜백 안에서는 예전 검색어를 보고 있을 수 있다.

이럴 때는 의존성 배열을 올바르게 주거나, 요청 취소와 최신 요청만 반영하는 로직을 함께 두는 편이 좋다.

### 3. 이벤트 리스너를 한 번만 등록해 둔 경우

윈도우 이벤트나 외부 라이브러리 콜백을 한 번만 등록해 두면, 그 콜백이 처음 렌더의 값만 기억하는 일이 생길 수 있다.

이런 경우는 `useRef`로 최신 값을 저장해 두고 리스너 내부에서 `ref.current`를 읽게 만드는 패턴이 자주 쓰인다.

## 어떻게 해결하냐

### 함수형 업데이트 사용

이전 상태를 기반으로 다음 상태를 계산할 때는 `setState(value)`보다 `setState((prev) => ...)`가 더 안전하다.  
특히 타이머나 비동기 콜백에서는 거의 습관처럼 떠올리는 게 좋다.

### 의존성 배열을 정확히 관리

`useEffect`, `useCallback`, `useMemo`에서 의존성 배열을 빼먹으면 오래된 값을 캡처하기 쉽다.  
면접에서는 "eslint 경고를 무시하지 않는다"까지 덧붙이면 실무 감각이 느껴진다.

### ref로 최신 값 분리

렌더링을 다시 일으키지 않아도 최신 값을 읽어야 할 때는 `useRef`가 유용하다.  
다만 모든 문제를 `ref`로 덮으면 상태 흐름이 숨겨져서 더 헷갈릴 수 있으니 필요한 곳에만 써야 한다.

## 면접에서 자주 나오는 꼬리질문

### 클로저는 좋은 건가, 나쁜 건가

좋은 개념이다.  
문제는 React처럼 렌더가 반복되는 환경에서 언제 어떤 값을 캡처했는지 의식하지 않으면 버그가 생긴다는 점이다.

### stale closure와 무한 렌더는 같은 문제냐

같지 않다.  
무한 렌더는 주로 상태 변경과 렌더 트리거가 반복될 때 생기고, stale closure는 콜백이 오래된 값을 보는 문제다.  
둘 다 `useEffect`에서 많이 보이지만 원인은 다르다.

## 면접 답변 예시

> 클로저는 함수가 선언될 때 외부 스코프를 함께 기억하는 자바스크립트 특성입니다. React에서는 렌더링이 반복되면서 함수도 새로 만들어지는데, 예전 렌더 시점의 값을 캡처한 콜백이 계속 살아 있으면 stale closure 문제가 생길 수 있습니다. 예를 들어 `setInterval`이나 비동기 콜백에서 오래된 state를 읽는 경우가 대표적입니다. 보통은 함수형 업데이트를 쓰거나, `useEffect` 의존성 배열을 정확히 관리하고, 필요한 경우 `useRef`로 최신 값을 읽게 해서 해결합니다.

## 장점

- 클로저 자체는 상태 유지와 캡슐화에 매우 유용하다.
- stale closure를 이해하면 React 비동기 버그를 설명하기 쉬워진다.
- 단순 문법 설명을 넘어 렌더링 모델 이해까지 보여 줄 수 있다.

## 단점

- 처음에는 "왜 최신 값이 아닌가"가 직관적이지 않다.
- 의존성 배열과 비동기 흐름이 섞이면 디버깅이 어려워질 수 있다.
- `ref` 남용으로 문제를 덮으면 코드 가독성이 떨어질 수 있다.

## 주의사항 / 실무 팁

- 이전 값 기반 갱신이면 먼저 함수형 업데이트부터 떠올리는 게 안전하다.
- `eslint-plugin-react-hooks` 경고를 습관적으로 무시하면 stale closure 버그가 늘어난다.
- 외부 이벤트 리스너나 타이머는 "언제 등록됐고 어떤 값을 잡았나"를 같이 봐야 한다.
- 면접에서는 클로저 정의만 말하지 말고 React에서 왜 문제가 되는지까지 이어서 설명하는 편이 훨씬 좋다.
