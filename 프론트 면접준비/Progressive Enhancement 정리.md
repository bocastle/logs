# Progressive Enhancement 정리

## 핵심 요약
- `Progressive Enhancement`는 기본 기능이 먼저 동작하게 만들고, 가능한 환경에서만 자바스크립트나 고급 UI를 덧붙이는 설계 방식이다.
- 핵심은 "좋은 환경에서 더 좋게"이지, "좋은 환경에서만 동작하게"가 아니다.
- 접근성, 네트워크 불안정, 저사양 기기 대응 측면에서 장점이 크다.
- 면접에서는 "핵심 사용 흐름을 먼저 보장하고, 이후 향상 기능을 추가하는 전략"이라고 설명하면 전달이 잘 된다.

## 개념 설명
프론트엔드 개발을 하다 보면 인터랙션이 많아질수록 자바스크립트 의존도가 커진다. 문제는 브라우저 환경이 항상 이상적이지 않다는 점이다. 스크립트 로딩이 늦을 수도 있고, 일부 기능은 브라우저 지원이 부족할 수도 있으며, 보조 기술 사용자에게는 예상과 다른 방식으로 동작할 수도 있다.

`Progressive Enhancement`는 이런 상황을 전제로 한다. 먼저 HTML만으로도 핵심 콘텐츠와 기본 동작이 가능하도록 만들고, 그 위에 CSS와 자바스크립트로 경험을 점진적으로 향상시킨다. 예를 들어 기본 제출이 가능한 폼을 먼저 만들고, 이후 자바스크립트가 가능할 때만 실시간 유효성 검사나 자동 저장을 붙이는 식이다.

핵심은 "기본값이 견고해야 한다"는 점이다. 인터랙션이 풍부한 UI를 만드는 것과 별개로, 사용자가 최소한의 기능을 끝까지 수행할 수 있어야 한다는 철학이 들어 있다.

## 예시
회원가입 폼을 생각해보자.

- 기본 단계
  - HTML `form`으로 입력과 제출이 가능하다
  - 서버가 필수값 검증과 에러 메시지를 처리한다
- 향상 단계
  - 자바스크립트로 비밀번호 강도 표시를 붙인다
  - 이메일 중복 확인을 비동기로 보여 준다
  - 입력 중간 상태를 로컬에 임시 저장한다

이 구조라면 스크립트가 실패해도 회원가입 자체는 가능하다. 반대로 처음부터 자바스크립트 전제 구조로만 만들면, 스크립트 오류나 로딩 지연이 곧 기능 장애로 이어질 수 있다.

## 면접 답변 예시
- 질문: "`Progressive Enhancement`는 무엇인가요?"  
  답변: "`Progressive Enhancement`는 먼저 모든 사용자가 접근 가능한 기본 기능을 제공하고, 브라우저 성능이나 지원 기능이 충분할 때 자바스크립트와 고급 UI를 추가하는 설계 방식입니다. 저는 핵심 사용자 흐름이 스크립트 실패에도 유지되는지를 중요하게 봅니다."
- 질문: "왜 실무에서 중요하다고 보나요?"  
  답변: "네트워크 지연, 스크립트 에러, 브라우저 지원 차이 같은 현실적인 변수를 고려할 수 있기 때문입니다. 접근성과 복원력을 같이 챙길 수 있고, 핵심 플로우가 더 안정적으로 유지됩니다."

## 장점
- 자바스크립트 실패나 지연 상황에서도 핵심 기능을 유지하기 쉽다
- 접근성과 복원력 측면에서 설계 품질이 좋아진다
- 서버 렌더링, 폼 처리, 기본 HTML 구조의 중요성을 자연스럽게 살릴 수 있다

## 단점
- 처음부터 화려한 인터랙션만 기준으로 설계할 때보다 구현 사고가 더 필요하다
- 기본 동작과 향상 동작을 분리해 관리해야 해서 구조 설계가 약간 번거로울 수 있다
- 팀이 이 철학에 익숙하지 않으면 "왜 두 번 구현하느냐"는 오해를 받을 수 있다

## 주의사항 / 실무 팁
- 핵심 플로우가 자바스크립트 없이도 끝까지 진행되는지 먼저 점검하는 편이 좋다.
- 서버 검증을 생략하고 클라이언트 검증만 믿으면 `Progressive Enhancement`의 장점이 크게 줄어든다.
- 면접에서는 접근성, 복원력, 네트워크 불안정 대응과 연결해서 말하면 답변이 탄탄해진다.
- 실무에서는 검색, 결제, 회원가입처럼 실패 비용이 큰 화면부터 이 원칙을 적용하는 것이 효과적이다.
