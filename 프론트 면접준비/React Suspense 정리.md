# React Suspense 정리

## 핵심 요약

- `Suspense`는 비동기적으로 준비되는 UI가 있을 때, 준비가 끝날 때까지 대체 화면을 보여 주는 React의 메커니즘이다.
- 면접에서는 "로딩 스피너 하나 띄우는 기능"으로만 말하지 말고, "비동기 경계를 선언적으로 다룬다"는 관점으로 설명하는 게 좋다.
- 코드 스플리팅과 함께 쓸 때도 많이 나오고, 최근에는 데이터 패칭 흐름과 연결해서 질문이 이어질 수 있다.
- 장점은 로딩 상태 처리를 컴포넌트 경계에서 깔끔하게 나눌 수 있다는 점이고, 단점은 잘못 쓰면 로딩 경험이 갑자기 뚝 끊기거나 디버깅이 어려워질 수 있다는 점이다.

## Suspense를 한 문장으로 설명하면

`Suspense`는 "아직 준비되지 않은 자식 UI가 있으면, 그동안 대신 보여 줄 화면을 선언하는 경계"라고 설명하면 깔끔하다.

예를 들어 어떤 컴포넌트가 늦게 로드되거나, 비동기 데이터가 준비돼야 렌더링할 수 있다면 그 바깥에 `Suspense`를 두고 fallback UI를 보여 줄 수 있다.

즉, 로딩 상태를 컴포넌트 바깥에서 감싸는 식으로 다룰 수 있게 해 주는 도구다.

## 어디에 많이 쓰이나

### 코드 스플리팅

가장 익숙한 예시는 `React.lazy()`와 함께 쓰는 경우다.  
큰 화면 컴포넌트를 즉시 번들에 넣지 않고 필요할 때 불러오되, 로딩 중에는 fallback을 보여 준다.

```tsx
<Suspense fallback={<div>불러오는 중...</div>}>
  <LazyPage />
</Suspense>
```

이 패턴은 번들 크기를 줄이고 초기 로딩 부담을 낮추는 데 자주 활용된다.

### 데이터 준비 대기

최근 React 흐름에서는 데이터 패칭도 Suspense와 연결해서 설명하는 경우가 있다.  
핵심은 "데이터가 준비되기 전까지 어떤 UI를 보여 줄지 경계 단위로 결정한다"는 점이다.

면접에서는 "코드 스플리팅에서 가장 익숙하게 쓰이지만, 개념적으로는 비동기적으로 준비되는 UI 전반에 적용되는 경계"라고 말하면 좋다.

## 왜 굳이 필요한가

전통적으로는 각 컴포넌트 안에서 `isLoading`, `error`, `data`를 일일이 관리했다.  
이 방식도 맞지만, 트리가 깊어질수록 로딩 조건문이 여기저기 퍼지기 쉽다.

Suspense는 그 로딩 처리를 "어디까지 기다릴 것인가"라는 UI 경계 관점으로 끌어올린다.  
그래서 상위에서 로딩 경험을 더 일관되게 설계하기 좋아진다.

## Error Boundary와는 뭐가 다르냐

둘 다 경계처럼 보이지만 역할은 다르다.

- `Suspense`: 준비되지 않은 비동기 UI를 기다리는 동안 fallback을 보여 준다.
- `Error Boundary`: 렌더링 과정에서 발생한 에러를 잡아서 대체 UI를 보여 준다.

면접에서는 "로딩 대기와 에러 복구를 각각 다른 경계로 나눠서 본다"는 식으로 구분하면 된다.

## 잘못 쓰기 쉬운 포인트

### fallback 범위를 너무 크게 잡는 경우

페이지 전체를 한 번에 감싸면 작은 일부만 늦어도 화면 전체가 로딩 UI로 바뀔 수 있다.  
사용자 입장에서는 멀쩡히 보이던 화면이 갑자기 사라진 느낌이 날 수 있다.

그래서 실무에서는 너무 큰 단위보다, 실제로 기다릴 가치가 있는 영역에만 경계를 두는 편이 낫다.

### 모든 비동기 상황을 Suspense로만 해결하려는 경우

Suspense가 있다고 해서 모든 로딩 상태를 한 방식으로 처리하는 건 아니다.  
사용자 입력 직후의 미세한 로딩, 백그라운드 리프레시, 점진적 로딩은 다른 전략이 더 자연스러울 수도 있다.

## 면접 답변 예시

> React Suspense는 비동기적으로 준비되는 UI가 있을 때 그 바깥 경계에서 fallback UI를 선언적으로 보여 주는 메커니즘입니다. 가장 흔한 예시는 `React.lazy`와 함께 쓰는 코드 스플리팅이고, 개념적으로는 데이터가 준비되기 전까지 어떤 UI를 보여 줄지도 경계 단위로 관리할 수 있습니다. 장점은 로딩 상태 처리를 개별 컴포넌트 안에 흩뿌리지 않고 상위 경계에서 정리할 수 있다는 점입니다. 다만 경계를 너무 크게 잡으면 화면 전체가 갑자기 fallback으로 바뀌어 사용자 경험이 어색해질 수 있어서 범위를 잘 나눠야 합니다.

## 장점

- 로딩 상태를 UI 경계 단위로 선언적으로 관리하기 좋다.
- 코드 스플리팅과 결합했을 때 초기 번들 부담을 줄이기 쉽다.
- 면접에서 로딩 UX, 경계 설계, Error Boundary와의 차이까지 연결해 말하기 좋다.

## 단점

- 개념을 얕게 이해하면 단순 스피너 도구처럼만 설명하게 된다.
- fallback 범위를 크게 잡으면 사용자 경험이 오히려 거칠어질 수 있다.
- 데이터 패칭과 함께 쓸 때는 라이브러리나 렌더링 전략 이해가 부족하면 헷갈리기 쉽다.

## 주의사항 / 실무 팁

- 면접에서는 `React.lazy` 예시를 먼저 들고, 그다음 "비동기 UI 경계"로 개념을 넓히면 설명이 안정적이다.
- `Suspense`와 `Error Boundary`를 같은 것으로 말하지 않도록 역할을 분리해 두는 게 좋다.
- 로딩 경계를 너무 크게 두지 말고, 사용자가 기다린다고 느껴도 자연스러운 영역만 감싸는 편이 실무적으로 낫다.
- "상태 플래그를 없애 준다"보다 "로딩 상태를 경계 단위로 재구성한다"라고 말하면 더 깊이 있어 보인다.
