# 서브리소스 무결성(SRI) 정리

## 핵심 요약
- `SRI(Subresource Integrity)`는 브라우저가 외부 스크립트나 스타일 파일을 받을 때, 미리 지정한 해시값과 실제 파일 내용이 같은지 검증하는 보안 기능이다.
- CDN이 변조되었거나 예상치 못한 파일이 내려오면 브라우저가 해당 리소스 실행을 막을 수 있다.
- 특히 외부 CDN의 `script`, `link` 리소스를 사용할 때 공급망 공격 위험을 낮추는 데 도움이 된다.
- 면접에서는 "외부 정적 리소스가 내가 기대한 파일인지 브라우저 차원에서 검증하는 장치"라고 설명하면 이해가 쉽다.

## 개념 설명
프론트엔드에서 라이브러리를 CDN으로 가져오면 배포는 편해지지만, 그 CDN 파일이 바뀌거나 변조되면 우리 서비스도 영향을 받을 수 있다. 예를 들어 외부 스크립트 한 줄이 악성 코드로 바뀌면 사용자 브라우저에서 그대로 실행될 수 있다.

`SRI`는 이런 위험을 줄이기 위해 HTML 태그에 리소스의 해시값을 함께 적어 둔다. 브라우저는 파일을 내려받은 뒤 실제 내용으로 해시를 계산하고, 태그에 적힌 값과 다르면 해당 리소스를 신뢰하지 않고 차단한다.

즉, `SRI`의 핵심은 "리소스의 출처만 믿는 것이 아니라 내용까지 검증한다"는 점이다. 도메인이 같거나 CDN이 유명하더라도, 파일 내용이 다르면 실행하지 않게 만드는 보호막이라고 이해하면 된다.

## 예시
예를 들어 아래처럼 CDN의 CSS 파일을 불러온다고 해보자.

```html
<link
  rel="stylesheet"
  href="https://cdn.example.com/app.css"
  integrity="sha384-..."
  crossorigin="anonymous"
/>
```

이 상태에서 CDN 파일이 원래 내용과 다르게 바뀌면 브라우저는 해시 검증에 실패하고 해당 CSS를 적용하지 않는다. 자바스크립트도 같은 방식으로 막을 수 있다.

## 면접 답변 예시
- 질문: "`SRI`는 왜 쓰나요?"  
  답변: "`SRI`는 외부 CDN에서 내려받는 스크립트나 스타일이 내가 배포 시점에 기대한 파일과 같은지 브라우저가 해시로 검증하게 해 주는 기능입니다. 그래서 CDN 변조나 공급망 공격에 대한 방어에 도움이 됩니다."
- 질문: "CSP와는 어떤 차이가 있나요?"  
  답변: "`CSP`는 어떤 출처의 리소스를 허용할지 제어하는 정책이고, `SRI`는 허용된 출처에서 내려온 파일의 내용이 맞는지 검증하는 기능입니다. 둘은 목적이 다르고 같이 쓰면 보안성이 더 좋아집니다."

## 장점
- 외부 CDN 리소스 변조에 대한 기본 방어선을 만들 수 있다
- 브라우저 차원에서 검증하므로 클라이언트 보호 효과가 직접적이다
- 공급망 보안 관점에서 설명하기 좋은 실무형 주제다

## 단점
- 파일이 조금만 바뀌어도 해시를 다시 생성해야 해 운영이 번거로울 수 있다
- 해시가 자주 바뀌는 동적 리소스에는 적용하기 어렵다
- 리소스 무결성만 보장할 뿐, 애초에 취약한 코드 자체를 안전하게 만들지는 못한다

## 주의사항 / 실무 팁
- CDN 버전을 고정하지 않고 최신 버전을 가리키면 해시 불일치가 자주 발생할 수 있다.
- `crossorigin` 설정이 필요한 경우가 많아 함께 확인하는 습관이 좋다.
- 면접에서는 `CSP`, `공급망 공격`, `CDN 사용 전략`과 묶어서 설명하면 깊이가 생긴다.
- 실무에서는 자주 바뀌는 자산은 자체 호스팅하고, 외부 의존성이 꼭 필요한 경우에만 `SRI`를 붙이는 식으로 관리하는 경우가 많다.
