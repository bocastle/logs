# Popover API와 dialog를 구분해서 쓰는 법 정리

## 핵심 요약

- 면접에서 "Popover API와 dialog를 구분해서 쓰는 법" 질문을 받으면 핵심은 "사용자가 실수하거나 키보드만 사용할 때도 흐름을 회복할 수 있는가?"이다.
- 작은 API처럼 보여도 브라우저 지원, 실패 상태, 접근성까지 같이 봐야 답변이 자연스럽다.
- 예시는 실제 화면 흐름이나 운영 지표와 연결해서 말하는 편이 좋다.

## 개념 설명

폼과 접근성 관련 주제는 사용자가 실수했을 때 다시 흐름을 회복할 수 있는지가 핵심이다.

키보드 이동, 포커스 복귀, 오류 안내, 자동완성처럼 작은 디테일이 실제 사용성 차이를 크게 만든다.

"Popover API와 dialog를 구분해서 쓰는 법" 관련 질문을 받으면 용어 정의에서 멈추기보다 적용 조건과 실패 시나리오를 같이 잡아야 한다.

## 예시

```html
<label for="email">이메일</label>
<input id="email" name="email" autocomplete="email" aria-describedby="email-error" />
<p id="email-error" role="alert">이메일 형식을 확인해 주세요.</p>
```

코드는 짧지만 면접에서는 이 흐름이 실패했을 때 어떤 상태가 남는지까지 이어서 설명하는 것이 좋다.

## 면접 답변 예시

> "Popover API와 dialog를 구분해서 쓰는 법"에 대해 묻는다면 저는 키보드 이동, 포커스, 오류 안내, 자동완성처럼 사용자가 실제로 만나는 흐름을 기준으로 검증한다고 답하겠습니다. 기능 이름만 말하기보다 어떤 사용자 문제를 줄이는지, 실패했을 때 화면이 어떻게 유지되는지, 적용 후 어떤 지표로 확인할지까지 연결해서 설명하겠습니다.

## 장점

- 입력 실패 후 사용자가 다시 시도하기 쉬워진다.
- 키보드와 보조 기술 사용자의 흐름을 안정적으로 만든다.
- 모바일 입력 경험도 함께 좋아질 수 있다.

## 단점

- 상태 안내가 과하면 오히려 화면 읽기 흐름을 방해할 수 있다.
- 브라우저와 보조 기술 조합마다 동작 차이가 있다.
- 디자인 요구와 접근성 요구가 충돌할 때 조율이 필요하다.

## 주의사항 / 실무 팁

- 지원하지 않는 환경의 기본 흐름을 먼저 정하고 최신 기능을 얹는다.
- 면접에서는 포커스 이동, 오류 안내, 키보드 조작을 반드시 같이 말한다.
- 작은 데모 코드와 함께 실제 확인 지표를 하나 붙이면 답변이 좋아진다.
