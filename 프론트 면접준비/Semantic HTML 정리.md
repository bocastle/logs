# Semantic HTML 정리

## 핵심 요약

- `Semantic HTML`은 태그 이름만 꾸미는 문제가 아니라, `콘텐츠의 의미와 역할에 맞는 마크업을 선택하는 것`이다.
- 핵심은 `div를 줄이는 것`보다 `구조와 의도를 브라우저, 검색엔진, 보조기기에 정확히 전달하는 것`에 있다.
- 면접에서는 접근성, SEO, 유지보수성을 함께 묶어서 설명하면 좋다.
- 무조건 새 태그를 많이 쓰는 게 아니라, 상황에 맞는 의미 있는 구조를 만드는 게 중요하다.

## Semantic HTML이 뭐냐

HTML에는 단순히 화면을 감싸는 용도 이상의 의미가 있다.  
예를 들어 `header`, `main`, `article`, `nav`, `button` 같은 태그는 각각 역할을 담고 있다.

Semantic HTML은 이런 의미를 살려서 문서 구조를 표현하는 방식이다.

즉 `보이는 모양`이 아니라 `이 요소가 무엇인지`를 기준으로 태그를 고른다고 보면 된다.

## 왜 중요한가

가장 큰 이유는 해석 주체가 사람만이 아니기 때문이다.

- 브라우저는 문서 구조를 더 잘 이해할 수 있다.
- 스크린 리더 같은 보조기기는 요소의 역할을 더 정확하게 전달할 수 있다.
- 검색엔진도 콘텐츠 구조를 파악하기 쉬워진다.
- 개발자 입장에서도 마크업 의도가 분명해져 유지보수가 편해진다.

그래서 Semantic HTML은 단순 문법 취향이 아니라 `접근성과 구조 설계의 기본기`에 가깝다.

## div만 많이 쓰면 왜 아쉽나

`div` 자체가 나쁜 건 아니다.  
문맥상 특별한 의미가 없는 래퍼에는 여전히 유용하다.

문제는 버튼 역할인데 `div`에 클릭 이벤트만 달거나, 내비게이션인데 그냥 `div`로만 감싸는 식의 사용이다.

이렇게 하면

- 키보드 접근성이 떨어질 수 있고
- 스크린 리더가 역할을 제대로 알기 어렵고
- 추가 ARIA나 이벤트 보정이 더 필요해진다

즉 원래 HTML이 기본 제공하던 의미를 스스로 버리는 셈이다.

## 대표적인 시맨틱 태그 예시

면접에서는 모든 태그를 외울 필요는 없고, 자주 쓰는 것만 정확히 알면 된다.

- `header`: 소개 영역이나 상단 영역
- `nav`: 내비게이션 링크 묶음
- `main`: 문서의 주된 콘텐츠
- `section`: 주제별 구획
- `article`: 독립적으로 의미 있는 콘텐츠
- `footer`: 하단 정보
- `button`: 클릭 가능한 동작

여기서 중요한 건 `모양`이 아니라 `역할`이다.  
예를 들어 버튼처럼 보인다고 다 `button`은 아니고, 실제로 사용자 동작을 실행하는 요소일 때 `button`이 적절하다.

## 접근성과는 어떻게 연결되나

Semantic HTML을 잘 쓰면 기본 접근성이 좋아진다.

예를 들어 `button`은 기본적으로 포커스 가능하고, Enter나 Space 같은 키보드 동작도 자연스럽게 지원한다.  
반면 `div`로 버튼을 만들면 이런 동작을 직접 보완해야 한다.

스크린 리더도 시맨틱 태그를 기반으로 페이지 구조를 더 잘 읽는다.  
그래서 접근성을 신경 쓰는 팀일수록 Semantic HTML을 먼저 챙기고, 정말 부족한 부분만 ARIA로 보완하는 편이다.

## SEO와는 어떤 관계가 있나

검색엔진은 콘텐츠 구조를 이해하려고 한다.  
문서의 제목, 본문, 내비게이션, 독립 콘텐츠 같은 구획이 명확하면 해석이 쉬워진다.

물론 시맨틱 태그만 쓴다고 SEO가 바로 좋아지는 건 아니다.  
하지만 `문서 구조를 명확하게 전달하는 기반`이라는 점에서는 분명 도움이 된다.

## 면접 답변 예시

> Semantic HTML은 화면을 감싸기 위해 아무 태그나 쓰는 게 아니라, 콘텐츠의 의미와 역할에 맞는 HTML 요소를 선택하는 방식이라고 생각합니다. 이렇게 하면 브라우저, 검색엔진, 스크린 리더가 문서 구조를 더 잘 이해할 수 있어서 접근성과 유지보수성에 도움이 됩니다. 예를 들어 클릭 동작이면 div보다 button을 쓰고, 내비게이션이면 nav를 쓰는 식으로 기본 의미를 먼저 살리고, 필요한 경우에만 ARIA로 보완하는 방향을 선호합니다.

## 장점

- 접근성, SEO, 유지보수성을 한 번에 연결해서 설명하기 좋다.
- 기본 HTML 이해도가 있다는 인상을 주기 좋다.
- ARIA와 네이티브 요소의 차이를 이야기하기 좋은 출발점이 된다.

## 단점

- 태그 이름만 나열하면 답변이 피상적으로 들릴 수 있다.
- div를 쓰면 안 된다는 식으로 과하게 말하면 오히려 부정확하다.
- 시맨틱 구조와 실제 인터랙션 역할을 구분하지 못하면 혼란스러울 수 있다.

## 주의사항 / 실무 팁

- 면접에서는 `의미에 맞는 태그 선택`이라는 문장을 먼저 깔고 가는 게 좋다.
- `button 대신 div` 사례처럼 안 좋은 예시 하나를 들면 설명이 쉬워진다.
- Semantic HTML은 ARIA의 대체재가 아니라, 먼저 챙겨야 할 기본값이라고 말하면 좋다.
- 레이아웃용 래퍼까지 전부 시맨틱 태그로 바꾸려 하기보다, 실제 역할이 있는 곳부터 정확히 쓰는 게 실무적이다.
