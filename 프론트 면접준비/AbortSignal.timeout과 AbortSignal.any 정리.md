# AbortSignal.timeout과 AbortSignal.any 정리

## 핵심 요약
- `AbortSignal.timeout()`은 일정 시간이 지나면 자동으로 취소되는 시그널을 만드는 기능이다.
- `AbortSignal.any()`는 여러 취소 조건 중 하나라도 만족하면 함께 취소되도록 시그널을 합성하는 기능이다.
- 면접에서는 "비동기 작업 취소를 더 선언적으로 조합하게 해 주는 API"라고 설명하면 핵심이 잘 잡힌다.
- 네트워크 요청, 사용자 화면 이탈, 컴포넌트 정리(cleanup) 같은 여러 취소 이유를 한 흐름으로 묶을 때 특히 유용하다.

## 개념 설명
프론트엔드에서 비동기 작업을 취소해야 하는 상황은 생각보다 많다. 사용자가 다른 화면으로 이동했을 때, 특정 시간 안에 응답이 오지 않았을 때, 혹은 더 최신 요청이 발생했을 때 이전 요청을 중단해야 한다.

기존에는 `AbortController`를 직접 만들고, `setTimeout`으로 취소 타이머를 걸고, 컴포넌트 정리 시점에 `abort()`를 수동 호출하는 코드가 자주 나왔다. 이런 방식도 가능하지만 취소 이유가 늘어날수록 코드가 흩어지기 쉽다.

`AbortSignal.timeout()`은 "몇 초 뒤 자동 취소"를 더 짧고 명확하게 표현하게 해 준다. `AbortSignal.any()`는 사용자 취소, 라우트 변경, 타임아웃처럼 여러 취소 조건을 하나의 시그널로 합쳐 준다.

즉, 이 두 기능은 단순한 문법 설탕이 아니라 "취소 조건을 선언적으로 조합하는 방식"을 제공한다는 데 의미가 있다.

## 예시
예를 들어 검색 자동완성 요청을 보낸다고 해 보자.

- 사용자가 입력을 바꾸면 이전 요청은 취소돼야 한다
- 3초 안에 응답이 없으면 타임아웃 처리도 해야 한다
- 화면을 떠나면 남아 있는 요청도 정리해야 한다

이때 하나의 `AbortController`만 수동으로 관리하면 조건이 많아질수록 흐름이 복잡해진다. 반면 `AbortSignal.timeout(3000)`으로 시간 제한을 만들고, 사용자 취소용 시그널과 `AbortSignal.any([...])`로 묶으면 "어떤 이유든 취소되면 이 요청을 멈춘다"는 의도가 더 선명해진다.

## 면접 답변 예시
- 질문: "`AbortSignal.timeout()`은 무엇인가요?"  
  답변: "일정 시간이 지나면 자동으로 abort되는 시그널을 만드는 기능입니다. 네트워크 타임아웃 같은 케이스를 더 선언적으로 표현할 수 있습니다."
- 질문: "`AbortSignal.any()`는 언제 쓰나요?"  
  답변: "여러 취소 조건을 하나로 합치고 싶을 때 씁니다. 예를 들어 사용자 직접 취소, 라우트 변경, 타임아웃 중 하나라도 발생하면 같은 요청을 중단하도록 묶을 수 있습니다."
- 질문: "왜 유용한가요?"  
  답변: "비동기 취소 로직이 분산되지 않고, 어떤 조건에서 작업이 중단되는지 코드 의도가 더 잘 드러나기 때문입니다."

## 장점
- 타임아웃과 복수 취소 조건을 간결하게 표현할 수 있다
- 취소 로직이 분산되지 않아 코드 의도가 선명해진다
- 요청 정리 누락으로 인한 메모리 낭비나 오래된 응답 반영 문제를 줄이는 데 도움이 된다

## 단점
- 지원 환경을 확인해야 한다
- 취소 이유가 많아질수록 디버깅 시 어느 조건이 발동했는지 추적이 필요하다
- 기존 `AbortController` 패턴과 섞이면 팀 내 이해 차이가 생길 수 있다

## 주의사항 / 실무 팁
- 면접에서는 `AbortController` 기본 개념을 먼저 짚고, 그 위에 `timeout`과 `any`가 추가된다고 설명하면 자연스럽다.
- 취소는 에러가 아니라 정상 흐름일 수 있다는 점을 같이 말하면 좋다. 즉, abort를 예외 상황만으로 보지 않는 태도가 중요하다.
- React나 SPA 환경에서는 컴포넌트 언마운트, 최신 요청 우선 처리, 네트워크 타임아웃을 함께 묶는 예시를 들면 실무성이 높아진다.
- 브라우저 호환성이 애매한 환경이라면 폴리필 또는 기존 패턴 대안을 같이 준비해 두는 것이 안전하다.
