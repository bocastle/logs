# Passive Event Listener 정리

## 핵심 요약
- `passive event listener`는 브라우저에 "이 이벤트 핸들러에서는 `preventDefault()`를 호출하지 않겠다"고 미리 알려 주는 방식이다.
- 주로 `touchstart`, `touchmove`, `wheel` 같은 스크롤 관련 이벤트에서 성능 개선에 도움이 된다.
- 면접에서는 "브라우저가 스크롤을 더 빨리 진행할 수 있게 도와 주는 이벤트 리스너 옵션"이라고 설명하면 핵심이 잘 잡힌다.
- 무조건 쓰는 옵션은 아니고, 실제로 기본 동작을 막아야 하는 경우에는 오히려 쓰면 안 된다.

## 개념 설명
브라우저는 사용자가 화면을 스크롤하거나 터치할 때, 자바스크립트 이벤트 핸들러가 `preventDefault()`를 호출할지 먼저 확인하려고 잠깐 기다릴 수 있다. 이 대기 때문에 메인 스레드가 바쁘면 스크롤이 버벅이거나 입력 반응이 느리게 느껴질 수 있다.

이때 이벤트 리스너를 등록하면서 `{ passive: true }`를 주면 브라우저는 "이 핸들러는 기본 스크롤 동작을 막지 않겠구나"라고 판단하고, 자바스크립트 실행이 끝나기 전에 스크롤을 더 빨리 진행할 수 있다. 즉 핵심은 이벤트 자체를 빠르게 만드는 것이 아니라, 브라우저가 기본 동작을 더 자신 있게 먼저 처리하게 돕는 것이다.

특히 모바일 웹에서 손가락 스크롤이나 트랙패드 스크롤이 많은 화면에서는 체감 차이가 날 수 있다. 반대로 직접 제스처를 제어하거나 기본 스크롤을 막아야 하는 UI라면 `passive: true`가 맞지 않는다.

## 예시
상품 목록 페이지에서 `wheel` 이벤트를 감지해 간단한 로그를 남기거나 통계 수집만 한다고 가정해 보자.

- 스크롤 자체는 브라우저 기본 동작으로 자연스럽게 흘러가야 한다
- 이벤트 핸들러는 추가 정보만 기록한다
- 이 경우 `passive: true`를 주면 브라우저가 스크롤을 덜 주저하고 처리할 수 있다

반대로 커스텀 슬라이더에서 터치 이동을 가로채고 기본 스크롤을 막아야 한다면 `preventDefault()`가 필요할 수 있으므로 passive 옵션을 신중히 봐야 한다.

## 면접 답변 예시
- 질문: "`passive event listener`는 왜 쓰나요?"  
  답변: "브라우저가 스크롤 관련 이벤트에서 자바스크립트가 기본 동작을 막을지 기다리지 않도록 힌트를 주기 위해 씁니다. 그래서 `touchmove`나 `wheel` 같은 이벤트에서 스크롤 성능과 반응성을 개선하는 데 도움이 됩니다."
- 질문: "언제 쓰면 안 되나요?"  
  답변: "핸들러 안에서 `preventDefault()`가 필요한 경우에는 쓰면 안 됩니다. `passive: true`면 기본 동작을 막을 수 없어서 의도한 제어가 깨질 수 있습니다."

## 장점
- 스크롤 성능 최적화 주제로 짧고 명확하게 설명하기 좋다
- 브라우저 동작과 자바스크립트 실행 순서를 이해하고 있음을 보여 주기 좋다
- Core Web Vitals나 입력 반응성 주제와 연결해서 답변을 확장하기 쉽다

## 단점
- 옵션 이름만 외우고 실제로 왜 빨라지는지 모르면 답변이 얕아진다
- 모든 이벤트에 적용하면 된다고 오해하기 쉽다
- `preventDefault()`가 필요한 상황과 아닌 상황을 구분하지 못하면 실무 답변이 약해진다

## 주의사항 / 실무 팁
- `passive: true`는 성능 옵션이지 만능 최적화가 아니다. 스크롤 관련 이벤트에서 특히 의미가 크다.
- 라이브러리나 프레임워크가 이벤트를 감싸는 경우 실제로 passive 옵션이 어떻게 적용되는지 확인하는 습관이 필요하다.
- 면접에서는 "`wheel`, `touchmove`에서 주로 이야기한다" 정도를 같이 말하면 더 자연스럽다.
- 실무에서는 성능 개선 전후를 브라우저 Performance 패널이나 실제 사용자 반응성 지표와 함께 보는 게 좋다.
