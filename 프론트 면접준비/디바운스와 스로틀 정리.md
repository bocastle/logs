# 디바운스와 스로틀 정리

## 핵심 요약

- 디바운스는 이벤트가 연속으로 발생할 때 마지막 호출만 실행되게 만드는 방식이다.
- 스로틀은 일정 시간 동안 한 번만 실행되게 제한하는 방식이다.
- 둘 다 과도한 이벤트 처리 비용을 줄이기 위한 기법이지만, 쓰는 목적이 다르다.
- 면접에서는 "검색어 자동완성은 디바운스, 스크롤/리사이즈는 스로틀" 정도까지 연결해서 말하면 이해도가 좋아 보인다.

## 왜 필요한가?

브라우저에서는 생각보다 이벤트가 매우 자주 발생한다.

- `input` 입력
- `scroll`
- `resize`
- `mousemove`

이런 이벤트마다 API 호출이나 무거운 계산을 바로 실행하면, 서버 요청이 너무 많아지거나 화면이 버벅일 수 있다.

그래서 "이벤트는 자주 오더라도 실제 로직 실행은 적당히 제어하자"라는 목적으로 디바운스와 스로틀을 쓴다.

## 디바운스란?

디바운스는 이벤트가 계속 발생하는 동안에는 기다리다가, 일정 시간 동안 추가 이벤트가 없을 때 한 번만 실행하는 방식이다.

예를 들어 사용자가 검색창에 `react`를 입력한다고 해보자.

- `r`
- `re`
- `rea`
- `reac`
- `react`

입력할 때마다 바로 API를 호출하면 다섯 번 요청이 나간다. 그런데 보통은 사용자가 입력을 멈춘 뒤 최종 검색어 기준으로 한 번만 요청해도 충분하다. 이럴 때 디바운스를 쓴다.

즉, 디바운스는 "마지막 행동이 끝난 뒤 실행"에 가깝다.

## 스로틀이란?

스로틀은 이벤트가 아무리 많이 발생해도 일정 주기마다 최대 한 번만 실행되게 만드는 방식이다.

스크롤 이벤트가 대표적이다. 사용자가 페이지를 내리는 동안 이벤트는 매우 자주 발생하는데, 이때 매 이벤트마다 무거운 계산을 하면 성능이 나빠진다.

이럴 때 100ms에 한 번, 혹은 200ms에 한 번만 처리하게 제한하면 부하를 줄일 수 있다.

즉, 스로틀은 "계속 발생하는 이벤트를 일정 간격으로 끊어서 처리"하는 개념이다.

## 둘의 차이

둘 다 호출 횟수를 줄인다는 공통점은 있지만, 실행 타이밍이 다르다.

- 디바운스: 이벤트가 멈춘 뒤 한 번 실행
- 스로틀: 이벤트가 진행되는 동안에도 일정 간격으로 실행

그래서 사용자 경험 관점에서도 용도가 갈린다.

검색 자동완성은 사용자가 입력을 멈춘 뒤 결과를 보여주는 편이 자연스럽다. 반면 스크롤 위치 추적은 스크롤 중에도 중간중간 값이 갱신돼야 하므로 스로틀이 더 맞다.

## 간단한 예시

### 디바운스 예시

```js
let timer;

function onInput(value) {
  clearTimeout(timer);
  timer = setTimeout(() => {
    search(value);
  }, 300);
}
```

입력이 이어지는 동안 타이머를 계속 초기화하고, 300ms 동안 멈췄을 때만 `search()`가 실행된다.

### 스로틀 예시

```js
let lastExecuted = 0;

function onScroll() {
  const now = Date.now();

  if (now - lastExecuted >= 200) {
    updateScrollPosition();
    lastExecuted = now;
  }
}
```

스크롤 이벤트가 계속 발생해도 200ms마다 한 번만 실행된다.

## 언제 무엇을 쓰면 좋을까?

디바운스가 잘 맞는 경우:

- 검색 자동완성
- 입력값 검증 API 호출
- 필터 조건 변경 후 결과 재조회

스로틀이 잘 맞는 경우:

- 스크롤 위치 계산
- 무한 스크롤 트리거
- 윈도우 리사이즈 처리
- 마우스 이동 기반 UI 효과

## 실무에서 자주 놓치는 부분

디바운스와 스로틀을 적용했다고 끝이 아니다.

예를 들어 검색 API를 디바운스로 줄였더라도, 이전 요청이 늦게 도착해서 최신 검색 결과를 덮어쓸 수 있다. 그래서 실무에서는 `AbortController`로 이전 요청을 취소하거나, 응답 시점에 최신 요청인지 확인하는 로직을 같이 두는 경우가 많다.

또 React에서는 디바운스 함수를 렌더링마다 새로 만들면 타이머가 의도대로 유지되지 않을 수 있어서, 참조 안정성도 같이 신경 써야 한다.

## 면접 답변 예시

디바운스와 스로틀은 둘 다 이벤트가 너무 자주 발생할 때 호출 횟수를 줄여서 성능과 네트워크 부하를 관리하는 기법입니다. 디바운스는 연속 이벤트가 끝난 뒤 마지막 한 번만 실행하는 방식이고, 스로틀은 이벤트가 계속 발생하더라도 일정 시간마다 한 번씩 실행되게 제한하는 방식입니다.

그래서 검색 자동완성처럼 사용자가 입력을 멈춘 뒤 한 번만 처리하면 되는 경우에는 디바운스를 많이 쓰고, 스크롤이나 리사이즈처럼 진행 중에도 주기적으로 상태를 갱신해야 하는 경우에는 스로틀을 많이 씁니다.

## 장점

- 불필요한 함수 실행을 줄여 성능을 개선할 수 있다.
- API 과호출을 막아 서버 부하를 낮출 수 있다.
- 사용자 입력이 많은 화면에서 버벅임을 줄이는 데 도움이 된다.
- 이벤트 처리 의도를 더 분명하게 표현할 수 있다.

## 단점

- 잘못 적용하면 응답이 늦게 느껴질 수 있다.
- 디버깅 시 "왜 바로 실행되지 않지?" 같은 혼란이 생길 수 있다.
- 비동기 요청과 함께 쓰면 오래된 응답 처리 문제를 따로 관리해야 한다.
- 타이머 정리나 컴포넌트 unmount 시점 처리까지 챙겨야 한다.

## 주의사항 / 실무 팁

- 검색창은 디바운스, 스크롤/리사이즈는 스로틀이라는 기본 감각을 먼저 가져가면 좋다.
- 디바운스는 지연 시간이 너무 길면 UX가 답답해질 수 있으니 보통 200~500ms 정도에서 조정한다.
- 비동기 요청과 함께 쓸 때는 취소 처리나 최신성 보장을 같이 고려해야 한다.
- 면접에서는 둘의 정의만 말하지 말고, "어떤 상황에서 왜 이걸 쓰는지"까지 이어서 설명하는 게 좋다.
