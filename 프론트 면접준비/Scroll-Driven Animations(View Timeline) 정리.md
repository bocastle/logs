# Scroll-Driven Animations(View Timeline) 정리

## 핵심 요약
- `Scroll-Driven Animations`는 스크롤 위치를 기준으로 애니메이션 진행도를 연결하는 웹 플랫폼 기능이다.
- 그중 `View Timeline`은 특정 요소가 뷰포트 안으로 들어오고 나가는 과정을 기준으로 애니메이션을 제어한다.
- 면접에서는 "스크롤 이벤트를 직접 계산하지 않고도 선언적으로 스크롤 연동 애니메이션을 만들게 해 주는 기능"이라고 설명하면 좋다.
- 시각 효과만 강조하지 말고 성능, 브라우저 지원, 과도한 연출의 위험도 함께 짚어야 실무 감각이 드러난다.

## 개념 설명
예전에는 스크롤 기반 애니메이션을 만들려면 `scroll` 이벤트를 듣고 현재 위치를 계산한 뒤, 자바스크립트로 스타일 값을 계속 갱신하는 경우가 많았다. 이 방식은 구현이 유연하지만, 계산이 많아지면 성능 부담이 커지고 코드도 복잡해진다.

`Scroll-Driven Animations`는 스크롤 자체를 타임라인으로 사용할 수 있게 해 준다. 그중 `View Timeline`은 "특정 요소가 뷰포트에 얼마나 들어왔는가"를 기준으로 애니메이션 진행도를 연결한다. 예를 들어 카드가 화면 아래에서 올라오면서 서서히 나타나는 효과를 보다 선언적으로 만들 수 있다.

핵심은 "스크롤 위치를 직접 계산하는 imperative 방식"에서 "브라우저가 제공하는 timeline에 애니메이션을 연결하는 declarative 방식"으로 옮겨간다는 점이다. 그래서 코드가 간단해지고, 브라우저 최적화를 더 기대할 수 있다.

다만 모든 애니메이션을 이런 방식으로 바꾸는 것이 정답은 아니다. 복잡한 사용자 상호작용, 비즈니스 로직과 결합된 상태 변화는 여전히 자바스크립트 제어가 더 나을 수 있다.

## 예시
예를 들어 긴 랜딩 페이지에서 섹션별 카드가 화면에 들어올 때 자연스럽게 등장하는 효과를 만든다고 해 보자.

- 예전 방식은 `Intersection Observer`나 `scroll` 이벤트로 위치를 계산하고 클래스를 토글한다
- `View Timeline`을 쓰면 카드 요소가 뷰포트에 들어오는 구간을 타임라인으로 잡고, 그 구간에서 `opacity`나 `transform` 애니메이션을 실행할 수 있다
- 결과적으로 구현 코드가 더 선언적이고, CSS 중심으로 정리되기 쉽다

이런 점 때문에 콘텐츠 중심 페이지나 스토리텔링 UI에서 자주 거론된다.

## 면접 답변 예시
- 질문: "`View Timeline`은 무엇인가요?"  
  답변: "스크롤 기반 애니메이션에서 특정 요소가 뷰포트에 들어오고 나가는 구간을 타임라인처럼 사용할 수 있게 하는 기능입니다. 스크롤 위치를 자바스크립트로 직접 계산하지 않고 선언적으로 애니메이션을 연결할 수 있습니다."
- 질문: "기존 방식과 무엇이 다른가요?"  
  답변: "기존에는 `scroll` 이벤트나 옵저버로 위치를 계산해 상태를 바꿨다면, 이 기능은 브라우저가 스크롤 진행도를 타임라인으로 제공해서 CSS 중심으로 애니메이션을 구성할 수 있다는 차이가 있습니다."
- 질문: "주의할 점은 무엇인가요?"  
  답변: "브라우저 지원을 확인해야 하고, 애니메이션이 많아지면 사용자 경험을 해칠 수 있어서 필요한 곳에만 써야 합니다."

## 장점
- 스크롤 연동 애니메이션을 더 선언적으로 작성할 수 있다
- 자바스크립트 계산량을 줄여 코드 복잡도를 낮추는 데 도움이 된다
- 콘텐츠 등장 연출이나 스토리텔링 UI를 구조적으로 구현하기 좋다

## 단점
- 브라우저 지원 범위와 폴백 전략을 고려해야 한다
- 모든 상호작용을 대체할 정도로 만능은 아니다
- 과하게 사용하면 성능보다도 가독성과 사용성이 먼저 나빠질 수 있다

## 주의사항 / 실무 팁
- 면접에서는 "`scroll` 이벤트 남발 대신 브라우저 타임라인을 활용하는 방식"이라고 설명하면 차별점이 분명해진다.
- 단순 등장 효과는 `Intersection Observer`로도 충분할 수 있으니, 왜 `View Timeline`이 더 적합한지 목적을 같이 말해야 한다.
- 실무에서는 `opacity`, `transform` 위주로 구성해 레이아웃 비용을 줄이는 편이 안전하다.
- 디자인이 화려하다고 좋은 것은 아니다. 핵심 콘텐츠를 읽는 흐름을 방해하지 않는 선에서 써야 한다.
