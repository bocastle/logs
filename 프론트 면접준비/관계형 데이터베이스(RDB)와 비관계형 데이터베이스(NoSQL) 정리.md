# 관계형 데이터베이스(RDB)와 비관계형 데이터베이스(NoSQL) 정리

## 핵심 요약

- **관계형 데이터베이스(RDB)**: 테이블 기반, **고정 스키마**, 데이터 **일관성/관계 표현/조인**에 강점 (예: MySQL, PostgreSQL, Oracle)
- **비관계형 데이터베이스(NoSQL)**: 유연한 모델(키-값/문서/그래프 등), **스키마 유연**, **수평 확장**에 강점 (예: MongoDB, Cassandra, Redis)
- 선택 기준은 “데이터 구조 변경 빈도”, “관계/조인 필요성”, “일관성 요구 수준”, “확장 방식”에 따라 달라집니다.

## 관계형 데이터베이스란?

관계형 데이터베이스는 데이터를 **테이블(Table)** 형식으로 저장하고 관리하는 데이터베이스입니다.

- 각 테이블은 **고정된 스키마**를 가집니다.
  - 행(Row): 개별 레코드
  - 열(Column): 속성
- 데이터 타입과 구조가 엄격하게 정의되어 있어, 정형화된 데이터를 다루는 데 유리합니다.
- 대표 예시: `MySQL`, `PostgreSQL`, `Oracle`

### 관계 표현 예시

사용자(User)와 주문(Order) 데이터를 각각 테이블로 만들고, 사용자 ID를 **외래키(FK)** 로 설정해 두 테이블을 연결할 수 있습니다.  
이를 통해 특정 사용자의 주문 내역을 정확히 조회하는 등 **관계 기반 조회/조인**이 용이합니다.

## 비관계형 데이터베이스(NoSQL)란?

비관계형 데이터베이스는 전통적인 테이블 기반 구조가 아닌, 보다 **유연한 데이터 모델**을 사용하는 데이터베이스입니다.

- 유형 예시: **Key-Value**, **Document**, **Graph**
- **스키마가 고정되어 있지 않**아, 데이터 구조가 일관되지 않아도 저장할 수 있고 필요에 따라 속성을 유동적으로 추가할 수 있습니다.
- 대표 예시: `MongoDB`, `Cassandra`, `Redis`

### NoSQL이 유리한 경우

초기 개발 단계처럼 데이터 구조가 자주 바뀌는 프로젝트에서 유리할 수 있습니다.  
또한 수평 확장이 쉬워 대용량 처리나 트래픽 급증 상황에서 안정적인 성능을 유지하는 데 도움이 됩니다.

## 장점

### 관계형 데이터베이스 장점

- 미리 정의된 타입/구조 검증으로 **데이터 일관성 유지**에 유리합니다.
- 데이터 간 **관계를 명확히 표현**할 수 있습니다.
- **복잡한 조건의 조회/조인**을 처리하기 용이합니다.

### 비관계형 데이터베이스(NoSQL) 장점

- 스키마가 고정되지 않아 **유연성**이 높습니다.
- **수평적 확장**이 비교적 쉬워 대규모 트래픽/데이터 처리에 유리합니다.
- 문서 기반 NoSQL은 한 객체에 필요한 데이터를 묶어 담아 **한 번의 조회로 필요한 정보**를 가져오기 쉬울 수 있습니다.

## 단점

### 관계형 데이터베이스 단점

- 스키마가 고정되어 있어 **유연성이 떨어집니다.**
  - 필드 추가/구조 변경 시 테이블 수정과 마이그레이션이 필요해 복잡해질 수 있습니다.
- **수평적 확장**이 상대적으로 어렵습니다.
  - 조인이 많은 구조에서 분산 저장 시 성능 저하가 발생할 수 있어 확장성이 한계로 작용할 수 있습니다.

### 비관계형 데이터베이스(NoSQL) 단점

- 스키마가 자유로운 만큼 **데이터 일관성 유지가 어렵**습니다.
  - 같은 컬렉션/저장 단위 내에 구조가 다른 데이터가 섞일 수 있어 가공/검증 시 오류가 생기기 쉽습니다.
- **조인 기능이 제한적**이라 관계 표현에 한계가 있습니다.
  - 여러 저장 단위의 데이터를 합쳐 조회해야 한다면 애플리케이션 단 로직이 늘어날 수 있습니다.

## 주의사항 / 실무 팁

- “관계가 중요하고 조인이 많다” → RDB가 유리한 경우가 많습니다.
- “데이터 구조 변경이 잦고 빠른 확장이 필요하다” → NoSQL이 유리한 경우가 많습니다.
- NoSQL을 쓰더라도, 팀 내에서 **문서 스키마 규칙/검증 전략**(예: 애플리케이션 레벨 validation)을 정해두면 일관성 문제를 줄이는 데 도움이 됩니다.
- RDB에서 스키마 변경이 잦다면, 마이그레이션 전략(버전 관리, 롤백 계획)을 초기에 갖추는 편이 좋습니다.
