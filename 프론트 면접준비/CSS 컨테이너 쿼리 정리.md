# 프론트 면접준비: CSS 컨테이너 쿼리 정리

## 핵심 요약
- 컨테이너 쿼리는 요소의 부모 컨테이너 크기에 따라 스타일을 분기합니다.
- `미디어 쿼리`가 **브라우저 뷰포트**를 기준으로 한다면, 컨테이너 쿼리는 **컴포넌트 자체 맥락**을 기준으로 합니다.
- 재사용 가능한 컴포넌트 구성에서 반응형 설계를 더 정확하게 만들 수 있습니다.

## 개념 설명
기존 반응형은 `@media`로 뷰포트 폭을 기준으로 했습니다. 하지만 실제로는 같은 뷰포트에서 컴포넌트가 배치되는 부모 공간이 다르면 같은 스타일이 적절하지 않을 수 있습니다.

컨테이너 쿼리는 대상 요소의 **부모 컨테이너 크기**를 기준으로
- 폰트 크기
- 여백
- 그리드 열 수
- 표시/숨김 규칙
을 바꿉니다.

핵심 키워드는 `container-type` 입니다.  
예를 들어 카드 목록 컴포넌트가 사이드바 안에서는 1열, 본문 영역에서는 3열로 깔끔하게 바뀌도록 만들 때 유용합니다.

## 실용 예시
```css
.card-list {
  container-type: inline-size;
  container-name: cardArea;
}

@container cardArea (min-width: 600px) {
  .card-list {
    display: grid;
    grid-template-columns: repeat(3, 1fr);
  }
}

@container cardArea (max-width: 599px) {
  .card-list {
    display: block;
  }
}
```

면접 포인트는 `@container`가 어떤 상황에서 미디어 쿼리를 대체/보완하는지 설명하면 됩니다.

## 면접 답변 예시
"미디어 쿼리는 뷰포트 단위라 컴포넌트 재사용이 어려울 수 있습니다. 반면 컨테이너 쿼리는 배치된 박스의 실제 크기 기준으로 스타일을 바꾸기 때문에, 모듈화된 UI에서 반응형 규칙의 의도를 더 정확하게 구현할 수 있습니다. 특히 위젯/카드/사이드바 조합에서 강점이 큽니다."

## 장점
- 컴포넌트 단위 반응형 구현이 가능
- 같은 컴포넌트를 여러 레이아웃에서 일관되게 재사용 가능
- 중첩 레이아웃에서 미디어 쿼리 의존도를 줄일 수 있음

## 단점
- 브라우저 지원 범위 확인 필요(신규 문법 특성 고려)
- 컨테이너 설정 규칙이 늘면 CSS 복잡도 증가
- 쿼리 대상 컨테이너를 의도대로 설정하지 않으면 예측 불가한 동작

## 주의사항 / 실무 팁
- `container-type`를 과하게 쓰지 말고 **의도한 컴포넌트 경계**에만 적용하세요.
- 네임드 컨테이너(`container-name`)를 사용하면 유지보수성이 좋아집니다.
- 기존 미디어 쿼리와 혼합할 땐, 우선순위와 겹침 규칙을 문서화하세요.
